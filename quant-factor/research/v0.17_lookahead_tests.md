# v0.17 研究笔记 — Lookahead 自动测试套件

## 动机

v0.16 暴露 MTF 有两阶段 lookahead bug，全靠肉眼看到"IC=+0.33 太诡异"才发现。
下次可能不会这么走运——需要机制化的防御。

## 测试分类

### 1. Oracle 测试（sanity check）
- 注入"知道未来"信号：`factor = fwd_return` 本身
- **预期**：IC = 1.0
- 目的：确保评估器本身没 bug

### 2. Noise 测试（lookahead 探测）
- 注入与 forward return **完全独立**的信号（固定种子高斯噪声）
- **预期**：IC ≈ 0（|IC| < 0.01 / √T 95%CI）
- 如果 IC 显著 > 0，说明流水线自己"泄漏"了某种信息

### 3. Permutation 测试（pipeline 因果性）
- 真数据上跑一次得 IC_A
- 把所有股票的时间戳随机 shuffle（breaks predictability），再跑得 IC_B
- **预期**：|IC_B| << |IC_A|；如果 IC_B 仍显著，说明 lookahead

### 4. Future-bar poisoning 测试（精准 lookahead 探测）
- 在 test 集范围内，把某只股票的 close 替换为"完美预测未来"值
- **预期**：该股票的因子值对 fwd_return 极度相关（IC > 0.9）
- 如果只有一股被 poison 但整个 universe 的 IC 飙高 → 有信息泄漏

### 5. MTF-specific: 跨 TF 边界泄漏
- 在 H1 base 数据上构造一个只在 03:00 bar close 有脉冲的信号
- 跑 H1+H4 MTF，检查在 H1=01:00 / 02:00 / 03:00 的因子值是否都受 03:00 脉冲影响
- **如果受影响 → H4 对齐时偷看了未来**（02:00 的因子不应该知道 03:00 的事）

## 设计选择

v0.17 实现 4 个核心测试：
1. `test_oracle_ic_is_one` — sanity
2. `test_noise_factor_has_zero_ic` — 基础 lookahead 探测
3. `test_mtf_no_future_leak` — v0.16 bug 专项
4. `test_combiner_shift_is_causal` — ic_weight 的 shift(1) 仍有效

### 工具框架
用纯函数 + assert，不依赖 pytest（减少依赖）。文件本身可 `python -m` 跑。

### 如何验证"测试能抓到 bug"
保存 fix 前的代码版本（git show HEAD~1），在那里跑测试套件，验证**测试失败**。
fix 后跑，验证**测试通过**。

## 不做（暂缓）

- 完整 CPCV 实现（计算开销大，留 v0.x）
- 对每个 alpha 单独做 oracle/noise 测试（太细，pool 级别够）
- Mutation testing（修改代码一小处看测试是否还能抓——过度工程）
