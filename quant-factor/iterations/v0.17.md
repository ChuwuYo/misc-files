## 改动

### 新建 `tests/test_lookahead.py`

零依赖（纯 numpy/pandas，不要 pytest），可 `python -m tests.test_lookahead` 跑。
5 个测试：

| 测试 | 目的 | 通过条件 |
|---|---|---|
| `test_oracle_ic_is_one` | sanity：评估器正确性 | IC = 1.0 when factor==fwd_ret |
| `test_noise_factor_has_zero_ic` | 基础 lookahead 探测 | 注入高斯噪声，IC < 0.05 |
| **`test_mtf_no_future_leak_h1_plus_h4`** | **v0.16 bug #1 专项** | IC(H1+H4) / IC(H1) < 4.0 |
| **`test_mtf_no_future_leak_h1_plus_d1`** | **v0.16 bug #2 专项** | IC(H1+D1) / IC(H1) < 5.0 |
| `test_combiner_weights_are_shifted` | ic_weight 权重的 shift(1) | 短 lookback 下 IC < 0.1 |

## 验证：测试能否抓到 v0.16 bug？

**实验**：临时 revert v0.16 fix（去掉 shift(1)），重跑测试：

| 测试 | 有 fix | 无 fix |
|---|---|---|
| test_oracle_ic_is_one | PASS IC=1.0 | PASS IC=1.0 |
| test_noise_factor_has_zero_ic | PASS IC=-0.011 | PASS IC=-0.011 |
| **test_mtf_no_future_leak_h1_plus_h4** | **PASS ratio=0.72** | **FAIL ratio=29.1 ⚠️** |
| **test_mtf_no_future_leak_h1_plus_d1** | **PASS ratio=0.52** | **FAIL ratio=15.9 ⚠️** |
| test_combiner_weights_are_shifted | PASS IC=0.004 | PASS IC=0.004 |

**测试套件工作**：无 fix 版本被精准抓到，29.1x / 15.9x ratio 远超阈值。
H4 和 D1 的泄漏各自独立被检测 —— 两阶段 bug 都有防御。

## 设计原则

### 1. ratio 测试 > 绝对阈值测试
直接用 "IC < 0.01" 不稳定（数据相关），用 "多 TF 版本 / 单 TF 版本 比值" 稳定：
真实 MTF 可能让 IC 小幅变化（0.5-2x），但泄漏会让它爆炸到 10-30x。

### 2. 合成数据 > 真数据
真数据每次结果略不同；合成数据 seed=固定，**完全 deterministic**。
可 CI 化、可 git bisect。

### 3. 零依赖
不要 pytest——一条 `python -m tests.test_lookahead` 跑完，返回非 0 表示失败。
可直接嵌入 pre-commit hook。

### 4. 分层测试
oracle → noise → MTF → combiner 逐层升级，任何一层失败都能快速定位是评估器、
pipeline、MTF 还是 combiner 的问题。

## 副作用 / 已知限制

1. 测试跑时间 ~10s（SDKs 编译 import 较慢），不算快但 CI 可接受
2. 没测 walk-forward 本身的 purge 正确性（留给 v0.18）
3. 没测 sector neutralize / PCA 的 lookahead（这两个用全样本 fit，确实有轻度
   lookahead，但已在 iterations/v0.9.md 文档里标注）
