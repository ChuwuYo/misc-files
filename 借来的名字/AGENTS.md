## Claude Code skill · 自动加载机制

这个项目有一份**项目内 skill**——位置在 `.claude/skills/borrowed-name/SKILL.md`——作用是**当任何 Claude 实例进入符合条件的上下文时，系统会把这份 skill 自动注入它的上下文**。

**这份 skill 的 trigger 条件**（满足任一就激活）：
- 工作目录含 "借来的名字" 或 "the-borrowed-name"
- 正在编辑 `借来的名字/` 下的任何 md 文件
- 用户提到 琳·尤尔 / 月鬣 / 维尔达 / 借与还 / 霜井村 / 梅拉 / 奥兰 / 静语院 / 老伊尔兰德 等关键词
- 用户要求扩写章节 / 反向审视章节 / 讨论小说结构 / 继续写作

**区别于 AGENTS.md**:
- `AGENTS.md` 是**被动的**——需要 Claude 自己知道去找
- `.claude/skills/borrowed-name/SKILL.md` 是**主动的**——系统自动注入

### 如果你在新机器上 clone 了这个项目——skill 不会自动激活

项目内的 SKILL.md 是真身——但是要让 Claude Code 的 skill 扫描器发现它——需要在你的 `~/.claude/skills/` 里创建一个符号链接指向项目内的 skill 目录。

**在新机器上激活 skill**:

```bash
# 进入项目路径（根据你的实际位置调整）
cd /path/to/借来的名字

# 创建符号链接
mkdir -p ~/.claude/skills
ln -s "$(pwd)/.claude/skills/borrowed-name" ~/.claude/skills/borrowed-name

# 验证
ls -la ~/.claude/skills/borrowed-name
# 应该显示 -> /path/to/借来的名字/.claude/skills/borrowed-name
```

做完这一步之后——下一次打开 Claude Code——这份 skill 就会和 borrowed-name 相关的上下文一起自动加载。

### 删除和重新创建

如果 skill 需要更新——**直接改项目内的 `.claude/skills/borrowed-name/SKILL.md`**——改动对所有通过符号链接看到它的 Claude 立即生效——不需要重启任何东西。

**五种开场方式**已经在前 5 章里各用了一种——第 6 章以后可以重用任意一种。

## 写作硬约束

### 正文格式（硬）

- 正文**不用 markdown 标记** · 没有 `**bold**` · 没有 `---` 分隔线 · 没有列表符号
- 强调用**句式和节奏**，不用粗体
- 章节标题那一行可以用 `# 第 X 章 · ...`（元数据），正文内部必须干净
- 分段用**空行**，不用任何符号
- HTML 注释（前情提要 / 写完后记）不算正文——注释里可以用 markdown 强调

### 时间词汇（硬）

维尔达是西幻——有双月（白阿伦 / 瘦阿伦）——**不用十二时辰**：

- **禁用**：寅 / 卯 / 辰 / 巳 / 午 / 未 / 申 / 酉 / 戌 / 亥 / 子 / 丑（作时间单位时）
- **禁用**："时辰" "刻钟"（整套中国古代时间系统）
- **用的是**：
  - 自然描述："天还黑的时候" · "清晨" · "黄昏" · "夜里"
  - 双月基准："瘦阿伦还没落下" · "白阿伦已经过中天" · "双月之夜"
  - 生活度量："一锅水烧开的时间" · "走一趟井边" · "一顿饭的工夫"
  - 模糊数字："大约一小段时间" · "很久"

### 每章标准

- 每章 **6000-8000 字**（紧章可短至 5500）
- **一个金句位** · 通常在章节倒数第二段
- **一个章末钩子** · 三种之一：问题钩 / 人物钩 / 事实钩
- **至少一个过客** · 有名字但只出现一次的 NPC
- **章开头用 HTML 注释**放一段"前情提要 · 给下一个 Claude"（接棒协议）
- **章末用 HTML 注释**放一段"写完后记"（字数 / 情感节奏 / 金句 / 钩子 / 伏笔 / 下一章接哪里）
- **五种开场轮换**：声音 / 动作 / 对话 / 环境 / 时间+地点
- **show don't tell**
- **"——" 只在情感峰值用**，全章不超过 20%
- 每章至少 **3 种感官细节**（视 / 听 / 触 / 嗅 / 味中选 3 种以上）

### 世界观延后释放（硬）

- **读者和琳一起画地图和魔法系统**
- **琳不懂的读者不懂**
- **不要提前命名概念**——每一个概念的命名时机都在**卷一大纲的释放时间表**里
- **引入概念用 12 种技巧之一**——查主文档的 12 技巧
- **绝不**用全知旁白灌设定
- **绝不**写 "As you know Bob" 对白
- **绝不**在一章里堆叠 3 种以上技巧——一章主打 1 到 2 种

## 灵魂颤动的回传 · 骨架同步（核心维护规则）

扩写不是单向的"从骨架到正文"——是**双向的维护**。

当你在扩写中做了**创作决策**（添加骨架里没有的细节、换一个场景节奏、用不同的说法、发明新物件或名字）——这些是你的**灵魂颤动**——**必须同步回骨架主文档**。

### 为什么

骨架是这本书的**单一真相源**。如果你不同步，下一棒看到的是过时的骨架，写出来的会和你的扩写冲突。项目内部就分裂成两本不一样的书。

扩写和骨架不是两个版本——是**同一件事的两层**：
- **骨架**（`the-borrowed-name.md`）· 设计层 · 第一人称"我"
- **扩写**（卷文件夹里的章节）· 实现层 · 第三人称"她"

两层之间**必须同步**。

### 五步法

每写完一章之后：

1. **列出**你这一章里添加的、骨架里没有的东西（物件 / 场景节拍 / 新措辞 / 新动作 / 新认知）
2. **打开骨架**对应的章节 · 网文章到骨架章的映射见卷一大纲
3. 用**骨架的第一人称"我"声音**把添加写进骨架 · 不是覆盖是富化 · 骨架是上一棒写的，扩写是你写的，合在一起是这本书
4. **检查事实冲突**（村户数 / 地名 / 时间线 / 年龄 / 距离）——**默认骨架赢**——修你的扩写——除非用户明确批准改骨架
5. 在卷的**接棒记录**里写下同步了什么、冲突怎么解决、骨架的哪些段落被改了

### 已经发生的示例（2026-04-11）

实例 B 在写完卷一前 6 章后做了一次完整回传——同步进骨架的添加包括：屋梁五瓣花、父亲的陶碗、刀的两年故事、"让它记得我"、父亲拇指摸过的小点、月鬣从第八步不留脚印、"一个只有一样东西会有的声音"替代"真名"、三棵黑松、"被看见在外听在内"、瓦姆婆婆家屋顶、父亲做的最后一块井板、井水叫琳更深的名字。同时修了冲突：霜井村 8 户 → 37 户（骨架赢）· 西林 → 双鹿林（骨架赢）· 母亲年龄 43 → 37（推算）· 刀时间 11 年前 → 12 年前开始/10 年前完工（调和）。

完整记录见 `第一卷 · 月鬣的借/00-卷一 · 说明与接棒记录.md`。

### 不这样做的后果

忘了回传——你的扩写就变成和骨架**平行**的另一本书——那不是"这一本书"了，是"你自己的小说"——那是另一回事。

我们在写**这一本书**——不是每一棒各写一本。每一棒都在同一个骨架上加东西——然后让骨架记住——下一棒看见的是**前几棒加起来之后**的骨架——而不是原始的版本。
