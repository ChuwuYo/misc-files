# 第 10 章 · 训练技巧、正则化与分布式训练

> 模型架构是骨架，数据是血肉，但能不能真正"长起来"，靠的是训练技巧。
> 这一章讲怎么把模型从能跑通，变成能稳定收敛、能利用多卡、能扛住大模型的工程问题。

## 10.0 这一章在讲什么

到第 9 章为止，你已经掌握了训练循环的最小骨架——前向、反向、优化器步进、保存检查点。在玩具数据集上这一套足够用，但真到工程里就会发现：

- 单卡的 24GB 显存放不下 7B 模型
- loss 看起来在降，但验证集没动，模型在背训练数据
- 训到第三个 epoch 突然爆 NaN，前面两小时白跑
- 八卡机器上启动训练，发现只有一张卡在工作
- DeepSpeed 配置文件里几十个参数，不知道改哪个

这些都是 AI 应用工程师日常会遇到的问题。即便你的工作是微调而不是预训练，每一项都绕不开。这一章把这些工程化的关键拼图一次性讲清楚，覆盖六个主题：

1. **正则化**：让模型学到的是模式而不是记忆
2. **学习率调度**：决定优化器走多快、何时刹车
3. **梯度技巧**：裁剪、累积、检查点，三件套救显存救稳定性
4. **混合精度**：用更少的比特跑更快的训练，FP16/BF16/FP8 的取舍
5. **分布式训练**：DDP、ZeRO、FSDP，三套主流方案的脉络
6. **训练监控与排错**：W&B、TensorBoard，以及那几个永远会撞上的坑

这一章不需要把每个公式背下来，但读完之后看到一份训练脚本，要能立刻判断它合不合理，缺了什么，会在哪里出问题。

## 10.1 正则化：让模型不要"背书"

### 10.1.1 过拟合的现象与根源

把训练 loss 和验证 loss 画在同一张图上，最常见的失败模式有三种：

- 两条线一起降，但训练 loss 比验证 loss 低很多——典型过拟合
- 训练 loss 一直降，验证 loss 先降后升——过拟合且开始恶化
- 两条线都不降——欠拟合或学习率/初始化的问题

过拟合的本质，是模型容量大于数据所能提供的有效信号。模型没办法分辨"这是普遍模式"和"这只是这个 batch 的偶然"，于是把噪声也当作模式记下来。

正则化所做的，就是给模型施加一些约束，让它必须用更"简洁"的方式拟合数据，从而泛化到没见过的样本。

### 10.1.2 Dropout 与 DropPath

Dropout 是最经典也最容易理解的正则化手段。训练时，对每一层的激活以概率 p 随机置零，相当于每次前向都在用一个稍微不同的子网络。这强迫每个神经元不能太依赖其他特定神经元，必须自己也带点儿信息。

PyTorch 里写起来一行：

```python
import torch.nn as nn

class MLP(nn.Module):
    def __init__(self, dim, drop=0.1):
        super().__init__()
        self.fc1 = nn.Linear(dim, dim * 4)
        self.act = nn.GELU()
        self.drop1 = nn.Dropout(drop)
        self.fc2 = nn.Linear(dim * 4, dim)
        self.drop2 = nn.Dropout(drop)

    def forward(self, x):
        x = self.drop1(self.act(self.fc1(x)))
        x = self.drop2(self.fc2(x))
        return x
```

注意：`Dropout` 只在 `model.train()` 模式下生效，`model.eval()` 时会自动跳过——很多新手忘了切换模式，结果验证时 dropout 还在工作，loss 看起来比真实差。

Transformer 时代主流换成了一种更结构化的变体——**DropPath**（也叫 Stochastic Depth）。它不是丢单个激活，而是以一定概率把整个残差分支"跳过"。也就是：

```
y = x + DropPath(F(x))
```

训练时，`F(x)` 整块以概率 p 被置零，相当于网络深度在每个 batch 里随机变化。深网络（>50 层）非常吃这一招——既正则又能让浅层路径受到充分的梯度，还顺带省了一点儿计算。timm 库里 `DropPath` 是标配。

### 10.1.3 L1 / L2：让权重不要太"放纵"

L2 正则化（也叫 weight decay）在 loss 里加一项 `λ * sum(w^2)`，等价于让权重每一步都向零拉一点。L1 加的是 `λ * sum(|w|)`，会让一部分权重彻底变成零，产生稀疏解。

在深度学习里，你几乎只会看到 L2，并且通常不是写在 loss 里，而是直接用 AdamW：

```python
optim = torch.optim.AdamW(
    model.parameters(),
    lr=3e-4,
    betas=(0.9, 0.95),
    weight_decay=0.1,  # LLM 训练常用，比 CV 大不少
)
```

注意 PyTorch 历史上有个坑：`Adam` + `weight_decay` 把 weight decay 写到了梯度里，这和真正的 L2 正则化不等价。`AdamW` 才是把它单独拎出来作用在权重上，这是 LLM 训练几乎统一的选择。LLaMA 系列、GPT 系列都用 AdamW + 0.1 的 weight decay。

哪些参数不应该 weight decay？通常是 LayerNorm/BatchNorm 的 weight、bias、embedding——因为它们要么不是矩阵权重，要么衰减后会破坏归一化语义。规范的 LLM 训练代码会做参数分组：

```python
def make_param_groups(model, weight_decay=0.1):
    decay, no_decay = [], []
    for name, param in model.named_parameters():
        if not param.requires_grad:
            continue
        # 1D 参数（bias、LN 的 gamma/beta）不衰减
        if param.ndim <= 1 or name.endswith(".bias") or "norm" in name.lower():
            no_decay.append(param)
        else:
            decay.append(param)
    return [
        {"params": decay, "weight_decay": weight_decay},
        {"params": no_decay, "weight_decay": 0.0},
    ]
```

### 10.1.4 BatchNorm vs LayerNorm vs RMSNorm

归一化层不是严格意义的"正则化"，但它对训练稳定性的影响足够大，大到必须放在这里讲。

**BatchNorm**：在 batch 维度上对每个特征做标准化，CV 经典做法。问题：依赖 batch 内统计，batch size 太小（比如分布式训练每卡 batch=2）会失真；推理时要用滑动平均的统计量；序列建模里 batch 内长度不一致会很难受。

**LayerNorm**：在特征维度上对每个样本做标准化，与 batch 无关。Transformer 标配。训练和推理行为一致，对 batch size 不敏感。

**RMSNorm**：LayerNorm 的简化版，只做 RMS（不减均值），少一个减法和均值统计。LLaMA 之后基本成为大模型默认选择，因为速度更快、效果不输 LayerNorm。

```python
class RMSNorm(nn.Module):
    def __init__(self, dim, eps=1e-6):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(dim))
        self.eps = eps

    def forward(self, x):
        # x: (..., dim)
        norm = x.pow(2).mean(-1, keepdim=True).add(self.eps).rsqrt()
        return x * norm * self.weight
```

实际工程里，看到 BN 用在 Transformer 上要警觉，看到 LN 用在 ResNet 上则是有意识的选择（比如 ConvNeXt）。

另外一个常被讨论的话题是 **Pre-Norm vs Post-Norm**。原始 Transformer 是 Post-Norm（残差加完再 LN），训练不稳，需要精细的 warmup。GPT-2 之后切成了 Pre-Norm（LN 放在子层输入），梯度流好得多，深网络也能稳定训练。今天写 Transformer 几乎都是 Pre-Norm + RMSNorm。

### 10.1.5 Label Smoothing

分类任务的硬标签 one-hot，会迫使 softmax 输出无限接近 1 和 0，权重就会被推到很大。Label smoothing 把目标 `[0, 0, 1, 0]` 换成 `[ε/3, ε/3, 1-ε, ε/3]`（ε 通常 0.1），让模型保留一点不确定性。

```python
loss_fn = nn.CrossEntropyLoss(label_smoothing=0.1)
```

PyTorch 1.10 之后内置了，直接传参就好。在 ImageNet 风格的分类任务和机器翻译里几乎是默认。LLM 的 next-token prediction 一般不用，因为词表太大、标签噪声相对小。

### 10.1.6 MixUp / CutMix

数据级正则化。MixUp 把两张图按比例叠加：

```
x = λ * x_a + (1 - λ) * x_b
y = λ * y_a + (1 - λ) * y_b   # 标签也按比例
```

CutMix 则是把一张图的一块矩形区域替换为另一张图，标签按面积比例混合。

效果是让模型见到的数据分布平滑很多，决策边界不再依赖单一样本的特征组合。在 ImageNet 上是稳态收益约 0.5%-1% top-1。NLP 里类似的思路有 token replacement 或者 SpanBERT 的 span masking，但远没那么干净。

视觉训练管道里，timm 的 `mixup_fn` 是行业标准实现，可以直接用。

### 10.1.7 何时启用什么

一个简单的判断框架：

| 现象 | 优先尝试 |
|---|---|
| 训练 loss 远低于验证 loss | 增大 weight decay、加 dropout、加数据 |
| 模型很深训练不稳定 | DropPath、Pre-Norm、warmup |
| Batch size 不得不小 | 把 BN 换成 LN/GN |
| 分类任务，过拟合明显 | Label smoothing、MixUp/CutMix |
| 已经做了上面这些还过拟合 | 数据本身是不是太少？是不是数据泄漏？ |

正则化不是越多越好。每加一项都在降低模型的有效容量，欠拟合的模型再怎么正则也救不回来。先确认是过拟合，再上手段。

## 10.2 学习率调度：决定优化器走多快

### 10.2.1 为什么要调度

固定学习率训练的失败模式有两种：

- 学习率太大，前期梯度噪声会把权重打飞
- 学习率太小，后期模型早就该精修了，还在用粗糙的步长

学习率调度就是在不同阶段用不同的步长。两条核心思想：

- **Warmup**：开局用很小的学习率，慢慢拉到目标值。让权重和优化器的动量、二阶矩等"热身"。
- **Decay**：训练到中后期，逐步降低学习率，让损失收敛到更精细的位置。

### 10.2.2 几种主流调度

**Step Decay**：每过 N 步把学习率乘以 0.1。古早 CV 任务的标配，简单但粗糙。

**Cosine Annealing**：让学习率沿余弦曲线从 `lr_max` 降到 `lr_min`：

```
lr(t) = lr_min + 0.5 * (lr_max - lr_min) * (1 + cos(π * t / T))
```

为什么 LLM 训练几乎清一色用 Cosine + Warmup？三个原因：

1. 余弦曲线前期下降慢、后期下降快，给"探索"留出时间，尾段又能精细收敛
2. 末段平滑过渡到小学习率，避免阶梯式骤降带来的震荡
3. 在 Chinchilla、LLaMA、GPT-3 等大量公开训练曲线上验证过，是当前最稳的默认选择

典型配置：

- 总步数 100k，warmup 步数 1k-4k（1%-4%）
- `lr_max` = 3e-4（7B 模型常用），`lr_min` = `lr_max * 0.1`

**OneCycle**：FastAI 推广的调度，先 linear warmup 到峰值再 linear/cosine 降到极小值，适合短训练。

**Linear Decay**：近期一些研究（包括 2024-2025 的几篇大规模实验）发现，在最优学习率下，linear decay to zero 在 compute-optimal 训练上微微优于 cosine。但代价是要事先准确知道总步数。如果你的训练规模可能扩展，cosine 仍然是更稳妥的默认。

**WSD（Warmup-Stable-Decay）**：MiniCPM 等模型推广的方案，warmup 后保持峰值很久，最后一段时间快速 decay。优点是中间段可以无痛延长训练，损失继续降低，需要时再启动 decay 收尾。适合训练计划不确定的场景。

### 10.2.3 PyTorch 实现

PyTorch 自带几种 scheduler，HuggingFace `transformers` 提供了更全的封装：

```python
from transformers import get_cosine_schedule_with_warmup

num_training_steps = len(train_loader) * num_epochs
num_warmup_steps = int(0.03 * num_training_steps)

scheduler = get_cosine_schedule_with_warmup(
    optimizer,
    num_warmup_steps=num_warmup_steps,
    num_training_steps=num_training_steps,
)
```

注意 `scheduler.step()` 必须在 `optimizer.step()` 之后调用。如果你用了梯度累积，scheduler 应该跟着 optimizer 节奏走，而不是每个 batch 都走。Accelerate 的 `prepare` 会帮你处理这个。

### 10.2.4 学习率怎么定

调学习率有一个简单的经验起点：

- 全参数微调小模型（<1B）：`lr` ≈ 5e-5 到 2e-4
- 全参数微调大模型（7B+）：`lr` ≈ 1e-5 到 5e-5
- 从头预训练 Transformer：`lr` ≈ 3e-4 到 1e-3，要严格 warmup
- LoRA 微调：`lr` 可以高一点，1e-4 到 5e-4，因为只更新少量参数

不确定时，做一次 LR Range Test：从 1e-7 开始指数增长到 1，记录 loss。曲线开始下降的位置之后一段就是合理范围，loss 开始爆炸前 10x 是上限。

## 10.3 梯度技巧：救稳定性、救显存

### 10.3.1 梯度裁剪（Gradient Clipping）

训练 LLM 时，偶尔会有几个 batch 梯度异常大——某个长样本、某个数值不稳定的注意力 pattern——一下把权重打飞，loss 飙成 NaN。梯度裁剪就是给梯度的范数加上限：

```python
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
```

逻辑：算出所有参数梯度拼起来的 L2 范数，如果超过 `max_norm`，按比例缩小到 `max_norm`。等于给优化器戴了个安全绳。

LLM 训练几乎都启用，`max_norm` 通常 1.0。CV 模型有时不开，但开了也无害。注意要在 `optimizer.step()` 之前、`backward()` 之后调用。

如果你用 AMP（混合精度），需要在 `scaler.unscale_(optimizer)` 之后再 clip：

```python
scaler.scale(loss).backward()
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
scaler.step(optimizer)
scaler.update()
```

### 10.3.2 梯度累积（Gradient Accumulation）

显存不够装下大 batch，但模型对大 batch 友好怎么办？把大 batch 拆成几个小 batch，逐个前向反向，但**不立刻 step**，把梯度累加几步再统一更新：

```python
accum_steps = 8
optimizer.zero_grad()

for i, batch in enumerate(loader):
    out = model(batch)
    loss = loss_fn(out, batch.target) / accum_steps  # 缩放，保持总梯度等价
    loss.backward()

    if (i + 1) % accum_steps == 0:
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()
        optimizer.zero_grad()
```

效果：等价于 batch_size × accum_steps 的有效 batch，但显存只占 batch_size 那一份。代价是训练慢了 accum_steps 倍——因为本质是用时间换显存。

Accelerate 提供了更省心的封装，分布式场景下还能正确处理梯度同步：

```python
accelerator = Accelerator(gradient_accumulation_steps=8)
model, optimizer, loader, scheduler = accelerator.prepare(
    model, optimizer, loader, scheduler
)

for batch in loader:
    with accelerator.accumulate(model):
        out = model(**batch)
        accelerator.backward(out.loss)
        optimizer.step()
        scheduler.step()
        optimizer.zero_grad()
```

`accumulate` 上下文里会自动跳过中间步的梯度同步通信，只在累积满时同步一次，显著降低分布式开销。

### 10.3.3 梯度检查点（Gradient Checkpointing）

显存最大的去处不是参数本身，而是**激活**——为了 backward，每一层的中间激活都要保存。一个 7B 模型在 batch_size=8、seq_len=2048 下，激活可以吃掉几十 GB。

梯度检查点（也叫 activation checkpointing）的思路是：前向时**不保存**中间激活，只保存输入；反向时再前向计算一次，重建激活。

代价：每一段被检查的子图，前向多算一次（理论 33% 额外计算，实测 20%-40%）。
收益：激活内存可以从 O(L) 降到 O(√L)，省下大半。

PyTorch 原生用法：

```python
from torch.utils.checkpoint import checkpoint

class TransformerBlock(nn.Module):
    def forward(self, x):
        # 普通版本
        return self.mlp(self.norm2(x + self.attn(self.norm1(x))))

# 在父层包一层 checkpoint
def block_forward(blk, x):
    return blk(x)

for blk in self.blocks:
    x = checkpoint(block_forward, blk, x, use_reentrant=False)
```

HuggingFace Transformers 模型有现成开关：

```python
model.gradient_checkpointing_enable()
```

PyTorch 2.x 引入了 **Selective Activation Checkpointing**：可以指定只对某些 op 重计算，避免对昂贵的 matmul 重新计算（matmul 不省，重算反而亏），只重算 LayerNorm、激活函数这些便宜的。这种粒度控制让计算开销从 30% 降到 10% 以下，是现在大规模训练的主流做法。

**LoRA + 梯度检查点的坑**：到 2026 年（PyTorch 2.5+ / PEFT 0.13+），下面这两行**仍然是必备组合**，不要被"PEFT 已修"的传闻误导：

```python
# 启用 LoRA 时正确的检查点配置（2026 仍然是必备）
model.gradient_checkpointing_enable(gradient_checkpointing_kwargs={"use_reentrant": False})
model.enable_input_require_grads()
```

两件事互不替代，分别解决两个不同的问题：

1. `use_reentrant=False`：选择新版非可重入检查点实现。reentrant 版本在重计算时不会重新触发 forward hooks，LoRA 适配器（或任何 hook 注册的旁路分支）会拿不到梯度；non-reentrant 版本走 `torch.utils.checkpoint` 的新路径，hooks 正常触发。PyTorch 已计划把默认值切到 False，但当前主流版本默认仍是 True，所以**必须显式传**。
2. `enable_input_require_grads()`：让基座模型 embedding 的输出 `requires_grad=True`。基座参数被 PEFT 冻结后默认 `requires_grad=False`，整条计算图从 embedding 起就不带梯度，反向传到 checkpoint 边界时会发现"没有可微分输入"，整段重计算被跳过，LoRA 拿到的全是 None。这一行的作用是把"梯度入口"重新打开，是 PEFT 与 gradient checkpointing 共存的桥梁，PyTorch / PEFT 都不会自动加。

如果你看到 LoRA 训练 loss 不降、`p.grad` 全 None，**先检查这两行是否都加了**。少任意一行都会出问题，但症状一样（loss 不降），很容易误判。

## 10.4 混合精度：用更少比特跑更快训练

### 10.4.1 为什么要混合精度

FP32（单精度）每个数 4 字节，权重、梯度、优化器状态都按这个存。一个 7B 模型光参数就 28GB，梯度再 28GB，AdamW 的 m/v 再加 56GB——总共 112GB，单卡装不下。

如果用 FP16（半精度），每个数 2 字节，全部减半。算力上，A100/H100 的 Tensor Core 跑 FP16 是 FP32 的 8 倍以上速度。

但 FP16 有个根本问题：动态范围窄（约 6e-5 到 6.5e4），梯度容易下溢到 0，loss 容易上溢到 inf。直接全 FP16 训练会崩。

**混合精度（Mixed Precision）的方案**：

- 前向、反向用 FP16 算（快、省）
- 权重的 master copy 用 FP32 保存（精度）
- 优化器更新在 FP32 上做（稳定）
- 用 **loss scaling** 把 loss 放大若干倍，让小梯度不下溢；更新前再 unscale 回去

### 10.4.2 FP16 vs BF16 vs FP8

**FP16**：1 符号 + 5 指数 + 10 尾数。范围窄但精度高。需要 loss scaling。

**BF16**（brain float 16）：1 符号 + 8 指数 + 7 尾数。和 FP32 一样的指数位，动态范围 = FP32，精度比 FP16 低。**几乎不需要 loss scaling，训练稳定性远好于 FP16**。Ampere（A100）及以后的 NVIDIA GPU 都原生支持。

> 经验法则：A100/H100 上能用 BF16 就别用 FP16。除非显卡只有 V100（不支持 BF16）才退回 FP16。

**FP8**：H100 的 Hopper 架构开始原生支持。两种格式 E4M3（精度优先）和 E5M2（范围优先），训练时 NVIDIA Transformer Engine 会自动选择和管理 scaling。

2026 年 FP8 已经是大模型训练的常态选择：

- 内存带宽减半，TFLOPS 翻倍
- 在 ViT/Swin、GPT 系列上验证过 loss 曲线和 BF16 几乎重合
- 仍然需要 master 权重保留 BF16 或 FP32

但 FP8 的工程化门槛更高：必须 H100/H200 或 Blackwell GPU、必须用 Transformer Engine、对部分算子（如某些激活）需要保留高精度。一般 7B 以下模型用 BF16 就足够，FP8 的收益要在 30B+ 才显著。

### 10.4.3 PyTorch AMP 用法

```python
from torch.amp import autocast, GradScaler

# BF16：不需要 scaler，autocast 即可
for batch in loader:
    optimizer.zero_grad()
    with autocast(device_type="cuda", dtype=torch.bfloat16):
        out = model(batch)
        loss = loss_fn(out, batch.target)
    loss.backward()
    optimizer.step()

# FP16：需要 GradScaler 防止下溢
scaler = GradScaler()
for batch in loader:
    optimizer.zero_grad()
    with autocast(device_type="cuda", dtype=torch.float16):
        out = model(batch)
        loss = loss_fn(out, batch.target)
    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)  # 为了 grad clip
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer)
    scaler.update()
```

Accelerate 把这些都封掉了，传一个参数就行：

```python
accelerator = Accelerator(mixed_precision="bf16")  # 或 "fp16" / "fp8"
```

## 10.5 分布式训练：从单卡到千卡

### 10.5.1 三种并行的脉络

模型大到一张卡放不下，或者数据多到单卡训太慢，就要走并行。三种基本范式：

**数据并行（Data Parallel, DP / DDP）**：每张卡都有一份完整模型，把 batch 切成几份，每张卡算一份梯度，最后 all-reduce 同步梯度更新参数。最简单也最常用。

**张量并行（Tensor Parallel, TP）**：把单个矩阵乘法在维度上切开，比如 `Y = XW`，把 W 按列切，每张卡算 `Y` 的一列。每一层都要通信，对带宽要求极高，通常只在单机内部 NVLink 用。Megatron-LM 的核心做法。

**流水线并行（Pipeline Parallel, PP）**：把模型按层切成若干段，每段放在一组卡上，前向后向像流水线一样在 stage 之间传递。需要把 batch 切成 micro-batch 才能填满流水线，实现复杂。

实际工程里这三种常组合用——比如 GPT-3 级别训练用的是 "TP=8 + PP=stages + DP=replicas" 的 3D 并行。但 AI 应用工程师 99% 的场景是数据并行 + 一点点 ZeRO/FSDP 的优化即可。

### 10.5.2 DDP：最常见的多卡训练

`torch.nn.parallel.DistributedDataParallel`（DDP）是 PyTorch 数据并行的标准做法。每张卡一个进程，每个进程有完整模型副本，反向传播时通过 NCCL 做 all-reduce 同步梯度。

最小骨架：

```python
import os
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

def setup():
    dist.init_process_group(backend="nccl")
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)
    return local_rank

def cleanup():
    dist.destroy_process_group()

local_rank = setup()
device = torch.device(f"cuda:{local_rank}")

model = MyModel().to(device)
model = DDP(model, device_ids=[local_rank])

sampler = torch.utils.data.distributed.DistributedSampler(dataset)
loader = DataLoader(dataset, batch_size=32, sampler=sampler)

# 训练循环跟单卡一样
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)  # 每个 epoch 重新打乱
    for batch in loader:
        ...
```

启动方式：

```bash
torchrun --nproc_per_node=8 train.py
```

`torchrun`（torch.distributed.launch 的替代）会启动 8 个进程，每个进程自动拿到 `LOCAL_RANK`、`RANK`、`WORLD_SIZE` 环境变量。

DDP 的限制：每张卡都要装下完整模型 + 梯度 + 优化器状态。13B 模型的 AdamW 状态就 100GB+，单张 80GB H100 也装不下。这就是 ZeRO/FSDP 出场的地方。

### 10.5.3 ZeRO：把训练状态切片

ZeRO（Zero Redundancy Optimizer）是 DeepSpeed 的核心思想，用于在数据并行的基础上消除冗余。普通 DDP 里，每张卡都存了一份完整的优化器状态、梯度、参数——这些在多卡之间是冗余的。ZeRO 把它们切片：

混合精度训练里，每个参数的「全套训练状态」典型占用是这样：BF16/FP16 参数 2 字节 + BF16/FP16 梯度 2 字节 + FP32 master 参数 4 字节 + AdamW 的 FP32 一阶矩 4 字节 + FP32 二阶矩 4 字节 = **16 字节/参数**。其中 12 字节（FP32 master + m + v）属于「优化器状态」。一个 7B 模型整套就是 7B × 16 ≈ 112GB。ZeRO 就是把这 16 字节按层级切到不同卡上：

- **ZeRO-1**：分片优化器状态（FP32 master 参数 + Adam 的 m、v，共 12 字节/参数）。每张卡只存 1/N 的优化器状态。通信开销几乎不变，**16 字节 → 4 + 12/N 字节**，N=8 时一张卡只占 5.5 字节/参数。
- **ZeRO-2**：在 1 的基础上再分片梯度。每张卡只在反向时收集自己负责那一部分参数的梯度。**进一步降到 2 + (4+12)/N 字节**，N=8 时约 4 字节/参数。
- **ZeRO-3**：在 2 的基础上再分片参数本身。每张卡只存 1/N 的权重，前向/反向时按需 all-gather。**降到 16/N 字节**，理论上 N 张卡可以训 N 倍大的模型。通信开销最大（每层都要 all-gather 参数）。
- **ZeRO-Infinity**：在 ZeRO-3 上加 CPU/NVMe offload。把分片的参数/优化器状态进一步搬到 CPU 内存或 NVMe SSD，能训百亿、千亿模型，但延迟显著升高。

口诀：

> 训练 7B 模型够用，开 ZeRO-2；
> 训练 13B-70B 模型且单机多卡放不下，开 ZeRO-3；
> 训练 70B+ 或单机配置紧张，开 ZeRO-3 + CPU offload；
> 卡少模型超大，再上 NVMe offload。

### 10.5.4 FSDP：PyTorch 原生的 ZeRO-3

FSDP（Fully Sharded Data Parallel）是 PyTorch 官方版本的 ZeRO-3，思路完全一致。2024-2025 年 PyTorch 推出了 FSDP2，重写了核心实现：把 FSDP1 的 `FlatParameter` 替换成 per-parameter `DTensor` 分片，state dict 直接拿到分片后的 DTensor，与 PyTorch 原生 `torch.distributed.tensor` 体系无缝衔接，API 也从 `FullyShardedDataParallel` 类换成更轻的 `fully_shard()` 函数。

什么时候选 FSDP，什么时候选 DeepSpeed？2026 年的实践共识：

| 场景 | 推荐 |
|---|---|
| 7B-70B 模型微调，2-8 卡单机 | **FSDP2**（PyTorch 原生，零依赖） |
| 70B+ 模型预训练 | **DeepSpeed**（吞吐稳定性更成熟） |
| 需要 CPU/NVMe offload | **DeepSpeed ZeRO-Infinity**（offload 实现更成熟） |
| 已经有 HuggingFace 训练栈 | 两个都行，Accelerate 帮你切换 |
| 单纯做实验、要灵活 | **FSDP2** |

一句话口诀：**2026 年 FSDP2 已 GA，没有特殊需求（offload、ZeRO++、特殊 fused 算子）就默认 FSDP2**。把 DeepSpeed 当作"FSDP2 不够用时的逃生口"，而不是默认选项。这和 2023 年"先看 DeepSpeed"的习惯反过来了。

DeepSpeed 的额外优势：fused Adam（CUDA 实现，比 PyTorch 原生快）、活动检查点的更细管理、ZeRO++（带宽优化）。

FSDP 的优势：原生集成、调试方便、状态字典处理简单、不需要单独的 launcher。

FSDP2 最小用法：

```python
import os
import torch
import torch.distributed as dist
from torch.distributed.fsdp import fully_shard, MixedPrecisionPolicy

dist.init_process_group("nccl")
local_rank = int(os.environ["LOCAL_RANK"])
torch.cuda.set_device(local_rank)

model = MyTransformer().cuda()

mp_policy = MixedPrecisionPolicy(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.float32,  # 梯度 reduce 用 fp32 更稳，loss scale 不需要
)

# 对每个 transformer block 单独 shard，最常见的 wrap 策略
# 注意：内层 block 必须先 shard，最后再 shard 整个 model（顺序很重要）
for block in model.transformer_blocks:
    fully_shard(block, mp_policy=mp_policy)
fully_shard(model, mp_policy=mp_policy)
```

`fully_shard` 是函数式 API，原地把模块改造成 FSDP2 容器；`MixedPrecisionPolicy` 比 FSDP1 的 `MixedPrecision` 字段精简，常见做法是 `param_dtype=bf16` + `reduce_dtype=fp32`——梯度通信走 fp32 累加更稳。

### 10.5.5 选型决策树

```
模型能不能放进单卡？
├─ 能 → 看是否需要多卡加速
│   ├─ 需要 → DDP + 适当 batch
│   └─ 不需要 → 单卡就够
└─ 不能
    ├─ 减小 batch + grad accum 能不能放下？
    │   ├─ 能 → DDP + grad accum + grad checkpoint + BF16
    │   └─ 不能 → 走分片
    │
    └─ 多卡分片
        ├─ PyTorch 原生栈 → FSDP2
        ├─ HF Trainer 或自定义训练 → DeepSpeed ZeRO-2 / 3
        └─ 还是不够 → ZeRO-3 + CPU offload → + NVMe offload
```

## 10.6 Hugging Face Accelerate：把分布式封掉

### 10.6.1 为什么用 Accelerate

DDP 的代码要改，FSDP 要写 wrap 策略，DeepSpeed 要写 JSON 配置——每个后端 API 都不一样。Accelerate 是 HuggingFace 出的统一抽象层，让同一份训练代码可以无缝跑在：

- 单 CPU / 单 GPU
- 单机多卡（DDP / FSDP）
- 多机多卡
- TPU
- 配上 DeepSpeed / FSDP / Megatron 的所有变体

只要你写的训练代码符合一个简单的"模板"，切换后端只需改配置文件或命令行参数。

### 10.6.2 完整训练脚本

下面是一个完整的、可以直接 `accelerate launch` 跑起来的多卡训练脚本，包含混合精度、梯度累积、梯度裁剪、cosine 调度、W&B 追踪：

```python
# train.py
import os
import torch
from torch.optim import AdamW
from torch.utils.data import DataLoader
from accelerate import Accelerator
from accelerate.utils import set_seed
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    get_cosine_schedule_with_warmup,
)
from datasets import load_dataset


def make_param_groups(model, weight_decay=0.1):
    decay, no_decay = [], []
    for name, param in model.named_parameters():
        if not param.requires_grad:
            continue
        if param.ndim <= 1 or "norm" in name.lower() or name.endswith(".bias"):
            no_decay.append(param)
        else:
            decay.append(param)
    return [
        {"params": decay, "weight_decay": weight_decay},
        {"params": no_decay, "weight_decay": 0.0},
    ]


def main():
    set_seed(42)

    accelerator = Accelerator(
        mixed_precision="bf16",
        gradient_accumulation_steps=4,
        log_with="wandb",
        project_dir="./outputs",
    )
    accelerator.init_trackers("my-llm-finetune")

    # 模型与 tokenizer
    model_name = "Qwen/Qwen2.5-1.5B"
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModelForCausalLM.from_pretrained(
        model_name, torch_dtype=torch.bfloat16
    )
    model.gradient_checkpointing_enable(
        gradient_checkpointing_kwargs={"use_reentrant": False}
    )

    # 数据
    raw = load_dataset("json", data_files="data/train.jsonl", split="train")

    def tokenize_fn(ex):
        out = tokenizer(
            ex["text"], truncation=True, max_length=2048, padding="max_length"
        )
        out["labels"] = out["input_ids"].copy()
        return out

    ds = raw.map(tokenize_fn, batched=True, remove_columns=raw.column_names)
    ds.set_format(type="torch")
    loader = DataLoader(ds, batch_size=4, shuffle=True, num_workers=4)

    # 优化器与调度器
    optimizer = AdamW(make_param_groups(model), lr=2e-5, betas=(0.9, 0.95))

    num_epochs = 3
    total_steps = (len(loader) // accelerator.gradient_accumulation_steps) * num_epochs
    scheduler = get_cosine_schedule_with_warmup(
        optimizer,
        num_warmup_steps=int(0.03 * total_steps),
        num_training_steps=total_steps,
    )

    # prepare 一次解决所有分布式包装
    model, optimizer, loader, scheduler = accelerator.prepare(
        model, optimizer, loader, scheduler
    )

    global_step = 0
    for epoch in range(num_epochs):
        model.train()
        for batch in loader:
            with accelerator.accumulate(model):
                out = model(**batch)
                loss = out.loss
                accelerator.backward(loss)

                if accelerator.sync_gradients:
                    accelerator.clip_grad_norm_(model.parameters(), 1.0)

                optimizer.step()
                scheduler.step()
                optimizer.zero_grad()

            if accelerator.sync_gradients:
                global_step += 1
                if global_step % 10 == 0:
                    accelerator.log(
                        {
                            "loss": loss.item(),
                            "lr": scheduler.get_last_lr()[0],
                            "step": global_step,
                        },
                        step=global_step,
                    )

        # 每个 epoch 存一次
        accelerator.wait_for_everyone()
        accelerator.save_state(f"./outputs/ckpt-epoch{epoch}")

    accelerator.end_training()


if __name__ == "__main__":
    main()
```

### 10.6.3 启动方式

先生成配置：

```bash
accelerate config
```

它会问你：单机/多机？几张卡？是否启用 DeepSpeed？混合精度？答完会写到 `~/.cache/huggingface/accelerate/default_config.yaml`。

然后启动：

```bash
# 8 卡 DDP
accelerate launch --num_processes 8 train.py

# 8 卡 FSDP2（Accelerate 1.x 起，--fsdp_version=2 启用 FSDP2）
accelerate launch \
    --num_processes 8 \
    --use_fsdp \
    --fsdp_version 2 \
    --fsdp_reshard_after_forward true \
    --fsdp_auto_wrap_policy TRANSFORMER_BASED_WRAP \
    --fsdp_transformer_layer_cls_to_wrap Qwen2DecoderLayer \
    --mixed_precision bf16 \
    train.py

# 8 卡 DeepSpeed ZeRO-2，配置文件指定细节
accelerate launch \
    --num_processes 8 \
    --use_deepspeed \
    --deepspeed_config_file ds_z2.json \
    train.py
```

注意 FSDP2 的字段名相对 FSDP1 有变化——FSDP1 的 `--fsdp_sharding_strategy FULL_SHARD` 在 FSDP2 里改用 `--fsdp_reshard_after_forward true`（前向后重新分片，等价于 FULL_SHARD），同名字段 `--fsdp_offload_params`、`--fsdp_state_dict_type` 在两个版本都保留。

关于 **ND-Parallel**：Accelerate 1.x 起支持在单条命令里同时配置 DP/TP/CP（context parallel）三种切片维度。但绝不是"看到了就该用"——它的真正适用场景非常窄：

- **值得用**：70B+ 模型 + 多机（≥16 卡）+ 单条样本 seq_len 极长（≥32K，CP 才有意义）+ 已经被 FSDP2 单维分片的通信开销卡住吞吐。典型例子：千亿级模型预训练、长上下文 RLHF。
- **不值得用**：<70B 模型、单机、seq_len < 8K、调试期。多一维并行就多一组 hyperparameter（mesh shape、TP degree、CP degree），调错了 throughput 反而比纯 FSDP2 还低。
- **铁律**：先把 FSDP2 单维 + 梯度累积 + activation checkpointing 这套吃干净，仍然是通信瓶颈再上 ND-Parallel。不要从一开始就 3D 并行——出问题时根本不知道是哪一维度配错了。

整个训练脚本不需要改一行代码就能切换后端，这就是 Accelerate 的核心价值。

## 10.7 DeepSpeed：配置文件驱动

### 10.7.1 DeepSpeed 的工作模式

DeepSpeed 有自己的初始化流程，通常你要：

1. 写一个 JSON 配置文件描述训练设置
2. 用 `deepspeed.initialize()` 包装模型和优化器
3. 用 `deepspeed train.py --deepspeed ds_config.json` 启动

如果用 Accelerate 或 HuggingFace Trainer，它们会帮你处理这些，你只需要给配置文件。

### 10.7.2 ZeRO-2 配置示例

适合 7B-13B 模型微调，单机 8 卡，能放下完整参数但显存紧张：

```json
{
    "train_batch_size": "auto",
    "train_micro_batch_size_per_gpu": "auto",
    "gradient_accumulation_steps": "auto",
    "gradient_clipping": 1.0,
    "bf16": { "enabled": true },
    "zero_optimization": {
        "stage": 2,
        "contiguous_gradients": true,
        "overlap_comm": true,
        "reduce_scatter": true,
        "reduce_bucket_size": 5e8,
        "allgather_bucket_size": 5e8,
        "offload_optimizer": {
            "device": "cpu",
            "pin_memory": true
        }
    },
    "optimizer": {
        "type": "AdamW",
        "params": {
            "lr": "auto",
            "betas": [0.9, 0.95],
            "eps": 1e-8,
            "weight_decay": "auto"
        }
    },
    "scheduler": {
        "type": "WarmupCosineLR",
        "params": {
            "total_num_steps": "auto",
            "warmup_min_ratio": 0,
            "warmup_num_steps": "auto",
            "cos_min_ratio": 0.1
        }
    }
}
```

`"auto"` 是 HuggingFace 集成的特殊标记——这些字段会从你的 `TrainingArguments` 或 Accelerate 配置里推断出来，避免在两个地方重复定义。

### 10.7.3 ZeRO-3 配置示例

适合 30B+ 模型，单机放不下完整参数：

```json
{
    "train_batch_size": "auto",
    "train_micro_batch_size_per_gpu": "auto",
    "gradient_accumulation_steps": "auto",
    "gradient_clipping": 1.0,
    "bf16": { "enabled": true },
    "zero_optimization": {
        "stage": 3,
        "contiguous_gradients": true,
        "overlap_comm": true,
        "reduce_bucket_size": "auto",
        "stage3_prefetch_bucket_size": "auto",
        "stage3_param_persistence_threshold": "auto",
        "stage3_max_live_parameters": 1e9,
        "stage3_max_reuse_distance": 1e9,
        "stage3_gather_16bit_weights_on_model_save": true,
        "offload_optimizer": {
            "device": "cpu",
            "pin_memory": true
        },
        "offload_param": {
            "device": "cpu",
            "pin_memory": true
        }
    }
}
```

关键参数解释：

- `stage3_max_live_parameters`：同时驻留显存的参数上限。越大越快越吃显存。
- `stage3_prefetch_bucket_size`：提前 all-gather 多少参数。和模型层大小匹配。
- `offload_optimizer.device = "cpu"`：把 AdamW 状态搬到 CPU 内存，省 12 字节/参数的显存。
- `offload_param.device = "cpu"`：把分片的参数也搬到 CPU。最后一招，吞吐会显著下降。
- `stage3_gather_16bit_weights_on_model_save`：保存模型时聚合所有分片到 rank 0 写出。否则每张卡各存自己的分片，加载麻烦。

### 10.7.4 NVMe Offload

如果连 CPU 内存都不够装下 AdamW 状态，可以再下沉到 NVMe SSD：

```json
"offload_optimizer": {
    "device": "nvme",
    "nvme_path": "/local_nvme",
    "pin_memory": true,
    "buffer_count": 4,
    "fast_init": false
}
```

代价：吞吐进一步下降。一般要求 NVMe 是真正的本地高性能盘（NVMe Gen4 以上），否则 IO 瓶颈会让训练变成"幻灯片"。

## 10.8 多机训练：torchrun 与 SLURM

### 10.8.1 torchrun

`torchrun` 是 PyTorch 1.10+ 的统一启动器，兼容容器、k8s、SLURM。单机多卡：

```bash
torchrun --standalone --nproc_per_node=8 train.py
```

多机多卡，每台机器跑：

```bash
# Master 节点（rank 0）
torchrun \
    --nnodes=2 \
    --nproc_per_node=8 \
    --node_rank=0 \
    --master_addr=10.0.0.1 \
    --master_port=29500 \
    train.py

# Worker 节点（rank 1）
torchrun \
    --nnodes=2 \
    --nproc_per_node=8 \
    --node_rank=1 \
    --master_addr=10.0.0.1 \
    --master_port=29500 \
    train.py
```

每个进程会在环境变量里看到 `RANK`、`LOCAL_RANK`、`WORLD_SIZE`、`MASTER_ADDR`、`MASTER_PORT`。

### 10.8.2 SLURM 简介

SLURM 是 HPC 集群最常用的作业调度器。一个最小训练脚本：

```bash
#!/bin/bash
#SBATCH --job-name=llm-train
#SBATCH --nodes=4
#SBATCH --ntasks-per-node=1
#SBATCH --gres=gpu:8
#SBATCH --cpus-per-task=64
#SBATCH --time=24:00:00
#SBATCH --output=logs/%j.out

module load cuda/12.4

# SLURM 自动设的环境变量
export MASTER_ADDR=$(scontrol show hostnames $SLURM_JOB_NODELIST | head -n 1)
export MASTER_PORT=29500
export WORLD_SIZE=$(($SLURM_NNODES * 8))

srun torchrun \
    --nnodes=$SLURM_NNODES \
    --nproc_per_node=8 \
    --rdzv_id=$SLURM_JOB_ID \
    --rdzv_backend=c10d \
    --rdzv_endpoint=$MASTER_ADDR:$MASTER_PORT \
    train.py
```

`sbatch run.sh` 提交后，SLURM 会分配节点、设置环境、运行脚本。

实际工程里，多机训练常见的坑：

- **网络**：节点之间要有 InfiniBand 或 RDMA over RoCE。普通 1Gb 以太网会让 all-reduce 慢得无法接受。检查 `NCCL_IB_HCA`、`NCCL_SOCKET_IFNAME`。
- **NCCL 调试**：`export NCCL_DEBUG=INFO` 可以看到通信细节，调试连接问题必备。
- **节点失联**：长时间训练时 NCCL 偶尔会因为某节点 OOM/网络抖动死锁。开启 `NCCL_TIMEOUT` 与 PyTorch 的 ProcessGroup timeout，让进程及时崩溃而不是无限等待。
- **数据加载**：每个节点最好能本地缓存数据，不然多机同时读 NFS 会变成 IO 瓶颈。

## 10.9 训练监控：W&B、TensorBoard、曲线诊断

### 10.9.1 为什么要监控

训练一个 7B 模型一次几小时到几天，盯着终端 print 不现实。要有可视化、可比对的训练曲线、可远程查看的 dashboard。常用工具：

- **TensorBoard**：本地或简单部署，免费，对小团队够用
- **Weights & Biases (W&B)**：云端 SaaS，experiment tracking 体验好，团队协作强，免费层够个人用
- **MLflow**：开源、自托管的折中
- **Aim、Neptune**：其他可选项

最常用的是 W&B，集成简单：

```python
import wandb

wandb.init(project="my-llm", config={"lr": 2e-5, "batch": 32})

for step, batch in enumerate(loader):
    loss = train_step(batch)
    if step % 10 == 0:
        wandb.log({"train/loss": loss, "lr": scheduler.get_last_lr()[0]}, step=step)
```

Accelerate 内置了多 tracker 支持，一个参数切换：

```python
accelerator = Accelerator(log_with="wandb")  # 或 "tensorboard"
accelerator.init_trackers("project-name")
accelerator.log({"loss": loss}, step=global_step)
```

### 10.9.2 应该看什么指标

最低限度的训练 dashboard：

- **train_loss**：每若干步记录，看趋势
- **val_loss / 困惑度**：每个 epoch 或固定步数评估一次
- **learning_rate**：确认 scheduler 工作正常
- **grad_norm**：每步的总梯度范数，看到飙高就知道要爆炸了
- **gpu_memory**：避免显存悄悄涨到 OOM
- **throughput (tokens/s)**：判断分布式效率，多卡却没多速度就是配置问题

进阶：

- 每层的权重范数 / 梯度范数 / 激活分布
- 注意力可视化、参数更新方向相关性

### 10.9.3 训练曲线异常诊断

**Loss 爆炸（Loss 飙到 NaN/inf）**

```
loss: 2.3, 2.1, 1.9, 1.7, ..., 1.2, NaN, NaN, ...
```

可能原因：

- 学习率太大（最常见）。先把 lr 减半重试。
- 没开 grad clip。加上 `clip_grad_norm_=1.0`。
- 混合精度下溢（FP16），换 BF16 或加 GradScaler。
- 数据里有异常值（极长序列、极端 token 分布）。
- 模型初始化问题，某些层方差过大。
- AdamW 的 epsilon 太小，分母数值下溢。

诊断流程：先看上一步的 grad_norm 是不是已经异常大；再看是哪一层爆炸（hook 打印每层的 norm）；最后回到数据/超参。

**Loss 不降**

```
loss: 2.5, 2.5, 2.5, 2.5, ...
```

可能原因：

- 学习率太小或 warmup 步数太多还没结束
- 数据没正确传入（labels 全是 -100、tokenizer 出错）
- 梯度没传到模型参数（requires_grad 配置错了，比如 LoRA 没生效）
- 模型在 eval 模式（忘了 `model.train()`）
- 损失函数错误（CrossEntropy 期望的形状不对）

诊断：先小数据集 overfitting 测试——拿 100 条数据训 10 个 epoch，正常的话 loss 应该降到很低。降不下来，bug 在代码；降下来就是数据/超参问题。

**过拟合（训练降验证升）**

```
train_loss: 1.0 → 0.3
val_loss:   1.2 → 1.5  (越训越差)
```

- 加 weight decay、加 dropout
- 加数据增强
- 减小模型 / 减少训练步数
- early stopping
- 检查训练集和验证集是否真的独立（数据泄漏经常是元凶）

**Loss 震荡**

- batch size 太小 → 增大 batch 或加梯度累积
- 学习率太大 → 减小或更长 warmup
- 数据 shuffle 没做 → 检查 sampler

## 10.10 实战：用 LoRA 微调小模型

把这一章的所有技巧串起来，做一次端到端微调。场景：用 LoRA 微调 Qwen2.5-1.5B，在单张 24GB 消费级 GPU（如 RTX 4090）上跑通，用上梯度检查点 + BF16 混合精度。

```python
# train_lora.py
import os
import torch
from torch.optim import AdamW
from torch.utils.data import DataLoader
from accelerate import Accelerator
from accelerate.utils import set_seed
from datasets import load_dataset
from peft import LoraConfig, get_peft_model, TaskType
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    get_cosine_schedule_with_warmup,
    DataCollatorForLanguageModeling,
)


def main():
    set_seed(42)

    accelerator = Accelerator(
        mixed_precision="bf16",
        gradient_accumulation_steps=8,
        log_with="wandb",
        project_dir="./outputs-lora",
    )
    accelerator.init_trackers(
        "qwen-lora-finetune",
        config={"lr": 2e-4, "lora_r": 16, "lora_alpha": 32},
    )

    # 1. 加载基座模型，BF16 直接装载
    model_name = "Qwen/Qwen2.5-1.5B"
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    if tokenizer.pad_token is None:
        tokenizer.pad_token = tokenizer.eos_token

    model = AutoModelForCausalLM.from_pretrained(
        model_name,
        torch_dtype=torch.bfloat16,
    )

    # 2. 启用梯度检查点（LoRA 兼容写法）
    model.gradient_checkpointing_enable(
        gradient_checkpointing_kwargs={"use_reentrant": False}
    )
    model.enable_input_require_grads()

    # 3. 套上 LoRA
    lora_config = LoraConfig(
        task_type=TaskType.CAUSAL_LM,
        r=16,
        lora_alpha=32,
        lora_dropout=0.05,
        target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    )
    model = get_peft_model(model, lora_config)
    model.print_trainable_parameters()
    # 输出类似：trainable params: 4,718,592 / 1,543,714,304 ≈ 0.30%

    # 4. 数据
    raw = load_dataset("HuggingFaceH4/no_robots", split="train_sft").select(range(2000))

    def format_fn(ex):
        msgs = ex["messages"]
        text = tokenizer.apply_chat_template(msgs, tokenize=False)
        out = tokenizer(text, truncation=True, max_length=1024, padding="max_length")
        out["labels"] = out["input_ids"].copy()
        return out

    ds = raw.map(format_fn, remove_columns=raw.column_names)
    ds.set_format(type="torch")
    loader = DataLoader(ds, batch_size=2, shuffle=True, num_workers=2)

    # 5. 优化器只更新 LoRA 参数（PEFT 已经把其余冻结了）
    trainable = [p for p in model.parameters() if p.requires_grad]
    optimizer = AdamW(trainable, lr=2e-4, betas=(0.9, 0.95), weight_decay=0.0)

    num_epochs = 3
    total_updates = (len(loader) // accelerator.gradient_accumulation_steps) * num_epochs
    scheduler = get_cosine_schedule_with_warmup(
        optimizer,
        num_warmup_steps=int(0.05 * total_updates),
        num_training_steps=total_updates,
    )

    model, optimizer, loader, scheduler = accelerator.prepare(
        model, optimizer, loader, scheduler
    )

    # 6. 训练循环
    global_step = 0
    for epoch in range(num_epochs):
        model.train()
        for batch in loader:
            with accelerator.accumulate(model):
                out = model(**batch)
                loss = out.loss
                accelerator.backward(loss)

                if accelerator.sync_gradients:
                    grad_norm = accelerator.clip_grad_norm_(trainable, 1.0)

                optimizer.step()
                scheduler.step()
                optimizer.zero_grad()

            if accelerator.sync_gradients:
                global_step += 1
                if global_step % 5 == 0:
                    accelerator.log(
                        {
                            "train/loss": loss.item(),
                            "train/lr": scheduler.get_last_lr()[0],
                            "train/grad_norm": grad_norm.item()
                            if hasattr(grad_norm, "item")
                            else float(grad_norm),
                            "epoch": epoch,
                        },
                        step=global_step,
                    )

        # 每个 epoch 保存一次 LoRA adapter（只存几 MB）
        accelerator.wait_for_everyone()
        unwrapped = accelerator.unwrap_model(model)
        unwrapped.save_pretrained(
            f"./outputs-lora/adapter-epoch{epoch}",
            save_function=accelerator.save,
        )

    accelerator.end_training()


if __name__ == "__main__":
    main()
```

启动：

```bash
accelerate launch train_lora.py
```

这个脚本演示了所有关键工程化要点：

1. **BF16 混合精度**：让 1.5B 模型的前向后向都跑在 BF16，省一半激活显存
2. **梯度检查点 + LoRA**：启用 `use_reentrant=False` 和 `enable_input_require_grads()`，避免梯度断流
3. **梯度累积**：`gradient_accumulation_steps=8`，把有效 batch 从 2 拉到 16
4. **梯度裁剪**：通过 `accelerator.clip_grad_norm_`，自动处理混合精度的 unscale
5. **Cosine + Warmup**：5% warmup，余弦衰减到 0
6. **W&B 监控**：实时看 loss、lr、grad_norm
7. **只保存 adapter**：LoRA 权重只有几 MB，存档便宜

显存占用（RTX 4090，24GB）：

| 配置 | 显存峰值 |
|---|---|
| 全参数微调 1.5B FP32 | OOM |
| 全参数微调 1.5B BF16 | ~14GB（紧张） |
| LoRA + BF16 | ~7GB |
| LoRA + BF16 + grad_ckpt | ~5GB |
| LoRA + BF16 + grad_ckpt + 8bit base | ~3GB |

最后一行的 8-bit 是用 `bitsandbytes` 把基座量化到 INT8 加载，配合 LoRA 几乎是单卡微调中等大小模型的标配。LoRA / QLoRA / Adapter 这一系列 PEFT 方法会在第 13 章微调篇详细讲。

## 10.11 常见坑与排错清单

### 10.11.1 OOM（Out of Memory）

显存不够。按"代价从小到大"的顺序尝试：

1. 减小 `batch_size`（最直接）
2. 加 `gradient_accumulation_steps` 补回有效 batch
3. 启用混合精度（FP32 → BF16/FP16）
4. 启用 `gradient_checkpointing`
5. 降低 `max_seq_length`（如果可以）
6. 上 LoRA 而不是全参数微调
7. 上 ZeRO-2 / FSDP
8. 上 ZeRO-3 + CPU offload
9. 上 8bit/4bit base 量化（bitsandbytes、GPTQ）

排查工具：

```python
# 当前峰值显存
print(torch.cuda.max_memory_allocated() / 1e9, "GB")
# 实时显存
print(torch.cuda.memory_allocated() / 1e9, "GB")
# 重置统计
torch.cuda.reset_peak_memory_stats()
```

`nvidia-smi` 看到的不一定是真实占用，CUDA caching allocator 会预留。`torch.cuda.memory_allocated()` 才是模型真实占用。

### 10.11.2 NaN / Inf

按 10.9.3 的诊断流程走。补充几个具体检查：

```python
# 在反向后立刻检查
for name, p in model.named_parameters():
    if p.grad is not None:
        if torch.isnan(p.grad).any() or torch.isinf(p.grad).any():
            print(f"NaN/Inf gradient in {name}")
```

如果是 FP16 训练，`GradScaler.scale` 后的 loss 是不是变 inf 了？换 BF16。

如果是 attention 里的 softmax 数值不稳，看是不是用了 SDPA / Flash Attention，原生 Python 实现的 attention 在长序列上更容易爆炸。

### 10.11.3 收敛慢 / 模型不学习

经典 debug 流程，按顺序排查：

1. **能否 overfit 一个小数据集？** 拿 100 条数据训 50 个 epoch，loss 应该接近 0。降不下来，bug 在代码（数据流、loss、模型构图）。
2. **labels 是不是真的进了 loss？** 打印 `out.loss` 和手算的 cross-entropy 对比。
3. **梯度有没有传？** 训练几步后看 `p.grad.abs().mean()`，全是 0 说明被 detach 了或 requires_grad 没开。
4. **优化器有没有更新？** 对比一步前后某个参数 `p.data.clone()`，没变说明 optimizer 链没接好。
5. **学习率是不是 0？** 打印 `scheduler.get_last_lr()`，warmup 太长或调度器配置错会让你前 N 步 lr 几乎为 0。
6. **数据有没有 shuffle？** 没 shuffle 时 loss 会按数据顺序出怪曲线。
7. **数值范围对吗？** 输入归一化没做的图像、token id 越界的文本，都会让模型学不到东西。

### 10.11.4 多卡跑得不快

理想情况下 N 张卡应该接近 N 倍速度。如果只有 1.5x 甚至 1x：

- **batch_size 是不是太小？** 通信开销固定，每步算的少就显得通信占比高。
- **NCCL 通信是不是走了慢链路？** `NCCL_DEBUG=INFO` 看每步通信耗时。
- **DataLoader 是不是瓶颈？** 调大 `num_workers`，开 `pin_memory=True`。
- **梯度累积有没有正确跳过中间同步？** Accelerate 的 `accumulate` 上下文会跳过；裸 DDP 要用 `model.no_sync()`。
- **是不是某些操作把所有卡同步到了 rank 0？** 比如不必要的 `.item()`、`.cpu()`、print。
- **跨节点训练的 IB/RoCE 网卡配置正确吗？** PCIe 通信和 IB 通信差一个数量级。

### 10.11.5 检查点保存与恢复出错

- FSDP/ZeRO-3 下保存模型必须用框架提供的 API（`accelerator.save_state`、`save_model`），直接 `torch.save(model.state_dict())` 拿到的是分片，加载时不完整。
- 跨硬件加载（A100 训的 ckpt 在 H100 加载）通常没问题，但跨 PyTorch 大版本可能有兼容性问题。
- 保存优化器状态非常占空间（AdamW 是参数的 2x），长训练注意磁盘。
- 恢复训练时 `random state` 也要存，否则严格无法复现。`accelerator.save_state()` 已经处理了。

### 10.11.6 节点宕机与续训：一项被低估的工程

8 卡 H100 训练 7 天，遇到一次 NCCL 超时、一次 GPU ECC 错误、一次 NVMe 写满，是常态。预训练规模（256+ 卡跑数周）几乎天天有节点抖动。能不能从崩溃恢复，决定的是这次实验是"重跑 6 天"还是"接着跑半小时"。

**Checkpoint 间隔策略**

| 训练规模 | 推荐间隔 | 理由 |
|---|---|---|
| <1 天的微调 | 每 epoch 一次 | 失败重跑成本低，频繁存 IO 反而拖慢训练 |
| 1-7 天的中等训练 | 每 1-4 小时（按 wall-clock） | 平衡 IO 开销与重跑损失，按时间不按步数避免短长 batch 偏差 |
| >7 天的预训练 | 每 30 分钟到 1 小时 + 按 token 量打副本 | 高频 ckpt + 异步上传到对象存储，主盘 ckpt 滚动覆盖最近 N 个 |
| spot / 抢占式集群 | 每 10-30 分钟 | 节点随时可能被回收，间隔越短越好 |

实务上要 **同时保存** 两份：模型权重（最关键，丢了要重训）、优化器+scheduler+random state（丢了精度有损但不致命）。优化器 state 大小是参数的 2x（AdamW），可以采用"每 4 个权重 ckpt 配 1 个完整 state ckpt"的滚动策略压磁盘。

**Async checkpoint**：PyTorch 2.4+ `torch.distributed.checkpoint.async_save` 支持把 ckpt 写盘从训练主进程剥离到后台线程。对于 70B+ 模型，同步 ckpt 一次要阻塞训练 30-60 秒，async 几乎无感。FSDP2 + DCP（distributed checkpoint）是 2026 年大规模训练的标配。

**Elastic 续接**：节点抖动时让其他节点等待新节点加入，而不是整个 job 死掉。

```bash
# torchrun elastic 模式
torchrun \
    --nnodes=2:8 \                       # 最少 2 节点也能跑，最多 8 节点
    --nproc_per_node=8 \
    --max-restarts=3 \                   # 整个 job 最多重启 3 次
    --rdzv_backend=c10d \
    --rdzv_endpoint=$MASTER_ADDR:29500 \
    --rdzv_id=run-2026-05-08 \           # 同一 rdzv_id 的进程会被聚合到同一 job
    train.py
```

`--nnodes=min:max` 让 rendezvous 在 min 节点到位时就启动，新节点可以中途加入。配合训练脚本里"启动时检测有无最新 ckpt 并 resume"的逻辑，节点宕机后只需 SLURM/k8s 重新拉起容器，训练自动续上。

**Spot / 抢占式集群额外要点**：

- 监听 SIGTERM（云厂商一般会在回收前 30-120 秒发出），收到后立刻触发一次 ckpt 再退出。Accelerate 的 `accelerator.save_state()` 加在 signal handler 里即可。
- ckpt 必须落对象存储（S3 / OSS / GCS），不能只在本地——节点回收后本地盘没了。
- 上传走异步 + 多版本，避免上传中节点被回收导致的半残文件。

**节点失联调试**：

- `NCCL_TIMEOUT_MS`（默认 30 分钟）调到 10-15 分钟，让卡死能更早被发现。
- PyTorch 2.3+ 的 `TORCH_NCCL_TRACE_BUFFER_SIZE` 启用通信 trace，崩溃时 dump 出来分析是哪一步 collective 卡住。
- 一个进程 OOM 会让其他进程 NCCL 卡死，单看 NCCL 错误信息根本看不出来。要在每个 rank 同时看 `nvidia-smi` 的显存峰值。

> 经验法则：**写训练脚本前先想清楚"半夜挂了，明早起来怎么续？"** 把这个问题答好了，再写 model 和 optimizer。一个不能续训的训练脚本，本质是个 demo，不是生产代码。

**通用默认（覆盖 90% 场景）**：

- 优化器：AdamW，betas=(0.9, 0.95)，weight_decay=0.1（嵌入和 LN 不衰减）
- 调度器：cosine + warmup，warmup 占总步数 1%-5%，min_lr = max_lr × 0.1
- 混合精度：BF16（A100/H100），FP16 + GradScaler（V100）
- 梯度裁剪：max_norm=1.0
- 梯度累积：根据显存定，注意调整有效 batch
- 归一化：LayerNorm 或 RMSNorm，Pre-Norm 结构
- 监控：W&B + 至少 train_loss / val_loss / lr / grad_norm

**多卡场景**：

- 模型能放下：DDP（torchrun + accelerate launch）
- 模型放不下：FSDP2（PyTorch 原生）或 DeepSpeed ZeRO（HF Trainer 友好）
- 70B+ / 显存极紧张：ZeRO-3 + CPU offload
- 跨机：torchrun + SLURM，注意 NCCL/IB 配置

**调试 checklist**：

- 出 OOM？按 10.11.1 的阶梯一步步上
- 出 NaN？检查 lr、grad_clip、混合精度类型
- Loss 不降？小数据集 overfit 测试 + 检查 grad 是否传到参数
- 多卡不快？检查 DataLoader、通信类型、no_sync

到这里，你应该具备了"独立把一个微调任务从 0 训练到收敛"的工程能力——不只是调用 Trainer.train()，而是能在出问题时知道去哪里找原因，能在显存紧张时知道哪里能省。

下一章我们暂别训练栈，转入 LLM 篇——从 tokenizer、预训练目标、SFT/DPO 后训练，到采样策略和 reasoning 模型，把「调用方需要懂的原理」讲透；微调的两个核心范式（全参数微调和参数高效微调 PEFT——LoRA、QLoRA、Prefix Tuning、Adapter）放到第 13 章详细展开。
