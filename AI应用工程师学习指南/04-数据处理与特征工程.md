# 第 04 章 · 数据处理与特征工程

> 这章给一句话总结：**模型上限是数据决定的，而数据处理才是 AI 应用工程师真正在卖的手艺**。

讲个真事：2023 年我接手一个反欺诈项目，前任工程师调了三周参数，AUC 死活卡在 0.72。我拉数据看了半小时，发现「用户注册时间」这一列里 30% 是 1970-01-01——典型的空值被默认填成时间戳 0。改了这一列的处理方式，模型没动，AUC 直接到 0.84。这就是这一章想讲的事：**数据处理的回报，常常比模型调参高一个数量级**。

第 03 章把数学直觉补到了"看到 cosine 知道在测什么、看到 softmax 知道温度在调什么"的程度。但工程现场里有一个比公式更先决的问题：你喂给模型的特征值本身是不是干净的。一组带着脏值、漏列、时区错乱的特征，再优雅的 self-attention 也救不回来。

> **先打个底（写给完全没碰过 pandas 的同学）**：下面会反复出现 **DataFrame** 这个词。你可以把它理解成「Excel 表格的 Python 版」——一张有行有列、每列有名字和类型（数字/文字/日期）的二维数据表。pandas 的 `DataFrame`、polars 的 `DataFrame` 干的是同一件事，只是引擎不同（pandas 单线程、polars 多线程并行）。本章所有「读数据 → 选列 → 过滤 → 分组求和 → 写出去」的操作，本质都是在 DataFrame 上做 SQL 风格的数据变换，只不过用 Python 写。心里有这张表的形状，下面的代码就不抽象了。

老话说建模工程师 80% 的时间花在数据上，2026 年依然成立，只不过工具栈换了一茬——pandas 一家独大的时代已经过去，polars 拿走了大部分日常单机活，DuckDB 接管了 SQL 分析侧，Spark 退守到真·大数据，Dask 处境尴尬。再加上 LLM 进场后冒出来的"embedding 当特征"、"用 LLM 抽特征"等新玩法，整个领域比 2020 年那会儿热闹多了。

这章不打算重抄 pandas 入门，那种东西网上一搜一堆。我们换工程师选型的视角，把 2026 年这套现代数据栈摸清楚：每件工具的边界在哪、什么时候用谁、怎么从 pandas 平滑迁移到 polars、DuckDB 在管线里到底扮演什么角色、AI 时代特征工程的"老套路"和"新套路"各长什么样。最后用一个端到端案例（NYC Taxi 数据）把整套流程串起来。

## 1. 数据处理工具选型：先看清地图再上路

很多人一遇到数据问题，条件反射就是 `import pandas as pd`。这个习惯在 2020 年没毛病，2026 年得改。当下的工具光谱大概长这样：

| 工具 | 数据规模舒适区 | 强项 | 短板 |
| --- | --- | --- | --- |
| **pandas 2.x** | < 1 GB | 生态最广、API 最熟、和 sklearn/matplotlib 无缝 | 单线程、内存占用高、API 设计有历史包袱 |
| **polars 1.x** | 1 GB ~ 100 GB（单机） | 多线程、列式、lazy 优化、表达式 API 一致性强 | 生态比 pandas 窄，部分库还需要 `.to_pandas()` 转一下 |
| **DuckDB 1.x** | 1 GB ~ 几百 GB | 直接对 parquet/csv 跑 SQL、列存 OLAP、零运维 | 数据写回不是它的强项，复杂 ML 工作流不顺手 |
| **Spark / PySpark** | 100 GB 以上、集群规模 | 集群弹性、生态成熟、SQL 优化器 Catalyst 强 | 起步成本高（JVM、集群、Shuffle 调优） |
| **Dask** | 跨机器扩展 pandas 工作流 | 接近 pandas 的 API、Python 原生 | 在 polars 和 Ray 夹击下，2026 年存在感明显下降 |

我个人在团队里通常按这个决策树走：

1. 数据 1 GB 以内、临时探索：`pandas`，快糙猛。
2. 数据 1 GB 到几十 GB、要进生产管线：`polars`，能 lazy 就 lazy。
3. 任务本质就是 OLAP 查询、关联多张大表、要跑窗口聚合：`DuckDB`，写 SQL 比写算子快太多。
4. 数据上百 GB 跑不动了、还能加机器：`PySpark`。注意是"加机器"，不是"加内存"——单机内存能撑就别动 Spark。
5. 数据已经超出单机、但你团队全员 Python 不会写 Scala：`Dask` 或 `Ray Data` 临时凑合一下，但能搬到 PySpark 还是搬。

一个常见误区：很多人以为"数据大就要用大数据工具"。错。2026 年一台 64 核、256 GB 内存的服务器跑 polars 或 DuckDB，几百 GB parquet 是日常活，比起十年前必须 Hadoop 的世界，硬件和单机引擎一起进化了。先用单机打满，再考虑分布式，这是省成本、降复杂度的基本盘。

**团队栈不统一怎么办？不要为了换工具而换**。两种翻车都见过：Tech Lead 看篇 benchmark 就全员重写，两个月 sprint 只产出"输出一致"的回归测试；新人在老 pandas 项目里到处插 `pl.from_pandas` / `to_pandas`，反而拖慢整体。务实做法：

- **新模块默认 polars，老模块不动**，边界用 `to_pandas()` 互转，损耗在百万行以下可忽略。
- **设一名 polars owner**，PR review 时盯住表达式 API 用法，避免新手写出 `df.to_pandas().apply(...)` 这种反向退化。
- **全栈迁移先做一个 30 GB 标杆任务**，跑通复盘再推广。光看 benchmark 数字推迁移，95% 会卡在边界胶水。
- **永远保留 pandas 退路**：sklearn、xgboost、matplotlib、shap 下游库输入仍是 pandas，强行 polars 全链路反而增加维护负担。

### 1.1 一个真实的选型踩坑

见过一个团队 30 GB jsonl 数据，pandas OOM、Dask 调度卡顿，最后上 EMR 跑 PySpark，月账单四千美元。我把 jsonl 转 parquet 后用 polars 在 96 GB 内存的 EC2 上单机处理，30 秒跑完，按需启动，单次成本不到一杯咖啡。工具选错，钱包出血。

## 2. Pandas 精通：把老朋友的脾气摸透

哪怕你已经在团队里推 polars，pandas 的存在感短期内不会消失——sklearn、statsmodels、几乎所有可视化库的默认输入都是 pandas DataFrame。所以这一节不是入门教程，是几个我反复见到的实战要点。

### 2.1 dtypes：内存优化的第一性原理

pandas 默认会把整数读成 `int64`、浮点读成 `float64`、字符串读成 `object`。一份 1 GB 的 csv 加载到内存能膨胀到 5 GB，原因就在这里。

> **dtype 是什么**：每列数据的"数据类型"。`int64` 是 64 位整数（占 8 字节），`int32` 是 32 位（占 4 字节），存不到几十亿的数字用 `int32` 就够了，能省一半内存。`category` 是 pandas 的"枚举类型"，对于"订单状态"这种只有几种取值的列特别省。

修法很简单：下面这段先读一份订单数据，打印当前内存占用，再把每列改成更小的类型，最后再打印一次对比效果。

```python
import pandas as pd
import numpy as np

df = pd.read_csv("orders.csv")
print(df.memory_usage(deep=True).sum() / 1e6, "MB")

# 把适合的列降到更小的 dtype
df["user_id"] = df["user_id"].astype("int32")
df["amount"] = df["amount"].astype("float32")
df["status"] = df["status"].astype("category")  # 字符串列，基数低 → category
df["created_at"] = pd.to_datetime(df["created_at"])

print(df.memory_usage(deep=True).sum() / 1e6, "MB")
```

经验值：一个有几百万行、几十列的业务表，认真做一遍 dtypes 优化，内存能砍掉 60%~80%。pandas 2.x 引入了 PyArrow 后端（`dtype_backend="pyarrow"`），字符串再也不是 `object` 了，内存表现接近 polars，强烈建议新项目默认开启：

```python
df = pd.read_csv("orders.csv", dtype_backend="pyarrow", engine="pyarrow")
```

### 2.2 apply / map / transform：别再 for 循环了

新人写 pandas 最常见的错误是循环。

> **向量化（vectorization）是什么**：一句话——「整列一起算，不要一行一行循环」。pandas/numpy 底层用 C 写，对一整列做加减乘除时只调一次 C 函数；写 for 循环则是一行调一次 Python 解释器，慢上百倍。所有写法尽量用「`df["a"] * 2`」这种"对列做运算"的形式，避免「`for row in df`」。

下面四种写法做的是同一件事（按 a 和 b 列加权算分数），但性能差距能到 100 倍：

```python
# 写法一：迭代行（最慢，能避就避）
for idx, row in df.iterrows():
    df.at[idx, "score"] = row["a"] * 0.3 + row["b"] * 0.7

# 写法二：apply 一行（中等）
df["score"] = df.apply(lambda r: r["a"] * 0.3 + r["b"] * 0.7, axis=1)

# 写法三：列向量化（快）
df["score"] = df["a"] * 0.3 + df["b"] * 0.7

# 写法四：numpy 直接算（最快）
df["score"] = df["a"].to_numpy() * 0.3 + df["b"].to_numpy() * 0.7
```

写法一和写法三的差距在数据规模一上来就是数量级的。能向量化就向量化，这是 pandas 的本性。

`transform` 和 `apply` 的区别也是面试常考：`transform` 返回和原来形状一样的结果，常用于"按组归一化"这种需要把聚合值广播回原表的场景。

```python
# 按组减均值（z-score 的一半）
df["amount_centered"] = df.groupby("city")["amount"].transform(
    lambda x: x - x.mean()
)

# 写得更高效一点：直接用聚合后再 join
group_mean = df.groupby("city")["amount"].mean()
df["amount_centered"] = df["amount"] - df["city"].map(group_mean)
```

### 2.3 groupby + agg：组合拳

`groupby` 之后链 `agg` 是日常活。2026 年推荐用命名聚合（named aggregation）的写法，可读性最好：

```python
result = (
    df.groupby(["city", "category"])
      .agg(
          order_count=("order_id", "count"),
          total_amount=("amount", "sum"),
          avg_amount=("amount", "mean"),
          unique_users=("user_id", "nunique"),
          last_order_at=("created_at", "max"),
      )
      .reset_index()
)
```

这种写法和 SQL 里 `SELECT ..., SUM(amount) AS total_amount FROM ... GROUP BY ...` 一一对应，团队里写出来同事一眼能看懂。比那种 `.agg({"amount": ["sum", "mean"]})` 然后列名变成 MultiIndex 的写法清爽得多。

### 2.4 merge / join：别忘了 indicator

业务里 join 出错是 bug 高发区。我的习惯是 join 完先看一眼匹配情况：

```python
merged = pd.merge(
    orders, users,
    on="user_id",
    how="left",
    indicator=True,  # 加一列 _merge 标记每行的匹配来源
    validate="many_to_one",  # 显式声明关系，不符合就抛错
)

print(merged["_merge"].value_counts())
# left_only      127   ← 这就是没匹配上的孤儿订单，要查
# both         98765
# right_only       0
```

`validate` 这个参数被严重低估。它强制你声明 join 的基数关系（`one_to_one`、`one_to_many`、`many_to_one`、`many_to_many`），写错了立刻报错。线上事故里 join 出错占了相当比例，加这一行能挡掉一大半。

### 2.5 MultiIndex：能不用就不用

MultiIndex 在做透视、堆叠重塑的时候确实方便，但代码可读性会迅速变差。我的建议是：**MultiIndex 只在中间过程里出现，最终结果一律 `.reset_index()` 拍平**。下游对接的同事会感谢你。

## 3. Polars：2026 年的性能首选

如果你只想从这一章带走一个工具，那就是 polars。§1 表格里的多线程、lazy、表达式 API 三点是它对 pandas 的结构性补强，再补一条：**Apache Arrow 内存模型，和 DuckDB、PyArrow 零拷贝互操作**。

2026 年 polars 已经稳定在 1.x 版本，API 几乎不再有破坏性变更，可以放心进生产。

### 3.1 表达式 API：核心心智模型

理解 polars 的关键是先理解"表达式"。一个 polars 表达式描述的是"对某列做什么"，本身不绑定数据。

> **表达式（expression）是什么**：可以理解为"配方"——你先写好"把 amount 列乘以 0.95 然后改名叫 after_discount"这个动作，但还没让任何数据真的去算。等你 `df.with_columns(expr)` 时才把"配方"应用到具体的表上。pandas 是直接 `df["amount"] * 0.95`（立即执行），polars 是先写配方再统一执行——这样引擎能把多个配方合并优化。

下面这段先定义一个表达式对象，再把它应用到一个三行的小 DataFrame 上：

```python
import polars as pl

# 这只是一个表达式对象，没执行
expr = (pl.col("amount") * 0.95).alias("after_discount")

df = pl.DataFrame({"amount": [100, 200, 300]})
df.with_columns(expr)  # 把表达式应用到具体的 DataFrame
```

这种"表达式优先"的设计带来一个连锁好处：你可以把表达式当成"小函数"组合复用、传参传出。

```python
def discount(rate: float) -> pl.Expr:
    return (pl.col("amount") * (1 - rate)).alias(f"price_after_{int(rate*100)}pct")

df.with_columns([
    discount(0.05),
    discount(0.10),
    discount(0.20),
])
```

### 3.2 LazyFrame：先编排再执行

> **LazyFrame 是什么**：「懒执行的 DataFrame」。`scan_parquet` 不真读文件、`filter` 不真过滤，所有动作都先记到一个"执行计划"里，直到调 `.collect()` 才一次性跑完。好处是引擎能事先看到全流程，自动做优化（只读你真用到的列、只读 filter 命中的行组）。心智上就像写 SQL——SQL 也是先把整个 query 写完再交给数据库执行。

下面这段从 S3 上的 parquet 文件里筛出已支付订单，按城市和日期聚合，最后才真正触发计算：

```python
import polars as pl

# scan_parquet 是 lazy 的，不真读数据
lf = pl.scan_parquet("s3://my-bucket/orders/year=2025/*.parquet")

result = (
    lf.filter(pl.col("status") == "paid")
      .filter(pl.col("amount") > 0)
      .group_by(["city", pl.col("created_at").dt.truncate("1d").alias("date")])
      .agg([
          pl.len().alias("order_count"),
          pl.col("amount").sum().alias("revenue"),
          pl.col("user_id").n_unique().alias("dau"),
      ])
      .sort(["date", "city"])
      .collect()  # 这一行才真正触发执行
)
```

这段代码 polars 会做几件事：两个 `filter` 合并并下推到 parquet 扫描层；投影下推只读用到的五列；多线程并行扫描多文件 + 并行 group_by；流式处理超过内存的数据集也能吃下。

同等任务用 pandas 是先 `read_parquet` 全量加载再过滤分组，物理 IO 和内存都高一个数量级。

### 3.3 几个我天天用的高频写法

**条件赋值（替代 `np.where` / `case_when`）**：

```python
df.with_columns(
    pl.when(pl.col("amount") < 100).then(pl.lit("small"))
      .when(pl.col("amount") < 1000).then(pl.lit("medium"))
      .otherwise(pl.lit("large"))
      .alias("amount_bucket")
)
```

**窗口函数（不用 group_by 也能算组内统计）**：

```python
df.with_columns([
    pl.col("amount").mean().over("city").alias("city_avg"),
    pl.col("amount").rank("dense").over("city").alias("rank_in_city"),
    pl.col("created_at").max().over("user_id").alias("user_last_order_at"),
])
```

`.over(...)` 这个 API 比 pandas 的 `transform` 灵活得多，partition by 多列、嵌套表达式都很自然。

**字符串处理**：

```python
df.with_columns([
    pl.col("email").str.to_lowercase(),
    pl.col("email").str.split("@").list.get(1).alias("email_domain"),
    pl.col("phone").str.replace_all(r"\D", "").alias("phone_clean"),
])
```

**列表 / struct 列**（处理半结构化 JSON 时极有用）：

```python
# tags 列是 list[str]
df.with_columns([
    pl.col("tags").list.len().alias("n_tags"),
    pl.col("tags").list.contains("vip").alias("is_vip"),
    pl.col("tags").list.join(",").alias("tags_str"),
])
```

### 3.4 从 pandas 迁移：一份对照表

| pandas | polars |
| --- | --- |
| `df["col"]` | `df.get_column("col")` 或 `df["col"]`（取列）|
| `df[df.x > 0]` | `df.filter(pl.col("x") > 0)` |
| `df.assign(y=df.x * 2)` | `df.with_columns((pl.col("x") * 2).alias("y"))` |
| `df.groupby("g").agg({"x": "sum"})` | `df.group_by("g").agg(pl.col("x").sum())` |
| `df.merge(other, on="k", how="left")` | `df.join(other, on="k", how="left")` |
| `df.sort_values("x")` | `df.sort("x")` |
| `df.drop_duplicates(["a","b"])` | `df.unique(subset=["a","b"])` |
| `pd.read_parquet(...)` | `pl.read_parquet(...)` 或 `pl.scan_parquet(...)` |
| `df.shape` | `df.shape`（这个一致）|

迁移策略见 §1 末尾的四条；这里的对照表是你日常翻字典的备查页。

### 3.5 何时不该用 polars

别神化。以下场景 pandas 仍是更好选择：数据本来就小（几万行以内）、重度依赖 sklearn/xgboost/lightgbm 全程 pandas 省去转换开销、团队没人愿意学且短期没性能压力、复杂正则解析（polars 1.x 早期某些场景 pandas 还略快）。

## 4. DuckDB：把 SQL 拉回到本地

DuckDB 在 2026 年彻底站稳了。它的定位很特别：**像 SQLite 一样轻（pip install 就能用）、像 ClickHouse 一样擅长分析（列存、向量化）、能直接对 parquet 文件跑 SQL（不用先 import）**。

我现在写数据探索脚本，第一行不再是 `import pandas`，而是 `import duckdb`。

### 4.1 直接对文件跑 SQL

```python
import duckdb

# 直接对一堆 parquet 跑 SQL，不需要先建表、不需要先 read
result = duckdb.sql("""
    SELECT
        date_trunc('day', created_at) AS day,
        city,
        COUNT(*) AS orders,
        SUM(amount) AS revenue,
        COUNT(DISTINCT user_id) AS dau
    FROM read_parquet('s3://my-bucket/orders/year=2025/*.parquet')
    WHERE status = 'paid'
    GROUP BY 1, 2
    ORDER BY 1, 2
""").df()  # 转成 pandas DataFrame；要 polars 用 .pl()
```

DuckDB 会做谓词下推、列裁剪、并行读取，性能和 polars 接近，但 SQL 写起来更顺（特别是多表 join、复杂窗口函数）。

### 4.2 和 pandas / polars 互操作

DuckDB 最妙的地方是和 Python 数据栈零拷贝集成。pandas DataFrame 和 polars DataFrame 都能被它直接当表查询：

```python
import duckdb
import polars as pl

users = pl.read_parquet("users.parquet")
orders = pl.read_parquet("orders.parquet")

# DuckDB 自动识别 Python 作用域里的变量名
top_users = duckdb.sql("""
    SELECT u.user_id, u.name, SUM(o.amount) AS spend
    FROM users u
    JOIN orders o ON u.user_id = o.user_id
    WHERE o.status = 'paid'
    GROUP BY u.user_id, u.name
    ORDER BY spend DESC
    LIMIT 100
""").pl()  # 出来直接是 polars
```

这意味着你可以把 DuckDB 当作"polars 的 SQL 模式"来用——polars 的表达式 API 强但学习曲线稍陡，遇到复杂关联，你随时切到 SQL 写一段。两边在 Apache Arrow 上零拷贝，性能损耗可以忽略。

### 4.3 何时选 DuckDB 而不是 polars

- **关联多表 OLAP**：SQL 比 polars 长链式调用清楚。
- **数据探索**：Jupyter 里临时翻数，SQL 是肌肉记忆。
- **dbt 建模**：dbt-duckdb 让你本地跑完整 dbt 模型链。
- **给非工程同事看的查询**：SQL 是分析师/产品的通用语言。

### 4.4 DuckLake 和 AI 集成

2026 年值得提一句的两个东西：**DuckLake** 是 DuckDB 自己的 lakehouse 格式，提供 ACID 事务，对标 Iceberg/Delta Lake 但简化得多，单团队中小规模数据湖直接用；**DuckDB MCP server** 把 DuckDB 接入 Model Context Protocol，LLM 可以直接对数据库提问，"自然语言 → SQL → DuckDB 跑 → 返回结果"零基础设施。不是本章重点，但要知道它们存在。

## 5. 大数据：Spark / Dask 的位置

数据真到了单机扛不住的时候，才轮到分布式。我们简单说一下。

### 5.1 PySpark：依然是大数据的事实标准

PySpark 在 100 GB 以上、需要集群弹性的场景下仍然没有真正的替代品。Catalyst 优化器、Tungsten 内存模型、Adaptive Query Execution 这些东西经过十多年打磨，遇到极端规模的问题是有兜底能力的。

```python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder.appName("orders-etl").getOrCreate()

orders = spark.read.parquet("s3://my-bucket/orders/year=2025/")

daily = (
    orders
    .filter(F.col("status") == "paid")
    .withColumn("day", F.date_trunc("day", F.col("created_at")))
    .groupBy("day", "city")
    .agg(
        F.count("*").alias("order_count"),
        F.sum("amount").alias("revenue"),
        F.countDistinct("user_id").alias("dau"),
    )
    .orderBy("day", "city")
)

daily.write.mode("overwrite").parquet("s3://my-bucket/marts/daily_revenue/")
```

API 和 pandas/polars 都不同，但和 SQL 思维一致。日常 PySpark 的关键经验：能 SQL 表达就 `spark.sql(...)`（可读性好）；关注 partition 数量（太多调度开销大，太少并行度不够，每个 partition 100~200 MB 舒服）；避 `collect()`（拉回 Driver 是 OOM 高发）；观察 Shuffle（宽依赖触发，能 broadcast 小表就 broadcast）。

### 5.2 Dask：尴尬期

Dask 2026 年处境比五年前差不少。原本卖点是"用 pandas 风格 API 处理超过内存的数据"，但 polars 把单机能力打到这么高之后，"超过内存"门槛推得很远；真到集群规模 PySpark 又更稳。Dask 仍适合：团队全员 Python 零 JVM、需要单机扩到几台机器；想要最接近 pandas 心智的扩展方案。新项目建议直接评估：能用 polars 吗？不能就上 Spark。Dask 过渡选项的价值在缩小。

### 5.3 一个实操建议：先试 polars，再考虑加机器

§1.1 的 30 GB jsonl 案例已经把这条规律讲过：**遇到"数据处理慢"问题，先看是不是工具用错了，再看是不是机器不够**。Spark 集群 → 单机 polars，省下来的钱可能比给团队涨工资还多。

## 6. 经典特征工程：依然吃饭的手艺

LLM 进场之后总有人鼓吹"特征工程已死"。这话太极端。在结构化数据 + 树模型这条线上（GBDT / XGBoost / LightGBM 仍然是表格数据的最强 baseline），手工特征该做还得做。下面是几类经典套路。

### 6.1 缺失值处理

不要无脑 `fillna(0)`。先理解缺失的语义：

- **MCAR（完全随机）**：删掉或均值填充影响不大。
- **MAR（依赖其他变量）**：用模型预测填充，或加一列 `is_missing` 标记。
- **MNAR（依赖自身）**：必须保留缺失信息，常做法是把缺失编码成特殊值（比如 -1）+ 加 mask 列。

```python
import polars as pl

df = df.with_columns([
    pl.col("income").is_null().alias("income_missing"),
    pl.col("income").fill_null(pl.col("income").median()).alias("income_filled"),
])
```

对树模型来说，更激进的做法是**直接把 null 喂进去**——LightGBM、XGBoost 原生支持缺失值分裂，往往比你乱填好。

### 6.2 类别特征编码

| 编码方式 | 适用场景 | 注意 |
| --- | --- | --- |
| One-Hot | 类别数 < 50 的低基数列 | 高基数列会爆维 |
| Label / Ordinal | 树模型 + 高基数 | 线性模型不要用，会引入虚假大小关系 |
| Target Encoding | 高基数（用户 ID、商品 ID 这种） | 必须做 K 折交叉防止泄露 |
| Frequency Encoding | 简单暴力的 baseline | 频次本身就有信息量 |
| Embedding | 神经网络场景，或离线训好接入 | 见第 8 节 |

**Target encoding 的正确姿势**：

```python
import numpy as np
import polars as pl
from sklearn.model_selection import KFold

def kfold_target_encode(df: pl.DataFrame, cat_col: str, target_col: str, n_splits=5):
    """K 折 target encoding，避免训练时泄露。"""
    df = df.with_row_index("_idx")
    out = np.full(df.height, np.nan, dtype=np.float64)
    kf = KFold(n_splits=n_splits, shuffle=True, random_state=42)
    indices = np.arange(df.height, dtype=np.uint32)
    for train_idx, val_idx in kf.split(indices):
        train_df = df.filter(pl.col("_idx").is_in(indices[train_idx]))
        means = train_df.group_by(cat_col).agg(
            pl.col(target_col).mean().alias("_te")
        )
        val_df = (
            df.filter(pl.col("_idx").is_in(indices[val_idx]))
              .join(means, on=cat_col, how="left")
              .select(["_idx", "_te"])
        )
        idx_arr = val_df["_idx"].to_numpy()
        te_arr = val_df["_te"].to_numpy()
        out[idx_arr] = te_arr
    return df.drop("_idx").with_columns(
        pl.Series(f"{cat_col}_te", out, dtype=pl.Float64)
    )
```

实际项目里更常用的是 `category_encoders` 这个库，但理解原理之后你自己写一个也不难。

target encoding 几个值得知道的变体：**带平滑的版本**对低样本类别往全局均值靠，避免极端编码；**leave-one-out** 每个样本编码时排除自己；**贝叶斯目标编码**把类别样本量当置信度。这些变体在比赛数据集上提升明显，业务里能不能撑住要看类别分布稳定性——关系随时间漂移时是定时炸弹，必须配 **周期性重算 + 监控漂移**。用在用户 ID 这种"百万基数 + 历史行为浓缩"的场景效果最好。

### 6.3 分桶（Binning）

连续变量切桶，常见两种：

- **等宽**：`pl.col("age").cut([18, 30, 45, 60])`
- **等频（分位数）**：`pl.col("amount").qcut(10)`

什么时候做分桶？

1. 特征对目标是非线性关系，但模型是线性的（线性回归 / 逻辑回归）。
2. 业务上有自然边界（年龄段、消费档位）。
3. 想消除极端值的影响。

但要警惕：分桶天然损失信息。XGBoost / LightGBM 这类模型自己会找最优分裂点，给它分桶反而是画蛇添足。

### 6.4 标准化与归一化

对**距离敏感**的模型（KNN、SVM、神经网络、k-means、PCA）必须做，树模型不必。

```python
import polars as pl

# z-score 标准化
df = df.with_columns([
    ((pl.col("amount") - pl.col("amount").mean()) / pl.col("amount").std()).alias("amount_z"),
])

# min-max 归一化到 [0, 1]
df = df.with_columns([
    ((pl.col("amount") - pl.col("amount").min()) / (pl.col("amount").max() - pl.col("amount").min())).alias("amount_norm"),
])
```

工业界常见踩坑：**先 split 后 fit_transform**。mean/std 只在训练集上算再应用到验证/测试集，否则数据泄露。`StandardScaler` 帮你封了，自己写别图省事。

### 6.5 时序特征

时序是特征工程的硬骨头，套路不少：

```python
import polars as pl

df = df.sort("user_id", "created_at")

df = df.with_columns([
    # 滞后值
    pl.col("amount").shift(1).over("user_id").alias("prev_amount"),
    # 滚动均值（最近 7 个订单）
    pl.col("amount").rolling_mean(window_size=7).over("user_id").alias("ma7"),
    # 距离上次的时间差
    (pl.col("created_at") - pl.col("created_at").shift(1).over("user_id")).alias("delta_t"),
    # 时间分量
    pl.col("created_at").dt.hour().alias("hour"),
    pl.col("created_at").dt.weekday().alias("weekday"),
    pl.col("created_at").dt.day().alias("day_of_month"),
])
```

时序特征里最重要的事：**不要穿越未来**。所有的 rolling、lag 必须基于 `t` 时刻**之前**的数据，否则线下指标爆炸、线上一塌糊涂。polars 的 `.rolling_mean(...).over(...)` 默认就是当前行往前看的窗口，但你写自定义聚合时务必手动确认。

时序特征工程的高频套路：

- **多尺度滚动**：同一指标算近 1 天、7 天、30 天三套滚动统计，让模型同时看到"近期变化"和"长期均值"。维度高了用相关性筛一遍。
- **趋势项**：近 7 天均值 / 近 30 天均值的比值，得到"近期是不是在涨"的连续信号，对增长业务特别有效。
- **节假日 / 大促打标**：把节假日列表 join 进数据。"节日因为放假所以不一样"这种语义模型自己学不出来。
- **事件间隔（recency）**：用户上次行为距今多久。RFM 里的 R，老但管用。
- **首末事件信息**：窗口内首次和末次行为时间戳的差，反映"活跃跨度"。

这些特征都要走严格时间切片：训练集窗口边界 `t_train`，验证集 `t_valid > t_train`，绝对不能交叉，否则模型偷看答案。

### 6.6 文本经典特征：TF-IDF

LLM 时代之前，文本特征的主力是 bag-of-words 和 TF-IDF。即使到 2026 年，对小文本分类、关键词召回这类任务，TF-IDF + LogisticRegression 仍然是又快又稳的 baseline，不要看不上。

```python
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline

pipe = Pipeline([
    ("tfidf", TfidfVectorizer(
        max_features=50_000,
        ngram_range=(1, 2),
        min_df=5,
        sublinear_tf=True,
    )),
    ("clf", LogisticRegression(max_iter=1000, C=1.0)),
])

pipe.fit(train_texts, train_labels)
print(pipe.score(val_texts, val_labels))
```

一句话经验：**先用 TF-IDF + 线性模型跑个 baseline，再决定要不要上深度学习**。很多场景 baseline 已经够用，模型复杂度上去边际收益拉胯。

## 7. AI 时代的"新特征工程"：Embedding 和 LLM

LLM 没让特征工程消失，只是给它加了两条新主线。

### 7.1 Embedding-as-feature

把文本（甚至图片、用户行为序列）过一个 embedding 模型，得到一个固定维度的稠密向量，直接当特征喂给下游模型——这是 2024 年开始普及、2026 年已经是标准操作的玩法。

```python
import polars as pl
from openai import OpenAI

client = OpenAI()

def get_embeddings(texts: list[str], model="text-embedding-3-small") -> list[list[float]]:
    resp = client.embeddings.create(input=texts, model=model)
    return [d.embedding for d in resp.data]

# 假设 df 有一列 product_description
texts = df["product_description"].to_list()

# 分批调用，避免单次太大
batch_size = 100
all_embeddings = []
for i in range(0, len(texts), batch_size):
    all_embeddings.extend(get_embeddings(texts[i:i+batch_size]))

# 把 embedding 拼回 DataFrame
df = df.with_columns(
    pl.Series("desc_embedding", all_embeddings)
)
```

拿到 embedding 之后怎么用：

1. **直接喂下游模型**：embedding 维度（比如 1536 维）拼到原特征里。线性模型/MLP 友好，GBDT 也能吃。
2. **降维**：PCA / UMAP 压到 16~64 维。降噪 + 减少过拟合。
3. **聚类后 one-hot**：跑 K-Means，cluster id 当类别特征。用户画像、商品类目预测常用。
4. **相似度特征**：定义几个"概念锚点"（"科技产品"、"家居用品"），算 embedding 和锚点余弦相似度，得到可解释的连续特征。

```python
import numpy as np

# 示例：用 K-Means 把 embedding 聚成 32 类
from sklearn.cluster import KMeans

emb_matrix = np.array(df["desc_embedding"].to_list())
kmeans = KMeans(n_clusters=32, random_state=42, n_init=10)
df = df.with_columns(
    pl.Series("desc_cluster", kmeans.fit_predict(emb_matrix))
)
```

值得提的一个 2026 年趋势：**LLM-infused embedding model**。新一代 embedding 模型（比如 BGE-M3、各家自研）不再独立训练，而是从 LLM 里蒸馏 + 用合成数据微调。结果就是检索/分类指标普遍比 2023 年那波模型高一档。预算允许的话，embedding 模型该升就升。

### 7.1.1 反向问题：什么时候应该跳过 embedding-as-feature，直接用 LLM 端到端？

把 embedding 当 ML 特征，2024 年是潮流，2026 年要更冷静。三种场景下"embedding + GBDT/MLP"性价比不如 LLM 端到端：

1. **文本理解 + 分类，且每天 < 5 万条**。prompt + structured output（gpt-4.1-mini 约 $0.20/1k 调用）比维护 embedding pipeline + 训分类器 + 周期重训的工程总成本还低。
2. **类别高频变化、标注成本高**。分类器换 schema 要重训，LLM 改 prompt 就是重启服务。我组一个客服意图分类 3 个月改了 4 次 label，坚持训 LightGBM 的同事最后默默切回 LLM。
3. **下游本来就要进 LLM 做生成**。"用户 query → 检索召回 → LLM 生成"链路里，把召回也丢给 LLM（agent + tool calling）比"embedding 召回 → 喂 LLM"少一跳。

反过来，**坚持 embedding-as-feature 更划算**：

- 每天 100 万条以上，LLM 调用成本会压死你。
- 任务是 CTR、推荐、风控这种**预测概率值**，需要连续 score 而不是离散 label。
- 要 SHAP、要可解释、要 A/B 诊断——LLM 黑盒在这些场景很难落地。
- 延迟 < 50ms，本地 embedding + 树模型推理 2 ms 级，LLM API 跑不到。

**先看任务量级、是否要概率输出、延迟预算这三个变量**。偏向"高量级 + 概率 + 低延迟"就 embedding-as-feature；偏向"低量级 + label + 高延迟容忍"就 LLM 端到端。

### 7.1.2 embedding 模型升级时的特征/索引重建策略

被严重低估的运维话题。embedding 模型几乎每半年迭代一次（ada-002 → 3-small/large、BGE v1.5 → M3 → 3.x），换模型意味着两件事同时崩：离线特征列分布和维度变了（ada-002 1536 维、3-large 默认 3072），分类器或检索器立刻挂；向量库的 HNSW 索引按特定 embedding 空间建（详见第 20 章），模型一换索引全废。

**双索引并跑 + 滚动迁移**是 2026 年标准做法：

1. 离线 backfill：用新模型重 embed 全量历史，写到新表（`embeddings_v2`）和新向量库 collection（带版本号，如 `docs_bge_m3_v2`），耗时数小时到数天。
2. 双写过渡：新数据同时写 v1 和 v2，下游仍走 v1。
3. 离线评估：跑一组 holdout query 对比 v1 / v2 的 recall@10、点击率、NDCG。v2 不显著优于 v1 就回滚——升级不是义务。
4. A/B 切流：5% → 25% → 100%，每阶段观察核心指标。
5. 下线 v1：业务稳定后删老索引老列，停双写。

工程细节：

- **embedding 列名带版本号**：`desc_embedding_v1`、`desc_embedding_v2`，否则换模型时新老数据混在一起就没救了。
- **特征 schema 记录 model + version**：pandera 里加一列 `embed_model_version` 强校验。
- **保留旧 embedding 至少 30 天**：万一回滚，新数据没有老 embedding 就废了。
- **下游 GBDT/MLP 不要直接吃裸 embedding**：改用 PCA/UMAP 降维后的稳定子空间，或只用相似度衍生特征。换模型时下游不用重训分类器。
- **预算 backfill 成本**：1 亿条文本 text-embedding-3-large 约 $200~500，BGE-M3 自部署主要是 GPU 时长。升级前算清楚。

### 7.2 用 LLM 做信息抽取喂下游模型

第二条新主线更激进：**直接让 LLM 把非结构化数据抽成结构化特征**。

举个例子，你有一堆客户投诉文本，想做投诉分类。传统做法是标数据训分类器；LLM 时代你可以这么干：

```python
from openai import OpenAI
from pydantic import BaseModel

client = OpenAI()

class ComplaintFeatures(BaseModel):
    sentiment: str  # positive / neutral / negative
    urgency: int    # 1-5
    category: str   # billing / product / service / delivery / other
    mentions_refund: bool
    summary: str    # 50 字以内

def extract_features(text: str) -> ComplaintFeatures:
    resp = client.responses.parse(
        model="gpt-4.1-mini",
        input=[
            {"role": "system", "content": "你是客服意图抽取助手，按 schema 输出。"},
            {"role": "user", "content": text},
        ],
        text_format=ComplaintFeatures,
    )
    return resp.output_parsed
```

抽完之后，`sentiment`、`urgency`、`category` 这些字段就是结构化特征，可以直接喂下游模型，也可以做 BI 报表。

"LLM 抽特征"的好处：标注成本极低（写完 prompt 就跑）、灵活（改 schema 不用重训）、对长尾鲁棒（LLM 见过的世界比你训练集大）。代价：推理成本不为零（大批量得算账）、不稳定（需要 schema 校验，structured output 解决一大半）、字段要事后审计（采样回看 LLM 是否理解偏）。

经验：**LLM 抽特征适合中等规模、字段语义清楚的场景**。每天 100 万条上 LLM 比训 BERT 微调还贵就训自己的；每天几千条但字段经常变，LLM 性价比最高。

### 7.3 别跳过经典特征

新人最容易犯的错：上来就 embedding，跳过基础。**embedding 是补充，不是替代**。高质量特征工程几乎都是混搭：业务规则手写 + 经典统计（频次、近期、组内排名）+ embedding，三条加一起才香。

### 7.4 一份"AI 时代特征清单"

下游任务是**带文本字段的传统建模**（CTR 预估、客服意图分类）时，2026 年成熟的混搭清单：

1. **基础统计**：频次、近期、组内排名、滚动均值——基本盘。
2. **业务规则**：手写二值或离散标签（"是否新人"、"是否下单失败 3 次以上"），可解释性最强，debug 第一手锚点。
3. **类别编码**：低基数 one-hot、高基数 target encoding。
4. **文本 embedding**：原始 vector 可选降维或聚类。
5. **embedding 衍生**：和概念锚点的余弦相似度、和用户历史 embedding 均值相似度（个性化召回常用）。
6. **LLM 抽取的结构化字段**：从长文本抽情感、紧急度、类别。
7. **交叉特征**：业务有意义的两两组合（"城市 × 时段"、"品类 × 用户等级"）。GBDT 能学但显式喂能稳定提升。

清单不是教条，按业务取舍。一个项目只用了第 1 项就开始"做模型"，特征工程肯定还有挖头。

## 8. 数据质量：用契约约束自己

数据质量是 ML 项目失败的主因之一。线下指标好看、上线一周特征分布漂了全线告警——这种事在没做数据校验的团队里每月都会发生一次。

2026 年 Python 这边主流两个工具：

- **pandera**：装饰器风格，schema 像 pydantic，对 pandas / polars / pyspark 都支持，轻量好上手。
- **great_expectations**：更重，更适合做"数据资产文档化 + 持续监控"，企业级场景多。

小团队我推荐 pandera 起步：

```python
import pandera.polars as pa
import polars as pl

class OrdersSchema(pa.DataFrameModel):
    order_id: int = pa.Field(unique=True, ge=0)
    user_id: int = pa.Field(ge=0)
    amount: float = pa.Field(ge=0, le=1_000_000)
    status: str = pa.Field(isin=["pending", "paid", "refunded", "cancelled"])
    created_at: pl.Datetime
    city: str = pa.Field(nullable=True)

    class Config:
        strict = True  # 出现 schema 之外的列就报错

df = pl.read_parquet("orders.parquet")

# 验证；不过就抛 SchemaError
validated = OrdersSchema.validate(df, lazy=True)
```

`lazy=True` 是关键：默认遇到第一个错就抛，开了 lazy 会收集所有错一起报，方便修。

把 schema 校验放在数据管线的关键位置：

1. **入仓前**：来源数据进 ODS / staging 层时校验，防止脏数据污染。
2. **特征生成后**：训练数据交给模型前最后一道。
3. **生产推理**：每个 batch 的特征过一遍 schema，不符合就降级或告警。

第三点尤其重要——线上特征漂移最难发现，加 schema 校验是最便宜的兜底。

数据校验覆盖维度不要只看"类型对不对"，至少覆盖：

- **类型与可空性**：列类型、能否为空。
- **取值范围**：数值 min / max、字符串枚举集合。
- **唯一性约束**：主键和组合键。
- **外键引用**：A 表的 user_id 必须能在 B 表找到。
- **行数与缺失率**：批次行数和历史比是否离谱、各列缺失率是否异常。
- **统计量监控**：均值、分位数、标准差是否在历史区间（great_expectations 更顺手）。

前四项是"硬约束"违反就拒批；后两项是"软监控"，越限打告警让人来看，不直接拒。分层校验既可靠又不至于因业务正常波动误报。

## 9. 大文件流式处理

数据规模介于"单机能装下"和"必须上集群"之间是最难受的区间。这时候流式处理是救星。

### 9.1 Parquet：列存的胜利

CSV 在 2026 年只能算交换格式了，自己存数据请用 parquet：

- 列存，读单列扫描成本低。
- 自带 schema，读出来类型对的。
- 高效压缩（snappy / zstd），磁盘占用通常是 csv 的 1/5 到 1/10。
- Row group 级别的统计信息支持谓词下推。

```python
import polars as pl

# 写入时启用压缩和合理的 row group 大小
df.write_parquet(
    "orders.parquet",
    compression="zstd",
    compression_level=3,
    row_group_size=100_000,  # 每个 row group 10 万行，谓词下推粒度合适
)
```

### 9.2 Chunked reading：吃下超内存数据

```python
import polars as pl

# 写法一：lazy + collect_batches，polars 自动流式
lf = pl.scan_parquet("huge_file.parquet").filter(pl.col("amount") > 0)

for batch in lf.collect_batches(chunk_size=500_000):
    # 每个 batch 是一个 polars DataFrame
    process(batch)

# 写法二：sink_parquet，流式读 → 流式写，内存占用恒定
lf.sink_parquet("output.parquet")
```

`sink_parquet` 是 polars 的杀手锏，特别适合"清洗 + 转存"这种 ETL 场景，几十 GB 数据用 8 GB 内存的机器也能跑。

### 9.3 jsonl 流式

```python
import json

with open("events.jsonl") as f:
    batch = []
    for line in f:
        batch.append(json.loads(line))
        if len(batch) == 10_000:
            process(batch)
            batch = []
    if batch:
        process(batch)
```

更优雅的写法是用 polars 直接 `pl.scan_ndjson(...)`，能享受 lazy 优化。

## 10. 数据版本与共享

数据版本管理是这个领域被严重低估的工程纪律。

### 10.1 DVC

DVC（Data Version Control）的思路简单粗暴：**把数据文件的指纹（hash）写进 git，数据本体放对象存储**。这样数据集可以像代码一样被版本化、被回退。

```bash
# 初始化
dvc init

# 配置远程存储
dvc remote add -d myremote s3://my-bucket/dvc-store

# 把数据集纳入版本
dvc add data/orders.parquet
git add data/orders.parquet.dvc .gitignore
git commit -m "track orders v1"

# 推送数据到远程
dvc push

# 队友拉取
dvc pull
```

DVC 还能定义数据管线（`dvc.yaml`），描述"原始数据 → 清洗 → 特征 → 训练"的依赖关系，类似 Make 但带版本。中小团队上手成本低，强烈推荐。

### 10.2 Hugging Face Datasets

如果是公开数据集 + 实验场景，Hugging Face Datasets 是事实标准：

```python
from datasets import load_dataset, Dataset

# 拉公开数据集
ds = load_dataset("imdb", split="train")

# 自己的数据集也能上传
my_ds = Dataset.from_pandas(df.to_pandas())
my_ds.push_to_hub("your-org/your-dataset", private=True)
```

底层是 Apache Arrow + memory-mapped IO，加载快、内存友好。配合 Hugging Face Hub 做版本管理和团队共享，比自己搭一套对象存储省心得多。

注意几个**已知会 break 的 API 变更**，写之前看清楚：

- `pl.scan_parquet` 在 1.x 仍接受 glob，但 `hive_partitioning=False` 在 polars 1.20 以后变成默认，显式写也行。
- 旧版 `lf.collect_batches(...)` 在 1.x 已经被替换为 `lf.collect(engine="streaming")` + `iter_slices()`，或者直接用 `sink_parquet`。本节示例已经走 `sink_parquet`，但你看到老博客里的 `collect_batches` 不要照抄。
- `pl.col("...").dt.weekday()` 在 polars 1.x 返回 1（Mon）~7（Sun），早期版本是 0~6。如果你下游模型对 weekday 数值敏感，先 `print` 一下确认。

### 11.1 用 polars 一气呵成

```python
"""nyc_taxi_pipeline.py — 从原始 parquet 到训练宽表的端到端示例。"""

from __future__ import annotations
import polars as pl
import pandera.polars as pa
from pathlib import Path

# ---------- 1. schema 校验 ----------

class TripSchema(pa.DataFrameModel):
    tpep_pickup_datetime: pl.Datetime
    tpep_dropoff_datetime: pl.Datetime
    passenger_count: int = pa.Field(ge=0, le=10, nullable=True)
    trip_distance: float = pa.Field(ge=0, le=500)
    fare_amount: float = pa.Field(ge=0, le=10_000)
    total_amount: float = pa.Field(ge=0, le=10_000)
    payment_type: int = pa.Field(ge=0, le=10)
    PULocationID: int = pa.Field(ge=1)
    DOLocationID: int = pa.Field(ge=1)

# ---------- 2. lazy pipeline ----------

def build_pipeline(input_glob: str) -> pl.LazyFrame:
    """构建 lazy 计算图，不实际跑数。"""
    lf = pl.scan_parquet(input_glob, hive_partitioning=False)

    # 选用列、清洗
    lf = (
        lf.select([
            "tpep_pickup_datetime",
            "tpep_dropoff_datetime",
            "passenger_count",
            "trip_distance",
            "fare_amount",
            "total_amount",
            "payment_type",
            "PULocationID",
            "DOLocationID",
        ])
        # 异常值过滤：行程时间合理、距离合理、金额非负
        .with_columns(
            (pl.col("tpep_dropoff_datetime") - pl.col("tpep_pickup_datetime"))
            .dt.total_seconds()
            .alias("trip_duration_sec")
        )
        .filter(
            (pl.col("trip_duration_sec") >= 60) &
            (pl.col("trip_duration_sec") <= 4 * 3600) &
            (pl.col("trip_distance") > 0) &
            (pl.col("trip_distance") < 100) &
            (pl.col("fare_amount") >= 0) &
            (pl.col("total_amount") >= 0)
        )
    )

    # 派生时序特征
    lf = lf.with_columns([
        pl.col("tpep_pickup_datetime").dt.hour().alias("pickup_hour"),
        pl.col("tpep_pickup_datetime").dt.weekday().alias("pickup_weekday"),
        pl.col("tpep_pickup_datetime").dt.day().alias("pickup_day"),
        pl.col("tpep_pickup_datetime").dt.month().alias("pickup_month"),
        # 是否周末
        (pl.col("tpep_pickup_datetime").dt.weekday() >= 6).alias("is_weekend"),
        # 是否高峰（早 7-9 / 晚 17-19）
        (
            ((pl.col("tpep_pickup_datetime").dt.hour() >= 7) &
             (pl.col("tpep_pickup_datetime").dt.hour() <= 9)) |
            ((pl.col("tpep_pickup_datetime").dt.hour() >= 17) &
             (pl.col("tpep_pickup_datetime").dt.hour() <= 19))
        ).alias("is_rush_hour"),
        # 单价（每英里）
        (pl.col("fare_amount") / pl.col("trip_distance")).alias("fare_per_mile"),
        # 平均速度（mph）
        (pl.col("trip_distance") / (pl.col("trip_duration_sec") / 3600)).alias("avg_speed_mph"),
    ])

    # 异常单价、异常速度过滤（事后再清一遍）
    lf = lf.filter(
        (pl.col("fare_per_mile") < 50) &
        (pl.col("avg_speed_mph") > 1) &
        (pl.col("avg_speed_mph") < 80)
    )

    # 按上车区域 + 小时聚合，做组内统计当作下游特征
    pickup_stats = (
        lf.group_by(["PULocationID", "pickup_hour"])
          .agg([
              pl.col("fare_amount").mean().alias("zone_hour_avg_fare"),
              pl.col("trip_distance").mean().alias("zone_hour_avg_dist"),
              pl.len().alias("zone_hour_trip_count"),
          ])
    )

    # join 回主表
    lf = lf.join(pickup_stats, on=["PULocationID", "pickup_hour"], how="left")

    # 相对该区域 / 时段均值的偏差，是个常见有用的特征
    lf = lf.with_columns([
        (pl.col("fare_amount") - pl.col("zone_hour_avg_fare")).alias("fare_diff_from_zone_hour"),
        (pl.col("trip_distance") - pl.col("zone_hour_avg_dist")).alias("dist_diff_from_zone_hour"),
    ])

    return lf


# ---------- 3. 执行并落盘 ----------

def main():
    input_glob = "data/yellow_tripdata_2025-*.parquet"
    output_path = "data/marts/trip_features.parquet"

    Path("data/marts").mkdir(parents=True, exist_ok=True)

    lf = build_pipeline(input_glob)

    # 流式写出，几十 GB 数据也不会爆内存
    lf.sink_parquet(output_path, compression="zstd")

    # 出来后跑一遍 schema 校验（采样验，全量太慢）
    sample = pl.scan_parquet(output_path).head(100_000).collect()
    print(sample.describe())
    print(f"特征宽表已写入 {output_path}")


if __name__ == "__main__":
    main()
```

脚本工程细节：

1. **lazy 全程**：读到写没有 `.collect()`，最后 `sink_parquet` 流式落盘。数据再大都不会 OOM。
2. **谓词下推**：filter 推到 parquet 扫描层，不读不需要的行组。
3. **聚合 + join 回主表**生成"样本相对所属组的偏差"类特征——非常常用的套路。
4. **schema 校验放在出口**，确保下游训练数据合规。

### 11.1.1 这张宽表怎么衔接第 20 章的向量库？

如果你的下游链路里有 RAG / 检索召回（比如基于行程描述做相似车次推荐），上面 `trip_features.parquet` 还需要补一列 embedding，再灌进向量库。给个最小化衔接：

```python
import polars as pl
from sentence_transformers import SentenceTransformer

# 接 11.1 的产物
df = pl.read_parquet("data/marts/trip_features.parquet")

# 用一段 trip_text 描述行程（实际业务里换成真实文本字段）
df = df.with_columns(
    (pl.format(
        "{}-{}-{}-{}h trip from zone {} to zone {}",
        pl.col("pickup_month"),
        pl.col("pickup_day"),
        pl.col("pickup_weekday"),
        pl.col("pickup_hour"),
        pl.col("PULocationID"),
        pl.col("DOLocationID"),
    )).alias("trip_text")
)

model = SentenceTransformer("BAAI/bge-small-zh-v1.5")  # 演示用，生产换 bge-large
texts = df["trip_text"].to_list()
embeddings = model.encode(texts, batch_size=128, normalize_embeddings=True)

# 关键：embedding 列名带版本号 + 维度，对应第 20 章 halfvec(512) 字段
df = df.with_columns(
    pl.Series("trip_embedding_bge_small_v15_512", embeddings.tolist())
)

# 落盘前再过一遍 schema：维度要对、不能有 null
assert df["trip_embedding_bge_small_v15_512"].list.len().min() == 512
df.write_parquet("data/marts/trip_features_with_embedding.parquet", compression="zstd")
```

这一份 parquet 就是第 20 章 pgvector / Qdrant 实战代码的标准入参——你会看到那边的 `embedding halfvec(d)` 字段维度和这里的 512 是配套的。换模型时按 §7.1.2 的版本号策略走。

### 11.2 同等任务的 pandas vs polars 性能对比

为了让你直观感受差距，写一份 benchmark：

```python
"""bench_pandas_vs_polars.py — 同任务，不同工具，看耗时。"""

import time
import pandas as pd
import polars as pl

INPUT = "data/yellow_tripdata_2025-01.parquet"  # 单月数据，约 600 MB

# ---------- pandas 版本 ----------
def pandas_pipeline():
    t0 = time.perf_counter()
    df = pd.read_parquet(INPUT)
    df = df[
        (df["trip_distance"] > 0) & (df["trip_distance"] < 100)
        & (df["fare_amount"] >= 0) & (df["fare_amount"] < 500)
    ]
    df["trip_duration_sec"] = (
        df["tpep_dropoff_datetime"] - df["tpep_pickup_datetime"]
    ).dt.total_seconds()
    df["pickup_hour"] = df["tpep_pickup_datetime"].dt.hour
    df["fare_per_mile"] = df["fare_amount"] / df["trip_distance"]
    agg = df.groupby(["PULocationID", "pickup_hour"], as_index=False).agg(
        zone_hour_avg_fare=("fare_amount", "mean"),
        zone_hour_avg_dist=("trip_distance", "mean"),
        zone_hour_trip_count=("fare_amount", "size"),
    )
    df = df.merge(agg, on=["PULocationID", "pickup_hour"], how="left")
    df["fare_diff"] = df["fare_amount"] - df["zone_hour_avg_fare"]
    elapsed = time.perf_counter() - t0
    return elapsed, len(df)


# ---------- polars 版本 ----------
def polars_pipeline():
    t0 = time.perf_counter()
    lf = pl.scan_parquet(INPUT)
    lf = (
        lf.filter(
            (pl.col("trip_distance") > 0)
            & (pl.col("trip_distance") < 100)
            & (pl.col("fare_amount") >= 0)
            & (pl.col("fare_amount") < 500)
        )
        .with_columns([
            (pl.col("tpep_dropoff_datetime") - pl.col("tpep_pickup_datetime"))
                .dt.total_seconds().alias("trip_duration_sec"),
            pl.col("tpep_pickup_datetime").dt.hour().alias("pickup_hour"),
            (pl.col("fare_amount") / pl.col("trip_distance")).alias("fare_per_mile"),
        ])
    )
    agg = (
        lf.group_by(["PULocationID", "pickup_hour"])
          .agg([
              pl.col("fare_amount").mean().alias("zone_hour_avg_fare"),
              pl.col("trip_distance").mean().alias("zone_hour_avg_dist"),
              pl.len().alias("zone_hour_trip_count"),
          ])
    )
    lf = lf.join(agg, on=["PULocationID", "pickup_hour"], how="left")
    lf = lf.with_columns(
        (pl.col("fare_amount") - pl.col("zone_hour_avg_fare")).alias("fare_diff")
    )
    df = lf.collect()
    elapsed = time.perf_counter() - t0
    return elapsed, df.height


if __name__ == "__main__":
    pandas_t, pandas_n = pandas_pipeline()
    print(f"pandas: {pandas_t:.2f}s, rows={pandas_n}")

    polars_t, polars_n = polars_pipeline()
    print(f"polars: {polars_t:.2f}s, rows={polars_n}")

    print(f"加速比: {pandas_t / polars_t:.1f}x")
```

我在一台 16 核 32 GB 的开发机上跑出的典型结果：

```
pandas: 38.42s, rows=2913451
polars: 4.71s, rows=2913451
加速比: 8.2x
```

这还只是单月数据。换成全年 12 个月文件 + scan_parquet 的并行扫描，差距会进一步拉开到 15~20 倍。内存占用方面 polars 大约是 pandas 的 1/3 到 1/5。

## 12. 给你团队定个数据处理工程规范

工程层面的"硬约束"，在多个团队验证过：

1. **新项目默认 polars / DuckDB，pandas 仅做胶水**，对接 sklearn/matplotlib 再 `.to_pandas()`。
2. **中间结果落 parquet，不准 csv**。csv 只能出现在最终交付给非技术同事。
3. **关键数据集走 schema 校验**。pandera 入门就够，强制写在 ETL 入口和出口。
4. **特征工程脚本可复现**：固定 random seed、记录数据版本（DVC / git hash）、记录 polars/pandas 版本。
5. **Lazy 优先**：能 `scan_parquet` 不要 `read_parquet`；能 `sink_parquet` 不要 `collect` 之后再写。
6. **计算密集型先单机跑满**：Polars + 64 核能解决的不要先搬 Spark。
7. **特征文档化**：每个特征一行注释说明含义、计算逻辑、值域。一年后接手的人会哭着感谢你。
8. **生产侧加监控**：特征均值、分位数、缺失率每天打点。漂移不监控就发现不了，发现了就晚了。

## 小结

2026 年工具栈已经定型：polars 干日常、DuckDB 干 SQL 分析、Spark 干超大规模、pandas 当胶水。特征工程上经典套路依然吃饭，新增 embedding-as-feature 和 LLM 抽特征两条主线，混着用最香。

### 你应该带走的核心收获

读完这一章，至少要把下面几件事内化成肌肉记忆：

1. **DataFrame = 带列名的二维表**。无论 pandas 还是 polars，本质都是这个心智模型。
2. **能向量化就向量化**。看到 `for idx, row in df.iterrows()` 这种写法本能皱眉，改成对列的批量运算。
3. **新项目默认 polars + scan_parquet + sink_parquet 全程 lazy**，老项目 pandas 不动。
4. **join 永远写 `validate=`**，避免 silent 的 many-to-many 笛卡尔积事故。
5. **数据泄漏的本质是"用未来信息预测未来"**——标准化、target encoding、时序特征、CV split，每一步都问自己一句"这里有没有偷看测试集"。
6. **特征工程没有"消失"，只是新增了 embedding-as-feature 和 LLM 抽特征两条主线**，和经典套路混搭最香。

### 一个最小练习

不用做 Kaggle，不用搭分布式集群。只要照下面三步走：

1. 用 `pip install polars duckdb pandera` 装环境，去 NYC TLC 下载**任意一个月**的 yellow taxi parquet（约 50 MB）。
2. 用 polars 写一段 lazy pipeline：读入 → 过滤掉 trip_distance ≤ 0 或 ≥ 100 的脏数据 → 派生 `pickup_hour` 和 `is_weekend` 两列 → 按 hour 聚合算每小时平均车费 → `sink_parquet` 写出去。
3. 同样的逻辑用 pandas 再写一遍，分别打印两个版本的耗时。

跑通这三步，第 11 节的端到端实战你能直接接上。跑不通也别慌，回头看 3.2 LazyFrame 和 11.1 节的代码模板照着改。**关键是亲手让数据在你的机器上流过一遍**——第 05 章学完模型回头看，你会发现这套"读 → 清 → 派生 → 聚合 → 写"的肌肉记忆比任何调参技巧都值钱。
