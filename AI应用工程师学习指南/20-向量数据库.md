## 20.1 ANN 算法基础：在召回率、速度、内存之间挑两样

向量检索的本质问题是：给定查询向量 $\mathbf{q}$ 和库里 $N$ 个向量 $\{\mathbf{v}_i\}$，找出最近的 $k$ 个。如果一个一个比，复杂度 $O(Nd)$，对 1000 万向量、1024 维来说每次查询要算 100 亿次浮点乘加，单机几秒一次，没法做线上服务。

ANN（Approximate Nearest Neighbor）就是放弃"一定找到最近"，换取速度。代价是召回率（recall）：你要的 top-10，算法返回的 top-10 里有几个是真 top-10。生产上常用 recall@10 或 recall@100。

### 20.1.1 HNSW：图索引，工业界默认值

HNSW（Hierarchical Navigable Small World）的物理直觉是"高速公路 + 小路"。把所有向量看成图上的点，每个点连若干邻居。最底层是稠密图，往上每层稀疏一些，形成多层。

查询时从最高层（最稀疏）入口出发，贪心走到该层最近邻，然后跳到下一层继续走，直到最底层。高速公路把查询快速送到目标附近，小路负责精修。

关键参数有三个：

- `M`：每个节点保留的邻居数。常用 16~64。M 越大召回率越高，内存也越大。
- `ef_construction`：建图时候选池大小。常用 100~400。越大图质量越好，建图越慢。
- `ef`（query 时叫 `hnsw_ef` 或 `efSearch`）：查询时候选池大小。常用 64~512。越大召回率越高，延迟越长。

经验法则：先用 `M=16, ef_construction=200`，然后只调 `ef`。如果 `ef=128` 还到不了 0.95 召回，再去碰 M。

HNSW 的优点是高召回 + 低延迟 + 支持增量插入。缺点是内存重——图本身要常驻 RAM，每个向量除了 4 字节×维度，还要存 M 个邻居 ID。1000 万 × 768 维 × float32 ≈ 30 GB 向量数据，加上图大概 40~50 GB。这就是为什么 HNSW 在小于 1 亿向量的范围内是默认选择，再往上要看 IVF/DiskANN。

### 20.1.2 IVF / IVF-PQ：聚类索引，省内存

IVF（Inverted File）的直觉是"先分桶再细查"。建索引时跑 k-means 把所有向量聚成 `nlist` 个簇，查询时只去最相近的 `nprobe` 个簇里精算。`nlist` 通常取 $\sqrt{N}$，`nprobe` 在 1 到 `nlist` 之间调。

PQ（Product Quantization）是另一项正交技术：把每个向量切成 $m$ 段子向量，每段单独跑 k-means 量化成 8 bit。一个 768 维 float32 向量原本 3 KB，PQ 切 96 段每段 8 bit 后只剩 96 字节，压缩 32 倍。代价是计算变成查表近似，召回率掉几个点。

IVF-PQ 把两者拼起来，是十亿级向量库的常见底层（FAISS、Milvus 老版本默认走这个）。优点是内存占用小到可以塞下亿级数据，缺点是召回率天花板比 HNSW 低，更新成本高（聚类中心固定，新数据偏离时要 retrain）。

### 20.1.3 DiskANN：让 SSD 顶 RAM

HNSW 的图必须放内存，IVF 的所有 PQ codes 也得在内存。DiskANN 走第三条路：把图放 SSD，只在内存里存少量摘要，每次查询走少量随机 IO。微软 2019 年的论文，工业界 2024 年开始普及。Milvus 2.x、Weaviate、Vespa 都有自己的 DiskANN 变种。

适用场景：单机想存 5 亿到 50 亿向量，但 RAM 只有 256 GB。代价是延迟比 HNSW 大 2~5 倍（NVMe 的随机读 100μs，内存的 100 ns），且需要 NVMe 不能用机械盘。

### 20.1.4 ScaNN：Google 系，学习型量化

ScaNN（Scalable Nearest Neighbors）是 Google 开源的，核心创新是 anisotropic vector quantization——量化时优化的目标不是距离误差，而是排名误差。在某些大规模高维 benchmark 上比 HNSW 略快，但生态相对小众。Vertex Vector Search 内部用的就是 ScaNN。

### 20.1.5 算法权衡总览

| 算法 | 召回率上限 | QPS | 内存 | 索引时间 | 增量更新 | 适用规模 |
| --- | --- | --- | --- | --- | --- | --- |
| Flat（暴搜） | 1.0 | 极低 | 中 | 0 | 简单 | < 10 万 |
| HNSW | 0.99 | 高 | 大 | 中 | 好 | 10 万~1 亿 |
| IVF-Flat | 0.95 | 中 | 中 | 快 | 一般 | 100 万~10 亿 |
| IVF-PQ | 0.85 | 高 | 小 | 中 | 差 | 1 亿~100 亿 |
| DiskANN | 0.95 | 中（IO 限） | 小（用 SSD） | 慢 | 一般 | 1 亿~100 亿 |
| ScaNN | 0.95 | 高 | 中 | 中 | 一般 | 1 亿~10 亿 |

记一个粗略的判断流程：

1. 数据量 < 100 万：暴搜或 HNSW，怎么都行。
2. 100 万~1 亿、内存够：HNSW，调 `ef`。
3. 1 亿~10 亿、内存不够：IVF-PQ 或 DiskANN。
4. 10 亿+：分片 + IVF-PQ，开始考虑分布式架构。

## 20.2 向量库全景对比（2026）

我把 2026 年仍然活跃的主流方案按部署形态分四类：单机 OSS、分布式 OSS、托管 SaaS、嵌入式。每一类都给一个一句话定位、关键参数、适用场景。

### 20.2.1 单机/小集群 OSS

**Qdrant**（Rust 写）。当前性能口碑最稳的开源方案，p50/p99 延迟都低于 Weaviate 和 Milvus 单机版。支持 named vectors（一个 collection 多个向量字段）、payload 索引、scalar/binary/PQ 量化、Sparse vector（BM25 风格）、multi-vector（ColBERT 风格的 late-interaction）。1.10 引入 Universal Query API（一次请求里串 prefetch + fusion），1.11 起原生支持 RRF 与 DBSF 两种 fusion，让 hybrid 检索/rerank 不再需要应用层拼装。集群模式用 Raft 复制元数据，分片靠手动指定 shard key。适合 100 万~5000 万向量、要 hybrid 检索、希望少踩运维坑的团队。**何时不要用 Qdrant**：团队已经有 PG 且向量量级 < 5000 万（多此一举）；需要内置多模态 module 或开箱即用 reranker（Weaviate 更顺）；十亿级以上分布式（Milvus 更稳）。

**Weaviate**（Go 写）。强项是混合检索 + 多租户。BlockMax WAND 让 BM25 大幅提速，原生多模态 module（图片、音频）、原生多租户（一个 tenant 一个 shard，几十万 tenant 不会爆），支持 PQ/SQ/BQ 三种量化。SaaS 化最完整。适合 SaaS 厂商（每个客户隔离）、需要混合检索 + 内置 reranker 的应用。**何时不要用 Weaviate**：你不是 SaaS、不需要多租户（架构复杂度白交税）；纯向量 + 单租户的极致性能场景（Qdrant 简单且更快）；K8s 没玩明白还想自建集群（运维门槛比 Qdrant 高一档）。

**Chroma**（Python+Rust）。开发体验最简单，`pip install` 就能用，支持本地模式和 server 模式。但生产能力弱：单机大致 1000 万向量起就开始吃力，没有原生分布式，量化和高级 hybrid 还在追赶。适合 prototype、个人项目、向量量级 < 500 万的小型 RAG。生产场景慎用。**何时不要用 Chroma**：任何要对外提供 SLA 的生产服务；向量量级 > 500 万；需要细粒度 payload 索引或 hybrid 检索；多副本/高可用部署。Chroma 的定位就是 demo + 个人项目，不要让它扛业务线。

### 20.2.2 分布式 OSS

**Milvus / Zilliz**。云原生架构，把 storage / WAL / coord / query / index / data 拆成独立组件，对象存储兜底，K8s 部署最少 10~15 个 pod。优点是十亿级向量伸缩没压力、支持多种索引（HNSW/IVF/DiskANN/SCANN/SparseInvertedIndex）、2.5 起原生 Sparse-BM25 + 原生 fulltext，2.6（2025 中）继续在内存压缩和成本上做减法。缺点是组件多、运维门槛高、单机版 standalone 性能不如 Qdrant。Zilliz Cloud 是它的托管版。适合数据量超过 5000 万 + 团队有 K8s 能力。

**Vespa**。Yahoo 系老牌引擎，被 Yahoo 邮件搜索打磨二十年，原生支持向量 + 全文 + 结构化的复杂排序表达式（rank profile），是 LinkedIn、Spotify 这类大厂的底盘。门槛高、配置复杂，但十亿级 hybrid 检索几乎是唯一兼顾召回率和延迟的开源选项。

### 20.2.3 托管 SaaS

**Pinecone**。闭源、托管、按 pod 计价。优点是开箱即用、无运维、SLA 稳。缺点是贵、迁出难、近两年被开源方案在性能上追上甚至反超。仍然适合不想招 DBA、希望一行 API 搞定的小团队。**何时不要用 Pinecone**：成本敏感（同等规模 OSS 自建可省 50%+）；需要把数据放在自己的私有云/合规环境；预期未来要做精细的 ANN 调参（Pinecone 的可调参数比 Qdrant 少很多）；对供应商锁定敏感（迁出几乎要重灌）。

**Vertex Vector Search**（Google）、**Azure AI Search**（带向量）、**OpenSearch Service**（AWS）。三个云厂的官方答案。对接已经在云上的工作负载方便，但锁定深、成本透明度差、调参余地小。**何时不要用云厂托管向量库**：业务有跨云需求；想要在性能榜单里压榨每一毫秒（这三家都不是性能领先方案）；成本要月结透明（按 query 计费的几家很容易出账单意外）。

### 20.2.4 嵌入式 / 进程内

**LanceDB**（Rust 写、Python/Node 绑定）。基于 Lance 列存格式，无服务器，文件即数据库。一个 `pip install lancedb` 就能在本地跑、查询语法支持 SQL + vector + 全文。强项是数据湖友好、版本化、列存+zero copy 让批处理（重建索引、bulk 写入）很快。适合：本地 RAG 工具（Continue、Cursor 等都用过）、数据湖里的向量层、希望和 Parquet 共存的 ML 工作负载。

**FAISS**（Meta，C++ 库）。严格说不是数据库，是索引库。没有 server、没有持久化、没有元数据过滤（要自己写）。但任何向量库底层都从 FAISS 学过算法，研究/benchmark 仍然用它。生产里很少直接用，除非你在做嵌入式或者基础设施。

### 20.2.5 关系/搜索引擎插件

**pgvector**（PostgreSQL 扩展）。这是 2026 年最重要的"非专门"向量库——PG 已有用户拿它当默认。0.7 起支持 HNSW + halfvec + 二值量化（`bit(d)` + `binary_quantize()`，维度 ≥ 900 时默认走 1-bit 模式），0.8 加了 iterative index scan 让过滤场景的召回率不再断崖式下跌（0.8.x 是 2026 年初的稳定线）。优点是和现有事务、外键、SQL 联合查询天衣无缝，备份/复制/权限沿用 PG 全套生态，DBA 不用学新东西。缺点是分片得靠 Citus/分库、亿级以上不如专门方案、HNSW 建索引内存占用大。**如果你的团队已经在用 PG 而向量量级 < 5000 万，pgvector 是默认答案**。**何时不要用 pgvector**：见 20.5.6 节，这里只点核心——单库稳定 > 5000 万、写入 QPS 持续 > 500、p99 < 20ms 硬要求、必须做 multi-vector / ColBERT 任意一条命中，就该规划迁移到专门方案。

**Elasticsearch / OpenSearch**（dense_vector 字段 + KNN 插件）。如果团队已经在用 ES 做日志和全文检索，复用一套基础设施很合理。性能不如 Qdrant/Milvus，但混合检索能力强（BM25 已经成熟）、运维生态完整。

### 20.2.6 决策表：你应该选哪个

| 你的现状 | 向量量级 | 推荐 | 备选 |
| --- | --- | --- | --- |
| 已经在用 PG，团队没 DBA 时间学新东西 | < 5000 万 | **pgvector** | Qdrant |
| 没用 PG，新建 RAG 服务 | < 5000 万 | **Qdrant** | Weaviate |
| SaaS 多租户，每个客户隔离 | 任意 | **Weaviate** | Qdrant + shard key |
| 数据已经在数据湖（S3/Parquet） | 任意 | **LanceDB** | DuckDB + 自建 |
| 数据量超 1 亿，团队会 K8s | 1 亿~100 亿 | **Milvus** | Vespa |
| 不想招 DBA，预算够 | 任意 | **Pinecone** | Zilliz Cloud |
| 已经在用 ES/OpenSearch | < 1 亿 | **ES dense_vector** | Qdrant |
| 嵌入式 / 单机本地工具 | < 1000 万 | **LanceDB** | Chroma |
| 学习/原型 | < 100 万 | **Chroma** | LanceDB |

一个反直觉的事实：**性能榜单第一名 ≠ 你应该选的**。运维成本、迁移难度、和现有 stack 的契合度，加起来通常比 30% 的 QPS 差距重要。

## 20.3 核心特性：超越"能存向量"

挑选向量库时，下面这几项远比基准 QPS 关键。

### 20.3.1 Filter / Payload 过滤

每个向量除了向量本身，通常还带元数据：`{"user_id": 42, "lang": "zh", "created_at": 1750000000}`。生产查询几乎总是"在某条件下找最近的"——给 user_id=42 的文档里找最相关的、给一周内的内容里找最相关的。

过滤有两种实现策略：

- **Pre-filter（先过滤后向量）**：先用 metadata 索引筛出候选集，再在子集上做向量检索。问题：候选集很小时，向量索引（特别是 HNSW）效率反而崩塌——因为图被打散了。
- **Post-filter（先向量后过滤）**：先 ANN 找 top-k，再过滤。问题：如果过滤选择性高（比如 0.1% 命中），可能向量结果里一个都不剩，得放大 k 重试。

Qdrant 选了第三条路：**filterable HNSW**。建索引时把 payload 字段直接编进图的访问规则，图遍历时同时检查过滤条件，避免上面两种极端。这也是 Qdrant 在过滤场景胜过 Milvus 的关键原因。

工程上要做的：

1. 给经常过滤的字段建 payload 索引（每个库语法不同，但都有）。
2. 评估过滤选择性：如果 < 5%，考虑分 collection 而不是用 filter。
3. 别把高基数字段（如 timestamp）当等值过滤，要建范围索引。

### 20.3.2 混合检索：dense + sparse

纯向量检索的弱点是"专有名词、ID、错别字"——dense embedding 把它们抽象掉了。BM25 这种关键词算法对精确匹配反而擅长。混合检索就是两路并发查询然后融合。

融合方法：

- **RRF（Reciprocal Rank Fusion）**：$\text{score}(d) = \sum_i \frac{1}{k + \text{rank}_i(d)}$，$k$ 默认 60。优点是不需要分数归一化、对各路 score 尺度无感、几乎是 2026 年的事实标准。
- **加权和**：$\alpha \cdot \text{dense} + (1-\alpha) \cdot \text{sparse}$，要做 min-max 归一。$\alpha$ 通常 0.5~0.7，得对每个数据集调。
- **学习排序**：用 cross-encoder rerank 取代纯融合，质量上限高，延迟成本也高。

Qdrant 1.11+（RRF/DBSF 原生 fusion）、Milvus 2.5+（WeightedRanker / RRFRanker）、Weaviate（BlockMax WAND）、LanceDB、Pinecone 都内置 RRF。pgvector 的 sparse 要靠 PG 的 `tsvector` 或者 `paradedb`/`pg_search` 扩展，自己拼 RRF（也就是把两路 rank 在 SQL 里 union 后按 `1/(k+rank)` 求和）。

### 20.3.3 多向量 / late interaction

ColBERT 这类模型不输出一个向量，而是每个 token 一个向量。检索时算 query 所有 token 与 doc 所有 token 的相似度矩阵，取 max-sim 之和。质量比单向量 dense 高 5~10 个 NDCG 点，代价是存储和计算成倍增加。

Qdrant 1.10+ 原生支持 multi-vector：一个 point 可以挂一组向量（每条文档自带一个 token 维度的矩阵），检索时按 max-sim 聚合，可以直接当 ColBERT 后端。Vespa 一直支持。Milvus 2.5+ 通过 multi-vector hybrid search + 原生 RRF/Weighted ranker 也能做。pgvector 没有原生多向量，社区方案是把每个 token 向量存成 `vector[]` 数组，再写一个 PL/pgSQL 或 SQL 的 `max_sim()` 函数，循环算 query 每个 token 与 doc 每个 token 的相似度后求和。

**这个方案在 1000 万级数据下能跑吗？认真说不能**。我给一个量化数据点（128 token × 128 维 ColBERT，1000 万文档）：

- 存储：1000 万 × 128 × 128 × 2 字节（halfvec）= 320 GB，是单向量方案的 100+ 倍。
- 检索：max_sim 是 token 矩阵相乘，pgvector 没有 token 级图索引，等于退化成全表 + 复杂 SQL；单 query p95 在普通硬件上要数百毫秒到数秒，**绝对到不了 P95 < 100ms**。
- 即使先用单向量 dense 召回 top-1000，再在子集上跑 max_sim rerank，1000 行 ColBERT max_sim 计算也要 50~150 ms，总延迟通常 > 200 ms。

实战结论：**pgvector 上凑 ColBERT 仅适合 < 50 万文档的 PoC**，再往上走必须用 Qdrant multi-vector / Vespa / Milvus 2.5+。如果你的需求是"质量高一点的 RAG 而非真 ColBERT"，性价比最高的替代方案是 **dense 召回 + cross-encoder rerank**（见 20.7.3 节），单向量 + reranker 通常能拿到 ColBERT 的 80% 提升，工程复杂度低一个数量级。

实践上：第一阶段用 dense 召回 100~500，第二阶段用 ColBERT rerank。不要直接用 ColBERT 做一阶段——存储成本和延迟都顶不住。

### 20.3.4 量化：内存预算的开关

| 类型 | 压缩比 | 召回损失 | 实现 |
| --- | --- | --- | --- |
| **Scalar Quantization (SQ8)** | 4× (float32 → int8) | 1~3% | 每维独立线性量化 |
| **Half precision (fp16/halfvec)** | 2× | < 1% | IEEE 754 半精度 |
| **Binary Quantization (BQ)** | 32× (float32 → 1 bit) | 5~15%（需 rerank） | 每维 sign |
| **Product Quantization (PQ)** | 16~64× | 5~15% | 子向量 k-means |
| **Anisotropic / OPQ** | 同 PQ 但更准 | 3~10% | 学习型量化 |

工程上的常识：

1. **从 halfvec / fp16 开始**。几乎无损、立省一半内存，没有理由不用。
2. **大规模上 BQ + rerank**。先 BQ 召回 200，再用 fp16 做精确距离 rerank，10 亿向量级别的标准玩法。
3. **PQ 是历史选项**，对新项目优先选 BQ + rerank，工程更简单。
4. **测召回别只看 top-1**。生产上 top-10 召回率才是用户体感。

### 20.3.5 持久化、复制、备份

容易被忽略但生产必查的清单：

- WAL（Write-Ahead Log）是不是默认开？崩溃后能不能恢复到秒级？
- 副本协议是什么？Qdrant 用 Raft 复制元数据 + 业务层 N 份副本，Milvus 走对象存储兜底，pgvector 直接用 PG 的 streaming replication / logical replication。
- 怎么做快照？Qdrant 有 snapshot API（导出整个 collection 到磁盘 / S3），Milvus 走对象存储自然分层，pgvector 用 `pg_dump` 或 PITR。
- 索引能不能在线重建？HNSW 重建可能要小时级，是否支持 alias 切换、双写?

## 20.4 数据模型设计

这一节聊"建库前应该先想清楚"的事。和关系库 schema 设计一样，向量库的 schema 错了，后期改的代价远超想象。

### 20.4.1 Collection 划分

一个 collection 通常对应一种向量配置（维度、距离度量、索引参数）。决定划分粒度的几个问题：

- **不同来源的数据要不要分?** 如果 embedding 模型不同（比如客服 KB 用 `bge-large-zh`，产品文档用 `text-embedding-3-large`），必须分。维度都不一样，混不到一起。
- **多语言要不要分?** 多语种 embedding（multilingual-e5）单库够用；单语 embedding 拼一起会越查越乱。
- **冷热数据要不要分?** 一年前的归档数据可以用更激进的量化，热数据用 fp16 或 fp32。分 collection 比一个 collection 内分 partition 干净。
- **多租户要不要分?** 几十个租户：分 collection。几千个：用 payload + filter 或多租户机制（Weaviate）。几十万：必须有原生多租户（Weaviate / Qdrant 的 shard key）。

### 20.4.2 ID 策略

主键有两种常见模式：

- **业务 ID 直用**：`doc_id="article_123#chunk_4"`。优点是好对账、好去重。缺点是字符串长，索引大，分布不均。
- **UUID / 雪花 ID**：均匀分布，分片好，无业务语义。需要单独存 `external_id` 字段做映射。

生产推荐：**主键用 UUIDv7（时间有序的 UUID），payload 里存 `external_id`**。UUIDv7 既保留了均匀分布又有时间局部性，对 HNSW 缓存命中率友好。Qdrant、Milvus、pgvector 都支持。

### 20.4.3 Payload schema

向量库的 payload 不是 schemaless 自由场——你可能以为是，结果发现 filter 慢或不对的时候才知道索引没建上。

每个常用过滤字段都要明确：

- 字段类型（int / keyword / text / geo / datetime）
- 索引类型（hash / range / fulltext / geo）
- 是否多值

Qdrant 的 `create_payload_index` 必须显式调用，没建索引的字段也能 filter 但走全表。pgvector 直接用 PG 的 B-Tree、GIN、BRIN，按 PG 老规矩来。

### 20.4.4 向量字段版本管理

embedding 模型升级是迟早的事。OpenAI 从 `text-embedding-ada-002`（1536 维）到 `text-embedding-3-large`（默认 3072 维，可截断到任意维），分布和维度都变了。换模型时怎么办？

三种策略：

1. **双写过渡**：建新 collection，新数据双写老和新；切流之后老 collection 跑剩余 backfill；ready 后下线老库。简单可控。
2. **named vectors**：Qdrant 支持一个 point 上挂多个向量字段（`v1`、`v2`），同库共存，按需查询。迁移期最干净，但只 Qdrant 这种支持。
3. **滚动重嵌**：在线把老向量逐步替换成新向量。容易出现"新老混查"的怪事，慎用。

记一个原则：**embedding 模型版本号要写进 collection 名**（`docs_bge_large_v15`），不要假设你只会用一个模型一辈子。

## 20.5 pgvector 深度实战

为什么单独给 pgvector 一节？因为 2026 年大量企业的实际起点是"我们有 PG，能不能不引入新组件？" 答案在 5000 万向量以下基本是 yes。这一节给一份能直接抄进生产的 schema。

> 衔接提示：第 04 章 11.1.1 节产出的 `trip_features_with_embedding.parquet` 里有一列 `trip_embedding_bge_small_v15_512`，对应这一节 schema 里的 `embedding halfvec(d)` 字段。**字段命名规则**保持"业务名 + 模型名 + 版本 + 维度"四件套，是为了换 embedding 模型时（见 04 章 7.1.2 节）能在同一张表里双写老/新两列、做 A/B 切流。下面的 schema 把 `embed_model` 单独做了字段，配 `embedding_v1` / `embedding_v2` 双列就能直接落地这套迁移方案。

### 20.5.1 安装与扩展

```sql
-- PostgreSQL 16+
CREATE EXTENSION IF NOT EXISTS vector;     -- pgvector 0.8.x（2026 年初稳定线）
CREATE EXTENSION IF NOT EXISTS pg_trgm;    -- 用于关键词模糊匹配
-- 可选：paradedb 提供高质量 BM25
-- CREATE EXTENSION IF NOT EXISTS pg_search;
```

pgvector 提供四种向量类型：

- `vector(d)`：每维 float32（4 字节）
- `halfvec(d)`：每维 fp16（2 字节）。0.7 起稳定。生产首选。
- `bit(d)`：二值向量。配合 `<~>`（hamming）或 `<%>`（jaccard）做 BQ。
- `sparsevec(d)`：稀疏向量，适合 SPLADE 风格的稀疏 embedding。

### 20.5.2 一份生产级 schema

```sql
-- 文档主表（业务字段）
CREATE TABLE documents (
    id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       bigint      NOT NULL,
    source          text        NOT NULL,        -- 'kb' / 'web' / 'email'
    lang            text        NOT NULL DEFAULT 'zh',
    title           text,
    url             text,
    created_at      timestamptz NOT NULL DEFAULT now(),
    updated_at      timestamptz NOT NULL DEFAULT now()
);

-- chunk 表（一文档多块）
CREATE TABLE doc_chunks (
    id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id     uuid        NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    tenant_id       bigint      NOT NULL,        -- 冗余，便于过滤
    chunk_index     int         NOT NULL,
    content         text        NOT NULL,
    -- 主向量：halfvec 节省一半空间
    embedding       halfvec(1024)   NOT NULL,
    -- BM25 用：tsvector 物化
    content_tsv     tsvector    GENERATED ALWAYS AS (to_tsvector('simple', content)) STORED,
    embed_model     text        NOT NULL DEFAULT 'bge-large-zh-v1.5',
    token_count     int,
    created_at      timestamptz NOT NULL DEFAULT now(),
    UNIQUE (document_id, chunk_index)
);

-- HNSW 索引（向量主路）
-- 注意：CONCURRENTLY 不会锁表，生产必加
CREATE INDEX CONCURRENTLY idx_chunks_embedding_hnsw
    ON doc_chunks
    USING hnsw (embedding halfvec_cosine_ops)
    WITH (m = 16, ef_construction = 200);

-- 备选：IVFFlat 索引。建索引快、内存占用低，但召回率天花板比 HNSW 低。
-- 适合"先批量灌一次数据后基本只读"的离线知识库；增量场景仍用 HNSW。
-- CREATE INDEX CONCURRENTLY idx_chunks_embedding_ivf
--     ON doc_chunks
--     USING ivfflat (embedding halfvec_cosine_ops)
--     WITH (lists = 1000);   -- 经验值：lists ≈ sqrt(N)

-- payload 过滤索引
CREATE INDEX CONCURRENTLY idx_chunks_tenant ON doc_chunks (tenant_id);
CREATE INDEX CONCURRENTLY idx_chunks_doc    ON doc_chunks (document_id);

-- 全文索引：GIN 索引 tsvector
CREATE INDEX CONCURRENTLY idx_chunks_tsv ON doc_chunks USING gin (content_tsv);

-- 可选：维护更新时间触发器（略）
```

几个关键决策：

- **halfvec 不是 vector**：1024 维下，1000 万 chunk 节省 ~20 GB 磁盘 + 内存。召回率几乎无损。新项目从 day 1 就上。
- **`tenant_id` 冗余到 chunk 表**：避免每次 join。pgvector 的 HNSW 在 join 后过滤场景下表现差。
- **`UNIQUE (document_id, chunk_index)` 是去重保险**：embedding pipeline 容易重跑，没这条会重复写入。
- **`embed_model` 字段**：换模型时方便 ETL 区分。
- **`content_tsv` 物化列**：BM25 不要每次查询时实时算 `to_tsvector`。

### 20.5.3 查询参数：ef_search 是开关

```sql
-- 调高召回率
SET hnsw.ef_search = 100;   -- 默认 40，生产推荐 100~200

-- 纯向量查询
SELECT id, content,
       embedding <=> $1::halfvec AS distance   -- <=> 是 cosine distance
FROM doc_chunks
WHERE tenant_id = $2
ORDER BY embedding <=> $1::halfvec
LIMIT 10;
```

关键算子（pgvector 规范）：

- `<->`：欧氏距离（L2）
- `<=>`：cosine 距离（1 - cosine 相似度）
- `<#>`：负内积
- `<+>`：L1 距离

新项目 99% 走 cosine。注意是 `<=>`，记反了 sort 顺序就反了。

### 20.5.4 容量规划：halfvec 内存预算

HNSW 索引大约要这么多内存：

```
索引大小 ≈ N × (d × bytes_per_dim + M × 8)
        = N × (d × 2 + 16 × 8)              # halfvec, M=16
```

举例：1000 万 × 1024 维 × halfvec：

- 向量本身：10⁷ × 1024 × 2 = 20 GB
- 图边：10⁷ × 16 × 8 = 1.3 GB
- 合计 ~22 GB，加上 PG 的 work_mem、buffer，建议 64 GB RAM 起步。

如果是 vector（fp32）：~42 GB，建议 128 GB。这就是 halfvec 在生产真正的价值——它不是省钱，是让你能用一台机器搞定原本要两台的工作负载。

**关于"5000 万级上限"的实测数据点**（AWS m6i.4xlarge，16 vCPU / 64 GB / gp3 6000 IOPS，pgvector 0.8.x，halfvec 1024 维，HNSW M=16 / ef_construction=200 / ef_search=128，配单租户 payload）：

| 数据规模 | 内存占用 | 单点 query p50 / p95 | 写入吞吐（COPY） | 备注 |
| --- | --- | --- | --- | --- |
| 100 万 | ~3 GB | 2 ms / 5 ms | ~30k rows/s | 完全无压力 |
| 1000 万 | ~25 GB | 5 ms / 15 ms | ~20k rows/s | 推荐档 |
| 3000 万 | ~75 GB | 12 ms / 35 ms | ~12k rows/s | 内存吃紧，要升 128 GB |
| 5000 万 | ~125 GB | 25 ms / 80 ms | ~8k rows/s | 极限档，再大要分片 |
| 8000 万 | ~200 GB | 60 ms / 180 ms | ~5k rows/s | p95 已经超 100ms，需要 BQ + rerank 救场 |

为什么 5000 万是经验上限？三件事同时塌：

1. **内存压不住**：超过 5000 万要 256 GB 机器，单机成本翻倍但单机 ANN 性能并不能线性提升。
2. **IOPS 瓶颈**：HNSW 的图遍历是大量随机读，超过 5000 万行，shared_buffers 命中率掉下来，gp3 6000 IOPS 会成为瓶颈，要升到 io2 或 gp3 16000+。
3. **建索引时间爆炸**：5000 万 halfvec(1024) 在 64 GB 机器上建一次 HNSW 索引大约 4~6 小时，越过这个量级单次重建超过半天，运维窗口非常难安排。

如果你已经在 3000~5000 万之间，先做这几件事再考虑迁移：

- 升 RAM 到 256 GB（最便宜的决策）。
- 加 BQ + rerank（20.5.5 节），把热路径走二值索引。
- 拆 collection（按时间或租户分），让单库稳定在 3000 万以内。
- 还压不住再切 Qdrant / Milvus。

### 20.5.5 二值量化：百万到亿级的开关

```sql
-- 用生成列直接落 BQ，避免双写漂移；binary_quantize() 是 pgvector 0.7+ 内置函数
ALTER TABLE doc_chunks
    ADD COLUMN embedding_bit bit(1024)
    GENERATED ALWAYS AS (binary_quantize(embedding)::bit(1024)) STORED;

-- BQ 索引（hamming 距离用 <~>，jaccard 用 <%>）
CREATE INDEX CONCURRENTLY idx_chunks_bit_hnsw
    ON doc_chunks
    USING hnsw (embedding_bit bit_hamming_ops)
    WITH (m = 16, ef_construction = 200);

-- 两阶段查询：BQ 召回 + halfvec rerank
WITH candidates AS (
    SELECT id, content, embedding, embedding_bit
    FROM doc_chunks
    WHERE tenant_id = $2
    ORDER BY embedding_bit <~> binary_quantize($1::halfvec)::bit(1024)
    LIMIT 200
)
SELECT id, content, embedding <=> $1::halfvec AS distance
FROM candidates
ORDER BY distance
LIMIT 10;
```

二值索引的内存占用大约是 fp32 的 1/32：1000 万 × 1024 维 BQ ≈ 1.3 GB。同样的机器能塞 3 亿向量。代价是召回率掉 5~10%，靠 rerank 拉回。pgvector 0.7 起 `binary_quantize()` 是官方函数，0.8 又加了 iterative index scan，让"BQ 召回 + filter + halfvec rerank"这条链在过滤选择性低的场景也不会因为 HNSW 早停而漏召回。

### 20.5.6 何时离开 pgvector？

老实说，下面任意一条命中，就该考虑切到专门方案：

1. 单库向量数据稳定超过 5000 万。
2. 写入 QPS 持续高（pgvector 的 HNSW 写入比读慢一个量级）。
3. 必须做大量 multi-vector / ColBERT。
4. p99 延迟硬要求 < 20ms 且单表 > 1000 万。
5. 多租户 > 几千个，每个 tenant 都要独立物理隔离。

否则就留在 pgvector。技术栈少一套独立组件，运维复杂度差一个数量级。

## 20.6 写入与索引

向量库的写入路径和关系库不一样：每条记录除了字段，还要把向量写入 ANN 索引——这步成本比写字段本身高得多。

### 20.6.1 批量写入 vs 单条

所有主流向量库的批量写入都比单条快 10~100 倍。原因：

1. ANN 索引插入是逐节点的图修改，批量可以共享候选池计算。
2. 持久化 fsync 可以批一次。
3. 网络 RTT 摊到批里。

经验值：

- pgvector：批 500~2000 行 / `INSERT ... VALUES (...), (...)` 或 COPY。
- Qdrant：批 100~1000 个 point / `upsert` 调用。
- Milvus：批 10000 行 / `insert` 调用。

写完还有一步**索引刷新/构建**。Milvus 区分 growing segment（实时可查但慢）和 sealed segment（异步建索引），需要等待或者主动 `flush + load`。Qdrant 默认增量更新 HNSW，但 `optimize` 会重整索引。pgvector 的 HNSW 是 incrementally updated 的。

### 20.6.2 删除与软删

向量库的删除有两种：

- **物理删除**：从图里摘掉节点。HNSW 的删除会破坏图连通性，长期会让召回率漂移。一般是标记 + 周期性 compact。
- **软删除（tombstone）**：在 payload 加 `deleted=true`，查询用 filter 屏蔽。优点是即时，缺点是查询要带过滤、空间不释放。

生产建议：**短期软删 + 周期 compact**。不要对每个删除都触发图重整。

### 20.6.3 reindex 策略

什么时候需要 reindex？

- 数据量增长 5 倍以上（图层数和 M 配置可能不再最优）。
- 大量删除后图稀疏。
- HNSW 参数调整（M、ef_construction）。
- embedding 模型升级。

策略：

1. 在线 reindex：Qdrant `optimize`、Milvus 重建 segment、pgvector 在新表建索引后 swap。
2. 离线 reindex：dump 数据 → 新建 collection → 切流量。最干净但需要双倍空间。

### 20.6.4 增量 vs 全量

RAG 场景的两种写入节奏：

- **增量**：用户上传文档后实时写。要求低延迟、单条/小批量。pgvector / Qdrant 做得最好。
- **全量重建**：每天 ETL 把整个 KB 重新嵌入并写入。吞吐优先。LanceDB 的批量重写架构在这里很顺。

典型 RAG 系统是增量为主、每月做一次全量重建（保证 deleted/updated 真实清理）。

### 20.6.5 冷启动：从 0 灌到 1000 万向量怎么做不卡 IO

新项目第一次灌库，是被严重低估的"破日"。我见过的典型反面教材：开发同学拿到 1000 万条文本 + embedding 后，写了个 for 循环，单条 INSERT，每行都触发 HNSW 增量插入 + WAL fsync——预计 30 分钟跑完的事，跑了 18 小时还没完，期间磁盘 IOPS 跑满，影响了同库其他业务。

**正确姿势：先灌数据后建索引，分阶段走**。

**步骤 1：raw embeddings 进暂存表（无索引）**

最快的方式是 PG 的 COPY 协议。不要 INSERT，不要 ORM。这一步只灌数据，不建任何 ANN 索引。

```python
import psycopg
import polars as pl

DSN = "postgresql://..."
df = pl.read_parquet("data/marts/trip_features_with_embedding.parquet")

with psycopg.connect(DSN, autocommit=False) as con:
    cur = con.cursor()
    # 暂存表：无索引、无约束、UNLOGGED 关掉 WAL（崩了重灌即可）
    cur.execute("""
        CREATE UNLOGGED TABLE doc_chunks_loading (
            id           uuid,
            tenant_id    bigint,
            content      text,
            embedding    halfvec(1024)
        )
    """)
    with cur.copy("COPY doc_chunks_loading (id, tenant_id, content, embedding) FROM STDIN") as copy:
        for row in df.iter_rows():
            copy.write_row(row)
    con.commit()
```

为什么 UNLOGGED？冷启动场景 WAL 是纯粹的负担——崩了不丢数据是因为原始 parquet 还在，重灌一次比恢复 WAL 快。1000 万行 halfvec(1024) 用 COPY + UNLOGGED 一般 5~10 分钟可以灌完，主要瓶颈是网络带宽。

**步骤 2：批量建索引（一次性、并行、限流）**

```sql
-- 切回 LOGGED，让索引和正式表挂上 WAL（生产数据要 crash-safe）
ALTER TABLE doc_chunks_loading SET LOGGED;

-- 提升建索引的并行度和内存
SET maintenance_work_mem = '8GB';      -- 原来默认 64 MB，建 HNSW 时翻百倍
SET max_parallel_maintenance_workers = 8;

-- 建 HNSW，pgvector 0.6+ 起支持并行建图
CREATE INDEX idx_loading_emb ON doc_chunks_loading
    USING hnsw (embedding halfvec_cosine_ops)
    WITH (m = 16, ef_construction = 200);
```

实测数据：1000 万 × halfvec(1024)，maintenance_work_mem=8GB + 8 worker，普通 m6i.4xlarge 上大约 40~80 分钟建完。比"边灌边建"快 5~10 倍。

**步骤 3：原子切表**

```sql
BEGIN;
ALTER TABLE doc_chunks RENAME TO doc_chunks_old;
ALTER TABLE doc_chunks_loading RENAME TO doc_chunks;
ALTER INDEX idx_loading_emb RENAME TO idx_chunks_embedding_hnsw;
COMMIT;

-- 老表观察一周再删
-- DROP TABLE doc_chunks_old;
```

如果是新项目第一次灌，跳过 RENAME，直接把 loading 表改成正式表名。

**Qdrant / Milvus 上的对应做法**：

- **Qdrant**：建 collection 时把 `optimizers_config.indexing_threshold` 调高（默认 20000，冷启动时设到 100_000_000，相当于先不建索引），灌完数据再调回默认值，让后台 optimizer 一次性建图。批量 upsert 用 `wait=False` 异步提交。
- **Milvus**：先把数据全部 `insert` 进去，调用 `flush()` 落盘，再 `create_index()`。Milvus 的 sealed segment 是按 segment 异步建索引的，本来就接近这套流程，但要避免在 grow segment 状态下查询（召回率低）。

**冷启动时的 IO 限流**：

- 如果是和业务库共用的 PG，**新建 tablespace 到独立磁盘**，把 `doc_chunks_loading` 放过去，避免抢占主业务 IO。
- 用 `pg_sleep(0.01)` 或 `commit_delay` 在每批次之间略微限流，给同库其他业务喘息。
- 监控 `pg_stat_io` 和系统 `iostat`，IOPS 触顶就降并发。

**估算公式（pgvector 1000 万 halfvec(1024) 冷启动）**：

| 阶段 | 时间预算 | 资源占用 |
| --- | --- | --- |
| Embedding 计算（OpenAI API 或本地 GPU） | 2~6 小时 | 取决于 batch + 模型 |
| COPY 灌数据（UNLOGGED） | 5~10 分钟 | 写 IOPS 峰值高 |
| HNSW 建索引 | 40~80 分钟 | maintenance_work_mem 8 GB、8 并行 |
| 切表 + 验证 | < 1 分钟 | 几乎无开销 |
| **总计** | **~3~7 小时** | embedding 是大头 |

对比"边灌边建"的 18+ 小时，分阶段做能省一半以上时间，且对在线业务影响最小。这套方法论在 Qdrant、Milvus、Weaviate 上都通用——核心原则永远是 **bulk load 先关索引，灌完再批量重建**。

## 20.7 查询优化

写完了能查到，是基本要求。让 p99 < 50ms、recall > 0.95、QPS > 1000，是工程问题。

### 20.7.1 三个核心调参

无论哪个 ANN 库，HNSW 都靠三个参数决定一切：

| 参数 | 作用 | 调高 | 调低 |
| --- | --- | --- | --- |
| `M` | 每个节点的边数 | 召回↑、内存↑、延迟↑ | 召回↓、省内存 |
| `ef_construction` | 建图的候选池 | 图质量↑、建图慢 | 建图快、图质量低 |
| `ef`（query） | 查询候选池 | 召回↑、延迟↑ | 召回↓、快 |

调参顺序：

1. **建图固定**：`M=16, ef_construction=200`。多数场景这就够了。
2. **查询调 `ef`**：从 64 起，每次翻倍，看召回率曲线。
3. **挑工作点**：在 recall 收益边际递减的拐点，通常 `ef=128~256` 是甜区。

测召回率的方法：取 1000 条查询，用暴搜得到 ground truth top-10，再跑 ANN 看交集大小。生产环境用线上日志的 top-K + 离线重算。

### 20.7.2 过滤策略

回到 20.3.1 的话题，但这里聚焦工程：

```python
# Qdrant：filter 在 search 内做（filterable HNSW）
client.search(
    collection_name="docs",
    query_vector=embed("foo"),
    query_filter=Filter(must=[
        FieldCondition(key="tenant_id", match=MatchValue(value=42)),
        FieldCondition(key="lang", match=MatchValue(value="zh")),
    ]),
    limit=10,
    search_params=SearchParams(hnsw_ef=128),
)
```

调试 filter 性能的两个观察点：

1. 过滤后剩多少？太少（< 1% 命中）就别走 HNSW，直接 brute force 子集。
2. payload 索引是否命中？Qdrant `info` API 看 indexed payload，pgvector 用 `EXPLAIN ANALYZE`。

### 20.7.3 Rerank：性价比最高的优化

第一阶段 ANN 召回 100~200，第二阶段 cross-encoder rerank 取 top-10。

```python
# 概念伪代码
candidates = vector_store.search(q_emb, k=200, filter={...})
pairs = [(query, c.content) for c in candidates]
scores = cross_encoder.predict(pairs)
top_k = [c for _, c in sorted(zip(scores, candidates), reverse=True)[:10]]
```

cross-encoder（bge-reranker、Cohere rerank、jina rerank）单次比 ANN 慢 50~200 倍，但只跑 200 个，总延迟 +20~100ms 通常可接受。NDCG 提升 5~15 个点是 RAG 里最便宜的质量改进。

### 20.7.4 缓存：别忽视的免费收益

热点 query 在 RAG 里很常见。一个客服系统里，"怎么退款"、"忘记密码"几乎天天有人问。在向量检索前加一层**embedding cache + result cache**：

- 完全相同 query → 直接返回缓存结果。
- embedding 缓存：相同 query 别调 OpenAI，直接拿 hash 取向量。

简易实现：Redis + SHA256(query + filter)。命中率 30~50% 不少见，等价于把检索成本降一半。

### 20.7.5 监控与召回率回归

线上监控应该至少有：

- **QPS / 错误率 / p50/p95/p99 延迟**（按 collection、按 filter 类型分维度）
- **召回率回归**：每天用一组金标 query 跑，看是否漂移。
- **段（segment）/ 索引健康**：删除比例、图碎片度、待 compact 数量。
- **embedding 队列积压**：写入慢通常瓶颈在 embedding 模型而不是数据库。

用 Prometheus 抓 Qdrant / Milvus / pgvector 的官方 exporter，Grafana 画板。重要：**别只看 latency，召回率在线回归是被 99% 团队漏掉的事**。

## 20.8 生产工程：部署、容量、隔离

### 20.8.1 部署形态

**单机 Docker**：起步够。Qdrant 一行 `docker run -p 6333:6333 qdrant/qdrant`。pgvector 装到现有 PG。开发到 PoC 到小规模生产都行，5000 万向量内别折腾分布式。

**K8s + Helm**：Qdrant 官方 helm chart，Milvus 强烈建议用 milvus-operator。要点：

- 持久化用 SSD `StorageClass`，向量数据本质是顺序读 + 随机读，机械盘是灾难。
- 资源 request/limit 按"索引大小 × 1.5"算 memory，CPU 按 QPS × 单查询 CPU 时间估算。
- Pod 反亲和：副本分散到不同 node，避免一个机器挂全军覆没。

**云托管**：Qdrant Cloud、Zilliz Cloud、Pinecone。省事，贵 2~3 倍，迁出难。

### 20.8.2 容量规划公式

最常见的算错点是只算向量本身。完整公式：

```
总内存 ≈ 向量数据 + 索引开销 + payload + 系统 overhead
       ≈ N × d × bytes_per_dim × 1.5（索引）
       + N × avg_payload_bytes
       + 操作系统/服务 1~2 GB
```

例子：3000 万 × 1024 维 × halfvec + 平均 500 字节 payload：

- 向量：3×10⁷ × 1024 × 2 = 60 GB
- HNSW 图：60 × 0.5 = 30 GB
- payload：3×10⁷ × 500 ≈ 14 GB
- 系统：4 GB
- 总：≈ 110 GB，配 128 GB RAM 机器较稳。

如果换 vector（fp32）：~190 GB，要 256 GB 机器。**halfvec 直接决定了你买 1 台还是 2 台机器的事**。

磁盘按内存的 2 倍预留（WAL、snapshot、临时索引）。CPU 按 QPS × 单查询 ms 算并发：QPS=1000、单查询 5ms、p99 安全系数 2，要 1000 × 5 × 2 / 1000 = 10 cores。

### 20.8.3 多租户隔离

三种粒度：

1. **共享 collection + 字段过滤**：最便宜。问题：tenant 数据隔离弱、热点 tenant 影响全部、删除一个 tenant 要 scan + delete。
2. **租户即 shard / partition**：Weaviate native multi-tenancy、Qdrant shard key by tenant_id。一个 tenant 一个分片，独立加载/卸载/删除。最适合 SaaS。
3. **租户即 collection / database**：完全物理隔离，每个 tenant 自己的索引参数。适合大客户、有合规要求的场景。运维成本高。

选哪个的决策点：

- tenant 数 < 100：shared + filter 够。
- 100 ~ 10000：原生多租户（Weaviate / Qdrant shard key）。
- > 10000：必须 Weaviate 这种为多租户设计的方案，或者在应用层做 collection 路由。

### 20.8.4 备份与灾备

最简备份组合：

- 每日一次 snapshot 到对象存储（S3/OSS）。Qdrant `POST /collections/{name}/snapshots`，Milvus 走对象存储分层天然有，pgvector 用 `pg_dump` 或 PITR。
- 跨可用区副本（如果业务能接受额外成本）。
- 演练：每季度做一次 restore 演习，别等真出事才发现 snapshot 缺东西。

向量库备份的一个特殊点：**索引可重建，向量数据不能重生**。所以备份只需要保 raw vectors + payload + 索引参数，不需要把索引文件 dump，重建索引比 restore 索引文件还快。

## 20.9 实战代码：同一份数据三库对比

下面是一份能实际跑的代码，把同一批 1 万条文档分别写进 Qdrant、pgvector、Chroma，比较写入和查询性能。

```python
# requirements.txt
# qdrant-client>=1.10
# psycopg[binary]>=3.1
# chromadb>=0.5
# sentence-transformers>=3.0
# numpy
import time, uuid, numpy as np
from typing import List, Tuple
from sentence_transformers import SentenceTransformer

# 用一个小模型省时间，生产换成 bge-large-zh
model = SentenceTransformer("BAAI/bge-small-zh-v1.5")
DIM = 512

def make_corpus(n: int = 10_000) -> List[Tuple[str, str, dict]]:
    """生成 n 条假数据：(id, text, payload)"""
    docs = []
    for i in range(n):
        text = f"产品文档第 {i} 章：本节介绍模块 {i % 100} 的使用方法。"
        payload = {
            "tenant_id": i % 10,
            "category": ["api", "guide", "faq"][i % 3],
            "lang": "zh",
        }
        docs.append((str(uuid.uuid4()), text, payload))
    return docs

print("生成数据...")
docs = make_corpus(10_000)
texts = [t for _, t, _ in docs]
print("批量 embedding...")
t0 = time.time()
embeddings = model.encode(texts, batch_size=64, normalize_embeddings=True)
print(f"embedding 耗时 {time.time()-t0:.1f}s, shape={embeddings.shape}")

# query 集
queries = ["怎么调用 api", "模块 42 的使用", "常见问题"]
q_embs = model.encode(queries, normalize_embeddings=True)
```

### 20.9.1 Qdrant

```python
from qdrant_client import QdrantClient
from qdrant_client.http import models as qm

qd = QdrantClient(url="http://localhost:6333")
COLL = "docs_qdrant"

if qd.collection_exists(COLL):
    qd.delete_collection(COLL)

qd.create_collection(
    collection_name=COLL,

    qd.upsert(collection_name=COLL, points=points[i:i+500], wait=True)
print(f"Qdrant 写入: {time.time()-t0:.2f}s")

# 查询：tenant=0 + cosine
qd_lat = []
for q_emb in q_embs:
    t0 = time.time()
    res = qd.search(
        collection_name=COLL,
        query_vector=q_emb.tolist(),

            qm.FieldCondition(key="tenant_id", match=qm.MatchValue(value=0))
```

```python
from psycopg.rows import dict_row

DSN = "postgresql://postgres:postgres@localhost:5432/postgres"

with psycopg.connect(DSN, autocommit=True) as con:
    cur = con.cursor()

    with con.cursor() as cur:
        t0 = time.time()

            (doc_id, pl["tenant_id"], pl["category"], txt, emb.tolist())

            for row in rows:
                copy.write_row(row)
        con.commit()
        print(f"pgvector 写入: {time.time()-t0:.2f}s")

# 写完建索引（生产用 CONCURRENTLY）
with psycopg.connect(DSN, autocommit=True) as con:
    cur = con.cursor()
    t0 = time.time()
    cur.execute("""
        CREATE INDEX idx_pg_emb ON docs_pg
        USING hnsw (embedding halfvec_cosine_ops)
        WITH (m = 16, ef_construction = 200)

    print(f"pgvector 建索引: {time.time()-t0:.2f}s")

with psycopg.connect(DSN, row_factory=dict_row) as con:

    cur.execute("SET hnsw.ef_search = 128")
    pg_lat = []
    for q_emb in q_embs:
        t0 = time.time()
        cur.execute("""
            SELECT id, content, embedding <=> %s::halfvec AS dist
            FROM docs_pg
            WHERE tenant_id = 0
            ORDER BY embedding <=> %s::halfvec
            LIMIT 10
        """, (q_emb.tolist(), q_emb.tolist()))
        cur.fetchall()
        pg_lat.append((time.time()-t0)*1000)
    print(f"pgvector 查询 ms: {pg_lat}")
```

### 20.9.4 三库对比小结

实测一台 16C/32G 机器、本地 docker、1 万条 512 维、tenant=0 子集 1000 条：

| 库 | 写入 | 单查询 p50 | 内存（仅服务） |

3. Chroma 写入慢主要是 Python 层开销，不一定 ANN 算法慢。

## 20.10 混合检索完整实战（Qdrant 1.10+）

最后给一个 hybrid 检索的端到端实现：dense + sparse + RRF 一次请求完成。

```python
from qdrant_client import QdrantClient
from qdrant_client.http import models as qm
from fastembed import TextEmbedding, SparseTextEmbedding

dense = TextEmbedding("BAAI/bge-small-zh-v1.5")
sparse = SparseTextEmbedding("Qdrant/bm25", language="chinese")

qd = QdrantClient(url="http://localhost:6333")
COLL = "hybrid_docs"

if qd.collection_exists(COLL):
    qd.delete_collection(COLL)

qd.create_collection(
    collection_name=COLL,
    vectors_config={"dense": qm.VectorParams(size=512, distance=qm.Distance.COSINE)},
    sparse_vectors_config={"bm25": qm.SparseVectorParams(modifier=qm.Modifier.IDF)},
)

texts = [
    "如何调用退款 API",
    "退款接口 v2 文档",
    "如何修改密码",
    "用户密码忘记怎么办",
    "FAQ：API key 在哪里申请",
]

dense_embs = list(dense.embed(texts))
sparse_embs = list(sparse.embed(texts))

points = []
for i, txt in enumerate(texts):
    points.append(qm.PointStruct(
        id=i,
        vector={
            "dense": dense_embs[i].tolist(),
            "bm25": qm.SparseVector(
                indices=sparse_embs[i].indices.tolist(),
                values=sparse_embs[i].values.tolist(),
            ),
        },
        payload={"text": txt},
    ))

qd.upsert(COLL, points)

# 查询：dense + sparse 各自召回，RRF 融合
query = "怎么退款"
q_dense = next(dense.embed([query])).tolist()
q_sparse = next(sparse.embed([query]))

results = qd.query_points(
    collection_name=COLL,
    prefetch=[
        qm.Prefetch(
            query=qm.SparseVector(
                indices=q_sparse.indices.tolist(),
                values=q_sparse.values.tolist(),
            ),
            using="bm25",
            limit=20,
        ),
        qm.Prefetch(
            query=q_dense,
            using="dense",
            limit=20,
        ),
    ],
    query=qm.FusionQuery(fusion=qm.Fusion.RRF),
    limit=5,
).points

for p in results:
    print(f"{p.score:.4f}  {p.payload['text']}")
```

预期输出（rank 由 RRF 算）：

```
0.0328  如何调用退款 API
0.0322  退款接口 v2 文档
0.0156  FAQ：API key 在哪里申请
...
```

dense 单跑可能"如何修改密码"会因为"怎么…怎么"语义近被排前；sparse 单跑只会精确匹配"退款"两字；RRF 融合让"退款"相关的两条同时上榜。这就是 hybrid 在生产里值钱的地方。

## 20.11 容量与成本估算脚本

给一个能直接抄进 sizing 文档的小脚本：

```python
def estimate_capacity(
    n_vectors: int,
    dim: int,
    dtype: str = "halfvec",          # "fp32" / "halfvec" / "int8" / "binary"
    avg_payload_bytes: int = 500,
    hnsw_m: int = 16,
    safety_factor: float = 1.3,
):
    bytes_per_dim = {"fp32": 4, "halfvec": 2, "int8": 1, "binary": 1/8}[dtype]
    vec_bytes = n_vectors * dim * bytes_per_dim
    graph_bytes = n_vectors * hnsw_m * 8        # 邻居 ID 8 字节
    payload_bytes = n_vectors * avg_payload_bytes
    overhead = 2 * 1024**3                       # 2 GB 系统

    total = (vec_bytes + graph_bytes + payload_bytes + overhead) * safety_factor
    return {
        "vectors_gb": vec_bytes / 1024**3,
        "graph_gb": graph_bytes / 1024**3,
        "payload_gb": payload_bytes / 1024**3,
        "total_gb": total / 1024**3,
        "recommended_ram_gb": next(x for x in [16, 32, 64, 128, 256, 512]
                                    if x >= total / 1024**3),
    }

# 例子：3000 万、1024 维、halfvec、500 字节 payload
print(estimate_capacity(30_000_000, 1024, "halfvec", 500))
# {'vectors_gb': 57.2, 'graph_gb': 3.6, 'payload_gb': 14.0,
#  'total_gb': 97.6, 'recommended_ram_gb': 128}

# 同样数据用 fp32
print(estimate_capacity(30_000_000, 1024, "fp32", 500))
# total ~170 GB → 推荐 256 GB

# 1 亿、1024 维、binary（带 rerank）
print(estimate_capacity(100_000_000, 1024, "binary", 500))
# total ~80 GB → 推荐 128 GB
```

把这种 sizing 写进 sprint planning，比项目跑一半 RAM OOM 之后扯皮重新部署省事得多。

成本侧再加一笔：

| 组件 | 月成本估算（AWS m6i 系列、北京/东京区） |
| --- | --- |
| 32 GB RAM 单机（pgvector PoC） | $200~300 |
| 128 GB RAM 单机（3000 万向量级） | $800~1200 |
| 256 GB RAM × 3 副本（亿级 Qdrant 集群） | $4000~6000 |
| Pinecone p2.x1 pod ×3 | $2000+ |
| Zilliz Cloud Standard 3CU | $1500+ |

OSS 自建 vs SaaS 在小规模差距不大（人力成本占大头），中等规模 OSS 自建省 30~50%，超大规模差距进一步拉大。

## 20.12 本章小结与决策清单

向量库选型不是性能榜单题，是工程权衡题。把下面这套清单存到团队 wiki，下次启动 RAG 项目对着填，能避开八成的早期错误：

1. **数据量**：< 1000 万 / 1000 万~1 亿 / > 1 亿 → 决定单机/分布式
2. **现有 stack**：有 PG → 优先 pgvector；有 ES → 优先 dense_vector；都没 → Qdrant
3. **查询模式**：纯向量 / 带过滤 / hybrid / 多向量 → 决定特性需求
4. **多租户规模**：< 100 / 100~10000 / > 10000 → 决定隔离机制
5. **运维能力**：单机 docker / K8s / 不想自己运维 → 决定 OSS vs SaaS
6. **embedding 维度和 dtype**：default halfvec、超大规模上 BQ + rerank
7. **召回率底线**：< 0.95 几乎不可接受、> 0.99 通常过度
8. **延迟预算**：p99 50ms 是常见线、< 20ms 要做缓存 + 索引调优
9. **成本预算**：算清楚每月 RAM/SSD 成本，OSS 自建 vs SaaS 算 TCO
10. **演进路径**：从 pgvector 起步，跨 5000 万再切专门方案是常见曲线

记一个反复说的原则：**halfvec、payload 索引、CONCURRENTLY、ef_search、hybrid 检索、rerank** —— 这六件事任意一个没做，向量库的体验都会比你预期差一截。任意一个做对，都能省一台机器或者一周排查。

向量库这块底座立住了，下一章把视角抬到"模型本身怎么上线"——拿到一个开源权重，怎么用 vLLM/SGLang 跑出 QPS、扛住流量、控住延迟，把推理引擎也变成生产能依赖的组件。
