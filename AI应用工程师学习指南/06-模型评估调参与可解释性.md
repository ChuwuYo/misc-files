# 第 06 章 · 模型评估、调参与可解释性

> 做 AI 应用，第一道门槛不是会调模型，而是会评估模型。
>
> 一个工程师如果只会盯着 `accuracy=0.92` 拍板上线，那基本可以认为他还没入门。
> 因为他既不知道这个 0.92 在业务上意味着什么，也不知道明天数据漂移以后这个数字还能不能站住，更不知道当用户问"模型为什么把我拒了"时该怎么回答。
>
> 这一章讲三件事：**怎么衡量模型的好坏**、**怎么把模型调到更好**、**怎么向人解释模型为什么这么决策**。三件事环环相扣：评估指标定义"好"，调参把模型推向"好"，可解释性告诉你这个"好"是真的好还是过拟合假象。

> **先给零基础读者两段"行话→人话"翻译**，免得后面遇到术语就懵：
>
> - **Accuracy（准确率）**：100 道题猜对了几道。直白，但样本不平衡时会骗人（参见 6.2）。
> - **Precision（精确率）/ Recall（召回率）**：报警准不准 / 该抓的有没抓到。一个怕冤枉，一个怕漏。
> - **F1**：Precision 和 Recall 的"调和平均"，一个折中分。
> - **AUC（ROC 曲线下面积）**：模型把"正样本"排在"负样本"前面的概率，1.0 是完美，0.5 是瞎猜。
> - **NDCG（推荐/搜索常用指标）**：排序质量打分，第一名错的比第五十名错的扣得狠（参见 6.4）。
> - **SHAP**：一种"给每个特征算一个贡献值"的可解释性方法，告诉你"这条预测里，年龄拉高了 0.2，收入降低了 0.05"（参见 6.9）。
> - **漂移（drift）**：训练时的数据分布和上线后看到的数据分布不一样了，模型会慢慢变废。
>
> 这些词在下面会反复出现。你不需要现在就完全理解，先有个直观印象就行，每节会展开讲。

上一章我们用 sklearn / XGBoost / LightGBM / CatBoost 把模型本身训出来了，但训出来不等于能上线。线下 `accuracy=0.92` 看着漂亮，业务真正会问的是：在我们的样本不均衡场景里这个数字算高还是低、调参时怎么避免在测试集上"作弊"、模型为什么把这条样本判错、上线三个月之后还成不成立。这一章把这些问题挨个拆开。

## 6.1 为什么评估比训练更难

训练是已经被自动微分和高质量框架解决得差不多的工程问题。给定数据和损失函数，让 GPU 转两小时，loss 一定会下降。任何一个本科二年级学过反向传播的人，照着 PyTorch 教程都能在三天内训出一个能跑的模型。

评估不是。评估的难处在于：

1. **指标和业务的距离**。模型层面的 0.95 召回率，可能在业务层面意味着每天多放过几千条欺诈交易。一个 AUC 看起来很优秀的违约预测模型，如果它把"未来一年违约的客户"和"未来五年违约的客户"混在一起识别，业务用起来仍然是灾难。指标的数值高低只是开始，把数值和真实业务损益对齐才是评估的核心难题。
2. **数据分布会变**。线下评估集是历史快照，线上是流动的现实。漂移一旦发生，离线再漂亮的指标都不作数。"模型上线时 AUC 0.91，三个月后实际命中率掉到 60%"——这种事在生产环境里几乎是常态，而不是异常。
3. **对生成式任务，没有标准答案**。LLM 给出的回答可以有十万种合理表达方式，BLEU 算出来 0.2，可能比 0.6 的版本更好。一句"巴黎是法国的首都"和"法国的首都是巴黎"，BLEU/ROUGE 上分数差很多，但语义完全等价；反过来"巴黎是英国的首都"和"巴黎是法国的首都"BLEU 上几乎只差一个 token，但事实正确性是相反的。
4. **可解释性是"评估"的一部分**。如果你不知道模型为什么对，就没法预测它什么时候会错。一个用"医院 ID"作为高重要性特征预测肺炎死亡率的模型，离线 AUC 可以漂亮到 0.95，但它学到的根本不是医学，而是"重症医院"和"普通医院"的转诊模式——换个医疗系统就崩。
5. **统计显著性常被忽视**。两个模型 0.823 和 0.835，差异是不是真的？基于一次 K 折的均值就拍板，等于把噪声当信号。任何严肃的模型对比都需要显著性检验或置信区间。
6. **多目标的权衡是政治问题**。准确率、延迟、成本、公平性、可解释性——这些目标常常冲突，最终的取舍不是数学问题而是组织问题。工程师能做的是把每个目标的权衡曲线画清楚。

所以这一章的整体顺序是：先把指标分类讲清楚（6.2~6.5），让你知道在哪种业务下应该看哪种数；再讲评估方法（6.6 交叉验证、6.10 显著性检验），保证你看到的数是稳定可信的；然后讲怎么调参把指标推上去（6.7~6.8）；接着讲怎么从"指标对了"上升到"我知道模型为什么对"（6.9 可解释性）；最后讲上线后的监控（6.11 漂移），并把全套流程串成实战代码（6.12）。

写在最前面的几条总原则，会贯穿整章：

- **永远不要用单一指标做决策。** 任何"最佳指标"都有它的盲区，多指标交叉验证才能排除盲点。
- **永远要带误差棒。** 0.85 ± 0.005 和 0.85 ± 0.05 是两件完全不同的事。
- **永远把训练-验证-测试三层分清。** 测试集只在最终汇报时使用一次，调参/选模型/特征工程都不能碰。
- **永远把离线指标当作"必要不充分条件"。** 离线赢不一定线上赢，离线输大概率线上输。线下是过滤器，不是裁判。

## 6.2 分类指标：从混淆矩阵开始

混淆矩阵是所有分类指标的根。两类问题里只有四个数：TP（真阳性）、FP（假阳性）、TN（真阴性）、FN（假阴性）。所有衍生指标都是这四个数的比例游戏。

> **TP/FP/TN/FN 的"人话"对照**（以"反欺诈：是不是欺诈"为例）：
>
> - **TP（真阳性）**：欺诈，模型也判欺诈。抓对了。
> - **FP（假阳性）**：正常交易，模型判欺诈。冤枉好人。
> - **FN（假阴性）**：欺诈，模型判正常。漏掉了。
> - **TN（真阴性）**：正常交易，模型也判正常。放对了。
>
> 后面所有指标公式都是这四个数怎么除来除去。看不懂公式时，回头对一下这张表。

| 指标 | 公式 | 业务含义 | 适用场景 |
|------|------|----------|----------|
| Accuracy | (TP+TN)/总数 | 整体猜对的比例 | 类别均衡、错误代价对等 |
| Precision | TP/(TP+FP) | 报警的准不准 | 误报代价高，如垃圾邮件判定 |
| Recall | TP/(TP+FN) | 该抓的有没抓到 | 漏报代价高，如癌症筛查、欺诈识别 |
| F1 | 2·P·R/(P+R) | P 和 R 的调和平均 | 类别不均衡、要兼顾两端 |
| AUC-ROC | ROC 曲线下面积 | 排序能力 | 阈值未定、需要比较模型整体 |
| AUC-PR | PR 曲线下面积 | 正样本极少时的排序能力 | 极不均衡（如 1:1000） |
| Log Loss | -Σ y·log(p) | 概率校准质量 | 需要可信概率，如风险定价 |

几条经验：

**Accuracy 是初学者陷阱。** 一个 99% 都是负样本的数据集，模型只要永远猜负就能拿 99% 准确率。任何时候报告 Accuracy，都得先报告类别分布。在欺诈检测、医疗筛查、广告 CTR 这类自然不均衡的领域里，Accuracy 几乎不传递信息。一个常见笑话：用一段时间统计了一下"模型正确率 99.7%"，老板拍板上线，三周后才发现模型对所有正样本都判错了——因为正样本只占 0.3%。

**Precision 和 Recall 是矛盾的。** 通常你只能选一边重要。风控场景倾向高 Precision（不要冤枉好人，免得用户投诉、客服爆炸）；急诊筛查倾向高 Recall（宁可虚惊不可漏诊，漏诊代价是人命）。F1 是没办法时的折中，但工程师应该警惕：F1 是几何平均的妥协，业务真正需要的往往是"在 Precision >= 0.9 的前提下最大化 Recall"这种带约束的目标，而不是 F1。这种情况下应当画 PR 曲线、自己定阈值，而不是机械地优化 F1。

**F-beta 是带偏好的 F1。** F-beta = (1+β²)·P·R / (β²·P + R)。β=1 就是 F1；β=2 把 Recall 看得比 Precision 重 2 倍（适合医疗筛查）；β=0.5 反过来（适合精准营销，宁可少打也要打准）。业务方如果说"我们更看重不漏掉"，你就该报 F2 而不是 F1。

**AUC-ROC 在极不均衡数据下会骗人。** 因为 ROC 曲线的横轴 FPR 在负样本巨多时分母太大，FP 增加几个百分点都看不出来，AUC 就显得特别高。一个具体例子：1:1000 的反欺诈场景，模型把欺诈样本都排到前面意义不大，因为前面还混着大量误报。这种场景应该看 AUC-PR——它的基线是正样本比例，更敏感地反映正样本排序质量。

**Log Loss 衡量的是概率本身的质量。** 如果你的下游业务要拿模型输出概率去做期望值计算（比如出价、风险定价、保费精算），Log Loss 比 AUC 更值得关注，因为它会惩罚"排序对但概率扭曲"的模型。两个模型 AUC 都是 0.85，但一个输出 [0.45, 0.55] 区间，另一个输出 [0.05, 0.95]，做期望值计算时差别巨大。Log Loss 还提示你需要做**概率校准**（calibration），常见方法有 Platt scaling、Isotonic regression，sklearn 里 `CalibratedClassifierCV` 一行搞定。

**类别不均衡的额外指标**：

- **Balanced Accuracy**：(TPR + TNR) / 2，让两类样本权重相等。
- **Matthews Correlation Coefficient (MCC)**：把混淆矩阵全部四个数综合成一个 [-1, 1] 的相关系数，是不均衡场景下被严重低估的指标。公式：MCC = (TP·TN − FP·FN) / √((TP+FP)(TP+FN)(TN+FP)(TN+FN))。任意一个分母项为 0 时 sklearn 会返回 0。
- **Cohen's Kappa**：和"随机猜"对比的提升度，在多分类标注一致性场景里也常用。

阈值的选择是工程实践里最容易被忽略的环节。`predict_proba > 0.5` 是默认值，但极少是最优值。正确做法是：在验证集上扫描阈值，根据业务约束（比如 "Precision >= 0.95"）反向定阈值，并且把阈值连同模型一起版本化保存。线上调参时改阈值往往比改模型更安全也更见效。

```python
from sklearn.metrics import (
    accuracy_score, precision_score, recall_score, f1_score,
    roc_auc_score, average_precision_score, log_loss,
    confusion_matrix, classification_report,
)

# y_true: 真实标签, y_pred: 0/1 预测, y_proba: 正类概率
print(classification_report(y_true, y_pred, digits=4))
print("AUC-ROC :", roc_auc_score(y_true, y_proba))
print("AUC-PR  :", average_precision_score(y_true, y_proba))
print("LogLoss :", log_loss(y_true, y_proba))
print("混淆矩阵:\n", confusion_matrix(y_true, y_pred))
```

多分类时上述指标都需要 `average` 参数。`macro` 是各类无加权平均（少数类被平等对待），`weighted` 是按样本数加权（多数类主导），`micro` 是把所有类的 TP/FP/FN 累加再算。报告的时候三种最好都给出来，不然容易被某一种掩盖问题。

## 6.3 回归指标：MAE、RMSE、R²、MAPE

回归指标相对简单，但选择仍有讲究。

- **MAE**（平均绝对误差）：直观、对离群点不敏感，单位和原始变量一致。
- **MSE / RMSE**：平方放大大误差，对离群点敏感。RMSE 单位和原始变量一致，比 MSE 好读。
- **R²**：模型解释了多少方差，1 是完美，0 是和"永远预测均值"持平，负数说明你不如均值。
- **MAPE**：相对误差，跨尺度可比，但当真实值接近 0 时会爆炸。

```python
from sklearn.metrics import (
    mean_absolute_error, mean_squared_error,
    r2_score, mean_absolute_percentage_error,
)
import numpy as np

mae = mean_absolute_error(y_true, y_pred)
rmse = np.sqrt(mean_squared_error(y_true, y_pred))
r2 = r2_score(y_true, y_pred)
mape = mean_absolute_percentage_error(y_true, y_pred)
```

业务上选哪个：如果错 100 块和错 10 块伤害相当（线性），用 MAE；如果错 100 块的伤害是错 10 块的几十倍（凸），用 RMSE；汇报老板时用 R² 或 MAPE，因为它们无量纲、易理解。

几个少为人知但有用的回归指标：

- **MSLE（Mean Squared Log Error）**：对数空间的 MSE，适合预测目标跨多个数量级的场景（销量、收入、播放量）。它惩罚低估比惩罚高估更重，所以保守预测会被压制。
- **SMAPE（Symmetric MAPE）**：MAPE 的对称版本，分母是真实值和预测值的均值，避免了"真实值近 0 时爆炸"的问题，时间序列预测圈子里常用。
- **Quantile Loss / Pinball Loss**：当你想预测分位数（比如 P90 销量上限）而不是均值时使用，是分位数回归的损失函数。

回归任务里另一个被忽视的事情是**残差分析**。光看一个 RMSE 是不够的，应该把残差分布画出来：是不是接近正态？是不是同方差（在 y_pred 各区间残差方差稳定）？是不是有系统性偏差（残差均值不为零）？如果残差和某个特征仍然强相关，说明模型还有结构没学到，调参不会救你，需要回去做特征工程。

## 6.4 排序指标：NDCG、MAP、MRR

推荐和搜索场景里，排第 1 的错误比排第 50 的错误严重得多，分类指标这时就力不从心。

> **一句话先理解三个名字**（以搜索场景为例）：
>
> - **MRR**：搜十次，问"第一条对的结果排在第几？"取倒数再求平均。只看"第一对"的位置。
> - **MAP**：每个搜索都计算 P@1、P@2、…… 的平均，再对所有搜索求平均。看整张返回列表的好坏。
> - **NDCG**：考虑每个结果的"相关度等级"（不是 0/1，而是"非常相关 / 相关 / 一般 / 无关"）和位置折扣（越靠后权重越低）。最贴近真实搜索/推荐的指标，但公式最复杂——下面会拆。

- **MRR**（Mean Reciprocal Rank）：第一个正确结果排名倒数的平均。只关心"首个正确答案在哪"，适合问答这类只要一个答案的场景。
- **MAP**（Mean Average Precision）：每个位置的 Precision 平均，再对所有 query 平均。综合衡量整个返回列表。
- **NDCG@k**（Normalized Discounted Cumulative Gain）：考虑相关度等级（不是 0/1，而是 0/1/2/3）和位置折扣，是搜索/推荐里最主流的离线指标。
  - DCG@k = Σ_{i=1..k} (2^{rel_i} − 1) / log₂(i + 1)
  - NDCG@k = DCG@k / IDCG@k（IDCG 是把所有 item 按真实相关度降序排出来时的 DCG）
  - sklearn 用更早的 Järvelin-Kekäläinen 形式（gain = rel_i 而不是 2^{rel_i}−1），两种都常见，报告时注明用的是哪种。

```python
from sklearn.metrics import ndcg_score
# y_true: shape (n_queries, n_items), 每个位置的相关度等级
# y_score: shape (n_queries, n_items), 模型给每个 item 的分数
print(ndcg_score(y_true, y_score, k=10))
```

排序指标的一个反复出现的坑：离线 NDCG 涨了 3%，上线后业务指标毫无变化。原因往往是离线评估集的标注不能反映真实用户偏好——比如训练数据里把"被点击"等同于"相关"，但用户点击是有标题党 bias 的。所以排序系统最后还是要 A/B。

排序场景下还有几个常被引用的概念：

- **Hit Rate@k / Recall@k**：top-k 里有没有命中目标。最简单粗暴的指标，但只看 0/1 不看排序顺序。
- **MRR vs NDCG 选哪个**：如果用户只看一个答案（问答机器人），MRR 更贴合；如果用户会看完整列表（搜索结果页），NDCG 更贴合。推荐场景里 NDCG@10 / NDCG@20 几乎是标配。
- **Coverage / Diversity**：排序系统不能只看准不准，还要看推得够不够多元。一个永远只推几个热门 item 的系统短期 NDCG 高，长期会让长尾内容生态萎缩。这类指标通常在业务侧另算。
- **Beyond-accuracy**：新颖性（novelty）、惊喜度（serendipity）、公平性（fairness）。在内容分发、电商推荐里越来越重要，但量化方法尚未统一。

## 6.5 LLM 评估：旧指标的局限与新范式

> **本节与第 24 章的分工**：本节讲"评估方法本身是什么、为什么这样设计、单次评估怎么跑"，是离线视角的方法论。**第 24 章 LLMOps** 把这些方法做成生产闭环——trace 接入、在线异步评分、dataset 版本化、CI 回归门禁、failure → dataset → eval 闭环、多 judge 集成、A/B、告警。两章在 ragas、LLM-as-judge bias、漂移监控上有交集但不重复：本节是"工程师独立跑通一次评估"，24 章是"团队让评估永远在跑"。读完本节如果只关心离线评估，可以暂缓 24 章；要把评估上线就必须读 24。

### 6.5.1 传统指标为什么不够

BLEU、ROUGE、METEOR、BERTScore 都是从机器翻译/摘要时代沿用下来的：

- **BLEU**：n-gram 重叠率。翻译里能用，但生成式问答里"猫坐在垫子上"和"垫子上有只猫"语义相同 BLEU 却很低。
- **ROUGE**：关注召回（参考答案的 n-gram 是否在模型输出里出现）。摘要里好用，开放问答里失效。
- **METEOR**：考虑了词形和同义词，比 BLEU 更鲁棒一点，但仍然是表面匹配。
- **BERTScore**：用 BERT 的上下文嵌入算余弦相似度，比 n-gram 强很多，但有两个问题：一是分数压缩在 0.7~0.95 之间难拉开差距，二是评估不了事实正确性（说错了但说得流畅，分数照样高）。

核心问题是：这些指标都假设存在"黄金参考答案"，而 LLM 任务里参考答案可以有无穷种。一篇好的产品介绍可以从产品功能切入，也可以从用户痛点切入，没有谁是标准答案。

### 6.5.2 现代评估方法

**LLM-as-judge**：让一个更强的 LLM 来打分。给它评分准则（rubric）、待评样本、参考答案（可选），让它输出分数和理由。优点：能评估事实性、语气、风格、有用性这类难量化的维度；缺点：贵、有 bias（偏好长答案、偏好自己家模型）。当前实践认为 LLM-as-judge 与人类专家的一致率约在 80% 左右，0–3 或 0–4 的整数刻度比 0–10 更稳定。

**人工标注**：黄金标准，也是最贵的标准。一般做法是用 LLM-as-judge 做日常监控，每周抽 100~500 条做人工复核，校准 judge 的偏差。

**RAG 专用：Ragas**。Ragas 把 RAG 系统拆成"检索"和"生成"两段，分别给指标：

- **Faithfulness（忠实度）**：回答里的每个声明是否都能从检索到的上下文推出？衡量幻觉。
- **Answer Relevancy（答案相关性）**：回答是否切题？反向构造问题，看和原问题相似度。
- **Context Precision（上下文精确率）**：检索回来的上下文里，有用片段排在前面吗？
- **Context Recall（上下文召回率）**：标准答案里的信息，检索上下文都覆盖到了吗？

```python
# ragas 0.2.x 推荐用法：EvaluationDataset + 类实例 metrics
from ragas import EvaluationDataset, evaluate
from ragas.metrics import (
    Faithfulness, AnswerRelevancy,
    LLMContextPrecisionWithReference, LLMContextRecall,
)
from ragas.llms import LangchainLLMWrapper
from ragas.embeddings import LangchainEmbeddingsWrapper
from langchain_openai import ChatOpenAI, OpenAIEmbeddings

samples = [{
    "user_input":     "What is the capital of France?",
    "response":       "Paris is the capital of France.",
    "retrieved_contexts": ["France is a country in Europe. Its capital is Paris."],
    "reference":      "Paris",
}]
ds = EvaluationDataset.from_list(samples)

evaluator_llm = LangchainLLMWrapper(ChatOpenAI(model="gpt-4o-mini"))
evaluator_emb = LangchainEmbeddingsWrapper(OpenAIEmbeddings())

result = evaluate(
    dataset=ds,
    metrics=[
        Faithfulness(llm=evaluator_llm),
        AnswerRelevancy(llm=evaluator_llm, embeddings=evaluator_emb),
        LLMContextPrecisionWithReference(llm=evaluator_llm),
        LLMContextRecall(llm=evaluator_llm),
    ],
)
print(result.to_pandas())
```

> 注：ragas 0.1.x 老接口（`from ragas.metrics import faithfulness, ...` 这种小写模块常量 + `datasets.Dataset.from_dict({"question":..., "answer":..., "contexts":..., "ground_truth":...})`）在新版本中仍可用但已不被推荐，字段名也改成了 `user_input / response / retrieved_contexts / reference`。新项目直接用上面的写法。

四个指标的工程经验阈值（都在 [0, 1] 区间）：

| 指标 | 通用问答可接受 | 面向用户产品文档 | 金融/医疗/法律 |
|------|-----------------|-------------------|-----------------|
| Faithfulness | ≥ 0.80 | ≥ 0.85 | ≥ 0.90 |
| Answer Relevancy | ≥ 0.75 | ≥ 0.80 | ≥ 0.85 |
| Context Precision | ≥ 0.70 | ≥ 0.80 | ≥ 0.85 |
| Context Recall | ≥ 0.70 | ≥ 0.80 | ≥ 0.85 |

Faithfulness 永远是底线（幻觉直接砸品牌），Context Recall 是检索覆盖率（漏召回会让 Faithfulness 假高——回答里没说错只是因为没敢说）。两者必须配套看。

**LLM-as-judge 写作 prompt 的要点**：

1. 评分维度逐项独立打分，不要一起打总分。让模型分别给"事实正确性""有用性""语气合规"打 0–3 分，最后由你按业务权重加权。
2. 先输出理由（chain-of-thought），再输出分数。这样既能减少草率打分，也方便事后回查。
3. 用整数小刻度（0/1/2/3），减少模糊判断。0–10 刻度听起来精细，实际上 LLM 给 7 还是 8 几乎是随机的。
4. 给典型样例（few-shot），尤其是边界 case。"这种是 2 分""这种是 3 分"的对比示例比抽象描述有效得多。
5. 定期用人工标注做对齐（计算 judge_alignment）。当 judge 和人类分歧 > 20%，说明 prompt 该重写。
6. 警惕几种 bias：位置偏好（pairwise 比较时偏好第一个）、长度偏好（更长的答案天然得分高）、自家模型偏好（GPT judge 偏 GPT 答案，Claude judge 偏 Claude 答案）、版本漂移（同一 judge 升级后历史分数失去可比性）。位置偏好可以通过随机交换顺序消除；长度偏好通过在 prompt 里明确"长度不影响打分"缓解；模型偏好用多 judge 投票或换 judge 缓解；版本漂移要把 judge 锁死到具体快照（如 `gpt-4o-2024-08-06` 而非 `gpt-4o`）。
7. 中文应用尤其要警惕**文化与语言偏见**。英文训练为主的 judge（GPT-4、Claude）评估中文输出时常出现几类系统性偏差：（a）偏好"美式直接"的回答结构（先结论后论据），扣分扣给中文常见的"先铺垫后结论"风格，即便后者业务上更得体；（b）对中文敬语、委婉语判定为"啰嗦"，对英式礼貌客套（"I would be happy to..."）反而打高分；（c）专业术语中英混用时偏好全英文表述（"用 API 调 endpoint"得分高于"用接口调地址"），与目标用户偏好相反；（d）涉及中国本地业务（医保、户籍、节日、法规引用）时事实判定弱，常把"对的中文细节"判为"幻觉"。缓解：评估 rubric 里显式声明语言/风格中立条款（"文化表达差异不影响打分"）；中文场景优先选中文友好的 judge（DeepSeek、Qwen-Max、GLM-4 在中文 rubric 任务上的对齐度通常更高）；calibration set 必须包含中文母语者标注的对照样本，不能只拿英文 rubric 翻译过来用。

**LLM 评估的多层架构**：

成熟团队会用三层：

- **第一层：单元 / 回归测试。** 一组固化的 prompt + 期望响应规则（比如"回答必须包含 X""不允许包含 Y"），CI 跑得起，每次代码改动都跑。
- **第二层：自动评估（含 LLM-as-judge）。** 离线大批量评估，每天跑一次，跟踪关键指标趋势。Ragas、DeepEval、TruLens 都是这一层的工具。
- **第三层：人工评估 + 线上 A/B。** 抽样人工标注，做最终质量校准；线上 A/B 看真实业务指标。

只做第一层不够（覆盖太窄），只做第二层不够（judge 自己有偏），只做第三层太慢太贵。三层互补，不可省略任何一层。

**评估数据集的演化**：

刚开始评估数据集只有几十条手写样本是正常的。重要的是建立**回流机制**：

1. 线上发现的 bad case 进数据集（脱敏后）。
2. 用户主动反馈差评的样本进数据集。
3. 模型版本切换时新老版本分歧的样本，人工裁判后进数据集。

经过半年到一年，评估集会从"工程师拍脑袋写的"演化成"业务实际场景的代表性切片"。这个数据集本身就是公司的资产。

## 6.6 交叉验证：评估方法不对，指标都是空话

划分数据是基本功。"训练 80% / 测试 20%"是最弱的方法，因为单次划分的方差很大。

### 6.6.1 K-fold

把数据切 K 份，每次拿一份做验证，其余 K-1 份训练，循环 K 次。最后报告 K 次得分的均值和标准差。

```python
from sklearn.model_selection import KFold, cross_val_score
import numpy as np

cv = KFold(n_splits=5, shuffle=True, random_state=42)
scores = cross_val_score(model, X, y, cv=cv, scoring="f1_macro")
print(f"F1 = {scores.mean():.4f} ± {scores.std():.4f}")
```

报告时一定要带标准差。两个模型 0.83 vs 0.85 看起来后者赢，如果标准差都是 ±0.03，这个差距不显著。

### 6.6.2 StratifiedKFold

类别不均衡时必须用。普通 K-fold 可能把少数类全分到一个 fold 里。StratifiedKFold 保证每个 fold 的类别比例和整体一致。`cross_val_score` 在分类任务上默认就是 stratified。

### 6.6.3 TimeSeriesSplit

时间序列绝不能用普通 K-fold——会出现"用未来预测过去"的数据泄露。`TimeSeriesSplit` 保证训练集时间永远在验证集之前。

```python
from sklearn.model_selection import TimeSeriesSplit
tscv = TimeSeriesSplit(n_splits=5, gap=24)  # gap: 训练集和验证集之间空 24 个点（避免边界泄露）
for train_idx, val_idx in tscv.split(X):
    ...
```

### 6.6.4 GroupKFold

数据有"组"结构时使用：同一个用户的多条样本、同一台设备的多条记录、同一篇文章的多个句子。如果不按组划分，模型会"记住用户特征"而不是学习泛化模式。

```python
from sklearn.model_selection import GroupKFold
gkf = GroupKFold(n_splits=5)
for train_idx, val_idx in gkf.split(X, y, groups=user_ids):
    ...
```

经验法则：业务里只要你能想到任何"重复来源"，就得问一句要不要 GroupKFold。

### 6.6.5 嵌套交叉验证（Nested CV）

调参和模型评估必须分开，否则你会陷入"在评估集上调参"的陷阱——评估指标乐观偏差，上线后必崩。嵌套 CV 的结构是：

- **外层 CV**（5 折）：用来评估"模型 + 调参流程"的整体泛化能力。
- **内层 CV**（3~5 折）：在外层每个训练折内部，再切一次做调参。

只有最终上线模型才用全数据训练。

```python
from sklearn.model_selection import StratifiedKFold, GridSearchCV
outer = StratifiedKFold(n_splits=5, shuffle=True, random_state=0)
inner = StratifiedKFold(n_splits=3, shuffle=True, random_state=1)
outer_scores = []
for tr, te in outer.split(X, y):
    grid = GridSearchCV(estimator, param_grid, cv=inner, scoring="f1_macro")
    grid.fit(X.iloc[tr], y.iloc[tr])
    outer_scores.append(grid.score(X.iloc[te], y.iloc[te]))
print("Nested CV f1:", np.mean(outer_scores), "±", np.std(outer_scores))
```

嵌套 CV 计算量大，工业实践中很多团队没有耐心做完整版。一个折中方案是：用一次 train/val/test 三段切分，调参在 train+val 上 K 折交叉，最终评估只用 test 一次。

### 6.6.6 数据切分的现实复杂性

教科书 CV 假设样本独立同分布。现实中你常会遇到：

- **时间 + 用户的双重耦合**：电商场景里既有时间维度（早期/晚期），又有用户维度（重度/轻度）。理想做法是用时间切训练集和测试集，再在训练集内部用 GroupKFold by user 做交叉验证。
- **冷启动问题**：评估集里出现了训练集没见过的用户/物品。这才是真实的线上场景。普通 K-fold 评估出的指标会显著高于线上。
- **极不均衡的 group 大小**：某个用户有 10000 条样本，另一个用户只有 5 条，按 group 切会让某些 fold 严重不均衡。可以考虑分层抽样的 group 切分（StratifiedGroupKFold）。

数据切分的最高境界，是模拟未来线上调用模型时所见到的输入分布。任何"训练集和测试集长得太像"的设计，都会让你高估模型能力。

## 6.7 超参调优：从网格搜索到贝叶斯优化

### 6.7.1 Grid / Random Search 的局限

Grid Search 把所有组合穷举一遍。优点是简单可复现，缺点是组合爆炸——3 个超参各 5 个候选就是 125 次实验，10 个超参就是 9 百万次。而且大多数组合是"显然不好"的，浪费算力。

Random Search 在搜索空间里随机采样。Bergstra & Bengio 的经典论文证明：**当只有少数超参重要时，Random Search 比 Grid Search 高效得多**——因为 Grid 在不重要维度上浪费了大量采样，而 Random 把每个采样点的"自由度"全用上了。

但两者都有共同缺陷：**不利用历史信息**。试过 (lr=0.01, depth=5) 效果差，下一次还可能去试 (lr=0.011, depth=5)。

### 6.7.2 贝叶斯优化与 Optuna

贝叶斯优化用代理模型（surrogate model，常用 TPE 或高斯过程）拟合"超参 → 指标"的映射，再用采集函数（acquisition function）选下一个最有信息量的点。简单说就是：**让算法自己学会哪儿值得探索**。

Optuna 是当前 Python 生态最主流的贝叶斯优化框架，特点是：

- **define-by-run**：搜索空间在 objective 函数里动态定义，可以写条件分支（不同模型搜不同的参）。
- **TPE 默认采样器**：在大多数表格场景下表现稳定。
- **内置剪枝（pruner）**：训练中途看到不行就早停，省算力。
- **持久化和并行**：用 SQLite/PostgreSQL 做 storage，多机多进程同时跑同一个 study。

```python
import optuna
import lightgbm as lgb
from sklearn.model_selection import StratifiedKFold
from sklearn.metrics import roc_auc_score

def objective(trial):
    params = {
        "objective": "binary",
        "metric": "auc",
        "verbosity": -1,
        "learning_rate": trial.suggest_float("lr", 1e-3, 0.3, log=True),
        "num_leaves": trial.suggest_int("num_leaves", 15, 255),
        "max_depth": trial.suggest_int("max_depth", 3, 12),
        "min_child_samples": trial.suggest_int("min_child_samples", 5, 200),
        "reg_alpha": trial.suggest_float("reg_alpha", 1e-8, 10.0, log=True),
        "reg_lambda": trial.suggest_float("reg_lambda", 1e-8, 10.0, log=True),
        "feature_fraction": trial.suggest_float("feature_fraction", 0.5, 1.0),
        "bagging_fraction": trial.suggest_float("bagging_fraction", 0.5, 1.0),
    }
    cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
    scores = []
    for fold, (tr, va) in enumerate(cv.split(X, y)):
        dtr = lgb.Dataset(X.iloc[tr], y.iloc[tr])
        dva = lgb.Dataset(X.iloc[va], y.iloc[va])
        booster = lgb.train(
            params, dtr, num_boost_round=2000,
            valid_sets=[dva],
            callbacks=[lgb.early_stopping(50, verbose=False)],
        )
        pred = booster.predict(X.iloc[va])
        score = roc_auc_score(y.iloc[va], pred)
        # 中间汇报，让 pruner 决定是否提前结束
        trial.report(score, fold)
        if trial.should_prune():
            raise optuna.TrialPruned()
        scores.append(score)
    return sum(scores) / len(scores)

study = optuna.create_study(
    direction="maximize",
    sampler=optuna.samplers.TPESampler(seed=42),
    pruner=optuna.pruners.MedianPruner(n_startup_trials=10, n_warmup_steps=2),
    study_name="lgbm_tuning",
    storage="sqlite:///optuna.db",
    load_if_exists=True,
)
study.optimize(objective, n_trials=100, show_progress_bar=True)
print("Best AUC:", study.best_value)
print("Best params:", study.best_params)
```

### 6.7.3 Hyperband / ASHA / BOHB

资源高效调度的几种方法：

- **Hyperband**：一种 multi-armed bandit 思路。给所有候选少量预算，淘汰差的，给剩下的更多预算。Optuna 的 `HyperbandPruner` 就是它。
- **ASHA**（Asynchronous Successive Halving）：Hyperband 的异步版，更适合分布式。Ray Tune 默认调度器。
- **BOHB**（Bayesian Optimization + Hyperband）：贝叶斯采样 + Hyperband 调度的结合，是当前学术界 baseline 之一。

工程选型经验：

- 单机、表格数据、超参不到 20 个：Optuna + TPE + MedianPruner。
- 多机、深度学习、需要分布式调度：Ray Tune + ASHA，可选用 Optuna 作为 sampler。Ray Tune 把训练函数自动并行化、统一资源管理，大规模实验更省心；Optuna 的 TPE 在小到中等规模搜索空间里搜索质量更优。两者可以组合：用 Ray Tune 做调度，把 OptunaSearch 当 sampler 注入。
- 训练一次几小时以上：必须配 pruner，否则烧不起。
- 搜索空间高度结构化（条件嵌套）：Optuna 的 define-by-run 优势明显，能在 objective 函数里写 `if model_type == "xgb": ...` 然后才去 suggest XGB 特有参数。
- 多目标优化（如准确率和延迟同时优化）：Optuna 用 `directions=["maximize", "minimize"]`，会自动跑 NSGA-II 族算法，输出 Pareto 前沿。

**调参的反直觉真相**：

1. **大部分超参不重要。** 一个 LightGBM 通常 5~7 个超参就能覆盖 95% 的提升空间，剩下的微调收益微乎其微。先识别哪几个最重要，再集中算力调它们。
2. **学习率几乎永远是最重要的。** 不论模型类型，第一个该调的就是 learning rate（或等价物）。
3. **正则强度和模型复杂度要联调。** 单调一个 max_depth 没意义，得连 min_child_samples 一起搜。
4. **早停是免费的午餐。** 任何能早停的算法都该开早停，而且 early_stopping_rounds 应当随训练规模动态调整（小数据 30~50，大数据 100~200）。
5. **不要把测试集用于调参。** Optuna study 的 best_value 是 CV 得分，不是测试集得分。最终上线模型用最佳超参在 train+val 上重训一次，测试集只在最后报告时跑一次。

### 6.7.4 Random Search vs Optuna 收敛对比

下面这段代码可以把两种方法画在同一张图上对比。在大多数光滑/弱噪声的目标函数上，Optuna 收敛通常显著快于 Random Search，但在搜索空间几乎无结构（纯随机噪声）的极端情形下，两者差距会缩小：

```python
import optuna
import numpy as np
import matplotlib.pyplot as plt

def objective_func(params):
    # 假设这是一个昂贵的真实评估，这里用合成函数代替
    x, y = params["x"], params["y"]
    return (x - 0.3) ** 2 + (y + 0.7) ** 2 + 0.1 * np.sin(10 * x)

# Random Search
rng = np.random.default_rng(42)
random_history = []
best = float("inf")
for _ in range(100):
    p = {"x": rng.uniform(-1, 1), "y": rng.uniform(-1, 1)}
    v = objective_func(p)
    best = min(best, v)
    random_history.append(best)

# Optuna (TPE)
def optuna_obj(trial):
    p = {
        "x": trial.suggest_float("x", -1, 1),
        "y": trial.suggest_float("y", -1, 1),
    }
    return objective_func(p)

study = optuna.create_study(direction="minimize",
                            sampler=optuna.samplers.TPESampler(seed=42))
study.optimize(optuna_obj, n_trials=100, show_progress_bar=False)
optuna_history = np.minimum.accumulate([t.value for t in study.trials])

plt.plot(random_history, label="Random Search")
plt.plot(optuna_history, label="Optuna (TPE)")
plt.xlabel("Trials"); plt.ylabel("Best value so far")
plt.legend(); plt.grid(alpha=0.3)
plt.savefig("convergence.png", dpi=120)
```

## 6.8 过拟合诊断：学习曲线与验证曲线

调参之前先看症状。两条曲线工具人手必备。

**学习曲线**：横轴是训练样本量，纵轴是训练/验证得分。

- 训练分高，验证分低，且差距大 → 过拟合（高方差）。对策：加正则、加数据、降模型复杂度。
- 两条曲线都低，且接近 → 欠拟合（高偏差）。对策：换更强的模型、加特征。
- 验证分还在涨 → 数据加多还能涨，值得标更多数据。

```python
from sklearn.model_selection import learning_curve
import numpy as np, matplotlib.pyplot as plt

train_sizes, train_scores, val_scores = learning_curve(
    model, X, y, cv=5, scoring="f1_macro",
    train_sizes=np.linspace(0.1, 1.0, 8),
    n_jobs=-1, random_state=42,
)
plt.plot(train_sizes, train_scores.mean(axis=1), label="Train")
plt.plot(train_sizes, val_scores.mean(axis=1), label="Validation")
plt.fill_between(train_sizes,
                 val_scores.mean(axis=1) - val_scores.std(axis=1),
                 val_scores.mean(axis=1) + val_scores.std(axis=1),
                 alpha=0.2)
plt.xlabel("Training samples"); plt.ylabel("F1")
plt.legend(); plt.grid(alpha=0.3)
```

**验证曲线**：横轴是某一个超参（比如 max_depth），其余固定。看模型在这个超参下的偏差-方差权衡曲线。配合贝叶斯优化前用一下，能帮你定大致的搜索范围。

**Bias-Variance Tradeoff 速记**：

- Bias 大：模型学不动业务，无论多少数据都救不了——换模型、加特征。
- Variance 大：模型学到了噪声，更多数据 + 更强正则可以救。
- 现代深度学习里有"双下降（double descent）"现象，超大模型 + 足量数据可以让 variance 在过拟合区之后又降下来，但这不是 AI 应用工程师的日常情景，知道有这事就行。

**过拟合的微妙之处**：

很多新手看到训练分高、验证分低就喊过拟合，其实需要再分一层：

- **真过拟合**：训练集本身就是干净的、有代表性的，模型只是记住了训练样本。对策：正则、Dropout、数据增强、早停。
- **训练-验证分布不一致**：其实没过拟合，是验证集和训练集分布就不同（比如时间漂移、采样偏差）。这种情况下加正则没用，根本问题是数据切分错了。
- **标签噪声**：训练集标签有大量错的，模型记住了噪声。对策：清洗数据、用对噪声鲁棒的损失（label smoothing、focal loss）。
- **特征泄露（leakage）**：训练时见到了不该见到的信息。这不是过拟合，是 bug。学习曲线会显示训练分异常高，验证分异常高，上线后崩。

诊断顺序：先排除 leakage，再确认数据切分合理，最后才上正则化。顺序错了会浪费大量时间。

## 6.9 可解释性：从全局到局部

### 6.9.1 全局可解释性

**`feature_importances_`**：树模型自带的属性。算法是"分裂带来的不纯度下降之和"或"split count"。简单快，但有偏：高基数类别特征会被高估。

**Permutation Importance**：把某一列打乱，看模型表现下降多少。模型无关、不偏，但慢、相关特征间会有"信用分配"问题。

```python
from sklearn.inspection import permutation_importance
result = permutation_importance(
    model, X_val, y_val,
    n_repeats=10, random_state=42, scoring="roc_auc", n_jobs=-1,
)
sorted_idx = result.importances_mean.argsort()[::-1]
for i in sorted_idx[:10]:
    print(f"{X_val.columns[i]:30s} {result.importances_mean[i]:.4f} ± {result.importances_std[i]:.4f}")
```

### 6.9.2 局部可解释性：SHAP

SHAP 基于博弈论中的 Shapley 值：把每个特征看作合作博弈里的玩家，模型输出是合作收益，SHAP 值就是每个玩家的公平分配。它满足三条公理（local accuracy、missingness、consistency），是目前理论最严谨的特征归因方法。

> **一句"人话"理解 SHAP**：模型对某条样本预测出了一个分数（比如违约概率 0.72），SHAP 告诉你这个 0.72 是怎么"凑"出来的——"基线 0.30 + 年龄贡献 +0.15 + 收入贡献 -0.05 + 负债比 +0.32 …… = 0.72"。每个特征的贡献值就是它的 SHAP 值。正数把预测拉高，负数把预测压低。读 SHAP 图本质就是看"哪几个特征把这条预测推到了现在的位置"。

实战上有几种 explainer：

- `TreeExplainer`：树模型专用，精确算 Shapley 值，速度快。XGBoost、LightGBM、CatBoost、sklearn 树模型都支持。
- `DeepExplainer` / `GradientExplainer`：神经网络。
- `KernelExplainer`：模型无关，但慢，慎用大数据集。
- `Explainer`（统一入口）：会自动选择最合适的后端。

```python
import shap

explainer = shap.Explainer(model, X_train)  # 自动选 TreeExplainer
shap_values = explainer(X_val)

# 全局：平均绝对值条形图
shap.plots.bar(shap_values, max_display=15)

# 全局：beeswarm（每个点是一个样本，看分布和方向）
shap.plots.beeswarm(shap_values, max_display=15)

# 局部：单条预测的瀑布图
shap.plots.waterfall(shap_values[0], max_display=15)

# 依赖图：某个特征的 SHAP 值随特征值的变化（颜色编码交互项）
shap.plots.scatter(shap_values[:, "feature_x"], color=shap_values)
```

读 SHAP 图的几个关键点：

- 横轴是 SHAP 值（对预测的推动力，正向上抬概率，负向下拉）。
- beeswarm 上颜色是特征值（红高蓝低）。如果某特征红点都集中在右侧，说明"特征值越大 → 预测概率越高"。
- waterfall 从基线（base value，整体均值）出发，每个特征加加减减堆到最终预测。这是给业务方解释单条决策最好的图。

### 6.9.3 LIME

LIME 的思路是局部线性近似：在你要解释的样本附近做扰动，用线性模型拟合扰动样本的输出，把线性模型的系数当解释。

优势：模型完全无关、思路直观、单点速度比 KernelSHAP 快。
劣势：相同样本两次解释结果可能不同（采样随机性）；不像 SHAP 那样严谨；只能做局部不能做全局。

```python
import lime.lime_tabular
explainer = lime.lime_tabular.LimeTabularExplainer(
    X_train.values,
    feature_names=X_train.columns.tolist(),
    class_names=["neg", "pos"],
    mode="classification",
)
exp = explainer.explain_instance(
    X_val.iloc[0].values,
    model.predict_proba,
    num_features=10,
)
exp.show_in_notebook()
```

实战选型：表格 + 树模型 → SHAP 优先。需要给非技术同事看单条解释，又怕 SHAP 数学解释成本高 → LIME 的简单线性图也是选项。深度学习模型 → SHAP 的 Deep/Gradient explainer 或 Integrated Gradients。

**SHAP 的几个高阶用法**：

- **依赖图找非线性**。`shap.plots.scatter` 可以揭示一个特征的 SHAP 值不是简单单调，可能呈 U 型或阶梯型，这种发现往往直接指导特征工程（比如该特征要分桶、要交叉）。
- **交互项**。`shap.TreeExplainer.shap_interaction_values` 可以算二阶交互，能识别出"特征 A 的影响依赖于特征 B 的取值"这种关系，是树模型挖掘交互特征的利器。
- **聚类比较**。`shap.plots.bar(cohorts={"高收入": ..., "低收入": ...})` 把不同人群分开看 SHAP 重要性排序，能发现"对高收入用户来说房龄重要，对低收入用户来说位置重要"这种业务洞察。
- **特征选择**。SHAP 平均绝对值排序后，只保留 top-N 特征重训模型，往往能在精度几乎不掉的前提下大幅降低线上推理成本。比基于 Pearson 相关或 mutual_info 的特征选择更可靠。

**SHAP 的常见误用警告**：

1. **SHAP 解释的是"模型怎么决策"，不是"世界怎么运作"。** SHAP 重要的特征不一定有因果关系。如果训练数据里"用户名包含数字"和"违约率"相关，SHAP 会把它高亮，但你显然不能基于此做业务决策。因果推断是另一个领域。
2. **大数据集要采样**。在 100 万条样本上算 SHAP 太慢，常见做法是用 KMeans 聚成 100 个代表点，或随机采样 1000~10000 条来算全局重要性，单条解释才精确算。
3. **TreeSHAP 在多树模型上是精确的，KernelSHAP 是近似的。** 报告 SHAP 值时要注明用的哪种 explainer，否则数字含义不一样。
4. **SHAP 不擅长解释类别变量过多的特征。** one-hot 后的特征会把"一个语义概念"的重要性分散到多个 SHAP 值上，需要事后聚合。CatBoost 这类直接处理类别的模型会缓解这个问题。

### 6.9.4 反事实解释（Counterfactual）

"如果用户的月收入再多 2000 元就能放贷了"——这种解释最贴近业务用户的直觉。开源工具有 DiCE（Diverse Counterfactual Explanations）。反事实解释也是 GDPR 等监管语境下"算法决策可质疑性"的主要技术手段之一。

单棵决策树或 GBDT 里的某一棵：

```python

import matplotlib.pyplot as plt
plt.figure(figsize=(20, 10))

          class_names=["0", "1"], filled=True, rounded=True, max_depth=3)
```

## 6.10 模型比较与显著性检验

跑了两个模型，A 的 F1 是 0.821，B 是 0.835。能不能说 B 更好？取决于差距是不是显著。

### 6.10.1 McNemar 检验

适用场景：两个分类器，同一个测试集，比较错误模式是否一致。

构造 2x2 列联表：

|  | B 对 | B 错 |
|---|---|---|
| A 对 | n00 | n01 |
| A 错 | n10 | n11 |

只关心不一致的格子（n01, n10）。统计量：(|n01 - n10| - 1)² / (n01 + n10) 服从 χ²(1)。

```python
from statsmodels.stats.contingency_tables import mcnemar
import numpy as np

a_correct = (pred_a == y_test)
b_correct = (pred_b == y_test)
table = np.array([
    [(a_correct & b_correct).sum(),   (a_correct & ~b_correct).sum()],
    [(~a_correct & b_correct).sum(), (~a_correct & ~b_correct).sum()],
])
result = mcnemar(table, exact=False, correction=True)
print(f"chi2 = {result.statistic:.4f}, p = {result.pvalue:.4f}")
```

p < 0.05 通常认为差异显著。McNemar 的优势是不需要重复实验，单测试集就能比较。

### 6.10.2 配对 t 检验 / Wilcoxon

如果你有 K 折交叉验证的得分序列，可以做配对 t 检验（参数）或 Wilcoxon signed-rank（非参）来比较两个模型。注意 K 折之间不独立，严格意义上 t 检验的假设不完全成立，业内通常用 Nadeau-Bengio 修正或者直接用更保守的 Wilcoxon。

### 6.10.3 A/B 测试基础

线下显著不等于线上显著。A/B 测试时几个工程师常踩的坑：

1. **样本量算错**。要根据基线指标、想检测的最小 effect size、显著性水平 α 和功效 1-β，提前算样本量。一个常见错误是边跑边看，发现 p<0.05 就停止，这叫 peeking，会显著放大假阳性率（接近 30%~40%）。正确做法是预先定好样本量再跑完。
2. **多重检验**。同时看十个指标里有一个 p<0.05，可能只是偶然。Bonferroni 或 BH 校正一下。或者预先定好"主指标"，副指标只做参考。
3. **新颖性效应**。新功能上线后用户因为新鲜而点击率上涨，几周后回落。短期 A/B 容易高估收益。一般至少跑 1~2 周覆盖一个完整工作周期。
4. **辛普森悖论**。整体看 B 赢，分人群看 A 全赢——分层观察永远是必要的。常见原因是新老用户、不同设备、不同地域的分布在 A/B 组中不一致。
5. **网络效应（spillover）**。社交、电商场景里 A 组用户可能影响 B 组（信息扩散、共享购物车），实验组之间不独立。需要 cluster-level 实验设计或精心设计的隔离机制。
6. **新陈代谢效应**。模型上线初期表现好，但因为它的输出会影响后续训练数据（曝光偏差、自我强化），长期可能塌缩。这种问题离线评估和短期 A/B 都看不出来，需要长期观察 + 反事实分析。

**离线-线上一致性诊断**：

如果离线指标提升 5%，线上 A/B 没提升甚至下降，依次排查：

1. 评估集是否代表真实流量分布。
2. 离线指标（比如 NDCG）是否和业务指标（CTR、GMV）有相关性历史证据。
3. 是否有 selection bias：训练数据来自当前模型推荐的内容，新模型推荐的内容用户可能根本没见过类似的，转化率不可比。
4. 是否触发了延迟、稳定性、成本等线上独有的约束。

A/B 测试是评估的最终裁判，但它不是免费的——曝光给用户的代价、对其他业务的潜在影响、组织协调成本都不低。所以离线评估要尽可能可靠，A/B 才能聚焦在真正值得验证的变化上。

## 6.11 数据漂移与概念漂移

模型上线后世界不会等你。两个核心概念：

- **数据漂移（covariate shift）**：P(X) 变了。比如疫情期间用户消费品类突然变了，但"高消费用户=低风险"这件事还成立。
- **概念漂移（concept drift）**：P(y | X) 变了。同样的特征，标签的含义变了。比如反欺诈里黑产换了打法。

数据漂移不一定让模型失效，但通常是概念漂移的早期信号。

### 6.11.1 检测方法

数值特征：KS（Kolmogorov-Smirnov）检验、PSI（Population Stability Index）、Wasserstein 距离。
类别特征：Chi-square、JS 散度。
全特征联合分布：用一个分类器去区分"训练集"和"线上数据"，AUC 显著高于 0.5 说明分布有差异。

```python
# 简易 PSI 实现
import numpy as np
def psi(expected, actual, buckets=10):
    breakpoints = np.percentile(expected, np.linspace(0, 100, buckets + 1))
    breakpoints[0], breakpoints[-1] = -np.inf, np.inf
    e_pct = np.histogram(expected, breakpoints)[0] / len(expected)
    a_pct = np.histogram(actual, breakpoints)[0] / len(actual)
    e_pct = np.where(e_pct == 0, 1e-6, e_pct)
    a_pct = np.where(a_pct == 0, 1e-6, a_pct)
    return float(np.sum((a_pct - e_pct) * np.log(a_pct / e_pct)))

# 经验阈值: <0.1 稳定, 0.1~0.25 轻微漂移, >0.25 显著漂移
```

### 6.11.2 工具：Evidently

Evidently 是当前开源 ML 监控的事实标准。它内置 20 多种漂移检测算法，能生成交互式 HTML 报告，也支持持续监控。

```python
# Evidently 0.6 / 0.7 当前推荐写法
from evidently import Report
from evidently.presets import DataDriftPreset, ClassificationPreset

report = Report([
    DataDriftPreset(),
    ClassificationPreset(),
])
# 第一个参数是当前数据（current），第二个是参考数据（reference）
snapshot = report.run(current_data=prod_df, reference_data=ref_df)
snapshot.save_html("drift_report.html")
# 也可以导出 JSON / dict
# snapshot.save_json("drift_report.json")
```

> 注：Evidently 的 API 在 0.4.x → 0.6.x → 0.7.x 经过两轮重构。0.4 之前的 `Report(metrics=[...])` + `Report.save_html()` 老写法已经不工作；0.6+ 的统一入口是 `Report([...]).run(current, reference)` 返回 snapshot，再由 snapshot 负责持久化。遇到 ImportError 先确认本地版本。

### 6.11.3 监控告警的工程实践

1. **三层告警**：数据漂移（最早期）、预测漂移（中期）、性能下降（最晚但最确定）。三层之间通常有时间差，数据漂移领先性能下降数天到数周不等，发现得越早，应对越从容。
2. **基线选择**：滚动基线（过去 N 天）vs 固定基线（训练集）。前者跟得上慢慢的业务演化，后者能发现长期偏移。两者都要建议同时监控，能形成互补视角。
3. **告警阈值动态化**。固定 PSI=0.25 在不同业务上意义不同，需要先跑几个月历史数据校准。理想做法是基于历史数据的 PSI 分布，取 P95 或 P99 当告警阈值，并允许人工调整。
4. **触发动作**。漂移触发不一定要重训。可以：先加权重训、加入新数据 fine-tune、回退到上一版本、人工介入。重训未必稳赢，新数据可能携带新噪声，盲目重训反而引入新问题。
5. **闭环**。监控发现的漂移样本应该回流到训练集（注意不要污染评估集）。
6. **可解释性 + 监控**。漂移告警别只报"PSI=0.31"，最好附带"是哪几个特征漂移最严重，对应的 SHAP 重要性是多少"。如果漂移特征不重要，可以暂缓处理；如果漂移特征是 SHAP top-3，必须立刻应对。

### 6.11.4 LLM 应用的漂移监控

LLM 应用的"漂移"和传统 ML 不一样：

- **输入分布漂移**：用户问题主题在变。可以监控用户 query 的 embedding 分布。
- **基础模型漂移**：调用的是 OpenAI/Anthropic API，他们悄悄升级模型版本，behavior 可能变。对策：固定 model snapshot（如 `gpt-4-2024-08-06`）、跑回归测试集检测变化。
- **检索质量漂移**：RAG 系统里知识库新增/删除文档，命中分布会变。监控召回率和 Ragas context_precision。
- **用户满意度漂移**：人工反馈、点踩率、对话轮数都是 proxy 指标，长期跟踪。

LLM 监控比传统 ML 更需要日志原始 prompt 和 response，否则事后无从分析。注意脱敏与合规。

**和传统漂移检测的方法论桥接**：6.11.1 讲的 PSI / KS / Wasserstein 不是只能用在数值特征上。在 LLM 场景里，把"用户 query 的 embedding 向量"看作一个高维特征，就能套用同一套思路：

- **embedding 维度 PSI**：取 query embedding 的每一维（或 PCA / UMAP 降维后的前几维），按 6.11.1 的分桶 PSI 计算，单维 PSI > 0.25 提示分布偏移。这比单纯统计 query 数量更早期。
- **MMD（Maximum Mean Discrepancy）**：embedding 空间的分布距离指标，比逐维 PSI 更敏感，能捕捉到"每一维分布看着差不多但联合分布变了"的情况。`alibi-detect` 库直接提供。
- **judge score 的 KS 检验**：把 LLM-as-judge 给出的分数序列当作分布，参考期 vs 当前期做 KS 检验，p < 0.05 提示评分分布偏移。这个偏移可能是模型变好/变坏，也可能是 judge 本身漂了，需要交叉对比 calibration set。
- **PSI 的告警阈值经验**：embedding 维度 PSI > 0.1 是注意区，> 0.25 是告警区，和 6.11.1 表格的阈值一致。重要的是基线的选取——LLM 应用的输入分布在产品早期变化极快，固定基线一周就过时，建议用滚动 30 天基线 + 训练 / 评估集快照基线两份并跑。

第 24 章会把这些 LLM 监控信号嵌入 trace + dashboard 的工程实现，本节先把方法论从传统 ML 桥到 LLM。**漂移检测的原理是不变的——分布对比、距离度量、显著性检验——变的只是输入空间从结构化特征换成了 token 或 embedding。**

## 6.12 端到端实战：Optuna 调参 + SHAP 解释 + 漂移检测

把这一章学的东西串起来。用 sklearn 自带的乳腺癌数据集（演示用，业务里换成你自己的）：

```python
import numpy as np
import pandas as pd
import optuna
import shap
import lightgbm as lgb
from sklearn.datasets import load_breast_cancer
from sklearn.model_selection import train_test_split, StratifiedKFold
from sklearn.metrics import roc_auc_score, classification_report

# ===== 1. 数据 =====
data = load_breast_cancer()
X = pd.DataFrame(data.data, columns=data.feature_names)
y = pd.Series(data.target, name="label")
X_tr, X_te, y_tr, y_te = train_test_split(
    X, y, test_size=0.25, stratify=y, random_state=42)

# ===== 2. Optuna 调参 =====
def objective(trial):
    params = {
        "objective": "binary", "metric": "auc", "verbosity": -1,
        "learning_rate": trial.suggest_float("lr", 1e-3, 0.3, log=True),
        "num_leaves": trial.suggest_int("num_leaves", 15, 127),
        "max_depth": trial.suggest_int("max_depth", 3, 10),
        "min_child_samples": trial.suggest_int("min_child_samples", 5, 100),
        "reg_alpha": trial.suggest_float("reg_alpha", 1e-8, 10, log=True),
        "reg_lambda": trial.suggest_float("reg_lambda", 1e-8, 10, log=True),
        "feature_fraction": trial.suggest_float("ff", 0.6, 1.0),
        "bagging_fraction": trial.suggest_float("bf", 0.6, 1.0),
        "bagging_freq": trial.suggest_int("bag_freq", 0, 5),
    }
    cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
    aucs = []
    for fold, (tr, va) in enumerate(cv.split(X_tr, y_tr)):
        dtr = lgb.Dataset(X_tr.iloc[tr], y_tr.iloc[tr])
        dva = lgb.Dataset(X_tr.iloc[va], y_tr.iloc[va])
        model = lgb.train(params, dtr, num_boost_round=2000,
                          valid_sets=[dva],
                          callbacks=[lgb.early_stopping(50, verbose=False)])
        pred = model.predict(X_tr.iloc[va])
        score = roc_auc_score(y_tr.iloc[va], pred)
        trial.report(score, fold)
        if trial.should_prune():
            raise optuna.TrialPruned()
        aucs.append(score)
    return float(np.mean(aucs))

study = optuna.create_study(
    direction="maximize",
    sampler=optuna.samplers.TPESampler(seed=42),
    pruner=optuna.pruners.MedianPruner(n_startup_trials=10, n_warmup_steps=2),
)
study.optimize(objective, n_trials=80, show_progress_bar=True)
print(f"Best AUC (CV) = {study.best_value:.4f}")
print("Best params  :", study.best_params)

# ===== 3. 用最优参在全训练集上训练终模型 =====
best_params = {**study.best_params,
               "objective": "binary", "metric": "auc", "verbosity": -1}
final_model = lgb.train(
    best_params,
    lgb.Dataset(X_tr, y_tr),
    num_boost_round=500,
)
y_proba = final_model.predict(X_te)
y_pred = (y_proba > 0.5).astype(int)
print(f"Test AUC = {roc_auc_score(y_te, y_proba):.4f}")
print(classification_report(y_te, y_pred, digits=4))

# ===== 4. SHAP 解释 =====
explainer = shap.TreeExplainer(final_model)
shap_values = explainer(X_te)

# 全局：哪些特征最重要
shap.plots.bar(shap_values, max_display=10, show=False)
# 全局分布：每个特征的影响方向
shap.plots.beeswarm(shap_values, max_display=10, show=False)
# 局部：随便挑一条样本解释
idx = 0
shap.plots.waterfall(shap_values[idx], max_display=10, show=False)
print(f"样本 {idx} 真实标签={y_te.iloc[idx]}, 模型预测概率={y_proba[idx]:.3f}")

# ===== 5. 漂移检测 =====
# 模拟"线上数据"：制造一个 covariate shift（把某些特征加噪）
rng = np.random.default_rng(0)
X_prod = X_te.copy()
X_prod["mean radius"] *= 1.15  # 整体偏移 15%
X_prod["mean texture"] += rng.normal(2.0, 0.5, len(X_prod))

def psi(expected, actual, buckets=10):
    bp = np.percentile(expected, np.linspace(0, 100, buckets + 1))
    bp[0], bp[-1] = -np.inf, np.inf
    e = np.histogram(expected, bp)[0] / len(expected)
    a = np.histogram(actual, bp)[0] / len(actual)
    e = np.where(e == 0, 1e-6, e); a = np.where(a == 0, 1e-6, a)
    return float(np.sum((a - e) * np.log(a / e)))

drift_report = []
for col in X.columns:
    psi_val = psi(X_tr[col].values, X_prod[col].values)
    flag = "OK" if psi_val < 0.1 else ("WARN" if psi_val < 0.25 else "DRIFT")
    drift_report.append((col, psi_val, flag))
drift_df = pd.DataFrame(drift_report, columns=["feature", "psi", "flag"])
print(drift_df.sort_values("psi", ascending=False).head(10))
```

把这套流程跑通，你就有了 AI 应用工程师的标准评估闭环：**调参得到模型 → 解释模型在做什么 → 监控线上数据是否还在原分布**。任何一个环节缺失都会让"模型上线"变成"模型上线即裸奔"。

## 6.13 评估指标作弊：业务团队和工程团队的"指标内卷"

写完上面的端到端流程，必须把一个不太体面但极其常见的现实摆出来：**评估指标本身会被博弈**。当 KPI 和奖金、晋升、续约挂钩之后，调整指标定义比改进模型容易得多。这不是道德问题，是组织问题，但工程师若假装看不见，就会在某次复盘里发现自己签字背书的"AUC 提升 5%"实际上是被人换了分母。

常见手法分四类：

1. **悄悄换评估子集**。"上线前两周指标涨了 8%"——细究之下评估集做了重采样：去掉了几个公认困难的 cohort（新用户、低消费、特殊地域），剩下的样本天然简单。子集换了，老指标看着还在涨。对策：评估集每次变更都必须 PR + diff，PR 描述里写清楚加/删了哪些 cohort，分数报告同时给"原集合"和"新集合"两份。
2. **改阈值不改模型**。`predict_proba > 0.5` 改成 `> 0.4`，Recall 立刻涨 5 个点，Precision 当然掉，但报告时只挑 Recall 给老板。对策：报告强制要求 Precision/Recall 一对成对出现，PR 曲线随报告附上；阈值变更必须有独立 PR、独立 owner。
3. **指标定义偷偷漂移**。"用户满意度"原本是"thumbs up rate"，季度末变成"thumbs up + 复制 + 停留 30 秒任一项"，分母不变分子膨胀。对策：核心指标的定义要写在公司 wiki 上，版本化，任何修改走 review；评估报告里 metric 名字带定义版本号（如 `satisfaction_v3`）。
4. **用 LLM-as-judge 的 rubric 调来调去**。Rubric 里加一句"风格符合品牌调性"，所有回答得分集体 +0.1。对策：rubric 也是版本化资产；rubric 改动必须重跑历史 baseline 才能宣称"提升"。

更隐蔽的一类是**"以告警为目标的拟合"**：监控大盘上某个指标变红了，团队不去查根因，而是把告警阈值调宽，或者把"问题样本"从计算口径里排除。这种行为在 KPI 压力大的团队里发生频率惊人，且通常没有恶意——大家觉得"反正没人投诉"。直到几个月后真用户开始流失，才发现问题早就有信号，只是被一次次"优化"掉了。

工程上能做的几件事：

- **评估集的 git 化**。所有评估集进 git（或 dataset registry，如 Langfuse / Phoenix），每次 PR 都能 diff。
- **指标定义独立 PR**。指标定义、阈值、rubric 三者改动不能和模型改动塞进同一个 PR。混在一起评审者根本分不清"提升"是来自哪儿。
- **双盲对照报告**。版本对比报告强制要求"老指标在老评估集"、"老指标在新评估集"、"新指标在新评估集"三档都给出来。哪一项缺失，直接打回。
- **审计留痕**。每次 release 把当时的评估集快照、指标版本、rubric 版本一起存档，至少留 2 个版本以上，回查根因要用。

评估的可信度最终是组织文化问题，但工程纪律可以把作弊成本提高到大多数人不愿意付。**指标变得太好看的时候，先怀疑指标自己变了，再怀疑模型真变了。**

## 6.14 章末小结：评估的工程纪律

1. **不报告标准差的指标都是耍流氓。** 一个 0.85 ± 0.05 和 0.85 ± 0.005 是两回事，差距不显著的"提升"很可能是噪声。
2. **指标要和业务指标挂钩，不能只看技术指标。** 离线 AUC 涨 1% 上线 GMV 涨多少？没量化过就别敢说"模型变好了"。每个项目开始时都应该做一次"指标 → 业务"的映射会议，把双方期望对齐。
3. **类别不均衡场景慎用 Accuracy 和 AUC-ROC**，优先 AUC-PR、Recall@k、F-beta、MCC。哪个指标和你的业务损益最贴合，哪个就是主指标。
4. **生成式任务别只盯 BLEU/ROUGE**。配 LLM-as-judge + 人工抽样 + 业务指标三件套，单一指标都会骗人。
5. **超参调优永远要在 CV 上做，不要在测试集上做。** 测试集只在最后用一次。Optuna 报告的是 CV 得分，不是测试得分。
6. **可解释性不是合规附加题，是工程师自己的诊断工具。** SHAP 图先给自己看，再给老板看。如果 SHAP 显示 top 特征里有"用户 ID""时间戳"这种诡异东西，多半是数据泄露或特征工程错误。
7. **上线不是终点，监控才是。** 三层告警（数据漂移→预测漂移→性能下降）、回流闭环、再训练触发，缺一不可。
8. **统计显著性是底线纪律。** 两个模型差异要做 McNemar 或配对检验，A/B 实验要算样本量、防 peeking、做多重比较校正。否则跑出来的数字只是"看起来在工作"。
9. **离线-线上一致性需要持续校准。** 离线赢线上输是常态，要建立离线指标和业务指标的相关性档案，离线评估改版时也回测一遍这个相关性是否仍然成立。
10. **建立评估资产，而不是一次性脚本。** 评估数据集、漂移监控基线、SHAP 解释模板、A/B 实验流程，这些应当沉淀成可复用资产，而不是每个项目从零开始造一遍。
