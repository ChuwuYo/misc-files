# 第 22 章 · 性能优化与成本控制

> 目标：把同一个应用的月账单从 $50K 砍到 $5K，且不让用户察觉变化。
>
> 本章面向已经把应用跑起来、开始为账单和延迟头疼的工程师。读完之后，你应该能把账本从「一个总额」拆成「按用户、按特性、按模型、按缓存命中率」的多维矩阵，并对每一项都有降本方案。

上一章把推理服务从 0 拉到 1：模型上了线、请求能进出、p99 在控制范围内。但"能跑"只是入场券，"跑得起"才是产品能不能继续做下去的事。这一章接着讲后半件——同一套服务，账单怎么砍、延迟怎么压、GPU 怎么榨，并且不掉用户体验。

「AI 应用是烧钱的代名词」——这句话在 2024 年还很流行，到 2026 年只剩一半成立。过去两年里，前沿模型 API 价格几乎每年腰斩一次：GPT-5 系列发布时把 GPT-4 同档价格直接砍到 1/4，DeepSeek V4 Flash 把开源旗舰拉到 $0.14/M 输入，Gemini 2.5 Flash 把多模态价格干到了 $0.30/M 输入。账面价格降了，可大量团队的月账单反而在涨——因为模型变强了，调用次数变多了，agent 一来一回十几轮，长上下文从 8K 涨到 200K，向量库塞了几亿条 embedding。

这一章把「真实账单结构」摊在桌上，按从大到小拆。读完你会发现：把成本砍到 1/10 不是营销话术，而是一组确定性的工程动作叠加的结果——单独一个 prompt cache 救不了你，但 prompt cache + batch + 路由 + 压缩 + 多级缓存 + RAG 优化 + 可观测性，叠起来很容易做到 8–10 倍。

## 22.1 你账单到底花在哪儿

在动手优化之前先停一下，问自己一个问题：你确定知道钱花在哪儿了吗？

绝大多数团队的回答是「知道」，但当你让他们打开账单做归因时，能拆到「按特性 + 按用户 + 按模型 + 按缓存命中率」四维矩阵的，十中无一。这一节先把成本结构讲清。

### 三大头：API、GPU、向量库

按 2026 年中位数 SaaS 应用的真实数据，AI 部分的总成本基本由三块构成：

| 成本项 | 占比（典型 SaaS） | 关键变量 | 单位价格变化趋势 |
|--------|------------------|----------|------------------|
| 模型 API 调用 | 55–75% | tokens、模型档位、缓存命中率 | 持续下降，但调用量增长更快 |
| 自托管 GPU 推理 | 15–35% | GPU 时长、利用率、批处理大小 | H100 现货价回落 30–50% |
| 向量库 / 检索栈 | 5–15% | 向量数、查询 QPS、维度 | 趋于稳定 |
| 嵌入 / rerank API | 2–8% | embedding 调用次数 | 跟随主模型下降 |

如果你是纯 API 玩家（不养 GPU），第一条直接占到 80% 以上。如果你跑了 vLLM / SGLang 集群，GPU 时长会变成最大头。

很多团队第一次拉账单时会被一个事实震惊：**单条 chat 消息的成本中位数是 $0.001，但 P99 可能到 $0.5，相差 500 倍**。原因是 P99 那批请求——通常是「agent 跑了 30 轮工具调用」「上下文塞了一整本 PDF」「用户连点重试 5 次」——把均值拉到了一个完全不同的世界。优化的第一性原理是：先治尾巴，再治头部。

一个很有迷惑性的数字是「平均 token / 请求」。它对 cost 几乎无意义。同样是「平均 4000 token / 请求」，下面两种分布的成本差 10 倍：

- 分布 A：99% 的请求 200 tokens，1% 的请求 400K tokens（agent 跑长文档分析）。
- 分布 B：所有请求都是 4K tokens。

A 的成本被那 1% 的 long-context 请求主导，治理 A 应该砍长尾——「agent 上下文窗口压到 100K 上限」「连续工具调用 ≥ 10 次自动总结压缩」。B 的成本是均匀分布，治理 B 应该普惠优化——「prompt caching」「换便宜模型」。**先看 token 直方图，再决定打法**。

### 把账单拆成可归因的矩阵

任何优化前的第一步，是把账单按下面四个维度拆开：

1. **按特性**：客服问答 / 文档摘要 / 代码补全 / agent 任务……每个特性的单位经济学差异极大。
2. **按用户**：你大概率有 1% 的用户消耗了 30%+ 的 token。要么他们值这个钱，要么你的产品被滥用了。
3. **按模型**：你以为只用了 Sonnet，但日志拉出来发现 17% 的请求漏到了 Opus。
4. **按缓存命中率**：「我们开了 prompt cache」和「我们的 prompt cache 命中率 78%」是两件事。

只有这张矩阵在手，下面的所有优化才不是闭着眼睛抓药。22.9 节会给出一个能落地的成本归因中间件实现。

### 三个常见的「钱花得冤」反模式

在动手前，提前打几个针：

1. **隐性放大器**：retry / fallback / re-rank 这三件事在日志里看是「正常重试」，在账单里却是 2–3 倍乘数。任何 retry 策略上线前，先把它算进单价。某团队的 LangChain RetryWithFallback 默认 retry 3 次，外加 OutputParser 失败再 retry 2 次，单次失败 prompt 实际收 6 次钱。
2. **stale prompt**：年初写的 8K system prompt，一年里被加了 17 段「特殊情况处理」，每天被调用 500 万次。那 8K 里 60% 已经是死代码，从未触发过任何输出差异，但每天都在烧钱。**prompt 也要做版本审查 + 单测**。
3. **agent 的「沉默循环」**：agent 在错误工具上反复尝试，最终成功但消耗了 30 轮。在用户视角它「只是慢了一下」，在账单视角它是平时的 30 倍。给 agent 加 `max_steps` 和「连续两轮无新信息则中止」是基本卫生。

## 22.2 API 成本优化：把账面价格直接砍掉

这一节讲的是「不需要改产品形态、不需要改模型、不需要改业务逻辑，只改调用方式」就能省的钱。这些优化的共同特点是 ROI 极高——通常一周内能上线，立刻见效。

### Prompt Caching：把重复的前缀只算一次钱

Prompt caching 是 2024–2026 年 LLM API 最重要的一项变化。它的核心思想朴素到不像优化：**如果一段前缀（system prompt、长文档、few-shot 例子、对话历史前半段）被重复发送，服务端可以只算一次「填充」成本，后续请求按一个非常便宜的「缓存读」价格收费**。

各家实现差异不小。下面这张表是 2026 年中的现状：

| 提供方 | 缓存读单价（相对输入） | 触发方式 | 最小缓存粒度 | TTL 默认 |
|--------|------------------------|----------|--------------|----------|
| Anthropic Claude | 10%（直接 1 折） | 显式 `cache_control` 标记 | 4096 tokens（Opus 4.5+/Haiku 4.5）/ 2048（Sonnet 4.6）/ 1024（更早版本） | 5 分钟（可显式开 1 小时） |
| OpenAI GPT-5 系列 | 10%（直接 1 折） | 自动，前缀 ≥1024 tokens（128 增量） | 1024 tokens | ~5–10 分钟，最长 1 小时 |
| Google Gemini 2.5 | 10%（显式 1 折）/ 25%（隐式 ~75 折） | 显式 `cachedContent` 或自动 | 4096 tokens（Flash）/ 32768（Pro） | 1 小时（显式可设） |
| DeepSeek V4 Flash | ~2%（接近免费） | 自动 | 64 tokens（极小） | 数小时 |

#### Anthropic 的显式缓存

Anthropic 把控制权交给你。你需要在请求里挑出 1–4 个 `cache_control` 标记点，告诉服务端「这之前的内容请缓存」：

```python
import anthropic

client = anthropic.Anthropic()

SYSTEM = """你是一个专业法律助手。请严格根据提供的合同条款回答用户问题，
不得自行延伸。回答用中文，引用条款时标注序号。"""

CONTRACT = open("contract.txt").read()  # 假设有 8000 tokens

resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {"type": "text", "text": SYSTEM},
        {
            "type": "text",
            "text": f"<contract>\n{CONTRACT}\n</contract>",
            "cache_control": {"type": "ephemeral"},  # 缓存到这里
        },
    ],
    messages=[{"role": "user", "content": "第 5 条违约金上限是多少？"}],
)

print(resp.usage)
# CacheCreationInputTokens: 8123  （第一次：按 1.25× 写入溢价计费）
# CacheReadInputTokens: 0
# 第二次同样 system 起的请求：
# CacheCreationInputTokens: 0
# CacheReadInputTokens: 8123       （只收 10% 价格）
```

注意三个坑：

1. **第一次缓存写入按基础输入价的 1.25×（5 分钟 TTL）或 2.0×（1 小时 TTL）计费**——Anthropic 把它叫「写入溢价」。所以 5min TTL 缓存只用 1 次会亏 25%，至少复用 2 次（命中 1 次）才回本；1h TTL 至少要复用 3 次（命中 2 次）。
2. **缓存是按内容哈希的**。如果你在 system 里塞了当前时间或 request_id，命中率立刻归零。
3. **默认 TTL 是 5 分钟**（Anthropic 在 2026 年 3 月把默认从 1 小时改回 5 分钟）。低 QPS 场景需要在 `cache_control` 里显式加 `"ttl": "1h"`——1h 写入是基础输入价的 2.0×（相对 5min 的 1.25× 多 0.6×），但 TTL 长 12 倍，对低 QPS 服务通常立刻回本。或者主动 keep-warm。
4. **最小缓存粒度依模型而定**。Opus 4.5+/Haiku 4.5 必须 ≥ 4096 tokens，Sonnet 4.6 ≥ 2048 tokens，更早的 Sonnet/Opus 才是 1024 tokens。短于阈值的内容即使打了 `cache_control` 也不会被缓存。

#### OpenAI 的自动缓存

OpenAI 选择了「无脑模式」：你不需要标任何东西，只要前缀超过 1024 tokens，相同前缀的后续请求自动享受「cached input」价格——在 GPT-5 系列上 cached input 大致是标准输入价的 10%（即 9 折优惠）。命中以 128 tokens 为粒度递增。

代价是控制权弱：

- 你不知道命中率（除了从响应的 `usage.prompt_tokens_details.cached_tokens` 看）。
- TTL 不公开承诺，闲时大约 5–10 分钟，最长 1 小时。
- 不同 organization 之间不共享缓存。

实际工程经验是：**OpenAI 的自动缓存命中率天然比 Anthropic 低**，因为很多团队的 prompt 里塞了时间戳、UUID、动态信息。把这些东西挪到 prompt 末尾（user message 里），命中率能从 30% 蹦到 80%+。

#### 命中率优化的几条铁律

不管哪家，提升 prompt cache 命中率的工程动作是相同的：

1. **静态内容前置，动态内容后置**。System prompt → tools 定义 → few-shot → 长文档 → 对话历史 → 当前 user message。前面任何一个 token 变了，后面全部失效。
2. **删掉所有动态字段或挪到末尾**。「Current time: 2026-05-08 14:32:11」这种东西要么删掉，要么挪到 user 消息里。
3. **用 sticky session**：在 gateway 层把同一用户的请求路由到同一节点。Anthropic 的缓存是按节点的，跨节点不命中（虽然官方说是 region 级，但实测 sticky 仍能把命中率拉高 5–15%）。
4. **监控命中率**。`CacheReadInputTokens / (CacheReadInputTokens + InputTokens)` 是命中率的近似。健康值 > 60%，优秀 > 85%。

光这一项，对话型应用的成本通常降 40–60%。

#### 一个真实的命中率排查清单

如果你打开了 cache 但命中率只有 10–20%，按下面顺序排查：

1. **system prompt 里有时间戳吗**？哪怕是「Today is 2026-05-08」，每天 0 点全部失效。挪到 user message 里。
2. **tool definitions 里有动态枚举吗**？「可选状态值：[active, paused, ...]」如果状态列表会变，cache 会因为 1 个新值整体失效。考虑「状态」不入 prompt，而是工具 schema 的 enum。
3. **few-shot 是否每次按用户历史挑**？一旦动态选 example，cache 就死了。改静态。
4. **多 region 部署时是否做了 sticky**？用户被路由到不同 region，不共享 cache。
5. **是否有 A/B test 分流**？两组人看到不同的 system prompt，命中率被劈半。把 A/B 标识只放最后。
6. **cache breakpoint 放对位置了吗**（仅 Anthropic）？记得放在「内容确实重复」的边界上。把 cache_control 放在 user message 上是常见错误——user message 几乎不重复。

走完这六步，多数团队能把命中率从 20% 拉到 70%+。

#### Gemini 的隐式 + 显式双轨

Gemini 2.5 系列做了一个特别的设计：默认就有「隐式缓存」（implicit context cache），命中即享 75% 折扣（cached input 是标准输入的 25%），开发者完全不需要操心；同时也提供「显式缓存」（explicit `cachedContent`），用于那种「我知道这段巨长内容会被用一整天」的场景，TTL 可由你设，命中折扣更大（cached input 仅占标准输入 10%，即 9 折优惠），代价是要付一笔「按存储时长 + token 数」的存储费。

隐式缓存的好处是「白送」，但缺点是不可观测——你看不到命中率，也无法主动控制。生产里建议：

- 高 QPS 场景依赖隐式缓存就够了。
- 低 QPS 但有大块固定 context 的场景（比如「对一份 200K 的合同问 30 个问题」），创建一次显式 cachedContent，剩下 29 次按超低价走。
- 别写「Today is X」之类的动态前缀，否则隐式缓存也救不了你。

#### DeepSeek 的「全自动 + 极小粒度」

DeepSeek V4 Flash 的缓存很激进：64 token 起步、自动、命中价直接打到 cache miss 价的 ~2%（约 98% 折扣）。这种小粒度对工具定义、短 system prompt 都能命中，命中率天然高。代价是 API 控制权最少——你能做的事就是「把静态内容放前面」。

因为 DeepSeek 本身价格就极低（V4 Flash 输入 $0.14/M），缓存命中后实际单价 ~$0.0028/M，是 2026 年「成本下限」的事实标杆。很多团队的混合架构是：**主流量走 DeepSeek，关键路径走 Anthropic / OpenAI**——又一个模型路由的应用场景。

### Batch API：异步换 5 折

OpenAI、Anthropic、Gemini 都提供了 Batch API：你把一堆请求打包成 jsonl 文件上传，服务端在 24 小时内完成，价格统统打 5 折。Anthropic 的 Message Batches API 也是类似逻辑。

这个折扣**和 prompt caching 是叠加的**。也就是说，理论上一个静态 system + 缓存命中的 batch 请求，单价能压到 cache miss 同条请求的 5% 左右。

适用场景：

- **离线评估 / 标注**：跑 10 万条评测集，没人催着要结果。
- **批量摘要 / 分类**：每晚凌晨把当天的工单全部摘要一遍。
- **预热缓存 / 嵌入**：批量生成 embedding。
- **Agent 离线探索**：让 agent 提前把可能的查询都跑一遍，结果落到 cache。

不适用：在线对话、实时 API。Batch 没有 SLA 保证，最坏情况你需要等满 24 小时。

```python
# Anthropic Message Batches 例子
import anthropic, json

client = anthropic.Anthropic()
requests = [
    {
        "custom_id": f"req-{i}",
        "params": {
            "model": "claude-haiku-4-5",
            "max_tokens": 200,
            "messages": [{"role": "user", "content": question}],
        },
    }
    for i, question in enumerate(big_question_list)
]

batch = client.messages.batches.create(requests=requests)
# 轮询 batch.processing_status 直到 ended
# 然后下载 results url，逐行解析
```

经验法则：**任何不是用户在等的请求，都应该走 batch**。能做到这一点的团队，月账单立刻降 20–35%。

### 模型路由：让 Haiku 处理 70% 的请求

「我们的 agent 一律走 Opus」是 2025 年最常见的浪费。Opus 4.7 输入 $5/M、输出 $25/M，Haiku 4.5 输入 $1/M、输出 $5/M——5 倍差价。但你的 70%+ 请求其实是「分类」「抽取」「改写」「简单摘要」这种 Haiku 能稳定干好的活。

提醒一点：Opus 4.7 出厂换了一套新 tokenizer，相同输入文本相比 4.6 可能多生成最多 35% 的 token，单 token 价格不变但每请求实际开销可能涨上去。换型号上线时不仅要看名义价，还要在自家流量上跑一段评估校准。

模型路由的两个流派：

#### 静态路由（按特性硬编码）

最简单也最稳。基于业务场景手工划：

```python
def pick_model(task_type: str, ctx_tokens: int) -> str:
    if task_type in {"classification", "extraction", "translation"}:
        return "claude-haiku-4-5"
    if task_type == "code_generation":
        return "claude-sonnet-4-6"
    if task_type == "complex_reasoning" or ctx_tokens > 100_000:
        return "claude-opus-4-7"
    return "claude-haiku-4-5"
```

不需要训练，不需要新组件，立刻能上。覆盖 80% 的场景。

#### 动态路由（按难度 routing classifier）

RouteLLM、Martian、自研路由器都属于这一派。核心思想是：训练一个轻量分类器（几十 M 参数的 BERT-类，或者一个直接调 Haiku 的 zero-shot 分类器），在请求进来时先判断「这个问题强模型才能答好，还是弱模型够用」。

简化版自实现：

```python
ROUTER_PROMPT = """你是一个请求难度分类器。判断下面用户请求需要哪个档位的模型：
- TIER_1（小模型即可）：分类、简单抽取、翻译、改写、固定格式输出
- TIER_2（中型模型）：多步推理、代码生成、长文档摘要
- TIER_3（大模型）：复杂规划、多文档综合、需要深度推理的数学/科学问题

只输出 TIER_1 / TIER_2 / TIER_3，不要解释。

请求：{query}"""

async def route(query: str) -> str:
    # 用最便宜的模型做路由判断
    decision = await haiku.complete(ROUTER_PROMPT.format(query=query[:2000]))
    decision = decision.strip().upper()
    return {
        "TIER_1": "claude-haiku-4-5",
        "TIER_2": "claude-sonnet-4-6",
        "TIER_3": "claude-opus-4-7",
    }.get(decision, "claude-sonnet-4-6")
```

注意路由本身有成本：每次 Haiku 调用大约 $0.0001。所以路由只有在「平均节省 > 路由成本」时才划算——也就是说**别给 5 token 的简单请求做路由**，对它们直接走 Haiku 即可。

公开评测里 RouteLLM（LMSYS / UC Berkeley，ICLR 2025）给出的数字是「在 MT Bench 上保持 GPT-4 性能 95% 的水位，成本降 85%；MMLU 上降 45%；GSM8K 上降 35%」。实际生产部署多见 40–70% 节省，跨基准的成本节省差异很大，别把单一数字当通用承诺。

#### 路由器自身的工程问题

动态路由器是一段额外代码，它本身会出问题：

- **路由失误导致用户体验下滑**：把一个复杂请求误判为 TIER_1，用户拿到一个差答案。需要离线评测路由准确率，目标 > 90%。
- **路由器调用本身就是延迟**：增加 100–300ms 的「分流头部」，对实时 chat 不一定可接受。可以并行：路由器和 Haiku 同时启动，路由判定为 TIER_2/3 再切。
- **冷启动**：路由器训练数据需要持续更新。生产里建议做 1% 流量「全部走 Opus」的影子流，用它们训练新路由器。

经验上，**有评测数据的团队上动态路由都能成功，没评测的团队上完都被回滚**。这是个上线前必须能回答「我们怎么知道路由对了」的项目。

### Cascading：先小后大的「兜底」模式

Cascading 和路由的区别：路由是预先选模型，cascading 是「先用小模型试，不满意再升级」。

适用条件：你能廉价地验证答案对不对。比如：

- **结构化输出场景**：先让 Haiku 出 JSON，schema 校验失败才走 Sonnet。
- **代码生成**：先让 Haiku 写，单测跑过就接受，不过才升级。
- **数学题**：先让 Haiku 算，第二个小模型独立验证（self-consistency 简化版），不一致才走 Opus。

```python
async def cascade_answer(question: str) -> str:
    # 第一档
    cheap = await haiku.complete(question, response_format=JSON_SCHEMA)
    if validate(cheap):
        return cheap
    # 升档
    mid = await sonnet.complete(question, response_format=JSON_SCHEMA)
    if validate(mid):
        return mid
    # 兜底
    return await opus.complete(question, response_format=JSON_SCHEMA)
```

cascading 的代价是**首次失败那一份钱白花了**，所以只在小模型成功率 ≥ 70% 时才划算。否则你是在双重收费。

### 结构化输出：JSON schema 比自由文本省 token

当你需要的是结构化数据时（80% 的 agent / 后台 / 数据管道场景都是），强制 JSON schema 比让模型自由发挥便宜得多：

- 输出 token 通常少 30–60%（没有客套话、没有解释、没有 markdown 装饰）。
- 不需要后处理（不用 `json.loads(extract_first_json(text))` 这种脆弱代码）。
- 配合 OpenAI/Anthropic/Gemini 的原生 structured output，模型可以直接被约束在 schema 上，几乎 100% 解析成功。

一个反模式：在 prompt 里写「请用 JSON 格式回答，参考下面的示例：…，下面是 schema：…」然后塞 800 tokens 的 example。这种写法既贵又不稳。直接用 SDK 的 `response_format` 参数：

```python
from pydantic import BaseModel

class TicketTriage(BaseModel):
    category: str
    priority: int  # 1-5
    summary: str

resp = client.messages.create(
    model="claude-haiku-4-5",
    max_tokens=200,
    messages=[{"role": "user", "content": ticket_text}],
    tools=[{"name": "triage", "input_schema": TicketTriage.model_json_schema()}],
    tool_choice={"type": "tool", "name": "triage"},
)
```

### Few-shot：复用 vs 内联的取舍

Few-shot 例子是 prompt 中很容易吃 token 的部分。三个常见反模式：

1. **每次请求都内联 5 个例子**：每次都付 5 例 × 200 token = 1000 token 的入费。
2. **例子不在 prompt 顶部**：cache 不命中。
3. **例子是动态生成的**（比如「从用户历史里挑最相关的 3 条」）：cache 完全失效。

正确做法：

- **静态 few-shot 全部塞在 system / 顶部**，吃 cache 红利。
- **动态 few-shot 走外部检索 + 摘要**：在请求路径上做「k=3」的轻量检索，但**只取一行摘要**塞进 prompt，而不是整段。
- **能 fine-tune 的场景，把 few-shot 烧进权重**：当一个 few-shot prompt 每天被调用 100 万次时，fine-tune 一个小模型把那些例子内化掉，往往一年节省 10× 以上。

## 22.3 Prompt 压缩：把 token 干掉一半

如果调用方式优化都做完账单还是高，下一步是动 prompt 本身。压缩的核心假设是：**LLM 对自然语言的冗余有强容忍度**。一段 1000 token 的文档，删掉 50% 的「连接词、修饰词、重复信息」后，模型仍能基于剩下的 500 token 完成大多数任务。

### LLMLingua / LongLLMLingua：用小模型挑出关键 token

Microsoft 的 LLMLingua 系列是这个方向最成熟的工具。LLMLingua-2 用一个 BERT-级（约 270M 参数）的分类器对 prompt 中每个 token 打分，然后丢掉低分 token：

```python
from llmlingua import PromptCompressor

compressor = PromptCompressor(
    model_name="microsoft/llmlingua-2-xlm-roberta-large-meetingbank",
    use_llmlingua2=True,
)

long_doc = open("report.txt").read()  # 假设 8000 tokens
result = compressor.compress_prompt(
    long_doc,
    rate=0.4,                   # 保留 40%，压 2.5×
    force_tokens=["\n", "."],   # 关键标点不删
)

print(result["compressed_prompt"])         # 压缩后文本
print(result["origin_tokens"], "->", result["compressed_tokens"])
print("ratio:", result["rate"])
```

实测在 RAG 场景下：

- **2–5× 压缩（LLMLingua-2 论文推荐区间）**：质量几乎无损，下游模型回答 BLEU/ROUGE 仅降 1–3%；论文在 GSM8K 复杂 9 步 CoT prompt 上做到 14× 仍能保持相近性能。
- **10× 压缩**：开始有可见的细节丢失，对检索类问答影响大于摘要类。
- **20× 压缩**：这是初代 LLMLingua 的极限上限，仅适合极简任务，复杂推理普遍崩。

LongLLMLingua 是处理超长输入（>32K）的变种，加了「问题感知」机制：根据用户问题对 prompt 各段重要性做加权，相关段保留率高。RAG + 长文档场景下首选。

LLMLingua 自身也有成本（要跑那个 270M 的小模型），但本地 GPU 上一次压缩通常 < 50ms，对节省的主模型 token 来说极划算。一个公开案例：客服 SaaS 把 RAG 提示从 8K 压到 2K，月账单从 $42K 降到 $2.1K。

#### LLMLingua 的工程坑

把 LLMLingua 接入生产时容易踩的几个坑：

- **GPU 资源争用**：压缩模型（270M）虽然小，但放在和主推理一起的 GPU 上会争 KV cache 显存。建议单独 1 张小卡（甚至 CPU 推理也能凑合）。
- **延迟翻倍**：压缩本身要跑一次 forward。对 8K 文档大约 30–80ms。在线场景需要把它和检索并行做。
- **不适合中文比例高的内容**：原版 LLMLingua 用的是 Llama 类 tokenizer，对中文 token 的判定不准，可能误删重要信息。中文为主时建议用基于 BERT 中文版重训的 LLMLingua-2。
- **数字 / 代码 / 表格容易被破坏**：把 `force_tokens` 配齐（数字、标点、关键符号），或者直接在压缩前用正则把这些段落「冻结」。

### 摘要-检索-精排：先压缩文档库本身

LLMLingua 是请求时压缩，更彻底的做法是**离线预压缩文档库**：

1. 入库时为每个 chunk 生成两个版本：原文 + 压缩摘要。
2. 检索阶段先用摘要做初筛（top-100），再用原文做精排（top-5）。
3. 最终塞进 prompt 的是原文 top-5，但检索召回阶段已经把候选集瘦了 80%。

这种结构对**总成本**和**首字延迟**都友好——召回时算的是更短的向量，rerank 时跑的是更短的文本。

### 上下文截断的优雅做法

agent 类应用里，对话历史会无限增长。最 naive 的「保留最近 N 轮」往往粗暴——重要工具调用结果会被截掉。更工程化的策略：

1. **永远保留 system + 最近 K 轮**（K=4 是好默认）。
2. **更老的轮次用大模型摘成 200 token 的「会话摘要」放进 system 后面**。
3. **重要工具结果（写文件、执行命令）独立保留，附在「artifacts」段**。

```python
def trim_history(msgs, keep_recent=4, summarize_threshold=12):
    if len(msgs) < summarize_threshold:
        return msgs
    head, tail = msgs[:-keep_recent], msgs[-keep_recent:]
    summary = summarize(head)  # 调一次 Haiku 把 head 摘成 200 tokens
    return [{"role": "system", "content": f"<conversation_summary>\n{summary}\n</conversation_summary>"}] + tail
```

摘要本身要钱，但对长会话来说，省下来的后续上下文成本远超摘要成本。一次摘要 $0.0005，省下后续 50 轮每轮多算的 1K tokens = $0.075，杠杆 150×。

## 22.4 缓存层次：从精确到语义到 KV

缓存是 2026 年单项最高 ROI 的优化。前面说的 prompt caching 是「服务端 KV 缓存」，但你自己的应用层还有两层完全独立的缓存可以叠加。

### 第一层：精确匹配缓存（Redis）

最朴素也最有效。键是请求哈希，值是 LLM 响应：

```python
import hashlib, json, redis.asyncio as aioredis

r = aioredis.from_url("redis://localhost:6379")

def cache_key(model: str, messages: list, **params) -> str:
    payload = json.dumps(
        {"model": model, "messages": messages, "params": params},
        sort_keys=True, ensure_ascii=False,
    )
    return "llm:" + hashlib.sha256(payload.encode()).hexdigest()

async def cached_complete(model, messages, ttl=3600, **params):
    key = cache_key(model, messages, **params)
    if hit := await r.get(key):
        return json.loads(hit)
    resp = await llm.complete(model=model, messages=messages, **params)
    await r.setex(key, ttl, json.dumps(resp))
    return resp
```

适用场景比想象中多：

- **常见 FAQ**：同一句「你们的退款政策是什么」每天被问 1000 次。
- **embedding 调用**：同样文本生成同样向量，永远命中。
- **离线管道里的幂等步骤**。

注意三件事：

1. **只缓存 temperature=0 或确定性输出**。否则你在缓存掷骰子的结果。
2. **包含 system prompt 哈希**：system 改了，缓存必须失效，否则 bug 没法解释。
3. **TTL 不要无限**：业务规则会变，文档会更新。1 小时到 24 小时通常合理。

精确匹配的命中率上限取决于 query 重复度。客服场景常见 15–30%，agent 场景通常 < 5%。这时候需要语义缓存。

### 第二层：语义缓存（GPTCache / 自建）

语义缓存的思想：**两个 query 即使字面不同，语义相同也应该共享答案**。「退款政策」和「怎么退钱」「我能退款吗」应该命中同一个缓存项。

实现要点：

1. 入站请求 → 生成 embedding → 在向量库里找最近邻。
2. 如果余弦相似度 > 阈值（典型 0.92–0.97），返回缓存答案。
3. 否则正常调 LLM，把 query embedding + 答案存进向量库。

```python
import numpy as np
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct

qdrant = QdrantClient(url="http://localhost:6333")
qdrant.recreate_collection(
    collection_name="llm_cache",
    vectors_config=VectorParams(size=1024, distance=Distance.COSINE),
)

SIM_THRESHOLD = 0.95

async def semantic_complete(query: str, ttl=3600):
    vec = await embed(query)  # 1024-d 向量
    hits = qdrant.search(
        collection_name="llm_cache",
        query_vector=vec, limit=1,
        with_payload=True,
    )
    if hits and hits[0].score >= SIM_THRESHOLD:
        return hits[0].payload["answer"]

    answer = await llm.complete(query)
    qdrant.upsert(
        collection_name="llm_cache",
        points=[PointStruct(
            id=hashlib.sha256(query.encode()).hexdigest(),
            vector=vec,
            payload={"query": query, "answer": answer, "ts": time.time()},
        )],
    )
    return answer
```

阈值是关键的工程参数：

- **0.99**：几乎只在字面同义时命中，效果接近精确匹配但更贵（多了 embedding 调用）。
- **0.95**：典型生产值。命中率约 30–50%。
- **0.90**：激进，命中率 60%+，但开始出现「问 A 答 B」的 false positive，伤用户体验。

GPTCache 是 Zilliz 的开源实现，封装了上面这套加各种适配器。但生产里很多团队选择自建——GPTCache 的抽象有时会挡路，自建 80 行就能搞定，也更好观测。

**语义缓存的反例：千万别给 agent 工具调用做语义缓存**。「读 src/foo.py」和「读 src/bar.py」语义相似度 > 0.99，但答案完全不同。给 LLM 任务级缓存可以，给具体工具调用结果做精确缓存，别做语义。

### 第三层：vLLM 的 prefix cache（自托管）

如果你跑自家 GPU，vLLM 默认开了 automatic prefix caching：相同前缀的 KV cache 会在 GPU 显存里被复用，不需要重新前向。

效果：

- **首字延迟**降 50–90%（视前缀长度）。
- **吞吐**升 2–5×（同样 GPU 能服务更多并发）。
- **不要钱**：纯软件优化，开就完事。

```bash
# 启动 vLLM 时显式开（默认已开）
vllm serve meta-llama/Llama-3.3-70B-Instruct \
    --enable-prefix-caching \
    --max-model-len 128000 \
    --gpu-memory-utilization 0.92
```

注意 prefix cache 在显存里，重启服务就没了。多副本部署时，搭配 sticky 路由（按 system prompt 哈希挑节点）能进一步提升命中率。

### 三层缓存如何叠？一张请求路径图

实战里这三层不是相互替代而是流水线：

```text
请求进来
   │
   ▼
[1] 精确匹配缓存（Redis）         ── 命中 → 立返
   │ miss
   ▼
[2] 语义缓存（Qdrant，sim≥0.95）  ── 命中 → 立返（可选回写 [1]）
   │ miss
   ▼
[3] 调 LLM API（带 prompt cache）── prompt 前缀命中 → 9 折
   │
   ▼
  写回 [1] 和 [2]，返回用户
```

层间的两个工程细节：

- **写回顺序**：调 LLM 拿到答案后，**先写回精确缓存，再写回语义缓存**。语义缓存写入需要算 embedding，比 Redis SET 慢得多，让用户等待这一步划不来——可以异步 fire-and-forget。
- **失败回退**：如果某层缓存（例如 Qdrant）挂了，跳过它继续走下一层，不要让缓存挂了把整个链路打死。生产实测，至少 1% 的缓存写入会因 race / 超时失败，必须吞掉异常。

## 22.5 延迟优化：用户感知 vs 总耗时

延迟和成本是两个维度。一次请求可能花 10 秒、$0.01，也可能花 2 秒、$0.05。哪个更值，取决于场景。

### 流式输出：把「感知延迟」从 8 秒降到 200ms

LLM 生成 1000 token 大约要 3–10 秒（视模型）。如果你阻塞等到全部生成完再返回，用户看着 spinner 转 8 秒；如果你 streaming 输出，用户在 200ms 后就看到第一个字往外冒，主观延迟少了 95%。

```python
async def stream_chat(messages):
    async with client.messages.stream(
        model="claude-sonnet-4-6", max_tokens=1024, messages=messages,
    ) as stream:
        async for text in stream.text_stream:
            yield text
```

工程上的两个细节：

1. **首 token 时间（TTFT）才是用户感知的延迟**，而不是总时长。所以监控指标要分别拉：TTFT、TPOT（每 token 时长）、E2E 总时长。
2. **流式会和缓存配合困难**：你要么把整段流式响应聚合后再写入缓存（多了一次内存复制），要么放弃缓存。生产实践是聚合写。

### 推测解码：让小模型替大模型「预测下一句」

Speculative decoding 的核心：**用一个小 draft model 一次性预测多个 token，让大模型一次性验证一批**。如果 draft model 猜对了 k 个 token，本来要跑 k 次 forward，现在只跑 1 次。

vLLM 在 0.16+ 后陆续把 P-EAGLE、Eagle3、MTP 这些 draft head 路径接进了主分支（0.18 上 MTP + chunked prefill 已基本可用），配合训练好的 draft head，常见的提速：

- **算术 / 代码生成**：2–3× tokens/s（重复模式多，draft 命中率高）。
- **创意写作**：1.3–1.8×（多样性高，draft 命中率低）。
- **长 reasoning**：2–4×（推理链有大量结构化重复）。

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
    --speculative-model meta-llama/Llama-3.3-1B-Instruct \
    --num-speculative-tokens 5
```

API 玩家无需关心：OpenAI、Anthropic 的延迟优化都已经把这类技术内置在 serving 栈里。

#### 推测解码的「兼容矩阵」

并不是所有优化能并存。vLLM 的几个特性之间有微妙关系：

- **prefix cache + speculative**：vLLM 0.18 之后默认两者都开，且功能正确；但在部分早期版本（0.9.x）里 prefix cache 命中统计不会被记入日志，可观测性偏弱。也有报告称 MTP speculative 开启后 prefix cache 命中率会比纯 prefix cache 略降（实测 ~71%），上线前自己测一下。
- **chunked prefill + speculative**：推荐同时开，对长上下文场景的 TTFT 友好。MTP 在 0.18 上经过 chunked prefill 改造，DeepSeek 类模型已可稳定使用。
- **量化 + speculative draft**：draft model 也要量化，否则会卡住主模型的吞吐节奏。
- **speculative + tool-use 强格式**：强 JSON schema 下 draft 命中率会下降，因为 schema 约束让 token 分布变窄、变陡，draft model 难学。

启用前在「真实流量回放」上跑一次 24 小时 AB 比对，再决定是否上线。

### 模型蒸馏：把任务「烧进」小模型

蒸馏不再是研究院专利。当你有一个稳定调用量大的特定任务（比如「客服意图分类」「日志异常分级」），可以用大模型生成几万条标注，fine-tune 一个 7B / 3B 小模型替代它：

- **延迟降 5–10×**（小模型推理快）。
- **成本降 20–100×**（自托管 / 廉价 API）。
- **质量保 90–98%**（特定任务上小模型完全够）。

代价是：你需要一个 ML 工程能力，需要一个评测集，需要持续的数据回流和再训练。所以蒸馏只对**调用量极大、任务边界清晰**的场景划算。一个经验阈值：单个任务月调用量 ≥ 1000 万次时，蒸馏几乎一定划算。

### 延迟 SLO 怎么定才不亏

不同场景的延迟阈值差异很大，下面这张表整理了 2026 年常见的 SLO 基线：

| 场景 | TTFT 目标 | TPOT 目标 | E2E 目标 |
|------|----------|-----------|----------|
| 实时 chat（用户在等） | < 500ms | < 50ms/tok | 取决于输出长度 |
| Agent 子任务 | < 2s | < 100ms/tok | < 30s |
| 文档摘要（异步可流式） | < 3s | < 80ms/tok | < 60s |
| 代码补全（IDE 内联） | < 200ms | < 20ms/tok | < 1s |
| 离线 batch | 不关心 | 不关心 | < 24h |

定 SLO 的关键是分清楚「TTFT 用户极敏感」「TPOT 中等敏感」「E2E 多数用户不太在意」。所以**首字优化 > 整体优化**——花在 TTFT 上的工程时间几乎不会浪费。

## 22.6 吞吐优化：让每一秒钟更值

延迟是单请求维度，吞吐是系统维度。两者经常拉扯：增大 batch 提升吞吐，但增加单请求延迟。

### 批处理：把零散请求合并发出

API 维度的批处理：前面 22.2 讲的 Batch API，离线场景用。

应用层的批处理：把短时间内（比如 50ms 窗口）到达的同类请求合并成一个 prompt 一次发出。embedding API 是最典型的场景：

```python
import asyncio
from collections import defaultdict

class EmbeddingBatcher:
    def __init__(self, max_batch=64, wait_ms=50):
        self.queue: list[tuple[str, asyncio.Future]] = []
        self.max_batch = max_batch
        self.wait_ms = wait_ms
        asyncio.create_task(self._run())

    async def embed(self, text: str) -> list[float]:
        fut = asyncio.get_event_loop().create_future()
        self.queue.append((text, fut))
        return await fut

    async def _run(self):
        while True:
            await asyncio.sleep(self.wait_ms / 1000)
            if not self.queue:
                continue
            batch, self.queue = self.queue[: self.max_batch], self.queue[self.max_batch :]
            texts = [t for t, _ in batch]
            vecs = await embed_api(texts)
            for (_, fut), v in zip(batch, vecs):
                fut.set_result(v)
```

OpenAI / Voyage / Cohere 的 embedding API 单次最多支持 2048 条文本，按 batch 调用比逐条调用便宜不少（HTTP overhead 摊销 + 服务端 batch 优化）。

### 异步 + 限流：把并发拉满又不被踢

各家 API 都有 rate limit（RPM、TPM、并发请求数）。Naive 的 `asyncio.gather` 会瞬间打满然后报 429。正确做法是用信号量 + 令牌桶：

```python
import asyncio
from asyncio import Semaphore

class RateLimitedClient:
    def __init__(self, rpm: int, max_concurrent: int):
        self.sem = Semaphore(max_concurrent)
        self.interval = 60.0 / rpm
        self.last_call = 0.0
        self.lock = asyncio.Lock()

    async def call(self, fn, *args, **kwargs):
        async with self.sem:
            async with self.lock:
                now = asyncio.get_event_loop().time()
                wait = self.last_call + self.interval - now
                if wait > 0:
                    await asyncio.sleep(wait)
                self.last_call = asyncio.get_event_loop().time()
            return await fn(*args, **kwargs)
```

生产环境推荐直接上 `aiolimiter` 或者更专业的 gateway（Kong AI、LiteLLM、自建 Envoy 插件）。

#### 多 key / 多 provider 的容量套利

到达一定规模后，单一 API key 的 RPM/TPM 上限会成为瓶颈。两种破法：

- **多 key 轮转**：在公司账号下申请多个 key（或多个 organization），按可用 token 桶动态分配请求。
- **多 provider 灰度**：同一类任务跨 OpenAI、Anthropic、Gemini 三家并行，按实时成本和延迟自动选最优。LiteLLM、Portkey、OpenRouter 都做这件事。

多 provider 还能解决另一个问题：**单点故障**。某家 API 突然挂了，自动 failover 到备用 provider，不影响业务。代价是要维护多套 prompt（不同模型对 system prompt 的反应不一样）和多套评测。

### GPU 利用率：自托管的真正瓶颈

如果你跑自家 vLLM，账单不是按 token 算的，是按 GPU 时长。同一张 H100，吞吐能差 5×：

| 配置 | 典型吞吐（70B 模型） |
|------|---------------------|
| 单请求 / 不开 batch | 30–50 tokens/s |
| 默认 vLLM | 800–1500 tokens/s |
| + 量化（FP8） | 2000–3500 tokens/s |
| + speculative + prefix cache 热 | 3500–6000 tokens/s |

GPU 利用率优化是一个独立的大话题，对应用工程师来说核心动作是：

1. **打开 `nvidia-smi dmon` 或 DCGM**，看 SM 利用率，目标 > 80%。
2. **拉 batch 大小到显存允许的极限**（vLLM 会自动 max-num-seqs，但需要给够 KV cache 预算）。
3. **量化能用就用**：FP8 / INT8 在多数任务上质量差异 < 1%，吞吐 2–3×。

### 自托管 vs API：什么时候转自托管划算

很多团队会问：到了多少调用量该转自托管？2026 年中的成本测算（H100 现货时租 $2/小时，70B 模型大约 1500 tok/s 吞吐）：

| 月输入+输出 token | API 月成本（Sonnet 档） | 自托管 H100 月成本（70B 开源） |
|------------------|------------------------|-------------------------------|
| 100M | $1,800 | $1,500（1 张卡，利用率 30%） |
| 1B | $18,000 | $4,500（3 张卡，利用率 60%） |
| 10B | $180,000 | $24,000（16 张卡，利用率 75%） |

转自托管的临界点大致在 **月 token 量 > 500M** 时，且你需要：

- ML / SRE 工程能力（vLLM 调优、监控、KV cache 调度）。
- 评测能力（保证开源 70B 在你的任务上不输 Sonnet）。
- 容量规划能力（突发流量怎么扛、GPU 空闲怎么省）。

如果三项里有一项缺，请继续用 API。「省下的钱不够付加班费」是太多团队的真实写照。

## 22.7 RAG 成本优化

RAG 是「省 token」的双刃剑：相比把整本文档塞进 prompt，检索 top-k 显然便宜；但相比直接 0-shot，RAG 多了 embedding、向量库、rerank 三层成本。优化点散落在每一层。

### 检索 vs 上下文长度的权衡

2026 年的现实：

- 长上下文（200K+）足够便宜了。Gemini 2.5 Pro 的 1M context 价格平易近人。
- 但**长上下文延迟**仍然糟糕：实测 RAG 平均 1 秒响应，1M context 平均 45 秒。
- **「迷失在中间」效应**仍存在：太长的上下文里，模型对中间段的注意力下降。

经验法则：

| 文档总量 | 推荐方案 |
|----------|----------|
| < 50K tokens | 直接全塞，不要 RAG |
| 50K – 500K | 简单 RAG（向量检索 + top-5） |
| > 500K | 高级 RAG（hybrid + rerank + 多跳） |
| 多租户隔离 | 必须 RAG（不能跨租户拼接） |

### Embedding 缓存

embedding 调用的成本被严重低估。每次入库写入要算钱，但更隐蔽的是**查询时**的 embedding 调用——每个用户问题都要 embed 一次。

两层缓存：

1. **查询 embedding 缓存**：query 文本哈希 → 向量。命中率受 query 重复度影响，FAQ 类场景能到 30%+。
2. **文档 embedding 缓存**：分块文本哈希 → 向量。重新入库时（文档版本更新）只对变化的 chunk 重 embed，大部分跳过。

```python
async def embed_cached(text: str) -> list[float]:
    key = "emb:" + hashlib.sha256(text.encode()).hexdigest()
    if hit := await r.get(key):
        return json.loads(hit)
    vec = await embed_api(text)
    await r.setex(key, 86400 * 30, json.dumps(vec))  # 一个月
    return vec
```

embedding 模型不变，向量是确定性的，TTL 可以放很长。

### Rerank 必要性评估：贵不贵 vs 值不值

Rerank 通常是 RAG 里**单次成本最高**的组件——cross-encoder 或大模型 rerank 比向量检索贵 10–100×。问题是：rerank 真的需要每次都做吗？

判断准则：

- **检索 recall@20 已经 > 95%**？多半不需要 rerank，top-5 直接用。
- **用户问题对召回精度极敏感**（法律、医疗）？必须 rerank。
- **rerank 后 NDCG 提升 < 5%**？砍掉，省钱。

激进做法：用一个轻量分类器判断「这个 query 需要 rerank 吗」。复杂查询走 rerank，简单查询直跳。生产实测能砍掉 30–60% 的 rerank 调用。

另一个优化：**rerank 结果缓存**。同一 query + 同一候选集 → 同一排序，可以缓存。命中率视场景，FAQ 场景常见 > 40%。

### Matryoshka 维度：检索用 256，rerank 用 3072

OpenAI text-embedding-3 / Voyage 3.5 都支持 Matryoshka 表示——同一个 embedding 模型输出的 3072 维向量，截前 256 维仍然语义有效。

工程上的应用：

1. 入库时存 3072 维。
2. 检索阶段只用前 256 维做粗筛（向量库内存占用降 12×，检索速度升 3–5×）。
3. 拿到 top-100 后，用全 3072 维做精排（如果不上 cross-encoder）。

向量库总成本（内存 + QPS）能降 60–80%，检索质量几乎无损。

### Hybrid 检索：BM25 不是过时的东西

2024 年很多团队 all-in 向量检索，现在大量回流到 hybrid（BM25 + vector）的做法。原因是：

- **关键词类查询**（专有名词、代码 ID、错误码）BM25 召回明显优于向量。
- **BM25 几乎免费**：Elasticsearch / Qdrant Sparse / Tantivy 都有原生支持，无 GPU、无 embedding 调用。
- **结合 RRF（reciprocal rank fusion）合并两路结果**，下游 rerank 拿到的候选质量更高，可以减少 rerank 的 top-k。

工程经济学：原来 vector top-50 + rerank top-50 = $X 成本。改成 BM25 top-30 + vector top-30 + RRF 合并 + rerank top-20 = $0.6X 成本，召回率持平甚至更好。

### 文档分块的隐性成本

分块（chunking）是 RAG 里最少被讨论但成本影响巨大的一环：

- **chunk 太小**（256 tokens）：召回必须取 top-20 才覆盖一个完整答案，rerank + LLM 都按 top-20 收费。
- **chunk 太大**（4096 tokens）：召回 top-3 就够，但每块本身的 embedding 成本和检索精度都受影响。
- **chunk 不带 overlap**：经常切在句子中间，造成断章。
- **chunk 不带元数据**（标题、章节路径）：retriever 看不到结构信息。

经验值是 chunk size = 512–1024 tokens、overlap = 64–128 tokens、每个 chunk 头部带「文档标题 / 章节」前缀。这样的分块通常让 top-5 即可覆盖 95% 的有效上下文，下游全部按 top-5 算钱。

### 自建：成本归因中间件

如果你不想引平台，最少也要写一个中间件，把每次调用的成本落到结构化日志里。下面这段是一个生产可用的实现，覆盖 prompt cache 折扣计算：

```python
import time, json
from dataclasses import dataclass, asdict
from contextvars import ContextVar

# 价格表（2026 年中，单位 $/1M tokens；cache_write 取 5min TTL 的 1.25× 写入溢价）
PRICING = {
    "claude-opus-4-7":    {"input": 5.00, "output": 25.00, "cache_write": 6.25, "cache_read": 0.50},
    "claude-sonnet-4-6":  {"input": 3.00, "output": 15.00, "cache_write": 3.75, "cache_read": 0.30},
    "claude-haiku-4-5":   {"input": 1.00, "output":  5.00, "cache_write": 1.25, "cache_read": 0.10},
    "gpt-5.4":            {"input": 2.50, "output": 15.00, "cache_write": 2.50, "cache_read": 0.25},
    "gpt-5.5":            {"input": 5.00, "output": 30.00, "cache_write": 5.00, "cache_read": 0.50},
    "gemini-2.5-pro":     {"input": 1.25, "output": 10.00, "cache_write": 1.25, "cache_read": 0.125},
    "gemini-2.5-flash":   {"input": 0.30, "output":  2.50, "cache_write": 0.30, "cache_read": 0.03},
    "deepseek-v4-flash":  {"input": 0.14, "output":  0.28, "cache_write": 0.14, "cache_read": 0.0028},
}

# 当前请求上下文（user / feature / tenant）
ctx_user: ContextVar[str | None] = ContextVar("ctx_user", default=None)
ctx_feat: ContextVar[str | None] = ContextVar("ctx_feat", default=None)

@dataclass
class CostEvent:
    ts: float
    user: str | None
    feature: str | None
    model: str
    input_tokens: int
    output_tokens: int
    cache_read_tokens: int
    cache_write_tokens: int
    latency_ms: int
    cost_usd: float

def compute_cost(model, input_t, output_t, cache_read_t, cache_write_t):
    p = PRICING[model]
    return (
        input_t       * p["input"]       / 1_000_000
      + output_t      * p["output"]      / 1_000_000
      + cache_read_t  * p["cache_read"]  / 1_000_000
      + cache_write_t * p["cache_write"] / 1_000_000
    )

async def tracked_call(client, model, **kwargs):
    t0 = time.time()
    resp = await client.messages.create(model=model, **kwargs)
    u = resp.usage
    ev = CostEvent(
        ts=t0,
        user=ctx_user.get(),
        feature=ctx_feat.get(),
        model=model,
        input_tokens=u.input_tokens,
        output_tokens=u.output_tokens,
        cache_read_tokens=getattr(u, "cache_read_input_tokens", 0),
        cache_write_tokens=getattr(u, "cache_creation_input_tokens", 0),
        latency_ms=int((time.time() - t0) * 1000),
        cost_usd=compute_cost(
            model, u.input_tokens, u.output_tokens,
            getattr(u, "cache_read_input_tokens", 0),
            getattr(u, "cache_creation_input_tokens", 0),
        ),
    )
    # 落到 stdout（让 fluentbit / vector 收走），或直接写 ClickHouse
    print(json.dumps(asdict(ev)))
    return resp
```

落到日志后用 ClickHouse / DuckDB 跑聚合：

```sql

FROM cost_events
WHERE ts >= today() - 1
GROUP BY feature
ORDER BY cost DESC
LIMIT 10;

-- 高消耗用户（异常排查）

FROM cost_events
WHERE ts >= today() - 7
GROUP BY user
ORDER BY cost DESC
LIMIT 50;

-- prompt cache 命中率

       sum(cache_read_tokens) / nullIf(sum(input_tokens) + sum(cache_read_tokens), 0) AS hit_rate

GROUP BY model;
```

### 异常告警：成本激增检测

最朴素的告警：「日总成本 > 昨日 × 1.5」时报警。但这太钝。更有用的告警是分维度的：

- **单用户日消耗 > 历史 P99 × 3**：可能在被滥用。
- **某特性单请求成本中位数突变**：可能 prompt 改了、上下文变长了。
- **prompt cache 命中率 < 20%**：可能 prompt 里被塞进了动态字段。
- **Opus 调用占比突增**：可能路由器坏了。

简化的滑窗检测：

```python
from collections import deque

class AnomalyDetector:
    def __init__(self, window=24, threshold=3.0):
        self.history: deque[float] = deque(maxlen=window)
        self.threshold = threshold

    def check(self, value: float) -> bool:
        if len(self.history) < self.history.maxlen:
            self.history.append(value)
            return False
        mean = sum(self.history) / len(self.history)
        anomaly = value > mean * self.threshold
        self.history.append(value)
        return anomaly
```

接到 PagerDuty / Slack 即可。这一项绝对值得花两小时搭起来——曾有团队因为一个 bug 让 Opus 路由打开了一整晚，第二天看到账单时已经多花了 $18,000。

### 预算硬约束：不要只靠告警

告警是事后的，预算是事前的。生产实践中真正能避免「一夜烧光」的，是在 gateway / SDK 层做硬上限：

- **每 user 每日 token 上限**：超出立刻 429，附带升级提示。
- **每 feature 每分钟 USD 上限**：滑动窗口预算耗尽则降级（切到 Haiku、关闭 RAG、返回缓存）。
- **总账户日预算 cap**：到点全部熔断，触发人工解锁。

```python
class BudgetGuard:
    def __init__(self, redis, daily_usd_cap: float):
        self.r = redis
        self.cap = daily_usd_cap

    async def reserve(self, key: str, est_cost: float) -> bool:
        bucket = f"budget:{key}:{datetime.utcnow():%Y%m%d}"
        used = float(await self.r.get(bucket) or 0)
        if used + est_cost > self.cap:
            return False
        await self.r.incrbyfloat(bucket, est_cost)
        await self.r.expire(bucket, 90000)
        return True
```

调用前先 `reserve`，预估成本按「最长输出 + cache miss」上界算。这种硬约束比任何告警都靠谱——告警是叫醒工程师，硬约束是不让事情发生。

## 22.9 案例：把月账单从 $50K 砍到 $5K

这一节用一个改编自三个真实复盘合并的案例，把前面所有手段串成一条线。背景是一家 B2B SaaS 客服自动化产品，2025 年 Q4 月 LLM 账单 $50K，经过一个季度的优化，2026 年 Q1 月账单 $4.8K，业务量同期增长 35%。

### 第 0 步：得到管理层的「容许试错」承诺

复盘里很多团队漏了这一步：在 CTO / CFO 那里明确「我们要在三个月内动很多线上代码，可能短暂影响 P95 延迟、可能偶发质量回退，请给我们评测和回滚的空间」。**没有这个承诺，第一次回退就会被打回去**。

承诺的形式：

- 数字目标：三个月降 60%，质量评分维持 ≥ 4.1。
- 试错预算：允许「答案质量在某些维度短暂下降 1 分」，但 24 小时内必须修。
- 回滚机制：每个优化都通过 feature flag 控制，能秒级回退。
- 周报机制：每周三给 CFO 一份「累计省了多少 / 这周做了什么」。

### 起点：盘点账单

第一步是把账单做成多维矩阵。下面是优化前的月分布（数字略调）：

| 模块 | 月成本 | 占比 | 调用量 | 平均单价 |
|------|--------|------|--------|----------|
| Agent 编排（GPT-4 Turbo） | $22,000 | 44% | 1.1M | $0.020 |
| RAG 问答（GPT-4） | $14,000 | 28% | 0.9M | $0.016 |
| 工单分类（GPT-4） | $7,500 | 15% | 3.0M | $0.0025 |
| Embedding（ada-002） | $4,000 | 8% | 200M tokens | - |
| 日报生成（GPT-4） | $2,500 | 5% | 12K | $0.21 |

### 第一波（一周内）：低风险动作

- **工单分类降档**：GPT-4 → GPT-4o-mini，再 → 自训 BERT-tiny。月成本 $7,500 → $300。
- **Embedding 模型换代**：ada-002 → text-embedding-3-small（256 维，Matryoshka）。$4,000 → $400。
- **Batch API 启用**：日报生成走 batch，5 折。$2,500 → $1,250。
- **Prompt cache 上线**：所有模块的 system prompt + 工具定义打 cache 标。RAG 命中率 71%，agent 命中率 58%。

第一波之后：$50K → $32K，降 36%。一周。

### 第二波（一个月内）：模型路由 + 压缩

- **模型路由**：用 Haiku 4.5 做 zero-shot 分类，决定任务走 Haiku / Sonnet / Opus。Opus 调用占比从 100% 降到 8%。
- **RAG 检索结果用 LLMLingua-2 压缩**：4× 压缩，下游回答质量评测仅降 1.8 个点（人工评分）。
- **流式输出全面铺开**：用户感知延迟降 80%，没有省钱但救了 NPS。
- **结构化输出强制化**：所有 agent 工具输出走 schema-mode，输出 token 平均降 38%。

第二波之后：$32K → $11K。

### 第三波（持续优化）：RAG 重构 + 蒸馏

- **RAG 双层缓存**：精确缓存 + 语义缓存（阈值 0.96），整体命中率 41%。
- **Rerank 必要性判断**：只有「复杂查询」才走 rerank，覆盖率 23%。
- **Agent 子任务蒸馏**：高频小任务（实体抽取、状态判别）蒸馏到 Llama-3.3-3B 自托管。
- **工单分类 BERT 模型 self-host**：放到 CPU 推理，不再调 API。

第三波之后：$11K → $4.8K。

### 优化前后的指标对照

复盘的最后做了一张关键指标对照表，比账单更能说明「这次优化是不是真值」：

| 指标 | 优化前 | 第一波后 | 第二波后 | 第三波后 |
|------|--------|----------|----------|----------|
| 月 LLM 成本（USD） | 50,000 | 32,000 | 11,000 | 4,800 |
| Daily Active Users | 12,000 | 12,400 | 13,800 | 16,200 |
| ARPU 折算 LLM 成本（$/user/月） | 4.17 | 2.58 | 0.80 | 0.30 |
| 平均首字延迟（ms） | 2,800 | 1,200 | 600 | 480 |
| 平均整体响应（s） | 8.4 | 6.2 | 3.8 | 3.2 |
| Prompt cache 命中率 | 0% | 71% | 78% | 81% |
| 答案质量评分（1–5） | 4.21 | 4.20 | 4.18 | 4.17 |
| NPS | 42 | 44 | 51 | 53 |

**质量几乎没掉，延迟降一半，单用户成本降 14×**。这是真正可以拿去给 CFO 讲的故事。

值得注意的是「答案质量评分」从 4.21 微降到 4.17。这 0.04 的下降在统计噪声内，但团队仍然花了一周排查，最终定位到是「LLMLingua 压缩在数学题场景下偶发性丢数字」，加了个 `force_tokens` 规则修掉了。这种对质量的零容忍是优化能持续推进的关键——任何一次明显回退都会被立即发现并回滚，团队和老板的信任就建立起来了。

### 反模式回顾

复盘中也总结了几个被踩过的反模式，不少都很常见：

1. **过度蒸馏**：把一个 OK 的 GPT-4 流程蒸馏到 7B 模型，质量评分掉了 6 分但月省 $2K。三个月后客户流失率上升 3%，挽回成本远超那 $2K × 3。**蒸馏前必须有持续评测，掉指标就回滚**。
2. **激进语义缓存**：阈值压到 0.88 之后命中率涨到 70%，但客服 SLA 投诉率涨了 11%。回到 0.96。
3. **盲目 long context**：把整套文档塞进 Gemini 2.5 Pro 的 1M context，账面便宜但首字延迟 30 秒+，用户跑了。
4. **Opus 偷偷上岸**：路由表里漏了一个分支，所有「unknown task」都默认走 Opus。一周后才在成本归因报表里看到。**所有 fallback 必须显式指定模型**。
5. **缓存导致的 stale 数据**：FAQ 缓存 24 小时，但「营业时间」类问题在节假日变了，回答一直错。**业务相关问题需要 invalidation hook**。

### 优化过程中的关键评测

整个三波优化里，团队做对的最重要一件事是**每一波都跑一次端到端评测**：

- 评测集：500 条历史工单 + 50 条人工编写的边界 case。
- 自动指标：BLEU、答案抽取准确率、JSON 结构有效率。
- 人工指标：随机抽 50 条让两位高级客服按「正确性 / 风格 / 完整性」打 1–5 分。
- 通过线：综合分相对基线下降 ≤ 1.5 分。

每次优化上线前都跑一次，不通过的优化退回。这套机制在第二波时挡住了一个差点上线的「提示压缩到 8×」尝试——账面省 60%，但人工评分掉了 4 分。

### 隐藏的非账单成本

复盘里还有一项数据值得提：**优化过程本身的工程师工时成本约 $80K**（三人三个月）。直接月省 $45K，3 个月回本一次（当然之后是纯赚）。但要意识到这笔投入：如果你的 LLM 月账单只有 $8K，砍到 $1K 的优化项目工时回本期可能要 18 个月，未必划算。

## 22.10 优化的优先级与停止条件

写到这里你可能已经被各种手段砸晕了。最后一节给一个清晰的优先级：**按 ROI 从高到低做**。

### 推荐顺序

1. **可观测性先行**：成本归因中间件 + Langfuse / Helicone。一周。没有它，下面所有动作都是赌博。
2. **Prompt caching**：把 system / tools / 长文档前置，加上 cache_control。一两天。立刻 30–50% off。
3. **模型路由（静态版）**：按特性分 Haiku / Sonnet / Opus。一两天。20–40% off。
4. **结构化输出**：所有非对话场景强制 JSON schema。一周。10–20% off。
5. **Batch API**：所有非实时任务走批处理。两三天。15–25% off。
6. **精确匹配缓存**：FAQ + embedding。一周。10–30% off。
7. **语义缓存**：仅对低准确性敏感场景。两周。5–20% off。
8. **Prompt 压缩**：RAG 长上下文场景。两周。20–50% off（局部）。
9. **模型路由（动态版）/ Cascading**：在静态路由跑稳后再上。一个月。10–20% off。
10. **蒸馏 / 自托管**：大调用量、确定性任务。一两个月。50–95% off（局部）。

绝大多数团队做到第 6 步就能砍掉 70% 账单。第 7–10 步是从「省钱」到「极致省钱」的跨越，工程复杂度陡增。

### 何时该停

ROI 是会衰减的。当你已经把账单从 $50K 砍到 $5K，再砍到 $4K 需要的工程量可能是前面所有工作的总和。这时候问自己：

- **省下的钱能不能 cover 工程师工资**？三个工程师两个月省了 $1K/月——亏。
- **延迟是否还能压**？很多时候付更多钱换 30% 延迟是值的。
- **是否在引入风险**？激进缓存导致客户投诉、激进蒸馏导致质量塌陷，都是常见的「省了钱但赔了命」。
- **业务增长是否会让 unit cost 自然下降**？如果你的调用量在三个月内会涨 5 倍，把 prompt cache 上好就赢了，没必要现在搞蒸馏。

最朴素的停止条件：**当 LLM 成本占总收入 < 5% 时，去优化别的**。

### 一年的优化路径范例

一个可复制的全年节奏（适用于 ARR $5–50M、LLM 月成本 $20K+ 的团队）：

- **Q1**：观测 + 基础优化。上线 Langfuse / 自建归因 + prompt cache + 静态路由 + Batch API。目标月降 50%。
- **Q2**：缓存 + 压缩。上线精确缓存、语义缓存、LLMLingua、嵌入缓存、Matryoshka 维度。目标月降 30%（在 Q1 基础上再降）。
- **Q3**：高级 RAG + agent 调优。上线 hybrid 检索、按需 rerank、上下文摘要截断、agent 步骤上限。目标月降 20%。
- **Q4**：考虑蒸馏 / 自托管 / 跨 region 套利。这些都是大手笔工程，年底再上能避免「优化中途产品形态变了」的浪费。

每个季度做一次「成本评审会」，把上一季的优化效果对账：实际省了多少、估的省下来了吗、有没有意外回退。

## 22.11 二十条速查清单

把这一章的 actionable items 整理成一份在 PR review、架构评审时直接用的清单。每条都是「能立刻做」的：

**调用方式**

1. system / tools 静态内容是否前置？
2. 是否有任何动态字段（时间、UUID、用户 ID）混进了 system？挪到 user message。
3. 长文档是否打了 cache_control（Anthropic）？
4. 非实时任务是否走了 Batch API？
5. 是否按特性做了静态模型路由？
6. 输出是否结构化（schema）？

**缓存**

7. embedding 是否做了缓存？
8. FAQ / 常见 query 是否做了精确缓存？
9. 是否有语义缓存？阈值 ≥ 0.95？
10. 缓存失败是否会 fallback 而不是 fail？

**RAG**

11. 检索是否用 Matryoshka 维度（256 检索 / 3072 rerank）？
12. 是否在 hybrid（BM25 + vector）？
13. rerank 是否「按需」而非「每次」？
14. chunk size / overlap 是否合理（512–1024 / 64–128）？

**可观测性**

15. 每次调用是否打了 user / feature / model / cache 标签？
16. 是否有 cache 命中率监控？
17. 是否有日成本归因 dashboard？
18. 是否有异常告警（用户级、特性级、模型级）？
19. 是否有预算硬约束（每用户、每特性）？

**评测**

20. 上线优化前是否跑了端到端评测？

每一项都以「Y / N + 工时估算」形式填，能在两小时内做完一次成本审计。

## 22.12 写在最后：可持续的成本意识

我见过最糟糕的案例不是「账单太高」，而是「账单不可解释」——CTO 拿着账单去 board 解释为什么这个月又涨了 40%，自己也答不上来。

可持续的成本意识需要做到三件事：

1. **每一次代码变更都能被定价**。PR 模板里加一行「预期对 LLM 成本的影响」，迫使工程师思考。
2. **每一个新特性都先有 cost budget**。产品立项时就估「上线后月成本应该 < $X」，超就回头优化。
3. **每个工程师都看得到自己代码的账单**。把成本归因暴露在 Grafana 里，让所有人都能搜「我的服务 / 我的特性 / 我的 endpoint 上周烧了多少钱」。

成本不是 SRE 的事，也不是财务的事，是产品工程师每个 PR 都该过一遍脑子的事。把这种意识嵌进研发流程，比任何花哨的优化技术都更长效。

## 小结

这一章把「LLM 应用是烧钱代名词」这件事从神话回归到工程：

- **拆账单**比省钱重要。没有归因矩阵的优化都是猜。
- **API 优化**是 ROI 最高的：prompt caching + batch + 模型路由 + 结构化输出，叠加能砍 60–80%。
- **缓存**有三层：精确、语义、KV。每一层独立工作，叠起来命中率 50%+ 不难。
- **延迟**和**成本**不是同一个维度。流式输出几乎免费就能把感知延迟砍 80%。
- **可观测性**是前置条件，不是补丁。先上观测，再上优化。
- **优先级 ROI 由高到低**：观测 → 缓存 → 路由 → 压缩 → 蒸馏。

如果一句话总结：**先看清楚，再动手；先 90% 收益的低成本动作，再 10% 收益的高复杂动作；时刻盯着评测指标，省钱不能赔产品**。

最后再放一个我自己常用的「降本三问」，每个新优化项目立项前都得过：

1. **它能在两周内上线吗**？不能就拆，或者放到下季度。
2. **它会不会让答案质量下降**？有评测就上，没评测先做评测。
3. **它带来的省钱能持续多久**？产品形态半年就变了？那就别上重的优化。

工程是「持续做对的事」的事，成本优化也不例外。一年下来一步一步做，回头看账单已经掉了一个数量级，没有奇迹，只有积累。

成本控住了，但还有一个更大的问题在等：这套系统怎么持续可靠地运转下去？数据会漂、模型会老、上游 schema 会变、事故会来。从下一章起进入运维篇，先讲经典 MLOps——数据版本、实验管理、模型注册、流水线、监控、复盘——把「上线」这件事从一次性动作变成一条能持续跑的流水线。
