# 第 13 章 · LLM 微调实战：从 SFT 到 DPO

> 第 12 章带你把开源模型挑了一遍——但通用底座再强，也不会凭空学会你公司的产品口径、合同模板、客服话术。**当 RAG 解决不了「风格 / 格式 / 领域口吻」这一类问题时，下一手就是微调**。本章把 SFT、LoRA / QLoRA、DPO 串成一条单卡也能跑通的实战流水线，让你能在一张 RTX 4090（或者一张租来的 A100）上，把一个 7B 开源模型微调成业务里需要的那个味道，并把它塞进 Ollama 跑起来。
>
> 时间锚点：2026-05。基础模型默认 Llama 3.1 / Qwen3 / Mistral 系列；工具默认 TRL v1.0、PEFT、Unsloth、LLaMA-Factory。本章所有代码片段都按这个时间点能跑为准。

前置：[第 11 章 · LLM 原理](./11-LLM原理.md)（理解 SFT / DPO 的训练目标）、[第 12 章 · 开源大模型生态](./12-开源大模型生态.md)（选基座与显存账本）。后置：[第 14 章 · Prompt 工程化](./14-Prompt工程化.md)（微调跑不动时回头打磨 prompt）、[第 21 章 · 模型部署](./21-模型部署.md)（把训练完的 adapter 推上生产）。

## 0 学习目标

【必读】

读完这一章，你应该能：

1. 一句话讲清 SFT、LoRA、QLoRA、DPO 各自要解什么问题，以及它们排队的先后关系。
2. 估算给定一个 7B 模型、一张 24G 卡，自己有多少显存可以折腾，应该选 LoRA 还是 QLoRA、rank 取多少、哪些 module 要打补丁。
3. 自己造一份 1 千~1 万条规模的指令数据集，按 ChatML 格式写好，并知道怎么用强模型造数据补缺口。

7. 在客户问"我这个场景该不该微调"的时候，给出一个不掉链子的判断。

## 1 直觉先行：微调到底改了模型什么

【必读】

### 1.1 把模型想成一台调好味的中餐馆

你买回家一台 7B 开源模型，本质上是一份调好了的高汤。预训练阶段它喝了几万亿 token 的语料，学会了语法、世界知识、基本逻辑、写代码的肌肉记忆。但它不知道你这家店专做潮汕牛肉火锅，也不知道你的菜单怎么报价、客人投诉怎么回。

微调干的事情，就是在这锅高汤上撒一把你自己业务的料。撒得好，汤还是那锅汤，但喝起来就是潮汕味；撒得不好，汤就咸了、糊了、或者你以为自己加了盐其实只是看着模型在原地打转。

把这个直觉记牢，下面所有的方法、参数、坑，都是围绕"加多少料、加在哪、怎么加"展开的。

### 1.2 三条主线、四个阶段

业内一般把"训出一个能用的模型"拆成四个阶段，AI 应用工程师只关心其中后两个，但你得知道前两个干了什么否则会迷路：

| 阶段 | 干什么 | 谁干 | 你要不要管 |
|---|---|---|---|
| Pre-training | 喝几万亿 token，学语言和世界 | Meta / Qwen / Mistral 团队 | 不要 |
| Continued Pre-training (CPT) | 在领域语料上继续喝，比如全部医疗文献 | 大厂 / 垂类公司 | 一般不要 |
| **Supervised Fine-Tuning (SFT)** | 教模型按指令格式回答，灌业务知识 | **你** | **要** |
| **Preference Alignment（DPO/ORPO/KTO）** | 校准口味，让它在两个回答之间挑你喜欢的 | **你** | **要** |

> 业内黑话提醒：post-training 是 SFT + Alignment 的统称；fine-tuning 这个词在不同语境下既可以指狭义的 SFT，也可以指 SFT + DPO 整套流程，看场合判断。

三条主线贯穿全章：

1. **样本驱动的纠偏**：SFT 是给模型看「该这样答」的正例，DPO 是给它看「这样答比那样答好」的对比样本。
2. **参数高效**：你 99% 的时间都不会动全参数，会用 LoRA / QLoRA 之类只改 0.1% 参数的招式。
3. **数据质量碾压数据数量**：1000 条认真标的样本，吊打 5 万条机器爬来的脏数据。这条比任何超参都重要。

### 1.3 显存账本：先算清楚自己有多少弹药

在选方法之前，最该做的不是看论文，是打开 `nvidia-smi` 看看自己手里有多少 VRAM。下面这张表是按 7B fp16 模型估的，你换成 13B 就 ×2，换成 70B 就 ×10：

| 显存 | 能跑的方法 | 典型卡 |
|---|---|---|
| 8 GB | QLoRA-7B（max_seq=512, batch=1） | RTX 3060 |
| 12 GB | QLoRA-7B（max_seq=2048, batch=1） | RTX 3080 / RTX 4070 |
| 16 GB | QLoRA-13B 起步、QLoRA-7B 舒适 | RTX 4060 Ti / 笔记本 4080 |
| 24 GB | LoRA-7B 舒适、QLoRA-30B 起步 | **RTX 4090 / 3090 ← 个人开发者甜蜜点** |
| 40 GB | LoRA-13B、QLoRA-70B | A100-40G / L40 |
| 48 GB | DoRA-13B、LoRA-30B | RTX 6000 Ada |
| 80 GB | LoRA-70B、QLoRA-405B | A100-80G / H100 |

新手最常见的认知错误：「我 8G 卡不够，干脆放弃」。现实是 8G 卡用 QLoRA + 短上下文，跑通 Llama-3 8B 的指令微调没问题——慢但能跑。本章默认场景按 24G 来描述，你卡小就压 batch、压上下文；卡大就抬 rank、开 DoRA。

### 1.5 一句话理解每个名词

下面这些词你会在论文、博客、群聊里反复碰到，先建立模糊正确的印象，后面章节再细看：

- **Base 模型 vs Instruct 模型**：base 是只跑过预训练的"半成品"，instruct 是经过 SFT + RLHF 已经能正常对话的"成品"。**做 SFT 一般以 instruct 为起点**（已经会聊天，再灌业务知识），除非你做继续预训练。
- **PEFT**：Parameter-Efficient Fine-Tuning，参数高效微调的统称，LoRA / DoRA / VeRA / Adapter 都属于这个家族。
- **adapter**：泛指挂在基座上的小补丁（LoRA、DoRA 都算）；狭义指 Adapter Tuning 那一类插入层。
- **merge / 合并**：把 LoRA 矩阵 `B·A` 加回基座 `W` 上，得到一个不再需要 adapter 的完整权重。推理时合并版更快但失去了"切换 adapter"的灵活性。
- **chat template / 对话模板**：tokenizer 自带的一段 Jinja 模板，把 `messages` 列表渲染成模型能吃的 token 序列。**永远用 `tokenizer.apply_chat_template`，不要手拼**。
- **packing**：训练时把多条短样本拼到一个 max_seq_length 里，提速但要小心样本之间 attention 的污染。Unsloth / TRL 都有正确实现。
- **gradient accumulation**：显存吃不下大 batch 时，分多次小 batch 累积梯度再更新一次，等效大 batch。配 `per_device_train_batch_size × gradient_accumulation_steps × num_gpus = effective batch`。
- **gradient checkpointing**：训练时不存所有中间激活、forward 时按需重算，显存换时间。**几乎所有微调脚本都会开**。

## 2 微调方法分类：从全参到偏好对齐

【必读】

### 2.1 按"改多少参数"分

| 方法 | 改多少 | VRAM (7B 模型) | 质量天花板 | 适合 |
|---|---|---|---|---|
| Full Fine-tuning | 100% | ≥ 80 GB | 100% | 你是大厂 |
| LoRA | 0.1%–1% | ~24 GB | 90%–95% | 单卡，主流选择 |
| QLoRA | 0.1%–1% + 4bit base | ~8 GB | 80%–90% | 消费级显卡 |
| DoRA | 0.1%–1% | ~26 GB | 比 LoRA 高 1–4 分 | LoRA 不够时 |
| VeRA | 0.001% | ~22 GB | 略低于 LoRA | 极致省参数 |
| Adapter / Prefix Tuning | 0.5% | 中等 | 略低于 LoRA | 历史方法，2026 已少用 |

> 默认就 QLoRA。等你确认 QLoRA 跑出来质量不够、瓶颈在容量而不是数据，再升级到 LoRA 或 DoRA。

### 2.2 按"用什么样本"分

| 方法 | 输入样本形态 | 训练阶段 |
|---|---|---|
| SFT | (instruction, output) | post-training 第一步 |
| DPO | (prompt, chosen, rejected) | SFT 之后做一遍 |
| IPO | 同 DPO | DPO 不收敛时换 |
| KTO | (prompt, response, label∈{good, bad}) | 你只有点赞/点踩数据 |
| ORPO | (prompt, chosen, rejected) | 想把 SFT 和偏好合一步做 |
| SimPO | 同 DPO | DPO 跑一遍发现长度偏差大 |

下面挨个拆。

### 2.3 全参数 SFT：知道它就行

把模型的全部权重当成可学习参数，loss 是普通的 next-token cross-entropy。

```python
loss = - sum(log P(y_t | y_<t, x))    # x 是 instruction，y 是 output
```

听起来朴素，问题是：

- 7B 模型 = 70 亿 fp16 参数 = 14 GB 权重；加 Adam 优化器（每参数 8 bytes 状态）+ 梯度 + 激活 ≈ 80–100 GB VRAM。
- 训练完之后存一份 14 GB 的 checkpoint，每个客户一份就要花上百 GB 磁盘。
- 你必须走 DeepSpeed Zero-3 或者 FSDP 才能在多卡上摊开，工程难度陡增。

结论：**应用工程师阶段几乎不碰**。除非你做模型层公司、有 H100 集群、且业务确实需要从底层把语言能力都换掉。

### 2.4 LoRA：低秩补丁

LoRA 的全名是 Low-Rank Adaptation，2021 年微软提出。它的核心假设是：

> 微调时权重的变化量 `ΔW`，**不是满秩**的。也就是说，让一个 4096×4096 的矩阵全部参数都可调是浪费的，它真正"想改"的方向其实只占很低维度的一个子空间。

于是干脆把 `ΔW` 拆成两个小矩阵的乘积：

```text
W_new = W_frozen + ΔW
       = W_frozen + B · A
       
其中 W ∈ R^(d×d)
     A ∈ R^(r×d)   ← 小
     B ∈ R^(d×r)   ← 小
     r 是秩，一般 8/16/32/64
```

`W_frozen` 在训练时不动，只学 `A` 和 `B`。可调参数量从 `d²` 降到 `2dr`：

| d | r | 全参 | LoRA | 比例 |
|---|---|---|---|---|
| 4096 | 8 | 16.78 M | 65.5 K | 0.39% |
| 4096 | 16 | 16.78 M | 131 K | 0.78% |
| 4096 | 64 | 16.78 M | 524 K | 3.13% |

当你给所有的 attention 和 MLP 层都打上 LoRA 补丁时，整个 7B 模型的可训参数量会落在 5M–50M，相比 7B 不到 1%，但保留了 90%+ 的全参微调质量。

**rank（r）和 alpha 怎么选**：

- `r` 越大可训容量越大、过拟合风险越高。常见取值 8/16/32。**新手默认 16**。
- `alpha` 是缩放因子，前向时实际乘出来的是 `(alpha/r) · B·A`。**默认设 alpha = r**（也就是缩放为 1）；如果想加大效果，设 `alpha = 2r`。
- 任务复杂、数据量大、又有显存（>24 GB）：上 `r=32` 或 `r=64`，配合 `use_rslora=True`（rsLoRA）让缩放变成 `alpha/sqrt(r)`，对高 rank 更友好。
- 任务简单、就想灌一些行业术语：`r=8` 也够。

**target_modules 选哪几个**：

- 老经验：只打 `q_proj`、`v_proj`。便宜但不彻底。
- 现代经验（QLoRA 论文之后）：`target_modules="all-linear"`，把 attention 的 q/k/v/o 加上 MLP 的 gate/up/down 都打上。**这是 2026 的默认推荐**。
- 不要打 `embed_tokens` 和 `lm_head`，除非你在做词表扩展。

### 2.5 QLoRA：4-bit 量化 + LoRA

QLoRA 在 LoRA 基础上做了三件事：

1. **4-bit NormalFloat (NF4) 量化**：把冻结的基座模型从 fp16 压成 4-bit，内存直接砍 75%。NF4 是专门为正态分布权重设计的量化数据类型。
2. **Double Quantization**：连量化常数本身也再量化一次，省一点点内存。
3. **Paged Optimizers**：用 NVIDIA Unified Memory 把优化器状态在 VRAM 满时丢到 RAM，避免 OOM。

7B 模型用 QLoRA：基座 4-bit ≈ 4 GB + LoRA 参数 ≈ 100 MB + 激活 + 优化器 ≈ 总共 6–8 GB VRAM。**你的笔记本 RTX 3060（12G）就能跑**。

代价：训练速度比 LoRA 慢 30%–50%（因为每次 forward 要反量化），最终质量比 LoRA 略低 1–3 分。但它打开了消费级显卡微调 7B–13B 模型的大门，**新手必须从这里开始**。

### 2.6 DoRA：LoRA 的 1.5 代

DoRA（Weight-Decomposed Low-Rank Adaptation，NVIDIA 2024）观察到：

> 全参数微调时，权重的"幅度"和"方向"是被分开调整的，但 LoRA 把它们粘成一团调，限制了表达力。

于是 DoRA 把权重 `W` 拆成 `magnitude · direction`，对方向用 LoRA 调，对幅度单独学一个标量。

经验数据：

- 在 Llama 7B / 13B 上的 commonsense reasoning 任务上，DoRA 比 LoRA 高 3.7 / 1.0 分。
- Llama 3 8B 上高 4.4 分。
- 推理时和 LoRA 一样可以合回去，无额外开销。

**实操选择**：在 PEFT 里只需要 `use_dora=True`。但训练时间会比 LoRA 慢 20%–40%。如果你时间充裕，DoRA 是一个免费午餐。

### 2.7 VeRA、Adapter、Prefix Tuning：知道就好

- **VeRA**（Vector-based Random Adaptation）：所有层共享同一对随机投影矩阵，只学每层的两个对角向量。参数量再砍一个数量级，但质量略低。**适合极端边缘场景**——一台手机要装 100 个不同任务的 adapter。
- **Adapter Tuning**：在每层 transformer 之间塞一小段 MLP。问题是会引入推理延迟（无法合并回基座）。**2026 基本被 LoRA 取代**。
- **Prefix / P-Tuning**：在输入序列前面拼一段可学习的"软 token"。问题是会占 context 长度、且效果不稳定。**研究价值大于工程价值**。

> 一句话：除非你看论文要复现实验，AI 应用工程师只需要 LoRA、QLoRA、DoRA 三件套就够。

### 2.8 偏好对齐家族：DPO、IPO、KTO、ORPO、SimPO

SFT 教会了模型「这个问题这样答」。但在很多场景下，「这样答」和「那样答」都对，只是其中一个更礼貌、更安全、更符合品牌调性。这种「相对好坏」的信息，SFT 表达不了。

老办法是 RLHF：训一个 reward model，再用 PPO 强化学习。这套流程门槛高、不稳定、显存吃满。

**DPO（Direct Preference Optimization，2023）的关键洞察**：reward model 其实不是必需的。你可以从 chosen / rejected 的对比中**直接**导出梯度。它的 loss 形式：

```text
L_DPO = - log σ ( β · [ log π(y_w|x)/π_ref(y_w|x) - log π(y_l|x)/π_ref(y_l|x) ] )

其中 y_w 是 chosen（人偏好的），y_l 是 rejected（不偏好的）
     π 是当前模型，π_ref 是 SFT 模型（参考）
     β 是 KL 正则强度，越小越敢偏离参考
```

直观理解：让 chosen 的概率涨、rejected 的概率跌，但不能跑得离 SFT 太远。

DPO 之后的几个变体，都是在解 DPO 的某个具体毛病：

| 方法 | 解的毛病 | 关键差异 | 何时用 |
|---|---|---|---|
| **DPO** | RLHF 太复杂 | sigmoid loss + 参考模型 | 默认起点 |
| **IPO** (Identity Preference) | DPO 在过强偏好下会过拟合 | 用平方 loss 而不是 sigmoid | DPO 的 reward 越练越离谱时 |
| **KTO** (Kahneman-Tversky) | 你没有成对数据，只有单条带 like/dislike | 单边的二分类损失 | 线上点赞/点踩、违规反馈数据 |
| **ORPO** (Odds-Ratio PO) | DPO 必须先 SFT，两阶段麻烦 | 把 SFT loss + 偏好 loss 合一步 | 想一遍训完、训练资源紧 |
| **SimPO** | DPO 倾向于生成长回复 | 去掉参考模型，按长度归一 | 你的 reward 偏向啰嗦答案时 |

> 2026 的现实是：**SFT → DPO 仍然是默认套路**。SimPO 在 AlpacaEval 2 上比 DPO 高 6.4 分、Arena-Hard 高 7.5 分，但实战里要 case by case 看。一线团队会做 SFT → DPO → SimPO 的"组合拳"，但你别一上来就追新，先把 SFT + DPO 稳定跑通。

### 2.9 五个偏好对齐方法的损失函数对比

把它们的 loss 摆在一起你就能直观看出差别。约定：`x` 是 prompt，`y_w` chosen，`y_l` rejected，`π` 是当前模型，`π_ref` 是 SFT 后的参考模型。

```text
DPO:  L = -log σ( β · [log(π(y_w|x)/π_ref(y_w|x)) - log(π(y_l|x)/π_ref(y_l|x))] )

IPO:  L = ( log(π(y_w|x)/π_ref(y_w|x)) - log(π(y_l|x)/π_ref(y_l|x)) - 1/(2β) )²
       （平方损失，不会因极端偏好饱和）

KTO:  L = w · σ( β · (log(π(y|x)/π_ref(y|x)) - z_ref) )
       y 是单条 completion，label 决定 reward 是正还是负

ORPO: L = L_SFT + λ · log σ( log(odds(y_w|x) / odds(y_l|x)) )
       SFT loss 和偏好 loss 加权求和，单阶段训练

SimPO: L = -log σ( β/|y_w| · log π(y_w|x) - β/|y_l| · log π(y_l|x) - γ )
        无参考模型；按长度归一；γ 是 reward margin 目标
```

观察：

- DPO 用 sigmoid——优点是数值稳定，缺点是当 reward gap 已经很大时还在推，会过拟合。
- IPO 用平方——它认 `1/(2β)` 这个目标 gap，到了就停，不会推到极端。
- SimPO 没有 `π_ref`——好处是省一倍显存（不用同时加载参考模型），坏处是更难调。

### 2.10 还有别的吗：GRPO、RLVR、DAPO（一笔带过）

2025–2026 起，post-training 的前沿方向已经从单纯偏好对齐，转向**带可验证奖励的 RL**：

- **GRPO**（Group Relative Policy Optimization）：DeepSeek-R1 走红的那一套，对一个 prompt 采样多条回复、按相对好坏算 reward。比 PPO 简单、不需要 critic 网络。
- **RLVR**（RL with Verifiable Rewards）：reward 不是人标的，而是从规则、单元测试、数学验证器拿来的。在数学和代码任务上效果突破式。
- **DAPO**：阿里近期开源的 RL 算法，号称在 GRPO 基础上稳定性更好。

这些更靠近模型层公司的活，AI 应用工程师 90% 的场景里，**SFT + DPO 已经够用**，本章不展开。

## 3 LoRA 原理推导：把矩阵分解的直觉吃透

【建议】

光记口诀不行，遇到 OOM 或者 loss 不收敛的时候，你需要知道哪根线在哪。这一节给你一个能在白板上推一遍的最小推导。

### 3.1 为什么 ΔW 是低秩的

考虑一个 attention 层的投影矩阵 `W ∈ R^(d×d)`。在预训练里它被几万亿 token 推到了一个非常 informed 的位置。当你做下游任务微调时，**你的数据量远远小于预训练**，你能给它的"修改方向"必然是有限的——不可能在所有 d² 个维度上都给出有信号的更新。

数学上：你的样本通过反向传播给 `W` 的累计梯度 `Σ ∂L/∂W` 在 SVD 之后，奇异值会很快衰减。也就是说，更新的"内秩"远小于 d。Aghajanyan et al. 2020 用 intrinsic dimension 实验证实，BERT 在 GLUE 上微调的 intrinsic rank 通常在 200 以下，远小于 768。

LoRA 直接用这条经验：既然 ΔW 是低秩的，那我索性强制它低秩——`ΔW = B · A`，`rank(BA) ≤ r`。

### 3.2 可调参数量的精确计算

给定一个模型，假设：

- hidden_size = `d`（如 Llama-3 8B 是 4096）
- intermediate_size = `m`（Llama-3 8B 是 14336）
- num_layers = `L`（32）
- GQA 分组比 `g` = num_attention_heads / num_kv_heads（Llama-3 8B 是 32 / 8 = 4）
- 你给所有 q/k/v/o + gate/up/down 这 7 个矩阵都加 LoRA

每层每个矩阵的 LoRA 参数数：

| 矩阵 | 形状 | LoRA A | LoRA B | 合计 |
|---|---|---|---|---|
| q_proj | d × d | r×d | d×r | 2dr |
| k_proj | d × (d/g) | r×d | (d/g)×r | r(d + d/g) |
| v_proj | d × (d/g) | 同上 | 同上 | 同上 |
| o_proj | d × d | r×d | d×r | 2dr |
| gate_proj | d × m | r×d | m×r | r(d+m) |
| up_proj | d × m | 同上 | 同上 | 同上 |
| down_proj | m × d | r×m | d×r | r(d+m) |

对 Llama-3 8B（d=4096, m=14336, g=4，即 KV 投影输出维度 d/g = 1024），r=16：

```text
单层每个矩阵：
  q_proj : 2dr = 2·4096·16 = 131,072
  k_proj : r(d + d/g) = 16·(4096+1024) = 81,920
  v_proj : 同 k_proj = 81,920
  o_proj : 2dr = 131,072
  gate_proj/up_proj/down_proj : 各 r(d+m) = 16·(4096+14336) = 294,912

单层合计 = 131072·2 + 81920·2 + 294912·3 = 1,310,720 ≈ 1.31 M
32 层    ≈ 41.94 M 可训参数

占 8B 总参数 ≈ 0.52%
```

**你可以心算的小公式**：r=16, all-linear, 每 1B 基础参数 ≈ 4–5M 可训参数。

### 3.3 一个具体的 ΔW 拆解小实验

如果你想直观感受 LoRA 是如何近似 ΔW 的，可以做这个 5 分钟的小实验：

```python
# lora_intuition.py
import torch

torch.manual_seed(0)
d = 256

# 假装 W 已经预训练好
W_pre = torch.randn(d, d) * 0.02

# 假装"全参微调"会把 W 推到 W_full
# 我们手动构造一个低秩的"真实变化"
true_rank = 8
A_true = torch.randn(true_rank, d) * 0.05
B_true = torch.randn(d, true_rank) * 0.05
delta_W_true = B_true @ A_true               # 真实 ΔW，秩 = 8

W_full = W_pre + delta_W_true                # 全参微调结果

# 现在用 LoRA 不同 rank 去拟合（SVD 找到最佳低秩近似）
U, S, Vh = torch.linalg.svd(delta_W_true, full_matrices=False)
for r in [2, 4, 8, 16, 32]:
    delta_W_lora = U[:, :r] @ torch.diag(S[:r]) @ Vh[:r, :]
    err = (delta_W_lora - delta_W_true).norm() / delta_W_true.norm()
    print(f"r={r:2d}  相对误差: {err:.4f}")
```

跑出来你会看到：

```text
r= 2  相对误差: 0.6824
r= 4  相对误差: 0.4318
r= 8  相对误差: 0.0000   ← 完美匹配
r=16  相对误差: 0.0000   ← 多余的 rank 不会更好
r=32  相对误差: 0.0000
```

结论：**当 ΔW 真实秩 = k 时，LoRA 的 rank 必须 ≥ k 才能完美拟合**。再大就是浪费。

实际微调里你不知道真实秩，但经验上：

- 简单任务（格式调整、口吻修改）：实际 rank ~ 4–8
- 中等任务（领域术语 + 风格）：rank ~ 8–16
- 复杂任务（多种能力 + 大量新知识）：rank ~ 16–32
- 当 rank > 64 仍不够：说明你想灌太多东西进 LoRA，应该走 **CPT + SFT** 两阶段

### 3.4 alpha / r 的物理含义

PEFT 里前向时实际算的是 `W + (alpha/r) · BA`。`alpha/r` 是一个**常数缩放**，它的作用类似 learning rate 的二次调节——`alpha` 大模型学得急、`alpha` 小学得慢。

那为什么不直接调 lr？因为：

- 你换 r 时，理论上"等效梯度"会变（更高 rank 自然容量大），用 `alpha/r` 缩放可以让 r=8 和 r=32 在同一 lr 下行为接近。
- rsLoRA 论文进一步说，正确的缩放应该是 `alpha/sqrt(r)` 而不是 `alpha/r`。当你 r > 32 时打开 `use_rslora=True` 会更稳。

**实战建议**：

```python
LoraConfig(
    r=16,
    lora_alpha=16,        # 等于 r，scaling=1，最稳
    lora_dropout=0.05,    # 防过拟合
    use_rslora=False,     # r ≤ 32 时不开
    use_dora=True,        # 免费午餐，开
    target_modules="all-linear",
)
```

## 4 数据准备：决定上限的环节

【必读】

> 一句业内黑话：**Garbage in, garbage fine-tuned out.** 同样的代码、同样的超参、同样的基座，数据差一个数量级的质量，结果差三个数量级。

### 4.1 数据质量大于数量

LIMA 论文（Meta 2023）的实验：用 1000 条精心挑选的指令样本微调 Llama 65B，效果不输 Alpaca 用 5.2 万条机器生成数据。COIG-CQIA 在中文上做了类似的事情，4.8 万条精挑细选数据效果好过几十万条爬来的脏数据。

什么样的数据是"高质量"：

- **任务多样性**：覆盖你业务里 80% 的真实问题类型，每类至少 50 条。
- **答案规范**：长度合理（一般 100–800 字）、无 markdown 滥用、无"作为一个 AI"这种废话开头、**绝对不能有训练标签泄漏**（比如答案里出现"根据训练数据..."）。
- **难度分布**：30% 简单、50% 中等、20% 困难。全是简单题模型学不到能力，全是难题它会过拟合到"猜"。
- **格式一致**：全部 ChatML 或者全部 Alpaca，**不要混**。

**你应该花在数据上的时间，至少占总工作量的 60%**。新手最常犯的错就是花 80% 时间调超参。超参再调也填不上垃圾数据的坑。

### 4.2 指令数据格式：ChatML、Llama-3、Qwen

不同模型用不同的指令模板。**用错模板，模型表现会下降一个档**。

**ChatML（OpenAI 风格，被 Qwen / Yi / 大量国产模型继承）**：

```text
<|im_start|>system
你是一个礼貌的客服助手。
<|im_end|>
<|im_start|>user
我的订单一直没发货怎么办？
<|im_end|>
<|im_start|>assistant
您好，请提供订单号，我立刻为您查询。
<|im_end|>
```

**Llama-3 / 3.1 / 3.2 模板**（注意 begin_of_text 和 header_id）：

```text
<|begin_of_text|><|start_header_id|>system<|end_header_id|>

你是一个礼貌的客服助手。<|eot_id|><|start_header_id|>user<|end_header_id|>

我的订单一直没发货怎么办？<|eot_id|><|start_header_id|>assistant<|end_header_id|>

您好，请提供订单号，我立刻为您查询。<|eot_id|>
```

**Qwen3 模板**（兼容 ChatML，但加了 `<think>` 标签支持思考链）：

```text
<|im_start|>system
你是潮汕牛肉火锅店的店长助理。
<|im_end|>
<|im_start|>user
推荐三种锅底
<|im_end|>
<|im_start|>assistant
<think>
用户问锅底，我列三个最受欢迎的：清汤、沙茶、麻辣。
</think>
我们最受欢迎的三种锅底是：清汤（牛骨原汤）、沙茶（潮汕本味）、麻辣（川渝改良）。
<|im_end|>
```

**实操要点**：

1. 不要自己手拼模板字符串。用 `tokenizer.apply_chat_template(messages, tokenize=False)`，模板从 `tokenizer_config.json` 里读，永远和模型对得上。
2. 训练时通常 mask 掉 user 和 system 部分的 loss，只在 assistant 输出上算 loss。Unsloth 和 TRL 默认就是这样。
3. 多轮对话训练时，整个对话历史拼一条样本，但只有最后一轮 assistant 输出算 loss（或者每一轮都算，取决于库的实现）。

### 4.3 jsonl 格式实例

**SFT 单轮**：

**SFT 多轮**：

```jsonl
  {"role": "user", "content": "牛胸口能涮多久？"},
  {"role": "assistant", "content": "8 秒，看到边缘卷起来就行。"},
```

### 4.4 数据合成：用强模型造数据

你不可能手写 1 万条样本。现实做法是混合三种来源：

1. **种子数据（人写）**：100–500 条最核心的样本，覆盖业务里最常见的问题。这是质量基线，强模型造的数据照这个写。
2. **强模型扩写（Self-Instruct / Evol-Instruct）**：把种子数据扔给 GPT-4 / Claude / DeepSeek，让它「按相同风格再写 50 条变体」「把这条问题改写得更复杂」。
3. **真实日志清洗**：上线后从客服日志、API 调用记录里捞用户真实 query，过一遍人工/模型审核，加进训练集。这是最珍贵的数据来源。

**Evol-Instruct 一个简单的 prompt 模板**：

```text
你是一个资深的指令重写专家。给你一条指令，请你按下面 5 种方式各重写一条：
1. 增加约束（加一个限制条件）
2. 深化要求（让回答需要更专业的知识）
3. 具体化（用真实场景代替抽象描述）
4. 增加推理步骤（让回答需要多步思考）
5. 改成多轮对话的形式

原始指令：「{instruction}」

输出 5 条重写后的 (instruction, output) 对，每条 JSON 一行。
```

**用强模型造数据的三个底线**：

- **不要让弱模型造数据**：用 GPT-4o、Claude Sonnet、DeepSeek-V3 这种级别的，否则你只是在蒸馏一个比基座更弱的"老师"。
- **务必抽样人工 review 至少 5%**：不 review 你不知道里面有多少幻觉、多少风格漂移。
- **造完之后要去重**：用 simhash 或 embedding 算相似度，相似度 > 0.85 的删掉一条。重复样本会让模型在某些 query 上过拟合。

### 4.5 多轮对话训练数据的两个细节

很多新手在多轮数据上踩坑，主要两点：

**1. 只对最后一轮 assistant 算 loss，还是每一轮都算？**

两种做法都有，TRL 默认是每一轮 assistant 都算 loss（用 `assistant_only_loss=True` 控制）。优点是数据利用率高、模型能学多轮回应；缺点是如果中间轮的 assistant 输出质量参差，会污染模型。

经验：

- 如果你的多轮数据是真实人写的、每轮都精修过——全打 loss
- 如果是合成的、只有最后一轮认真——只打最后一轮
- LLaMA-Factory 里用 `mask_history: false`（默认）打全部，`mask_history: true` 只打最后一轮

**2. 用户输入要不要算 loss？**

绝对不要。loss mask 必须把所有 user 和 system 的 token 屏蔽。如果不屏蔽，模型会学着自己生成"用户问题"——上线后你会看到模型自言自语。

Unsloth 和 TRL 的 SFTTrainer 默认正确处理了，但你用别的库（比如自己写的训练循环）务必检查这一点。

### 4.6 数据清洗的 7 个动作

数据进训练前必走的清洗流程，每一步都直接影响效果：

1. **去重**：用 simhash / minhash / embedding 余弦，相似度 > 0.85 的删一条。重复样本让 loss 下降但泛化变差。
2. **截断长度过滤**：太短（< 10 token output）的样本删掉（信号弱）；太长（> max_seq_length）的截断或删除（截断会破坏完整性）。
3. **乱码 / 编码异常**：检查是否有未解码的 `\u` 转义、HTML 标签残留、PDF 解析的乱字符。
4. **角色泄漏**：assistant 输出里出现 `[user]`、`<|im_start|>` 之类的特殊 token——这些会让模型学会自己生成对话边界，灾难。
5. **AI 痕迹清理**：`As a large language model...`、`I cannot provide...`、`作为一个 AI 助手` 这种开头要全删。让模型学会人话。
6. **PII 脱敏**：手机号、身份证号、邮箱、地址。一是合规，二是避免模型背诵客户信息。用正则 + spaCy 抽实体过一遍。
7. **质量评分过滤**：用强模型给每条样本打 1–5 分，扔掉 ≤ 2 分的。这是 Self-Reward / RM 派生数据集的标准动作。

下面这段 Python 是一个最小清洗模板，按需扩展：

```python
import json, re, hashlib

def clean(rec):
    out = rec["messages"][-1]["content"].strip()
    # 删 AI 痕迹
    bad_starts = ["作为一个", "作为一位", "As an AI", "I cannot", "对不起，我不能"]
    if any(out.startswith(s) for s in bad_starts):
        return None
    # 删特殊 token 泄漏
    if any(t in out for t in ["<|im_start|>", "<|eot_id|>", "[INST]"]):
        return None
    # 太短
    if len(out) < 10:
        return None
    # 手机号脱敏
    out = re.sub(r"1[3-9]\d{9}", "[手机号]", out)
    rec["messages"][-1]["content"] = out
    return rec

# 简单 hash 去重
seen = set()
def is_dup(rec):
    key = rec["messages"][-2]["content"][:200]   # 取 user 输入前 200 字
    h = hashlib.md5(key.encode()).hexdigest()
    if h in seen: return True
    seen.add(h); return False

with open("raw.jsonl") as f, open("clean.jsonl", "w") as g:
    for line in f:
        rec = json.loads(line)
        rec = clean(rec)
        if rec and not is_dup(rec):
            g.write(json.dumps(rec, ensure_ascii=False) + "\n")
```

跑完前后样本数对比一定要看：清洗后留下 60%–80% 算正常，留下 < 30% 说明源数据极脏。

### 5.2 完整 SFT 脚本（QLoRA, Qwen3-8B）

```python
"""
sft_qwen3_qlora.py
单卡 24G 跑 Qwen3-8B QLoRA SFT
依赖：unsloth, trl, peft, datasets
"""

from unsloth import FastLanguageModel
from trl import SFTTrainer, SFTConfig
from datasets import load_dataset
import torch

# === 1. 加载基础模型（自动 4-bit 量化） ===
MAX_SEQ_LEN = 2048
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="unsloth/Qwen3-8B",            # Unsloth 官方预量化版本（Qwen3 原始版本统一了 thinking / non-thinking，不带 -Instruct 后缀；如需 2507 后切分版改成 unsloth/Qwen3-8B-Instruct-2507）
    max_seq_length=MAX_SEQ_LEN,
    dtype=None,                                # 自动选 bf16
    load_in_4bit=True,                         # QLoRA 关键
)

# === 2. 套上 LoRA 适配器 ===
model = FastLanguageModel.get_peft_model(
    model,
    r=16,                                      # rank
    lora_alpha=16,                             # alpha = r
    lora_dropout=0.05,
    bias="none",
    target_modules=[
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj",
    ],
    use_gradient_checkpointing="unsloth",      # Unsloth 优化版
    use_rslora=False,
    use_dora=True,                             # 免费午餐
    random_state=42,
)

# === 3. 加载并处理数据 ===
# 假设你的数据是本地 jsonl，每行 {"messages": [...]}
ds = load_dataset("json", data_files="data/train.jsonl", split="train")

def format_chat(example):
    text = tokenizer.apply_chat_template(
        example["messages"],
        tokenize=False,
        add_generation_prompt=False,
    )
    return {"text": text}

ds = ds.map(format_chat, remove_columns=ds.column_names)

# === 4. 训练配置 ===
training_args = SFTConfig(
    output_dir="./out/qwen3-8b-sft",
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,             # 等效 batch = 16
    num_train_epochs=3,
    learning_rate=2e-4,                        # LoRA/QLoRA 比全参高 1-2 个数量级
    lr_scheduler_type="cosine",
    warmup_ratio=0.05,
    weight_decay=0.01,
    optim="adamw_8bit",                        # Paged AdamW
    bf16=True,
    fp16=False,
    max_length=MAX_SEQ_LEN,                    # TRL ≥ 0.13 起 max_seq_length 已弃用，改用 max_length
    dataset_text_field="text",
    packing=False,                             # 短数据可开 True 提速
    logging_steps=5,
    save_strategy="steps",
    save_steps=100,
    save_total_limit=3,
    report_to="tensorboard",                   # 或 "wandb"
    seed=42,
)

trainer = SFTTrainer(
    model=model,
    processing_class=tokenizer,         # TRL ≥ 0.13 起 tokenizer 参数已重命名为 processing_class
    train_dataset=ds,
    args=training_args,
)

# === 5. 训！ ===
trainer.train()

# === 6. 保存 LoRA ===
model.save_pretrained("./out/qwen3-8b-sft/lora_final")
tokenizer.save_pretrained("./out/qwen3-8b-sft/lora_final")
```

### 5.7 几个真实场景的 loss 曲线长什么样

把脑子里的"健康曲线"建立起来，看一眼图就知道训练健不健康。下面是手感描述：

**健康的 SFT loss**（1 万条数据，3 epoch）：

```text
loss
2.5 ┤●
2.0 ┤ ●●
1.5 ┤   ●●●
1.0 ┤      ●●●●●
0.7 ┤            ●●●●●●●●●●●●●●●  ← epoch 1 末尾
0.5 ┤                              ●●●●●●●●●●  ← epoch 2-3
    └──────────────────────────────────────────  step
     0    100   500   1000  2000  4000
```

特征：epoch 1 内有明显大坡度下降，第 2、3 epoch 是平缓微降。eval loss 跟训练 loss 差距 < 0.1。

**异常 1：lr 太高**

```text
loss
2.5 ┤●
3.0 ┤ ●  ●        ←  Spike
1.5 ┤  ● ● ●  ●
2.5 ┤   ●   ●● ●  ←  震荡，不收敛
```

应对：降 lr 到原来的 1/5；或者增大 warmup_ratio 到 0.1。

**异常 2：过拟合**

```text
loss
2.0 ┤●●
1.0 ┤  ●●●
0.5 ┤      ●●●●●●●  train
0.3 ┤             ●●●●●●●  train
                       
eval                  ●●●  ← eval 反弹
0.8 ┤      ●●●●●●●  ●●●
```

应对：在 eval loss 触底点早停，减 epoch，或者加 dropout。

**异常 3：训练数据泄漏到 eval**

```text
train_loss ≈ eval_loss 全程贴着
```

eval 比 train 略高 0.05-0.15 才正常；完全相等说明 eval 集和 train 集有重叠。**严重时影响所有评估结论**。

### 5.8 显存炸了的紧急救援

按从轻到重的顺序生效：

```text
1. per_device_train_batch_size=1, gradient_accumulation_steps += ×2  → 等效 batch 不变
2. max_seq_length 砍半（看你的真实数据 p95 长度）
3. 开 use_gradient_checkpointing="unsloth"
4. optim="adamw_8bit"（必开）
5. load_in_4bit=True（QLoRA）
6. 关 eval_dataset 期间的 generation（默认就是关的，但要确认）
7. 砍 r：16 → 8
8. 减 target_modules：all-linear → q_proj/v_proj
9. 换更小的基座：14B → 7B
```

每一步都是显存 vs 质量的 trade-off。**Unsloth 已经把前 5 步默认开了**，所以直接用 Unsloth 能省你不少调试时间。

## 6 DPO 实战：教模型挑你喜欢的答案

【必读】

### 6.0 DPO 的数学直觉

把 DPO 损失展开看，每一步它在做的事是：

```text
∂L/∂θ ∝ (1 - sigmoid(β·margin)) · [β · ∇log π(y_w|x) - β · ∇log π(y_l|x)]
```

直观读：

- 当 `margin = log π(y_w)/π_ref(y_w) - log π(y_l)/π_ref(y_l)` 还很小（chosen 没显著超过 rejected）时，`(1 - sigmoid(...))` 接近 1，**梯度强**——模型被用力推向 chosen。
- 当 margin 已经很大时，`(1 - sigmoid(...))` 接近 0，**梯度衰减**——避免过拟合。

这就是 DPO 比 RLHF 简洁的地方：sigmoid 自动给出"已经学好就别再推了"的饱和效应，不需要 reward model + KL 约束的复杂工程。

但这个机制也是 IPO 论文的批评点：当数据有噪声（rejected 实际比 chosen 好的样本）时，DPO 的 sigmoid 仍会给出小但持续的梯度，把模型推向错误方向。IPO 用平方损失换掉 sigmoid，让"已达到目标 margin"的样本梯度直接归零。

### 6.1 什么时候用 DPO

SFT 之后，你的模型已经会按格式答了，但你看输出会皱眉头：

- 太啰嗦（应该简洁），或者太冷淡（应该热情）
- 总是漏掉某种安全提醒
- 在两个差不多的答案里偏向不喜欢的那种风格

这些是 **风格 / 偏好 / 边界类问题**，不是知识盲区。继续 SFT 喂样本是低效的——你需要的是「在两个回答之间挑」的对比信号。这就是 DPO 的位置。

### 6.2 偏好数据怎么造

三种常见来源：

1. **强模型 vs 弱模型**：同一个 prompt，用 GPT-4o 生成 chosen，用基座模型生成 rejected。最容易凑数。
2. **SFT 模型自采样 + 评判**：用刚训好的 SFT 模型对一个 prompt 生成 N 条回复（temperature=0.9），用 GPT-4 / 人工 / reward model 选出最好和最差的。**质量最高**。
3. **真实业务反馈**：上线后用户点踩的回复 = rejected，人工改写后的回复 = chosen。**最贵也最值**。

数据规模：10k–50k 偏好对就足以让 7B 模型有明显改观。1k 偏好对也能跑，但要看效果是否达预期。

格式（jsonl，messages 风格）：

```jsonl
{
  "prompt": [{"role": "user", "content": "我的订单一直不发货"}],
  "chosen": [{"role": "assistant", "content": "您好，非常抱歉给您添麻烦。请提供订单号，我立即为您查询并加急处理。"}],
  "rejected": [{"role": "assistant", "content": "可能仓库还没出货吧，再等等。"}]
}
```

### 6.3 完整 DPO 脚本

```python
"""
dpo_qwen3.py
在 SFT 模型基础上做 DPO，单卡 24G
"""
from unsloth import FastLanguageModel, PatchDPOTrainer
from trl import DPOTrainer, DPOConfig
from datasets import load_dataset

PatchDPOTrainer()  # Unsloth 给 TRL 打补丁

# === 1. 加载 SFT 后的模型，继续以 LoRA 形式训 ===
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="./out/qwen3-8b-sft/lora_final",  # SFT 产出
    max_seq_length=2048,
    load_in_4bit=True,
)
model = FastLanguageModel.get_peft_model(
    model, r=16, lora_alpha=16,
    target_modules=[
        "q_proj","k_proj","v_proj","o_proj",
        "gate_proj","up_proj","down_proj",
    ],
    lora_dropout=0.0,    # DPO 阶段一般不开 dropout
    use_dora=True,
    random_state=42,
)

# === 2. 加载偏好数据 ===
ds = load_dataset("json", data_files="data/dpo_train.jsonl", split="train")
eval_ds = load_dataset("json", data_files="data/dpo_eval.jsonl", split="train")

# === 3. DPO 配置 ===
dpo_args = DPOConfig(
    output_dir="./out/qwen3-8b-dpo",
    num_train_epochs=1,                    # DPO 一般 1 epoch 就够
    per_device_train_batch_size=2,
    gradient_accumulation_steps=4,
    learning_rate=5e-6,                    # DPO 比 SFT 低一两个数量级
    lr_scheduler_type="cosine",
    warmup_ratio=0.1,
    optim="adamw_8bit",
    bf16=True,
    
    # DPO 专属
    beta=0.1,                              # KL 强度，0.05–0.5 常见
    loss_type="sigmoid",                   # 默认 DPO；可选 "ipo" / "hinge" / "robust" / "sppo_hard" / "apo_zero" 等。注意 "kto_pair" 已在 TRL 新版被移除，做 KTO 改用 KTOTrainer
    max_length=2048,
    max_prompt_length=1024,
    
    # 性能
    precompute_ref_log_probs=True,         # 预算参考概率，省显存
    
    eval_strategy="steps",
    eval_steps=100,
    save_steps=200,
    logging_steps=5,
    report_to="tensorboard",
    seed=42,
)

trainer = DPOTrainer(
    model=model,
    ref_model=None,                        # None 时自动用未加 LoRA 的 base 当 ref
    args=dpo_args,
    train_dataset=ds,
    eval_dataset=eval_ds,
    processing_class=tokenizer,            # TRL 新版 API；老版本是 tokenizer=
)
trainer.train()
trainer.save_model("./out/qwen3-8b-dpo/lora_final")
```

### 6.4 DPO 关键超参与监控

| 参数 | 推荐 | 含义 |
|---|---|---|
| `learning_rate` | 5e-7 ~ 5e-6 | **比 SFT 低 100 倍**。DPO 对 lr 极敏感 |
| `beta` | 0.1 | KL 正则。**调大** = 更保守 = 更像 SFT；**调小** = 更激进 = 容易偏离基础能力 |
| `num_train_epochs` | 1 | DPO 极易过拟合，**几乎从不超过 2 epoch** |
| `loss_type` | "sigmoid"（DPO）/ "ipo"（IPO）/ "hinge" | DPO reward 起飞时换 IPO |

要盯的指标（DPOTrainer 会自动 log）：

- `rewards/chosen` 应该缓慢上升
- `rewards/rejected` 应该缓慢下降
- `rewards/margin = chosen - rejected` 应该上升
- `rewards/accuracies`（chosen reward > rejected 的比例）应该从 0.5 升到 0.7–0.85
- 但**不要追求 accuracy = 1.0**，那意味着已经过拟合。0.75–0.85 是健康区间。

### 6.5 SFT vs DPO 效果对比

举个客服场景的真实手感（数字仅作示意）：

| 阶段 | 业务相关任务得分 | 通用能力（MMLU 中文子集） | 风格符合度（人工打分） |
|---|---|---|---|
| 基座 | 62 | 71 | 4.2 / 10 |
| SFT 后 | 84 | 69 | 6.5 / 10 |
| SFT + DPO 后 | 86 | 68 | 8.4 / 10 |

观察：

- SFT 把任务相关能力提了 22 分，DPO 只提了 2 分——**DPO 不补知识**。
- 风格分 SFT 升到 6.5 已经能用，DPO 把它推到 8.4，**这是 DPO 的甜蜜区**。
- MMLU 略降 1–3 分是正常的灾难性遗忘，**只要别降超过 5 分就接受**。

### 6.6 ORPO：把 SFT 和 DPO 合一步

如果你嫌两阶段训练麻烦，可以试 ORPO（用 trl 的 `ORPOTrainer`）：

```python
from trl import ORPOTrainer, ORPOConfig

orpo_args = ORPOConfig(
    output_dir="./out/qwen3-8b-orpo",

    max_length=2048,

    model=model,
```

ORPO 的优点：一次跑完。缺点：在很多场景效果不如 SFT → DPO 两阶段。**新手先走两阶段稳妥；当训练资源紧时再考虑 ORPO**。

### 6.7 SimPO：当 DPO 让答案越来越啰嗦

DPO 的一个隐藏副作用：奖励信号其实在偷偷鼓励"长回复"。原因是 chosen 通常比 rejected 长，DPO 在最大化整段序列的概率比时，对长序列天然有一个累积优势。

SimPO（NeurIPS 2024）的修法：

1. 去掉 `π_ref`，让 loss 不依赖参考模型；
2. 用 `1/|y|` 把序列概率按长度归一；
3. 加一个 reward margin 目标 `γ`，控制差距。

实验数据：在 Mistral-7B 和 Llama-3-8B 上，SimPO 比 DPO 在 AlpacaEval 2 高 6.4 分、Arena-Hard 高 7.5 分，且生成长度没有膨胀。

```python
from trl import CPOTrainer, CPOConfig    # SimPO 集成在 CPOTrainer 里

cpo_args = CPOConfig(
    output_dir="./out/qwen3-8b-simpo",
    loss_type="simpo",                    # 关键
    beta=2.0,                             # SimPO 的 beta 比 DPO 大
    cpo_alpha=0.0,                        # SimPO 时设 0
    simpo_gamma=1.0,                      # reward margin 目标
    learning_rate=1e-6,
    num_train_epochs=1,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=4,
    bf16=True,
)
trainer = CPOTrainer(
    model=model,
    args=cpo_args,
    train_dataset=ds,
    processing_class=tokenizer,  # TRL 新版 API
)
trainer.train()
```

**实操经验**：先做 SFT → DPO 看效果，效果不够再考虑 SimPO；不要直接拿 SFT 模型上 SimPO，没有参考模型作锚的话偏移会很猛。

### 6.8 KTO：你只有点赞数据时

如果你的数据是从产品里收来的「这个回答用户点了赞 / 点了踩」（不是成对的），KTO 才是你需要的。`KTOTrainer` 用法跟 DPO 类似，但数据格式：

```jsonl
{"prompt": "...", "completion": "...", "label": true}
{"prompt": "...", "completion": "...", "label": false}
```

KTO 的好处是数据收集成本低——线上点赞数据天然就是这种形式。

## 7 评估：训完不等于训好

【必读】

> 99% 的微调失败在评估这一关。loss 降得再漂亮，最终能不能上线由评估说了算。

### 7.1 三层评估体系

| 层 | 看什么 | 工具 | 频次 |
|---|---|---|---|
| **任务级** | 业务相关指标（如订单识别准确率、医疗术语 F1） | 自定义评测脚本 | 每个 checkpoint |
| **通用能力回归** | MMLU / CMMLU / C-Eval / GSM8K 子集 | lm-evaluation-harness | 训前 + 训后各一次 |
| **人感 / 业务体验** | LLM-as-judge / 人工 ABtest | OpenAI Evals / Promptfoo | 终选 checkpoint |

任何一层都不能跳过：

- 跳了任务级你不知道训得有没有用；
- 跳了通用回归你不知道有没有"训傻"；
- 跳了人感你不知道用户是不是真的喜欢。

### 7.2 任务级评估示例

针对客服场景，可以这样构造评测集（200–500 条手工标注，**和训练集完全分离**）：

```python
# eval_business.py
import json
from openai import OpenAI    # 用 vLLM / Ollama 起本地服务后，OpenAI 兼容接口

client = OpenAI(base_url="http://localhost:8000/v1", api_key="x")

eval_data = [json.loads(l) for l in open("eval/business_test.jsonl")]
hits = {"intent": 0, "tone": 0, "format": 0}
for item in eval_data:
    resp = client.chat.completions.create(
        model="qwen3-8b-sft-dpo",
        messages=[{"role": "user", "content": item["query"]}],
        temperature=0.0,
    ).choices[0].message.content
    
    # 自定义校验逻辑
    if item["expected_intent"] in resp:
        hits["intent"] += 1
    if not any(bad in resp for bad in ["可能", "也许", "随便"]):  # 客服不能模糊
        hits["tone"] += 1
    if resp.startswith(("您好", "感谢")):                         # 必须有礼貌开场
        hits["format"] += 1

n = len(eval_data)
print({k: v/n for k, v in hits.items()})
```

### 7.4 LLM-as-judge

人工评估贵，LLM-as-judge 是中间地带。让 GPT-4o / Claude Sonnet 当裁判，按 rubric 对每条回复打 1–10 分。

模板示例：

```text
你是一个严格的客服质量评估员。请按下面三个维度对【回答】打分（每项 0–10）：

1. 准确性：是否回答了用户的真实问题
2. 礼貌度：是否符合客服规范（开场致意、不使用模糊词）
3. 完整性：是否包含必要的下一步指引（如要订单号）

【用户问题】: {query}
【回答】: {response}

只输出 JSON：{"accuracy": x, "politeness": y, "completeness": z, "reasoning": "..."}
```

**LLM-as-judge 的三个坑**：

1. **位置偏差**：A/B 对比时把 A 放前面，模型倾向选 A。**做对比评估时必须随机交换顺序，每条都跑两遍取均值**。
2. **冗长偏差**：长回复评分高出 10%–15%。**所以你看到 SFT 后回复变长不一定是好事**。
3. **自吹偏差**：用 GPT-4 评 GPT-4 蒸馏出来的模型，分数会虚高 5%–8%。**至少跑两个不同家的 judge 取均值**。

### 7.5 LLM-as-judge 的实操脚本

下面这段是一个能直接套到自己业务的二选一对比评估脚本，自动处理位置偏差：

```python
# llm_as_judge.py
import json, random
from openai import OpenAI

judge = OpenAI()    # 用 GPT-4o 当裁判

JUDGE_PROMPT = """你是严格的客服质量评审员。下面是一个用户问题，以及两个客服的回复 A 和 B。
请按下列三个维度给出哪个更好：
1. 准确性（是否答对真问题）
2. 礼貌度（开场致意、避免模糊词）
3. 完整性（是否引导下一步）

输出 JSON：{"winner": "A" | "B" | "tie", "reason": "..."}

【用户问题】: {query}
【回复 A】: {a}
【回复 B】: {b}
"""

def judge_pair(query, resp_old, resp_new):
    # 随机调换顺序消除位置偏差
    if random.random() < 0.5:
        a, b, swap = resp_old, resp_new, False
    else:
        a, b, swap = resp_new, resp_old, True
    out = judge.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": JUDGE_PROMPT.format(query=query, a=a, b=b)}],
        response_format={"type": "json_object"},
        temperature=0.0,
    ).choices[0].message.content
    res = json.loads(out)
    if res["winner"] == "tie":
        return "tie"
    if (res["winner"] == "A" and not swap) or (res["winner"] == "B" and swap):
        return "old"
    return "new"

eval_set = [json.loads(l) for l in open("eval/abtest.jsonl")]
counts = {"old": 0, "new": 0, "tie": 0}
for item in eval_set:
    counts[judge_pair(item["query"], item["resp_old"], item["resp_new"])] += 1
print(counts)
# 如果 new / (old + new) < 0.55，新模型没显著优势，慎重上线
```

判读门槛：

- new win rate > 60%：可以灰度
- 50%–60%：人工抽样确认
- < 50%：训坏了，回去查问题

## 8 部署：把 LoRA 变成能跑的服务

【必读】

训完得到的是一份 LoRA adapter（几百 MB），不是完整模型。要上线，三种路线：

### 8.1 路线 A：合并 LoRA，导 GGUF，丢 Ollama

适合本地部署、单机服务、客户内网环境。

```python
# merge_and_save.py
from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="./out/qwen3-8b-dpo/lora_final",
    max_seq_length=2048,
    load_in_4bit=False,                         # 合并必须是非量化
    dtype="bfloat16",
)

# 1. 合并 LoRA 到基座，保存为完整 HF 模型
model.save_pretrained_merged(
    "./out/qwen3-8b-dpo/merged",
    tokenizer,
    save_method="merged_16bit",                 # 也可以 "merged_4bit_forced"
)

# 2. 转成 GGUF（多种量化级别）
model.save_pretrained_gguf(
    "./out/qwen3-8b-dpo/gguf",
    tokenizer,
    quantization_method=["q4_k_m", "q5_k_m", "q8_0"],
)
```

跑完会生成几个 `.gguf` 文件。挑一个（`q4_k_m` 是常用平衡点）：

```text
qwen3-8b-dpo-q4_k_m.gguf  # 约 4.5 GB
```

写一个 `Modelfile`：

```text
FROM ./out/qwen3-8b-dpo/gguf/qwen3-8b-dpo-q4_k_m.gguf

TEMPLATE """{{ if .System }}<|im_start|>system
{{ .System }}<|im_end|>
{{ end }}{{ range .Messages }}<|im_start|>{{ .Role }}
{{ .Content }}<|im_end|>
{{ end }}<|im_start|>assistant
"""

PARAMETER temperature 0.3
PARAMETER top_p 0.9
PARAMETER stop "<|im_end|>"
PARAMETER stop "<|im_start|>"

SYSTEM """你是潮汕牛肉火锅店的店长助理。"""
```

注册并跑：

```bash
ollama create my-hotpot -f Modelfile
ollama run my-hotpot
```

或者起 OpenAI 兼容服务：

```bash
ollama serve            # 默认 http://localhost:11434
```

接入业务时直接当 OpenAI API 调（base_url 改成 `http://localhost:11434/v1`）。

### 8.1.1 Modelfile 写法的几个细节

很多人 Modelfile 写错，模型在 Ollama 里跑就胡言乱语。三个最容易踩的坑：

**1. TEMPLATE 必须和训练时的 chat_template 对得上**

最稳的做法是从 tokenizer 拿出训练时用的模板：

```python
from transformers import AutoTokenizer
tok = AutoTokenizer.from_pretrained("./out/qwen3-8b-dpo/merged")
print(repr(tok.chat_template))   # Jinja 模板
```

把 Jinja 模板转成 Ollama 的 Go template 语法。Unsloth 的 `save_pretrained_gguf` 会自动生成一份 Modelfile，**优先用它生成的，不要自己写**。

**2. STOP token 必须列全**

ChatML 系列至少要列 `<|im_end|>` 和 `<|im_start|>`；Llama-3 系列要列 `<|eot_id|>` 和 `<|end_of_text|>`。漏一个，模型在生成完一轮后会继续编下一轮的 user 提问。

**3. PARAMETER 设置和训练对齐**

如果训练时 max_seq_length=2048，Modelfile 里 `PARAMETER num_ctx 2048`；强行设大于训练长度，模型在长文里会乱来（除非你明确做了 long-context 训练）。

### 8.2 路线 B：vLLM + LoRA 热加载

适合云端服务、多租户、要切换不同微调版本。vLLM 0.4+ 支持运行时挂 LoRA：

```bash
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen3-8B \
  --enable-lora \
  --lora-modules \
    hotpot=./out/qwen3-8b-dpo/lora_final \
    insurance=./out/qwen3-8b-insurance/lora_final \
  --max-loras 4 \
  --max-lora-rank 16 \
  --port 8000
```

请求时指定 model：

```bash
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "hotpot",
    "messages": [{"role": "user", "content": "几点关门？"}]
  }'
```

**优势**：基座模型只装载一次，多个业务共享显存；切租户时只换 LoRA 几百 MB，秒级。

### 8.3 路线 C：合并后再蒸馏到更小模型

如果你的目标是手机端、边缘设备，把 7B 合并模型当老师，蒸馏到 1B–3B 的小模型上。这一块属于[第 21 章 · 模型部署](./21-模型部署.md)与[第 22 章 · 性能优化与成本控制](./22-性能优化与成本控制.md)的范围，本章不展开。

### 8.4 量化级别速查

| 量化 | 文件大小（7B） | 困惑度增量 | 推理速度 | 适用 |
|---|---|---|---|---|
| f16 | 14 GB | 0% | 慢 | 服务器、追求质量 |
| q8_0 | 7.2 GB | < 0.1% | 中 | 服务器、稳妥 |
| q5_k_m | 5 GB | ~0.5% | 快 | 高端消费级 |
| **q4_k_m** | 4.5 GB | ~1.5% | 快 | **默认选择** |
| q3_k_m | 3.5 GB | 3%–5% | 很快 | 边缘设备紧凑 |
| q2_k | 2.7 GB | 8%+ | 极快 | 不推荐生产 |

### 8.5 部署后必做的三件事

训完上线不是终点，是另一个起点。

**1. chat_template 和系统 prompt 对齐**

训练用的是 `Qwen3` 模板，部署时 Ollama 用了它生成的 Modelfile 模板。这两份必须完全一致——多一个 `\n` 都可能让生成质量掉一截。验证方法：

```python
# 训练后立刻验证一次
import json
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("./out/qwen3-8b-dpo/merged")
sample = [{"role": "user", "content": "测试"}]
print(tok.apply_chat_template(sample, tokenize=False, add_generation_prompt=True))
# 把这个字符串和 Modelfile 里的 TEMPLATE 渲染结果手动对照一遍
```

**2. 灰度策略**

不要一次切 100% 流量。常见做法：

- 第 1 周：5% 流量到新模型，监控延迟、错误率、用户满意度
- 第 2 周：根据指标扩到 25%
- 第 3 周：50%
- 第 4 周：100% 或回滚

期间老模型保持热备，回滚预案要 5 分钟内能回去。

**3. 收集线上反馈喂下一轮训练**

线上的「用户改写」「点踩 + 重新生成」「转人工」是金矿——这些都是天然的 (chosen, rejected) 偏好对。日志做好埋点，每两周做一次小规模 DPO 增训，让模型持续向用户偏好靠拢。这就是工业界俗称的 **continual alignment** 飞轮。

## 9 常见坑：训练时你会撞到的所有问题

【必读】

按出现频率从高到低排：

### 9.1 训练直接 OOM

最常见。检查清单：

1. `load_in_4bit=True` 开了吗？
2. `gradient_checkpointing` 开了吗？（Unsloth 用 `use_gradient_checkpointing="unsloth"`）
3. `per_device_train_batch_size` 压到 1 试试，用 `gradient_accumulation_steps` 凑等效 batch。
4. `max_seq_length` 是不是设太大？很多人盲目设 8192，实际数据 99% 都是 1024 以内，浪费显存。
5. 你是不是没设 `optim="adamw_8bit"` 而用了普通 AdamW？省显存差距巨大。
6. 别的进程在占显存？`nvidia-smi` 看一下。

### 9.2 Loss 不降 / 一直震荡

按可能性顺序排查：

1. **学习率不对**：QLoRA 默认 2e-4。设到 2e-5 这种「全参微调级别」就会几乎不动；设到 2e-3 就会爆炸震荡。
2. **target_modules 写错**：只挑了 `q_proj` 一个，可调容量太小。改成 `all-linear` 或全 7 个 module。
3. **数据格式不对**：tokenizer.apply_chat_template 没用，自己手拼漏了 `<|im_end|>`。打印一条样本人眼检查。
4. **Loss mask 不对**：在 prompt 部分也算 loss，导致信号被稀释。看 SFTTrainer 是否设了 `dataset_text_field` 而非 `messages`。
5. **基座模型选错**：拿了 base 模型（没经过指令微调）当 instruct 模型用。一般用 `*-Instruct` 后缀的模型作起点。

### 9.3 Loss 降了，但模型变弱了（灾难性遗忘）

症状：训完客服任务后，让它写个简单代码或者答个常识题，质量比训前还差。

成因：LoRA 把模型推得太远，基础能力被覆盖。

应对：

1. **降学习率**：从 2e-4 降到 5e-5。
2. **减 epoch**：3 → 1 或 2。
3. **加正则数据**：在你的训练集里掺 10%–30% 的「通用任务」样本（如 Alpaca、ShareGPT 子集），保持基础能力的输入信号。这叫 **data mixing / replay**。
4. **小一点的 r**：r=8 的更新空间小，遗忘也少。
5. **冷冻更多模块**：只打 attention（q/v），不打 MLP。
6. **别在 SFT 上做 5 epoch 以上**：经验值 1–3 epoch 是甜蜜区。

### 9.4 过拟合

症状：train loss 降到 0.1 以下，但 eval loss 在 epoch 2 之后开始往上爬。

应对：

1. **加 dropout**：lora_dropout 从 0 到 0.05–0.1。
2. **减 epoch**：直接停。
3. **数据去重**：相似度高的样本去掉。
4. **加数据**：再扩 2 倍样本通常就过了。
5. **早停**：用 `EarlyStoppingCallback`，看 eval loss。

### 9.5 训练正常但效果不及预期

最隐蔽。loss 降了、benchmark 没掉、人工抽样也还行，但上线后用户反馈不好。

诊断：

1. **训练数据和真实 query 分布偏离**：你的训练集是从内部文档造的，真实用户用方言问、用错别字问。修：从线上抽真实 query 做评估集。
2. **chat_template 训练和推理不一致**：训练时拼的模板和 Ollama / vLLM 推理时拼的不一样。修：保存 tokenizer 时一并保存 `chat_template` 字段，部署时也用同一份。
3. **温度设错**：训练用 temp=0 评估，部署用 temp=0.9，分布就漂了。修：评估和部署对齐温度。
4. **prompt 工程没跟上**：你以为 SFT 之后 system prompt 不重要了。其实 SFT 模型对 system prompt 仍然敏感，**部署的 system prompt 和训练里的尽量一致**。

### 9.6 DPO 后模型开始胡说

DPO 特有：rewards/margin 一直在涨，但生成出来的回复开始牛头不对马嘴、出现幻觉。

原因：DPO 把 chosen 概率推得太高、rejected 推得太低，破坏了语言流畅性。

应对：

1. **加大 beta**：从 0.1 调到 0.3。
2. **降 lr**：从 5e-6 降到 1e-6。
3. **减 epoch**：1 epoch 内早停。
4. **换 IPO loss**：`loss_type="ipo"`，对极端偏好更鲁棒。
5. **检查偏好数据质量**：rejected 是不是太差（比如直接是乱码）？太差的 rejected 让 loss 信号过强，模型学到的是「不要乱码」而不是「要更礼貌」。

### 9.7 评估失真

LLM-as-judge 给你打 9 分，人工抽看觉得也就 6 分。

原因：

1. judge 模型本身偏向某种风格。
2. 评估 prompt 没写清楚维度。
3. 样本量太小（< 50 条）置信度差。

应对：

- 评估集 ≥ 200 条
- 至少跑两个不同家的 judge（GPT-4o + Claude）
- 抽 10% 人工对齐

### 9.8 训练好看、上线翻车

具体症状：评估集所有指标都达标，灰度后第一天用户投诉就来了。

诊断步骤：

1. **抽 50 条线上 query 看分布**：是不是和你的训练集 query 分布差很远？常见情况：训练集都是规范化中文，线上是用户用拼音首字母打的、错别字多、混杂表情。
2. **看 system prompt**：训练时你用了什么 system prompt？部署时一样吗？大部分翻车都是因为这两个不同步。
3. **看 temperature 和 top_p**：训练评估时温度通常 0；线上为了多样性常用 0.7。SFT 模型在不同温度下行为差很多。
4. **多轮对话**：你用单轮训了，线上是多轮，模型不会处理上下文。修：训练数据里混 30% 多轮样本。
5. **token 限制**：你 max_seq_length=2048 训的，线上有些 query 拼上历史轮已经超了，被截断后回答出现幻觉。

修法是迭代的——每一轮翻车都是下一轮训练数据的来源。

### 9.9 量化掉点

合并 LoRA、导出 GGUF q4_k_m 后，发现质量比合并的 fp16 版本掉了 5-10 分。

原因：

1. **量化方法太狠**：q3、q2 系列在 7B 模型上掉点很明显。换成 q4_k_m 或 q5_k_m。
2. **校准数据集不合适**：llama.cpp 的 imatrix 量化需要校准数据，默认是 wikitext。如果你的业务在中文医疗，应该用中文医疗语料校准。
3. **模型太小经不起量化**：1B–3B 小模型 q4 量化掉点比 7B 严重。小模型尽量 q5 或 q6 起步。

应对：先用 q5_k_m 上线确认无问题，再尝试 q4_k_m 看能不能压。

### 9.10 GPU 训练正常但 CPU 推理结果不一样

LoRA merge 之后在训练机器上 generate 测试 OK，部署到客户机器（无 GPU）上跑结果不对。

原因：

1. dtype 不一致——训练 bf16，CPU 推理 fp32 升精度反而暴露了量化误差累积。
2. tokenizer 不一致——你忘了一起拷 tokenizer 文件。
3. system prompt 编码——一些场景里中文 system prompt 在不同编码下被切成不同的 token 序列。

部署 checklist：模型 + tokenizer + chat_template + system prompt + 推理参数（temperature/top_p/repeat_penalty）一份都不能少。

## 10 何时不该微调：一个老工程师的劝退清单

【必读】

> 业内一句话：**80% 的 "我们要微调" 项目，最后用 RAG + 好 prompt 就解决了**。

下面这些情况，**不要微调**：

### 10.1 数据量不够

少于 500 条高质量样本。这种规模微调出来的模型，效果不会比 few-shot prompt 好。

**替代**：把这 500 条做成 RAG 知识库，或者塞进 system prompt 当 examples。

### 10.2 你的需求只是知识更新

「让模型知道我们公司有 XX 产品」「让它认识新 API」——**这是 RAG 的活儿**。

微调适合教模型**怎么答**（风格、格式、推理路径），不适合教**它知道什么**。强行用微调灌知识：

- 知识幻觉率高（模型会编）
- 更新成本高（公司改产品名你要重新训）
- 容量上限低（一个 7B QLoRA 大约只能稳定记住几百条事实）

### 10.3 业务在快速迭代

需求每周变。微调 + 评估 + 部署一轮至少 3–5 天，等你训完业务又改了。

**替代**：先用 prompt 跑通业务、收集真实数据 3–6 个月、再谈微调。

### 10.4 你还没把 prompt 工程做完

90% 的人开口就要微调，本质上是 prompt 没调好。检查：

- 你试过 few-shot 了吗？
- 你试过 chain-of-thought 了吗？
- 你试过把 instruction 拆成多个 step 让模型分步答了吗？
- 你试过换更强的基座（Claude / GPT-4o / DeepSeek-V3）了吗？

**这些都做完还不行，再考虑微调**。

### 10.5 任务本身就是闭源大模型的强项

写代码、写营销文案、做数据分析这种任务，闭源模型 2026 年的水平，开源 7B 微调短期内追不上。

**替代**：直接调闭源 API，或者用 RAG / Tool Use 加强。

### 10.6 你是在追时髦

「公司要有自己的模型」「老板看新闻说要做大模型」——这种动机微调出来的项目几乎全部沉没。先把 KPI 想清楚：你期望微调让 **哪个具体业务指标** 提升 **多少个百分点**？答不出来就别开训。

### 10.7 该微调的场景长这样

反过来，下面这些是值得微调的：

- **垂直领域 + 强格式**：医疗病历摘要、法律合同条款抽取、SQL 生成。这类任务格式严苛、领域术语密集，prompt 拉不动。
- **品牌调性**：客服、IP 形象的对话风格。
- **隐私 / 合规要求本地部署**：金融、医疗、政府场景，必须断网，闭源 API 用不了。
- **海量调用场景的成本**：每月几亿次调用，把它从 GPT-4 换成自己的 7B 微调，成本能砍 90%+。
- **小语种 / 方言**：闭源模型在某些非英文语言上仍弱，你的训练数据反而能补这个空。

**判断口诀**：「这事我用 prompt 做过 100 次，每次都做对，但放规模成本受不了，且我的数据已经攒到几千条」——这种时候，微调。

### 10.8 一个判断流程图

把上面所有的"该不该微调"提炼成一张决策树，下次客户问起你照着走：

```text
                    "我们要不要微调？"
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
        是知识问题吗？          是风格/格式问题吗？
        (用户问 "什么是 X"     (希望模型按某种调性
         "公司有没有 Y")        或固定格式答)
              │                     │
              ▼                     ▼
            用 RAG               继续往下看 ↓
        别浪费时间训
                                    │
                    ┌───────────────┴───────────────┐
                    ▼                               ▼

            （高质量、有标签）              （或者全是脏数据）
                    │                               │
                    ▼                               ▼

                                          (3-6 个月线上日志)
                    │
```

### 10.9 客户来咨询时的标准应答模板

把上面这些总结成一段话，客户问"我们也要做大模型微调"时你可以这样答：

> 等下，先回答我三个问题：
>
> 1. 你的数据是知识，还是风格/格式问题？前者用 RAG，后者才考虑微调。
> 2. 你期望微调让哪一个具体业务指标提升多少？答不出来就是没想清楚。
> 3. 你最近 3 个月的日志攒下来了吗？训练数据从哪来？凭空造的不算。
>
> 这三个问题答得出来，咱们再聊技术选型。答不出来，先做 prompt 工程 + RAG，跑半年再回来。

这段话能挡掉 80% 不靠谱的微调需求，让你的精力留给真正值得做的项目。

## 11 实战案例：把 Qwen3-8B 训成一个客服模型

【建议】

把前面所有的零件拼成一个真实流程。背景设定：你是一家中等规模电商的工程师，要把通用大模型微调成符合品牌调性的客服助手。

### 11.1 阶段 0：搞清楚要解什么

老板的诉求是"自家模型代替 GPT-4 客服 API，省钱"。但你拆解之后真正的痛点是：

- GPT-4 答得太啰嗦（用户嫌麻烦）
- GPT-4 偶尔会拒答业务问题（说"建议联系客服"——你就是客服）
- 流量大，每月 API 费 8 万元

把这些翻译成 SLA：

- 答案首句必须是问候 + 直接答（不绕弯）
- 涉及订单 / 物流的问题 100% 给出"请提供订单号"的引导
- 涉及退换货走《消费者权益保护法》7 天无理由话术，禁止说"看公司政策"
- 平均响应 token 数 < 200（GPT-4 现在是 350）

这个 SLA 是后面所有评估的标尺。**没 SLA 就训不动**。

### 11.2 阶段 1：数据（占 60% 时间）

第一周全部砸在数据上：

**Day 1–2：种子样本**

抓 200 条历史人工客服对话日志，请客服 leader 改写出"理想答案"。这一步必须人工，不能让 AI 改——因为你要的是品牌真实声线。

**Day 3：分类**

把 200 条按 query 类型打标签，发现分布：

| 类型 | 占比 |
|---|---|
| 订单查询 | 35% |
| 物流问题 | 22% |
| 退换货 | 18% |
| 商品咨询 | 15% |
| 投诉抱怨 | 6% |
| 闲聊 | 4% |

**Day 4–6：扩写**

按 1:50 的比例用 GPT-4o 扩写。给 AI 的 prompt 里强调：

```text
你将看到一条客服对话样本。请保持以下不变：
- 答案首句格式（问候+直接答）
- 涉及订单类必加"请提供订单号"
- 退换货必引《消费者权益保护法》七天

请基于这条样本，生成 50 条同类型但表述不同的（query, response）对。
要求：query 来自不同情绪（着急、友好、抱怨），response 风格统一。
```

跑完得到 ~10000 条，但还要清洗。

**Day 7：清洗**

按 §4.6 的 7 步清洗，用上面的 Python 模板跑一遍。剩 7800 条。再人工抽 5%（约 400 条）review，发现 3% 有事实错误（GPT-4 编了一些不存在的优惠政策），全删。最终 7560 条。

切 9:1，得到 train 6804 条 / eval 756 条。

### 11.3 阶段 2：基线评估

先把基座模型在 eval 集上跑一次：

```python
# baseline.py
import json
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="x")
eval_set = [json.loads(l) for l in open("data/eval.jsonl")]

results = []
for item in eval_set:
    user_q = item["messages"][-2]["content"]
    expected = item["messages"][-1]["content"]
    resp = client.chat.completions.create(
        model="qwen3:8b",
        messages=[{"role": "user", "content": user_q}],
        temperature=0.0,
    ).choices[0].message.content
    results.append({"query": user_q, "expected": expected, "actual": resp})

# 简单规则计算 SLA
hit_greeting = sum(1 for r in results if r["actual"].startswith(("您好", "亲", "感谢")))
hit_orderid = sum(1 for r in results if "订单查询" in r["query"] and "订单号" in r["actual"])
avg_len = sum(len(r["actual"]) for r in results) / len(results)
print(f"问候达标率: {hit_greeting/len(results):.2%}")
print(f"订单引导达标率: {hit_orderid/sum(1 for r in results if '订单查询' in r['query']):.2%}")
print(f"平均长度: {avg_len:.0f}")
```

跑完得到基线（虚构数字）：

- 问候达标率：62%
- 订单引导达标率：71%
- 平均长度：347 字
- CMMLU 5-shot：71.4

记下来。

### 11.4 阶段 3：SFT

用 §5.2 的脚本，r=16, lr=2e-4, 3 epoch。在 RTX 4090 上跑 2 小时。

训完跑评估：

- 问候达标率：93% ✓
- 订单引导达标率：91% ✓
- 平均长度：187 字 ✓
- CMMLU：68.9（降 2.5 分，可接受）

业务指标已经达标，但抽样人工看，发现新的问题：

- 涉及退换货时偶尔答得很硬，缺人情味
- 投诉类的回复有点冷，没有"理解您的心情"这种共情

这是典型的 **SFT 学会了形但没学到味**的情况。该上 DPO 了。

### 11.5 阶段 4：DPO 偏好数据

造法：

1. 拿训练数据里"退换货 + 投诉"两类抽 800 条 prompt
2. 让 SFT 模型对每条 prompt 生成 4 条回复（temperature=0.9）
3. 用 GPT-4o 当 judge，挑出最好和最差的两条，构成 (chosen, rejected)
4. 同时塞 200 条人工标注的高难度 case

得到 1000 条偏好对。

### 11.6 阶段 5：DPO 训练

用 §6.3 的脚本，lr=5e-6, beta=0.1, 1 epoch。RTX 4090 上 40 分钟。

监控指标：

- rewards/margin 从 0 升到 0.8（健康）
- rewards/accuracy 从 0.5 升到 0.79（健康，未过拟合）

跑评估：

- 问候达标率：94%
- 订单引导达标率：92%
- 平均长度：195 字
- CMMLU：68.6（没掉）
- LLM-as-judge 共情度：从 6.2 升到 8.1 ✓

### 11.7 阶段 6：合并部署

用 §8.1 的脚本：

```bash
python merge_and_save.py        # 合并 LoRA, 导出 q4_k_m
ollama create kefu-v1 -f Modelfile
ollama serve
```

然后做 ABTest。先 5% 流量打到新模型，跟 GPT-4 对比。一周后看：

- 用户满意度（结束对话时点赞率）：从 78% 升到 85%
- 平均会话轮数：从 3.4 降到 2.1（用户不需要追问就答得清）
- API 成本：每月 8 万降到 1.5 万（电费 + 自有 GPU 折旧）

老板满意，扩量到 100%。

### 11.8 阶段 7：飞轮

线上跑起来后，每天日志里有：

- 用户点踩的回复（约 0.5%）
- 用户改写后的提交（约 1.2%）
- 转人工后人工的真实答案（约 2%）

每两周用这些数据做一次 mini-DPO（500–2000 条偏好对，1 epoch），让模型持续靠近用户偏好。这就是飞轮。

### 11.9 这个案例里你应该记住的事

- **SLA 比超参重要**：先把"成功"定义清楚，再训。
- **数据 7 天，训练 4 小时**：时间分配大头永远在数据。
- **SFT 解形，DPO 解味**：知道什么问题用哪个工具。
- **基线 → SFT → DPO 是默认路径**：不要跳步。
- **飞轮才是真护城河**：训一次只是入场券，持续训才是壁垒。

## 13 习题（含解答）

【建议】

**Q1.** 你拿到一台 RTX 4090（24G），需要微调 Qwen3-14B 用于客户的合同条款抽取。预计训练数据 8k 条。请给出方法选择和关键超参。

<details>
<summary>参考答案</summary>

- 方法：**QLoRA**（14B fp16 ≈ 28 GB，必须 4-bit）
- r=16, alpha=16, target=all-linear, use_dora=True, lora_dropout=0.05
- max_seq_length=2048（合同段落用 4096 也可，但 batch 要压）
- per_device_train_batch_size=1, gradient_accumulation_steps=16（等效 16）
- lr=2e-4, cosine, warmup 0.05, 3 epoch
- 评估集独立 500 条，跑业务级 + CMMLU 抽样 + GPT-4 judge
- 训练时间预计 3–5 小时

</details>

**Q2.** SFT 训完后，你在 MMLU 上发现成绩比基座掉了 8 分。给出至少三种修复方案。

<details>
<summary>参考答案</summary>

1. 训练集里掺 20% 通用样本（Alpaca-zh / ShareGPT），做 data replay，让基础能力的信号不丢
2. 降 lr：2e-4 → 5e-5；或者减 epoch：3 → 1
3. r 调小：16 → 8；或者只在 attention 上加 LoRA，不打 MLP（容量小，遗忘少）
4. 用 OPLoRA / orthogonal projection 类的更先进方法
5. 检查训练数据质量：是不是答案里大量出现领域 jargon 把模型推向了一个狭窄分布

</details>

**Q3.** 解释为什么 DPO 的学习率比 SFT 低 100 倍。

<details>
<summary>参考答案</summary>

DPO 的梯度信号本质是「让 chosen 概率上升、rejected 下降」，这个信号比 SFT 的"按 token 拟合"要强得多——SFT 每步都在按 cross-entropy 推参数，DPO 一步可能让一个回复的概率变化几个数量级。如果 lr 设和 SFT 一样大，模型会很快把 chosen reward 推到极端、把 rejected 推到极端，破坏语言流畅性，导致幻觉。低 lr 是为了让 DPO 的更新「细水长流」、不破坏 SFT 已经学到的通用能力。

</details>

**Q4.** 公司想做一个法律咨询助手，主要回答《民法典》相关问题。同事说要微调。你的判断和建议？

<details>
<summary>参考答案</summary>

**先不要微调**。理由：

- 法律是「知识更新型」需求——用户问的核心是事实知识，不是风格。这是 RAG 的强项。
- 法律内容更新频繁（司法解释、案例），微调每次更新成本高。
- 法律风险高，幻觉成本不可接受。RAG 能给出引用，更利于审计。

建议路径：

1. 把《民法典》全文 + 司法解释 + 高质量案例做成 RAG 知识库
2. 用强基座（DeepSeek-V3 / Qwen3-72B）+ 检索 + COT prompt
3. 跑 3 个月收集真实问答数据
4. 如果发现稳定有「风格 / 格式」问题（比如要求出案号、要求列法条编号），**再**做微调，目标是格式而不是知识

</details>

## 15 给团队的建议：从一个人到一个流水线

【建议】

最后说一些不写在论文里、但实际项目中决定生死的事。

### 15.1 团队角色

一个微调项目跑顺，至少需要三个人（可以一个人兼）：

- **数据 owner**：负责数据收集、清洗、标注规范、质量评分。这个角色的产出决定模型上限。
- **训练 owner**：负责调参、跑实验、维护训练管线。这个角色的产出决定能不能稳定复现。
- **评估 owner**：负责评估集设计、指标定义、ABTest 流程。这个角色的产出决定上线决策对不对。

新手最容易把三个角色挤到一个人头上，然后偏向"训练 owner"那一段——天天调 lr、调 r、调 epoch——结果数据没洗、评估没做，最后训出一堆 checkpoint 不知道哪个能上。

**忠告**：如果一定一个人做，把时间分配按 5:2:3 给数据 / 训练 / 评估，不要超过这个比例去抠超参。

### 15.2 实验管理

每跑一次训练，至少记录这些东西到一个表格（Wandb / Notion / Excel 都行）：

| 字段 | 例子 |
|---|---|
| 实验编号 | `2026-05-08_001` |
| 基座模型 | `Qwen/Qwen3-8B`（原始版本，含 thinking / non-thinking 双模式） |
| 方法 | QLoRA r=16 alpha=16 dora=true |
| 数据 | `train_v3.jsonl` (6804 条) |
| 关键超参 | lr=2e-4, 3 epoch, eff_batch=16 |
| 训练时长 | 2h 11m on RTX 4090 |
| 业务任务得分 | 87.3 |
| CMMLU | 68.9 |
| 上线决策 | 是/否，理由 |
| LoRA 路径 | `out/exp001/lora_final` |

跑过 30+ 次实验后，这张表就是你的"知识库"——你能直接看出哪种数据 + 哪种参数 = 哪种结果。没这张表，30 次实验跑完你也总结不出规律。

### 15.3 失败原因复盘清单

每次实验跑完不达预期，按这个顺序排查（按概率从高到低）：

1. 数据有问题（85%）：脏数据、格式不对、量不够、分布偏。
2. 评估方法有问题（10%）：评估集和训练集泄漏；指标定义不合理。
3. 超参不对（4%）：lr / epoch / r 偏离常识范围。
4. 代码 bug（1%）：loss mask 不对、tokenizer 不匹配。

**不要先怀疑超参**。新手 90% 时间都花在调超参上，但 90% 的失败原因其实是数据。

### 15.4 一些额外的工程实践

- **训练前在 100 条小样本上跑一遍**：完整跑 1 万条之前，先用 100 条跑 5 步，看 loss 是不是下降、看输出是不是正常。这能在 10 分钟内 catch 95% 的代码 bug。
- **保留 base model snapshot**：万一你训坏了想从头再来，至少别再下载一次基座（10–30 GB）。
- **存所有 checkpoint 但只保留 3 个**：每个 step 都存太占磁盘，但一次 train 至少留 3 个不同时间点（中期、末尾、最佳），方便回退。
- **训练 log 推到对象存储**：Wandb / S3 都行。本地磁盘炸了不会丢实验记录。
- **写 reproduce 脚本**：训练命令 + 数据版本 + 环境锁定（requirements.lock）。3 个月后老板说"再训一次"你能复现。

## 16 一句话总结

微调不是炫技、不是赶时髦，是把基座模型推到你业务边缘的一种**工程手段**。它的上游是数据和评估，下游是部署和反馈。所有"训练"那一段——LoRA 还是 QLoRA、DPO 还是 ORPO、r 取 16 还是 32——只是中间一小段。把数据和评估做扎实，剩下的细节用本章给你的默认值，绝大多数业务问题都能解到 80 分。剩下的 20 分，留给你做完三轮才会感受到的那种"哎，这模型确实是我们公司的"的手感。
