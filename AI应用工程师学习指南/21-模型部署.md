# 第 21 章 · 模型部署：vLLM、Ollama 与生产推理

> 训练一个模型，三天三夜烧 GPU，烧出来的权重是个文件。
>
> 把这个文件变成一个能扛 1000 QPS、p99 延迟 800ms 以内、半夜 OOM 自动恢复的服务，是另外一回事。
>
> 这一章只讲后半件事。前半件事的产物——一个 GGUF / safetensors / HF 仓库——是这一章的输入。
>
> 这一章给你的能力是：拿到一个开源权重，知道在你的机器上该跑哪个引擎、怎么压测、什么时候该量化、怎么扛住流量峰值、什么时候自建什么时候直接调 API。

---

## 21.1 推理服务的三个核心问题

把模型上线，工程上要回答三个问题：

1. **延迟要多低**。聊天场景用户对 TTFT（first token 到达时间）敏感，超过 1 秒就开始觉得慢；批处理场景没人盯着屏幕，整体吞吐才是钱。
2. **吞吐要多高**。同样一张 H100，跑 Llama-3.1-8B 可以做到 8000 token/s，可以服务几百个并发用户；如果你只想服务自己一个人，上 H100 是浪费。
3. **预算多少**。一张 H100 SXM 包月按云价大约 $2~3/小时，B200 按 $2~4/小时；自购一台 8 卡 H100 整机一百多万人民币。如果你的流量月不到十亿 token，几乎所有情况下都是直接调 API 划算。

这三个问题决定后面所有技术选型。把它们想清楚再去看引擎对比，不然容易陷入"哪个最快"的伪问题——没有"最快"，只有"在你的负载和预算下最划算"。

一个完整的推理服务，从外到内有四层：

```
[ 客户端 / 业务后端 ]
        ↓ HTTPS
[ 网关层（Nginx / Envoy）：TLS、限流、鉴权、日志 ]
        ↓
[ 路由层（vllm-router / Envoy weighted）：基于队列深度选实例 ]
        ↓
[ 推理引擎层（vLLM / SGLang / TensorRT-LLM）：N 个副本 ]
        ↓
[ GPU + 模型权重 ]
```

外两层和普通 web 服务没区别，难度是从路由层开始。LLM 请求和 HTTP API 请求很不一样——它的处理时间不是 ms 级而是 s 级、它的资源消耗是按 token 数动态变化的、它的输出是流式不是一次性的。这意味着一切传统 LB 假设（round-robin、least-conn、固定 timeout）都需要重新考虑。

下面先把整个推理引擎全景过一遍，再讲底层优化原理，再到实战。

## 21.2 推理引擎全景：六大流派

这两年推理引擎卷得厉害。按设计目标和适用场景，可以分成六类。

### 21.2.1 vLLM（生产首选，PagedAttention 鼻祖）

UC Berkeley 的 SkyLab 在 2023 年发了 PagedAttention 那篇论文之后，vLLM 就一路成了开源推理的事实标准。它的特点：

- **PagedAttention**：把 KV cache 切成固定大小的 page，像操作系统管内存一样管 GPU 显存，碎片率从原来的 60-80% 降到 4% 以下。这是 vLLM 第一次火起来的根本原因。
- **Continuous Batching**：迭代级调度。一个 batch 里的某个请求先生成完了，立即把这个槽位让给排队的新请求，不用等整个 batch 都做完。
- **OpenAI 兼容 API**：直接 `vllm serve <model>` 就起一个 `/v1/chat/completions` 端点，前端代码和调 OpenAI 几乎不用改。
- **V1 引擎**：2025 年默认开启，把 prefill 和 decode 阶段独立调度，配合 chunked prefill，长 prompt 不再阻塞短请求的 decode。
- **Multi-LoRA**：单卡可以同时挂几十甚至上百个 LoRA adapter，按请求路由不同 adapter，热替换。
- **生态**：HF 模型 Day 0 支持基本是 vLLM 兜底，Llama / Qwen / DeepSeek / GPT-OSS / Mistral 全部官方适配。

什么时候选 vLLM：你不知道选啥就选它。除非有特殊理由（比如要 RAG 共享前缀、要纯 NVIDIA 极致优化），vLLM 是 80% 场景的合理默认值。

### 21.2.2 SGLang（新秀，结构化输出和共享前缀强）

LMSYS 团队搞的，本来是给 ChatBot Arena 做后端的。它的杀手锏是 **RadixAttention**：把 KV cache 组织成 radix 树（按 token 索引），多个请求共享相同前缀时直接复用 KV，不用重新算。

- **多轮对话场景**：用户每发一条新消息，前面的整段对话历史都是共享前缀。SGLang 比 vLLM 在这种场景能快 10-30%。
- **RAG 场景**：长 system prompt + 检索到的文档块，跨请求高度重叠。SGLang 命中率比 vLLM 的 block-level prefix cache 更高。
- **结构化输出**：内置 grammar 约束的 JSON / regex / 正则文法解码，生成结构化数据时比 vLLM 的 outlines 集成更快。
- **吞吐**：在 H100 上和 LMDeploy 并列，跑 Llama-3.1-8B 大约 16000 token/s，比 vLLM 的约 12500 高 28%。
- **冷启动**：和 vLLM 差不多，60 秒级。

什么时候选 SGLang：RAG / Agent / 多轮对话场景，明显比 vLLM 快。或者你需要把生成结果约束成严格 JSON，又不想额外做后处理。

### 21.2.3 TensorRT-LLM（NVIDIA 极致优化）

NVIDIA 自己出的推理后端，是 TensorRT 的 LLM 版本。核心是把 PyTorch 模型编译成 NVIDIA 自家的 engine 文件，跑的时候完全脱离 Python，全是手写 CUDA kernel。

优点：
- 在 H100 / H200 / B200 上吞吐和延迟基本是天花板。比 vLLM 高 15-30%，开了 speculative decoding 可以再快 2-3.6 倍。
- 对 FP8 / FP4 量化、Multi-Head Latent Attention、各种新架构的 kernel 适配比开源圈子早一两个月。
- 集成 TensorRT-LLM Triton backend，和 NVIDIA 的整套企业部署栈无缝对接。

缺点：
- 编译模型要 20-30 分钟，模型一改就重编。
- 配置复杂，不是一句 `serve` 就完事，要写 engine config、要懂 plugin。
- 几乎只支持 NVIDIA 卡。
- 模型结构改了要等 NVIDIA 的 PR，社区贡献门槛比 vLLM 高。

什么时候选 TensorRT-LLM：你有稳定的模型版本（半年不换），单位 token 成本压到极致比开发体验更重要，团队有 NVIDIA 工程能力。否则，老老实实用 vLLM。

### 21.2.4 TGI（HuggingFace Text Generation Inference）

HuggingFace 的官方推理后端，2023 年早期是 vLLM 的主要竞品。

现在的状况：
- 核心特性都有：continuous batching、PagedAttention、streaming、量化支持。
- TPOT、TTFT 在多数 benchmark 里比 vLLM 慢一点（约 30-50%），社区资源向 vLLM 倾斜。
- 优势是和 HF 生态绑定深，HF Inference Endpoints、Spaces 全用 TGI 做后端。

什么时候选 TGI：你已经在 HF 生态里深度集成，团队对 HF 的工具链熟悉。新项目从零起步建议直接 vLLM。

### 21.2.5 llama.cpp / Ollama（端侧 / CPU / 小机器）

llama.cpp 是 Georgi Gerganov 一个人搞起来的 C++ 项目。最初的目标是在 MacBook 上跑 LLaMA，现在已经变成端侧推理事实标准。

它的特征：
- 纯 C/C++，零 Python 依赖，二进制几 MB。
- GGUF 格式 + K-quants 量化，把 7B 模型压到 4GB 以内，在 16GB 内存的笔记本上跑得动。
- Metal / CUDA / ROCm / Vulkan 全后端支持。
- Apple Silicon 上原生 Metal 加速，M4 Max 跑 70B 量化模型可以做到 30+ token/s。
- 适合单用户、低并发、本地隐私场景。

Ollama 是 llama.cpp 的"docker for LLM"封装：
- 一行命令拉模型：`ollama pull qwen3:8b`。
- 自带 OpenAI 兼容 API。
- 模型仓库托管，用 `Modelfile` 定义 system prompt 和参数。
- 支持 `OLLAMA_NUM_PARALLEL` 并发，单 4090 跑 8B 模型大约 100+ token/s，可以扛 4-8 个并发用户。
- 2025 年开始 Apple Silicon 上接 MLX 后端，Mac 性能再上一档。

什么时候用：本地开发、内网部署、单机服务几十个员工的场景。生产服务上百并发不要用 Ollama，用 vLLM。

### 21.2.6 MLC-LLM / MLX（移动端 / Apple）

MLC-LLM（CMU 出品）：基于 TVM 的跨平台部署，目标是把 LLM 跑到手机和浏览器上。WebGPU 后端能让你在 Chrome 标签页里跑 Llama-3.1-8B。

MLX（Apple 官方）：Apple 自己的端侧 ML 框架，类 PyTorch API，针对 Apple Silicon 的统一内存架构优化。在 M4 Max 上跑同款模型，MLX 能比 llama.cpp 快 2 倍以上，比 Ollama 默认后端快 3 倍。Apple 在 WWDC 2025 把 MLX 钦定为 Apple Silicon 上 LLM 推理的首选。

什么时候用：你要做手机 App 内的本地 LLM、浏览器侧的小型助手、或者你团队是 Mac 重度用户。

### 21.2.7 选型决策表

把上面六种引擎放到一张表上，按场景看：

| 场景 | 首选 | 备选 | 理由 |
|------|------|------|------|
| 通用生产 API 服务 | vLLM | SGLang | 生态、稳定、Day 0 模型支持 |
| RAG / 多轮对话 / Agent | SGLang | vLLM | RadixAttention 共享前缀复用 |
| 极致单位 token 成本（NVIDIA 重度用户） | TensorRT-LLM | vLLM | 编译优化天花板 |
| HF 生态深度集成 | TGI | vLLM | 工具链一致 |
| 单机本地部署、内网团队工具 | Ollama | llama.cpp | 简单、零运维 |
| 端侧 Mac 开发机 | MLX / Ollama-MLX | llama.cpp | 统一内存架构利用 |

| 国产卡（昇腾 / 寒武纪 / 海光） | vllm-ascend / lmdeploy | TGI | Day 0 适配能力 |

实战建议：**起步阶段一律 vLLM**，跑通整个链路（发版、压测、监控、扩缩容）后，再根据具体瓶颈换引擎。换引擎的成本比一开始就选错小得多，因为 OpenAI 兼容 API 抽象掉了大部分差异。

## 21.3 推理优化原理：六个核心技术

不懂原理只会调参数等于不会调。这一节把 LLM 推理服务的六个核心优化讲清楚——把原理弄懂，看到一个新引擎的 README 就能立刻判断它强不强。

### 21.3.1 KV Cache：为什么生成是 memory-bound 的

Transformer 的 attention 每次计算都要拿当前 token 的 query 去和**所有历史 token 的 key/value** 做点积。如果每生成一个 token 都把前面所有 token 重新过一遍，计算量是 O(n²)，完全不可行。

KV Cache 的做法是：每个 token 第一次计算出来的 K 和 V 缓存住，后面再生成新 token 时只算这一个新位置的 QKV，旧的直接读缓存。这样每生成一个 token 就只是 O(n) 的 attention 计算。

代价是显存。对 Llama-3.1-8B 来说，每个 token 的 KV cache 大约 128KB，一个 4K 上下文的请求就要 512MB，10 个并发请求要 5GB。70B 模型这个数字还要乘 10 倍。**显存对推理服务的瓶颈意义大于算力**——你的卡跑得多快，取决于能塞多少个并发请求进去，而不是 GPU TFLOPS。

具体公式记下来，做容量规划用得到。一个 token 的 KV cache 字节数：

```
kv_size_per_token = 2 * num_layers * num_kv_heads * head_dim * dtype_bytes
```

例子：Llama-3.1-70B 有 80 层，KV head 数 8（Grouped Query Attention），head_dim 128，FP16 是 2 字节，所以一个 token 占用 `2 × 80 × 8 × 128 × 2 = 327680 字节 ≈ 320KB`。一个 8K 上下文的请求就要 2.5GB KV cache。

这就解释了为什么 70B 模型在 H100 80G 上 BF16 精度下并发上不去——权重本身占 140GB（超出单卡），即使分到 2 卡上，每卡剩 70G - 模型权重，KV cache 预算非常紧。这也是为什么 H200 (141GB) 和 B200 (192GB) 比 H100 在长上下文场景能多一个数量级的并发：显存就是命脉。

### 21.3.2 PagedAttention：把 KV cache 当虚拟内存管

传统做法：给每个请求预分配一段连续显存放 KV cache，按 max_seq_len 留满。问题是大部分请求达不到 max_seq_len，剩下的显存被锁死，浪费率 60-80%。

PagedAttention 的做法是借鉴操作系统的虚拟内存：
- KV cache 切成固定大小（vLLM 默认 16 个 token 一块）的 page。
- 每个请求维护一个"页表"，把逻辑位置映射到实际物理 page。
- 物理 page 不要求连续。生成一个新 token 显存够用就分一个 page，不够用就 swap 或者抢占。
- 多个请求共享相同前缀时，page 可以共享（用 ref count）。

效果：KV cache 显存利用率从 20-40% 升到 96%+，单卡能塞的并发请求数翻 2-4 倍。

这是 vLLM 第一次起飞的关键。后来 SGLang 的 RadixAttention 是 PagedAttention 之上再加一层 token 级的前缀复用。

### 21.3.3 Continuous Batching：迭代级调度

最朴素的 batching：把 N 个请求凑成一个 batch，整个 batch 一起做完再放下一批。问题是 LLM 生成是迭代的，N 个请求生成长度不一样，先做完的那个要等最长的那个，GPU 利用率低。

Continuous Batching（也叫 in-flight batching）的做法是按**生成步**调度，不是按**请求**调度：
- 每一步迭代结束后，已经生成 EOS 的请求出 batch。
- 队列里排队的新请求立刻补进 batch 的空位。
- batch 的大小动态变化，但 GPU 始终满载。

这是 vLLM 和 TGI 都实现的标准技术。Continuous Batching 让吞吐量比传统 static batching 提升 2-5 倍。

### 21.3.4 Chunked Prefill：让长 prompt 不要阻塞短请求

LLM 推理有两个阶段：
- **Prefill**：把整个 prompt 一次性算 KV cache，是 compute-bound 的，并行度高。
- **Decode**：一次生成一个 token，每生成一个就和 KV cache 做 attention，是 memory-bound 的。

问题：一个 8K token 的长 prompt 进来，prefill 阶段要占用 GPU 几百 ms，期间所有正在 decode 的请求都被阻塞，TTFT 飙升。

Chunked Prefill 的做法：把长 prompt 切成小块（比如每块 512 token），分多步处理，每步处理一小块 prefill + 同时给所有正在 decode 的请求多生成一步 token。这样长请求不会饿死短请求，p99 TTFT 显著降低。

vLLM V1 默认开启，参数 `--enable-chunked-prefill`。

### 21.3.5 Speculative Decoding：让一次生成多个 token

LLM decode 是逐 token 串行的，每一步都要把 70B 参数从 HBM 读到 SM，这是 memory bandwidth 的瓶颈。

Speculative Decoding 的核心思路：
1. 用一个**小的 draft model**（或者目标模型本身的 lightweight head）一次生成 K 个 token 的"猜测"。
2. 用大模型一次性 verify 这 K 个猜测——只需要一次 forward pass。
3. 找到第一个被拒绝的 token，截断；接受的部分作为已生成。

数学上是无损的——只要 verify 用拒绝采样，最终输出分布和直接用大模型完全一致。

主流变体：
- **Vanilla Speculative**：用一个独立的小模型当 drafter（比如 Llama 70B 配 Llama 1B 当 draft）。
- **Medusa**：在目标模型上加几个 LM head，并行预测后面几个位置。
- **EAGLE-3**（2025-2026 的 SOTA）：用目标模型的内部 hidden state 训一个轻量自回归头，acceptance rate 可以做到 0.7+，整体加速 2-6 倍。

什么时候用：单用户低并发场景效果最显著（batch=1 时 GPU 闲置算力多，spec decoding 直接拿来用）。高并发场景（batch=64+）GPU 已经很忙，spec decoding 收益变小。

vLLM 开启：`--speculative_model meta-llama/Llama-3.2-1B --num_speculative_tokens 5`。

实测一组数据（H100 跑 Llama-3.1-70B + 1B drafter，ShareGPT 数据集）：
- batch=1：TTFT 不变，TPOT 从 22ms 降到 9ms，整体延迟降 2.4 倍。
- batch=8：TPOT 从 25ms 降到 14ms，加速 1.8 倍。
- batch=64：TPOT 从 35ms 降到 32ms，加速仅 1.1 倍。

所以 spec decoding 是**单用户/低并发杀器，高并发场景收益低**。生产里要不要开取决于业务画像：聊天助手类（很多并发但每个用户感知重要）开了往往值得；批处理类（吞吐优先）开了反而可能拖累。

### 21.3.6 Prefix Caching：跨请求复用 KV

如果你的服务是 RAG 或者 Agent，每个请求前面都有几千 token 的 system prompt + 工具说明，每次都重新算这一段 prefill 是巨大的浪费。

Prefix Caching 的做法：把已经算过的 KV cache 按 hash key 缓存住，新请求来的时候 prefix 部分直接命中缓存，跳过 prefill。

两种实现：
- **vLLM 的 block-level**：按 page（16 token）粒度 hash，命中要求边界对齐。
- **SGLang 的 RadixAttention**：按 token 级 radix 树索引，命中粒度细，shared prefix 自动检测。

实测：长 system prompt 的多轮对话场景，TTFT 能降 50-80%。开启方式：vLLM 加 `--enable-prefix-caching`，SGLang 默认开。

举一个具体数字感受下：一个 RAG 服务 system prompt 是 2000 token，每个用户 query 增加 1000 token 上下文。冷启动时第一个请求要算 3000 token 的 prefill，假设 H100 上 FP16 prefill 速度是 5000 token/s，TTFT 就是 600ms。命中前缀缓存后，只需要算新增的 1000 token，TTFT 降到 200ms。

更进一步，Agent 场景每一步都包含整个 ReAct 历史（system prompt + 工具说明 + N 轮思考与观察），每一步增量只有几十 token，prefix cache 命中率能到 95%+，TTFT 几乎只剩 decode 第一个 token 的时间（< 50ms）。这是为什么生产 Agent 服务**必须开 prefix caching**——不然每一步都重算几千 token 的历史，体验直接崩。

### 21.3.7 优化技术叠加效应

把上述六种技术放一起会出现非线性效果。给一组 H100 单卡跑 Llama-3.1-8B 的实测对比（数据来自 vLLM 官方 benchmark + 笔者实测合并）：

| 配置 | 吞吐 (token/s) | TTFT-p50 | TPOT-p50 |
|------|---------------|----------|----------|
| baseline（FP16，static batching，no KV cache）| 80 | 4500ms | 22ms |
| + KV cache | 1200 | 320ms | 18ms |
| + Continuous batching | 4500 | 280ms | 17ms |
| + PagedAttention | 7800 | 250ms | 16ms |
| + Chunked prefill | 8200 | 180ms | 16ms |
| + Prefix caching（命中场景）| 11500 | 50ms | 16ms |
| + FP8 quantization | 16800 | 30ms | 11ms |
| + Speculative decoding（batch=1）| - | 30ms | 4ms |

七层优化叠下来，对比最朴素实现是 200 倍以上的差距。这就是为什么开源推理引擎在过去三年迭代得这么凶——每一层优化都是工程师从论文里挖出来再死磕实现的。理解这些原理之后，你看 vLLM 的版本说明（"v0.7 默认开启 prefix caching"、"v0.10 GB200 上支持 disaggregated serving"）才能立刻判断对你的业务有没有意义。

## 21.4 量化：把模型压到能跑

量化是把模型权重从 FP16/BF16 压到 INT8 / INT4 / FP8 / FP4。两个目标：
1. **省显存**：FP16 → INT4 是 4 倍压缩，70B 模型从 140GB 降到 35GB，单卡 A100 80GB 就能跑。
2. **加速度**：INT4 在 Hopper 之前的卡上不一定真快（因为没有原生 kernel），但 FP8 在 H100/H200 上是硬件原生支持，吞吐直接翻倍。

这一节把主流量化方案捋一遍。

### 21.4.1 GGUF / K-quants（llama.cpp 系）

GGUF 是 llama.cpp 的模型文件格式，自带元数据 + tokenizer，单文件。量化方案：

| 量化等级 | 大小（7B 模型）| 质量 | 用法 |
|---------|--------------|------|------|
| Q2_K | 2.6 GB | 明显损失 | 极限省内存，质量优先放弃 |
| Q3_K_M | 3.3 GB | 有损失 | 边缘场景 |
| **Q4_K_M** | **4.1 GB** | **几乎无损** | **甜点位，强烈推荐** |
| Q5_K_M | 4.8 GB | 几乎无损 | 偏保守 |
| Q6_K | 5.5 GB | 无损 | 接近 FP16 |
| Q8_0 | 7.2 GB | 无损 | 量化基本只剩省显存意义 |

经验：Q4_K_M 是绝大多数本地部署场景的甜点位。perplexity 比 FP16 高 0.18，肉眼基本分辨不出来，体积压到 1/4。

I-quants（IQ2_S / IQ3_S 等）是 llama.cpp 后来加的方法，基于 importance matrix，同等大小下质量更好。Q5 以下推荐用 importance matrix 校准。

```bash
# 量化命令
./llama-quantize input-model-f16.gguf output-q4km.gguf q4_k_m
```

### 21.4.2 AWQ：激活感知权重量化

Activation-aware Weight Quantization。核心观察：不是所有权重都同等重要，被大激活值乘的那一小撮权重对模型质量影响巨大，量化时给这些权重保留更高精度。

特点：
- 主要做 INT4 weight-only 量化（激活仍是 FP16）。
- 校准过程比 GPTQ 快几倍。
- 在 vLLM / TensorRT-LLM 都有原生支持。
- 质量比 GPTQ 略好（多数 benchmark 上）。

什么时候用：你想把 70B 模型塞进单卡 H100 80G 又要保留质量。

### 21.4.3 GPTQ：基于二阶信息的逐层量化

利用 Hessian 矩阵的二阶信息逐层量化，理论上更精确。

特点：
- INT4 weight-only。
- 校准慢（几小时级别）。
- 工具链最成熟（autogptq、optimum）。
- 质量在多数 benchmark 上略低于 AWQ。

实际上 GPTQ 和 AWQ 是同一个生态位的两个选择，AWQ 较新一点，社区现在更倾向 AWQ。

### 21.4.4 SmoothQuant：W8A8 同时量化

把权重和激活都量化到 INT8（W8A8）。难点是激活的动态范围比权重大很多，直接量化掉点严重。SmoothQuant 的思路：把权重的"难量化性"和激活的"难量化性"互相平衡——给权重乘一个 scale，给激活除以同一个 scale，最终数学等价但量化更友好。

效果：INT8 的吞吐和精度都不错，但 H100 时代大家直接上 FP8 了，SmoothQuant 用得变少。

### 21.4.5 FP8：H100/H200/B200 时代的默认选项

FP8 是 8 位浮点，有两种格式：
- **E4M3**：4 位指数 + 3 位尾数，动态范围窄精度高，适合权重和激活。
- **E5M2**：5 位指数 + 2 位尾数，范围大精度低，适合梯度。

H100 / H200 / B200 都有原生 FP8 Tensor Core，吞吐是 FP16 的 2 倍。

FP8 量化的好处：
- 几乎无损（多数 benchmark 掉点 < 2%）。
- 显存减半（70B 从 140GB 降到 70GB）。
- 吞吐翻倍。
- 不需要复杂校准（per-tensor scale 就够）。

业内共识：**有 H100 及以上就上 FP8，没有疑问**。

vLLM 启动 FP8：
```bash
vllm serve meta-llama/Llama-3.1-70B-Instruct \
  --quantization fp8 \
  --tensor-parallel-size 2
```

### 21.4.6 NVFP4 / FP4：B200 时代

B200 引入了 FP4 Tensor Core（NVFP4 是 NVIDIA 自家变种）。比 FP8 再省一半显存、再快一倍。但 FP4 量化掉点比 FP8 大不少，需要 SmoothQuant 风格的校准 + per-block scale 才能压住质量。

2026 年初的状态：FP4 在 GPT-OSS-120B 这种大模型上效果可用，每 million token 成本可以压到 $0.02 级别（B200 上）；小于 30B 的模型用 FP4 性价比反而不如 FP8。

### 21.4.7 量化对效果的实际影响

不同量化对模型质量的影响（以 Llama-3.1-70B 为基准，跑 MMLU、GSM8K、HumanEval 平均掉点）：

| 量化 | 显存 | 质量损失 | 推荐场景 |
|------|------|---------|---------|
| FP16/BF16 | 140 GB | 基准 | 大显存集群 |
| FP8 | 70 GB | < 1% | H100/H200 生产 |
| AWQ INT4 | 35 GB | 1-2% | 单卡部署 |
| GPTQ INT4 | 35 GB | 1-3% | 单卡部署（替代 AWQ） |
| GGUF Q4_K_M | 38 GB | 1-2% | 本地 CPU/低端 GPU |
| FP4 | 35 GB | 2-4% | B200 大模型 |
| GGUF Q3_K_M | 28 GB | 3-6% | 极限省显存 |
| GGUF Q2_K | 22 GB | 8-15% | 不推荐生产 |

经验法则：**生产 H100 用 FP8，单卡 A100/4090 用 AWQ INT4，本地用 GGUF Q4_K_M**。Q2 / Q3 量化只有在你显存极端紧张又能接受质量退化时才用。

### 21.4.8 量化的实操流程

光知道方法不够，要会自己量化模型。下面是把一个 BF16 的 Llama-3.1-70B 量化到 FP8 的完整流程：

```bash
# 装 llm-compressor（vLLM 团队的官方量化工具）
uv pip install llmcompressor

# 写量化脚本
cat > quantize_fp8.py <<'EOF'
from llmcompressor.transformers import oneshot, SparseAutoModelForCausalLM
from llmcompressor.modifiers.quantization import QuantizationModifier
from transformers import AutoTokenizer

MODEL_ID = "meta-llama/Llama-3.1-70B-Instruct"
SAVE_DIR = "Llama-3.1-70B-Instruct-FP8"

# FP8 不需要校准数据集（dynamic activation quant）
recipe = QuantizationModifier(
    targets="Linear",
    scheme="FP8_DYNAMIC",
    ignore=["lm_head"],
)

model = SparseAutoModelForCausalLM.from_pretrained(MODEL_ID, torch_dtype="auto")
tokenizer = AutoTokenizer.from_pretrained(MODEL_ID)

oneshot(model=model, recipe=recipe, output_dir=SAVE_DIR)
tokenizer.save_pretrained(SAVE_DIR)
EOF

python quantize_fp8.py
# 70B 模型 FP8 量化大约 30 分钟（H100 单卡），产出 ~70GB safetensors

# 直接用 vLLM 加载
vllm serve ./Llama-3.1-70B-Instruct-FP8 \
  --quantization fp8 --tensor-parallel-size 2
```

AWQ INT4 量化要校准数据：

```python
from llmcompressor.transformers import oneshot
from llmcompressor.modifiers.quantization import GPTQModifier
from datasets import load_dataset

# 用 c4 数据集做校准（约 512 条样本足够）
ds = load_dataset("allenai/c4", "en", split="train", streaming=True)
calib = list(ds.take(512))
calib_text = [d["text"][:2048] for d in calib]

recipe = GPTQModifier(
    targets="Linear",
    scheme="W4A16",     # weight 4 bit, activation 16 bit
    ignore=["lm_head"],
    dampening_frac=0.01,
)

oneshot(
    model=MODEL_ID,
    dataset=calib_text,
    recipe=recipe,
    output_dir="./Llama-3.1-70B-W4A16",
    max_seq_length=2048,
    num_calibration_samples=512,
)
```

校准时间：70B 模型 W4A16 大约 2-3 小时（单 H100）。校准数据**要尽可能贴合你的业务分布**——如果你做中文场景就用中文校准集（COIG / OpenCSG 等），用纯英文 C4 校准的模型在中文上掉点会更明显。

### 21.5.3 完整生产启动脚本

下面是一个生产配置的启动脚本，每一行参数都讲了为什么。

```bash
#!/usr/bin/env bash
set -euo pipefail

# ============================================================
# vLLM 生产启动脚本
# 适用：8 卡 H100 服务器跑 Qwen3-72B FP8
# ============================================================

# 模型与硬件
MODEL="Qwen/Qwen3-72B-Instruct"
TP_SIZE=4              # 4 卡张量并行
DP_SIZE=2              # 2 路数据并行（共 8 卡）
GPU_MEM=0.92           # 给系统留 8% 显存

# 服务端口
HOST="0.0.0.0"
PORT=8000
API_KEY="${VLLM_API_KEY:-sk-local-dev}"  # 上线请用环境变量

# 性能参数
MAX_MODEL_LEN=32768            # 最大上下文长度
MAX_NUM_SEQS=256               # 最大并发数
MAX_NUM_BATCHED_TOKENS=8192    # 单步 batch 总 token 数

# 启动
exec vllm serve "$MODEL" \
  --host "$HOST" \
  --port "$PORT" \
  --api-key "$API_KEY" \
  --tensor-parallel-size "$TP_SIZE" \
  --data-parallel-size "$DP_SIZE" \
  --quantization fp8 \
  --kv-cache-dtype fp8 \
  --gpu-memory-utilization "$GPU_MEM" \
  --max-model-len "$MAX_MODEL_LEN" \
  --max-num-seqs "$MAX_NUM_SEQS" \
  --max-num-batched-tokens "$MAX_NUM_BATCHED_TOKENS" \
  --enable-prefix-caching \
  --enable-chunked-prefill \
  --disable-log-requests \
  --served-model-name qwen3-72b \
  --trust-remote-code
```

参数解读（背下来）：

- **`--tensor-parallel-size`**：张量并行卡数。把单层权重切到 N 张卡上，N 通常等于单节点 GPU 数。70B 模型 FP8 一张 H100 80G 装不下，要切 2-4 卡。
- **`--data-parallel-size`**：起 N 路同样的 vLLM 实例并行处理请求，每路独立负载。`TP×DP` 等于总 GPU 数。
- **`--quantization fp8`**：开启 FP8 量化，权重和激活都跑 8 位，吞吐翻倍。
- **`--kv-cache-dtype fp8`**：KV cache 也用 FP8 存（默认是 FP16），显存再省 50%，并发上限更高。
- **`--gpu-memory-utilization 0.92`**：vLLM 占 92% 显存，留 8% 给系统和其他进程。设太高（0.98）容易 OOM，太低浪费显存。
- **`--max-model-len 32768`**：硬性限制单请求最大 token 数。设太大每个请求 KV cache 预算大，并发上不去；设太小用户体验差。
- **`--max-num-seqs 256`**：最大并发请求数。受显存限制，跑起来后看 `vllm:num_requests_running` 实际能跑到多少。
- **`--max-num-batched-tokens 8192`**：单次迭代处理的总 token 上限。这个数大一点 prefill 快，但 decode 阶段 ITL 抖动；小一点 ITL 平稳但 prefill 慢。生产建议 4096-8192。
- **`--enable-prefix-caching`**：开 KV cache 跨请求复用，对 system prompt 长的场景必开。
- **`--enable-chunked-prefill`**：长 prompt 不阻塞 decode，TTFT p99 显著降。V1 默认开启，老版本要手动打开。

### 21.5.4 Python 客户端调用

```python
"""
vLLM 客户端：用 OpenAI 官方 SDK 直接调，异步流式输出。
"""
import asyncio
import os
from openai import AsyncOpenAI

client = AsyncOpenAI(
    base_url=os.getenv("VLLM_URL", "http://localhost:8000/v1"),
    api_key=os.getenv("VLLM_API_KEY", "sk-local-dev"),
)

async def chat_stream(prompt: str) -> str:
    """流式请求，边收边打印，返回完整结果。"""
    full = []
    stream = await client.chat.completions.create(
        model="qwen3-72b",
        messages=[{"role": "user", "content": prompt}],
        stream=True,
        temperature=0.7,
        max_tokens=1024,
    )
    async for chunk in stream:
        delta = chunk.choices[0].delta.content or ""
        full.append(delta)
        print(delta, end="", flush=True)
    print()
    return "".join(full)

async def main():
    await chat_stream("用三句话解释 PagedAttention。")

if __name__ == "__main__":
    asyncio.run(main())
```

### 21.5.5 多卡张量并行

单卡 H100 80G 装不下 70B FP16 模型（要 140G）。两种方案：

**方案 A：FP8 量化 + 单卡或 2 卡 TP**
```bash
vllm serve meta-llama/Llama-3.1-70B-Instruct \
  --quantization fp8 \
  --tensor-parallel-size 2
```

**方案 B：BF16 + 4 卡 TP**
```bash
vllm serve meta-llama/Llama-3.1-70B-Instruct \
  --tensor-parallel-size 4
```

经验：能 FP8 就 FP8，质量损失 < 1% 但显存和吞吐都赢。TP 数尽量是 attention head 数的因子（Llama-70B 是 64 head，TP 选 2/4/8 都行）。

跨节点的话再加 pipeline parallel：
```bash
# 2 节点 × 8 卡 = 16 GPU，TP=8（节点内），PP=2（跨节点）
vllm serve some-405b-model \
  --tensor-parallel-size 8 \
  --pipeline-parallel-size 2 \
  --distributed-executor-backend ray
```

PP 的代价是 TTFT 略增（请求要在 stage 间传递），优势是跨节点带宽要求低（不用 NVLink，普通 InfiniBand 够）。

### 21.5.6 Multi-LoRA 热替换

一个 base model + N 个 LoRA adapter，按请求路由不同 adapter。N 可以做到几十到上百，单卡 vLLM 切换 adapter 只要 5-10ms。

```bash
# 启动支持 LoRA 的 vLLM
VLLM_ALLOW_RUNTIME_LORA_UPDATING=True \
vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --enable-lora \
  --max-loras 8 \
  --max-lora-rank 64 \
  --lora-modules \
    sql-bot=/models/loras/sql-bot \
    code-bot=/models/loras/code-bot
```

调用时指定 adapter 名：
```python
resp = await client.chat.completions.create(
    model="sql-bot",   # 注意：这里写 LoRA 名，不是 base model
    messages=[{"role": "user", "content": "查最近 7 天注册用户数"}],
)
```

运行时新增 adapter（不重启服务）：
```bash
curl -X POST http://localhost:8000/v1/load_lora_adapter \
  -H "Content-Type: application/json" \
  -d '{
    "lora_name": "summary-bot",
    "lora_path": "/models/loras/summary-bot"
  }'
```

适用场景：SaaS 多租户，每个租户一个 LoRA；或者一个产品里有多种角色（写代码 / 写 SQL / 写文档），各自一个 LoRA。

### 21.5.7 生产里 vLLM 的几个老大难

把社区反复踩过的坑列出来，省得你重新踩：

**1. tokenizer 和模型不一致**

下了模型权重但 tokenizer 用的是 base model 的版本，输出是乱码。检查：`ls model_dir | grep tokenizer`，必须有 `tokenizer.json` 或者 `tokenizer.model`。HuggingFace 上有些社区量化版本忘了带 tokenizer，要从原仓库手动拷过来。

**2. trust_remote_code 风险**

很多新模型（Qwen、Yi 等）需要 `--trust-remote-code` 才能加载，因为模型仓库里有自定义 Python 代码。这个开关意味着 vLLM 会执行模型仓库里的任意代码——理论上是安全风险，生产环境务必先把模型仓库 fork 到你自己的私有 registry，code review 后再加载。

**3. max_model_len 和 rope scaling**

模型默认上下文长度（比如 Llama-3.1 是 128K）和你 `--max-model-len` 设的值要匹配。如果你设了一个超过模型原生支持的值，vLLM 会自动启用 rope scaling，但质量会掉。设小没问题，设大要小心。

**4. flash-attention 版本问题**

vLLM 依赖特定 flash-attn 版本。手动 `pip install` 装新版的 flash-attn 经常和 vLLM 冲突。建议直接用官方 docker 镜像，或者 `uv pip install` 时让它自己解依赖。

**5. 多卡 NCCL 死锁**

TP > 1 时如果 GPU 之间的 NCCL 通信出问题（典型场景：跨容器 / IPC / hostNetwork 没配对），vLLM 会卡死在加载阶段。排查：
- 容器加 `--ipc=host` 或者把 `/dev/shm` 挂大。
- K8s 里 pod 加 `hostIPC: true` 或者 dshm volume 至少 16Gi。
- 设 `NCCL_DEBUG=INFO` 看 NCCL 自己的报错。

## 21.7 生产部署：Docker、K8s、负载均衡

写完单机怎么跑，下一步是怎么扛流量。

### 21.7.1 Docker 镜像

vLLM 官方维护了 Docker 镜像：

```dockerfile
# Dockerfile
FROM vllm/vllm-openai:v0.11.1

# 拷贝启动脚本

RUN chmod +x /opt/entrypoint.sh

```

`entrypoint.sh` 就是 21.5.3 那个生产启动脚本。

`docker run` 跑起来：
```bash
docker run --gpus all \
  -p 8000:8000 \
  -v /models:/models \
  -e HF_HUB_OFFLINE=1 \
  -e VLLM_API_KEY=sk-prod-xxx \
  --shm-size=16g \
  --name vllm-qwen3 \
  vllm-qwen3:latest
```

注意点：
- `--gpus all` 或 `--gpus '"device=0,1"'` 指定 GPU。
- `--shm-size=16g`：vLLM 多进程通信用 `/dev/shm`，默认 64MB 不够，必须开大。
- 模型权重挂载本地，避免每次起容器从 HF 拉。
- `start-period=180s`：模型加载要 1-3 分钟，健康检查启动期要给够。

### 21.7.2 Kubernetes 部署

生产环境基本都在 K8s 上。一个 vLLM 部署清单的核心字段：


关键点：

- **NVIDIA GPU Operator** 必装，提供 `nvidia.com/gpu` 资源、设备插件、MIG 切分。装好之后 K8s 会自动给 GPU 节点打标签。
- **三种探针** 分开配：startup 给 10 分钟模型加载窗口；readiness 5 秒探一次决定流量切入；liveness 30 秒一次防止进程挂死。
- **节点亲和性** 把 H100 任务调度到 H100 节点，A100 任务到 A100 节点。
- **PVC 共享模型权重**：所有 pod 挂同一个 PV，避免每个 pod 重新下载。

### 21.7.3 自动扩缩容（HPA + KEDA）

K8s 原生 HPA 只看 CPU/内存，对 LLM 服务没意义。要看 vLLM 的业务指标——队列深度。

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: vllm-scaler
spec:
  scaleTargetRef:
    name: vllm-qwen3
  minReplicaCount: 2
  maxReplicaCount: 16
  pollingInterval: 15
  cooldownPeriod: 300        # 5 分钟内不缩容（避免抖动）
  triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus.monitoring:9090
        threshold: "10"      # 平均每 pod 排队 > 10 就扩
        query: |
          avg(vllm:num_requests_waiting{app="vllm-qwen3"})
```

为什么是 `num_requests_waiting`：
- 表示当前因为 GPU 忙而被排队的请求数。
- 直接对应"用户体验变慢"的瞬间。
- 比 GPU 利用率敏感（GPU 利用率 100% 不一定意味着排队，可能 batch 满了但延迟还可以）。

为什么 `cooldownPeriod=300s`：模型加载到 GPU 显存要 1-3 分钟，频繁缩扩容会让请求大量打到正在加载的 pod 上失败。给一个长缓冲。

### 21.7.4 K8s 上的高级编排选项

直接 Deployment + KEDA 是最朴素的方案，对中小团队够用。如果要做更复杂的事（disaggregated prefill/decode、prefix-aware 路由、跨 region 容灾），有几个生产级框架值得了解：

- **KServe**：CNCF 项目，K8s 原生 Serverless 推理框架，支持自动 scale-to-zero、流量切分、多模型版本。和 vLLM 集成成熟。
- **llm-d**：Red Hat 出品，把 vLLM + KServe 整合，专门为大模型推理优化，支持 prefill/decode 分离部署、智能路由。生产级开箱方案。
- **Ray Serve**：Anyscale 出品，分布式 Python 应用框架，跨节点 Actor 模型，适合复杂 pipeline（多模型协同 / RAG 后端）。
- **NVIDIA Triton Inference Server**：老牌推理服务器，多框架（TensorRT、ONNX、PyTorch、vLLM 当 backend），适合多模型混布、严格的企业场景。

选型建议：

| 你需要的 | 用什么 |
|---------|-------|
| 单一 LLM 模型，简单部署 | Deployment + KEDA |
| 多模型 / 多版本 / 灰度 | KServe |
| 极致大模型生产（70B+） | llm-d |
| 复杂 pipeline（ASR + LLM + TTS） | Ray Serve |
| 企业混合推理（CV + NLP + LLM） | Triton |

绝大多数业务从 Deployment + KEDA 起步，遇到具体瓶颈再升级。不要一上来就选 KServe / llm-d，复杂度爆炸。

### 21.7.5 多实例负载均衡：自写 Router

虽然 K8s Service 自带轮询，但 LLM 场景下"哪个实例当前最闲"才是更好的路由策略——因为不同请求的 KV cache 占用差别巨大。

下面是一个生产可用的 router，路由策略基于实时队列长度：

```python
"""
vLLM 多实例 Router
策略：
- 路由到当前队列最短的后端
- 健康检查：定期 ping /health，挂掉的实例自动剔除
- 流式透传：SSE / chunk 都直接转发，不落地
"""
import asyncio
import json
import os
import time
from typing import Optional

import httpx
from fastapi import FastAPI, Request, HTTPException
from fastapi.responses import StreamingResponse
import uvicorn

# ============================================================
# 后端实例配置
# ============================================================
BACKENDS = [
    {"url": "http://vllm-1:8000", "weight": 1},
    {"url": "http://vllm-2:8000", "weight": 1},
    {"url": "http://vllm-3:8000", "weight": 1},
    {"url": "http://vllm-4:8000", "weight": 1},
]

API_KEY = os.getenv("ROUTER_API_KEY", "sk-prod-xxx")
HEALTH_INTERVAL = 5  # 秒

# ============================================================
# 健康状态 + 队列长度跟踪
# ============================================================
class BackendState:
    def __init__(self, url: str):
        self.url = url
        self.healthy = True
        self.queue_depth = 0
        self.last_check = 0.0

    def __repr__(self):
        return f"<{self.url} healthy={self.healthy} queue={self.queue_depth}>"

states = {b["url"]: BackendState(b["url"]) for b in BACKENDS}

# ============================================================
# 健康检查 + 指标拉取
# ============================================================
async def poll_metrics(client: httpx.AsyncClient, state: BackendState):
    """拉 /metrics 取队列长度。失败标记为 unhealthy。"""
    try:
        # /health 快速健康检查
        r = await client.get(f"{state.url}/health", timeout=2.0)
        if r.status_code != 200:
            state.healthy = False
            return
        # /metrics 取业务指标
        m = await client.get(f"{state.url}/metrics", timeout=2.0)
        # 简单文本解析，找 vllm:num_requests_waiting + running
        waiting = 0
        running = 0
        for line in m.text.splitlines():
            if line.startswith("vllm:num_requests_waiting "):
                waiting = float(line.split()[-1])
            elif line.startswith("vllm:num_requests_running "):
                running = float(line.split()[-1])
        state.queue_depth = int(waiting + running)
        state.healthy = True
        state.last_check = time.time()
    except Exception:
        state.healthy = False

async def health_loop():
    async with httpx.AsyncClient() as client:
        while True:
            await asyncio.gather(*[poll_metrics(client, s) for s in states.values()])
            await asyncio.sleep(HEALTH_INTERVAL)

# ============================================================
# 路由策略：最短队列优先
# ============================================================
def pick_backend() -> Optional[BackendState]:
    healthy = [s for s in states.values() if s.healthy]
    if not healthy:
        return None
    return min(healthy, key=lambda s: s.queue_depth)

# ============================================================
# FastAPI App
# ============================================================
app = FastAPI()

@app.on_event("startup")
async def startup():
    asyncio.create_task(health_loop())

@app.get("/health")
async def health():
    healthy = sum(1 for s in states.values() if s.healthy)
    if healthy == 0:
        raise HTTPException(503, "all backends down")
    return {
        "status": "ok",
        "backends": [{"url": s.url, "healthy": s.healthy, "queue": s.queue_depth}
                     for s in states.values()],
    }

@app.api_route("/v1/{path:path}", methods=["GET", "POST"])
async def proxy(path: str, request: Request):
    # 鉴权
    auth = request.headers.get("authorization", "")
    if auth != f"Bearer {API_KEY}":
        raise HTTPException(401, "invalid api key")

    backend = pick_backend()
    if backend is None:
        raise HTTPException(503, "no healthy backend")

    body = await request.body()
    is_stream = b'"stream":true' in body or b'"stream": true' in body

    target_url = f"{backend.url}/v1/{path}"
    headers = {k: v for k, v in request.headers.items() if k.lower() != "host"}
    # 替换为后端的 API key
    headers["authorization"] = f"Bearer {os.getenv('BACKEND_API_KEY', 'sk-local-dev')}"

    async def stream_response():
        async with httpx.AsyncClient(timeout=300.0) as client:
            async with client.stream(
                request.method, target_url,
                content=body, headers=headers,
            ) as r:
                async for chunk in r.aiter_raw():
                    yield chunk

    if is_stream:
        return StreamingResponse(stream_response(), media_type="text/event-stream")
    else:
        async with httpx.AsyncClient(timeout=300.0) as client:
            r = await client.request(request.method, target_url, content=body, headers=headers)
            return StreamingResponse(iter([r.content]),
                                     status_code=r.status_code,
                                     media_type=r.headers.get("content-type"))

if __name__ == "__main__":
    uvicorn.run("router:app", host="0.0.0.0", port=8080)
```

这个 router 干了五件事：
1. 后台 5 秒一次拉所有后端的 `/metrics`，记录队列深度和健康状态。
2. 每个新请求到达时，挑队列最短的健康后端。
3. 流式请求（`stream=true`）逐 chunk 透传，不缓冲。
4. 简单 Bearer Token 鉴权。
5. 暴露自己的 `/health` 让上游 LB 看。

生产里这个 router 通常放在 Nginx / Envoy 之后，Nginx 负责 TLS + 限流 + 日志，router 负责智能路由。也可以直接用 vLLM 官方的 `vllm-router`（开箱即用）或者 LLM-D 项目（KServe + vLLM 整合）。

### 21.7.6 流式输出：SSE 还是 WebSocket

LLM 服务的流式输出几乎是标配。两种主流实现：

- **SSE（Server-Sent Events）**：HTTP 之上的单向流，用 `Content-Type: text/event-stream`。OpenAI 官方 API、vLLM、Ollama 都用这个。优点是简单（HTTP 之上）、能穿透代理、断线浏览器自动重连。缺点是单向（服务器推客户端，客户端不能再发消息）。
- **WebSocket**：双向流。优点是双向、低开销。缺点是要单独的连接管理、跨代理可能有问题、协议升级麻烦。

**99% 的场景选 SSE**。除非你做的是真正的多轮交互式 Agent（比如代码执行环境），需要中途打断或者发送中间消息，否则 SSE 完全够用。

SSE 的协议格式：
```
data: {"choices":[{"delta":{"content":"你"}}]}

data: {"choices":[{"delta":{"content":"好"}}]}

data: [DONE]
```

每条消息以 `data: ` 开头，`\n\n` 结尾，最后用 `[DONE]` 标记结束。客户端处理：

```python
import httpx

async def stream_chat(prompt: str):
    async with httpx.AsyncClient(timeout=300.0) as client:
        async with client.stream(
            "POST", "http://localhost:8000/v1/chat/completions",
            json={"model": "qwen3-72b",
                  "messages": [{"role": "user", "content": prompt}],
                  "stream": True},
            headers={"Authorization": "Bearer sk-xxx"},
        ) as r:
            async for line in r.aiter_lines():
                if not line.startswith("data: "):
                    continue
                data = line[6:]
                if data == "[DONE]":
                    break
                import json
                chunk = json.loads(data)
                delta = chunk["choices"][0]["delta"].get("content", "")
                if delta:
                    print(delta, end="", flush=True)
```

## 21.8 性能压测：先压再上线

任何 LLM 服务上线前必做压测。压测的目的是：
1. 确认它能扛住预期流量。
2. 找到它的崩溃点（precise breaking point），让你知道扩容预算。
3. 量化关键指标，给上下游业务做容量规划。

### 21.8.1 关键指标

| 指标 | 含义 | 目标值（聊天场景） |
|------|------|-----------------|
| **TTFT**（Time To First Token） | 请求发出到收到第一个 token 的时间 | p50 < 500ms，p99 < 2s |
| **TPOT**（Time Per Output Token，也叫 ITL） | 后续每个 token 的间隔 | p50 < 50ms（约等于 20 token/s 用户感知速度） |
| **Throughput** | 每秒总输出 token 数（所有用户加起来） | 越高越好，单 H100 通常 5000-15000 t/s |
| **Concurrent Requests** | 同时处理的请求数 | 越高越好，决定能服务的用户数 |
| **GPU Utilization** | GPU 实际使用率 | 75-90% 健康，> 95% 接近瓶颈，< 50% 浪费 |
| **KV Cache Utilization** | KV cache 占用率 | 70-85% 健康 |

经验：用户对 **TTFT 比 TPOT 敏感得多**。第一个字慢 1 秒明显慢，但每秒生成 20 字还是 30 字感受差异不大（因为人阅读速度跟不上）。优化优先 TTFT。

#### 自写压测脚本

需要更精细控制时（比如自定义 prompt 分布、阶梯压测）：

```python
"""
LLM 压测脚本
- 阶梯式提升并发：1, 4, 16, 64, 256
- 每个阶梯跑 60 秒
- 输出 TTFT、TPOT、QPS、错误率
- 区分流式和非流式
"""
import asyncio
import json
import os
import statistics
import time
from dataclasses import dataclass, field

import httpx

API_URL = os.getenv("API_URL", "http://localhost:8000/v1/chat/completions")
API_KEY = os.getenv("API_KEY", "sk-local-dev")
MODEL = os.getenv("MODEL", "qwen3-72b")

PROMPTS = [
    "用 200 字解释 PagedAttention。",
    "写一个 Python 装饰器实现函数耗时统计。",
    "翻译：The quick brown fox jumps over the lazy dog.",
    "推荐三本系统学习分布式系统的书。",
    "写一首关于秋天的现代诗。",
] * 20  # 100 条


@dataclass
class Result:
    ttft: float = 0.0
    tpot: list = field(default_factory=list)
    total_time: float = 0.0
    output_tokens: int = 0
    success: bool = False
    error: str = ""


async def one_request(client: httpx.AsyncClient, prompt: str) -> Result:
    r = Result()
    start = time.perf_counter()
    last_token_time = None
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": prompt}],
        "stream": True,
        "max_tokens": 256,
    }
    try:
        async with client.stream(
            "POST", API_URL,
            json=payload,
            headers={"Authorization": f"Bearer {API_KEY}"},
            timeout=120.0,
        ) as resp:
            if resp.status_code != 200:
                r.error = f"http {resp.status_code}"
                return r
            async for line in resp.aiter_lines():
                if not line.startswith("data: "):
                    continue
                data = line[6:].strip()
                if data == "[DONE]":
                    break
                chunk = json.loads(data)
                delta = chunk["choices"][0]["delta"].get("content", "")
                if not delta:
                    continue
                now = time.perf_counter()
                if r.ttft == 0:
                    r.ttft = now - start
                else:
                    assert last_token_time is not None
                    r.tpot.append(now - last_token_time)
                last_token_time = now
                r.output_tokens += 1
        r.total_time = time.perf_counter() - start
        r.success = r.output_tokens > 0
    except Exception as e:
        r.error = str(e)
    return r


async def run_stage(concurrency: int, duration: float) -> dict:
    print(f"\n--- 并发 {concurrency}，{duration}s ---")
    results: list[Result] = []
    stop_at = time.perf_counter() + duration
    sem = asyncio.Semaphore(concurrency)

    async def worker(idx: int):
        async with httpx.AsyncClient(http2=False) as client:
            while time.perf_counter() < stop_at:
                async with sem:
                    prompt = PROMPTS[idx % len(PROMPTS)]
                    r = await one_request(client, prompt)
                    results.append(r)
                    idx += 1

    await asyncio.gather(*[worker(i) for i in range(concurrency)])
    successful = [r for r in results if r.success]
    if not successful:
        return {"concurrency": concurrency, "qps": 0, "error_rate": 1.0}

    ttfts = [r.ttft * 1000 for r in successful]
    tpots = [t * 1000 for r in successful for t in r.tpot]
    total_tokens = sum(r.output_tokens for r in successful)
    elapsed = duration

    summary = {
        "concurrency": concurrency,
        "total_requests": len(results),
        "success": len(successful),
        "error_rate": (len(results) - len(successful)) / len(results),
        "qps": len(successful) / elapsed,
        "tok_per_s": total_tokens / elapsed,
        "ttft_p50": statistics.median(ttfts),
        "ttft_p99": statistics.quantiles(ttfts, n=100)[98] if len(ttfts) > 100 else max(ttfts),
        "tpot_p50": statistics.median(tpots) if tpots else 0,
        "tpot_p99": statistics.quantiles(tpots, n=100)[98] if len(tpots) > 100 else (max(tpots) if tpots else 0),
    }
    print(json.dumps(summary, indent=2, ensure_ascii=False))
    return summary


async def main():
    stages = [1, 4, 16, 64, 128]
    duration = 60.0
    all_summaries = []
    for c in stages:
        s = await run_stage(c, duration)
        all_summaries.append(s)
        await asyncio.sleep(5)  # 让服务回血一下
    print("\n========== 总报告 ==========")
    print(f"{'并发':<6}{'QPS':<10}{'tok/s':<10}{'TTFT-p50':<12}{'TTFT-p99':<12}{'TPOT-p50':<12}{'错误率':<8}")
    for s in all_summaries:
        print(f"{s['concurrency']:<6}{s.get('qps',0):<10.2f}{s.get('tok_per_s',0):<10.1f}"
              f"{s.get('ttft_p50',0):<12.1f}{s.get('ttft_p99',0):<12.1f}"
              f"{s.get('tpot_p50',0):<12.2f}{s.get('error_rate',0):<8.2%}")


if __name__ == "__main__":
    asyncio.run(main())
```

跑出来的结果大概长这样（H100 跑 Qwen3-72B-FP8）：

读法：从 64 并发开始 TTFT-p99 接近 2s（聊天场景红线），到 128 并发已经超 4.5s（用户开始抱怨）。**这台机器健康容量大约 50-80 并发**，超过就要扩容。

把硬件账算清楚。这一节讲哪种卡跑哪种模型最划算，以及自建 vs 云的成本边界。

### 21.9.1 NVIDIA 数据中心卡对比（2026 年初）

- **摩尔线程 MTT S4000**：自研 GPU，2025 年量产，社区工具链跟进。

实战建议：如果是国产化合规要求强制（金融、政企），昇腾 910B + vllm-ascend 是当前最成熟的开源方案。性能比 H100 弱 20-40%，但是能合规上线。其他场景仍以 NVIDIA 为主。

适用场景：个人开发、小团队内部工具、隐私优先的桌面 Agent。生产 API 服务不要用 Mac。

### 21.9.5 三个具体配置例子

**例子 1：内部知识库 Agent，约 50 用户，QPS 峰值 5**

负载特征：长 system prompt（含工具说明 5K token）+ 多轮对话，prefix 重叠率高。

推荐：单卡 H100 80G + vLLM + Qwen3-32B FP8 + prefix caching。月成本约 $1800（云）。如果走 API 月成本约 $300（用 GPT-5-mini）——这种场景 API 划算。

**例子 2：To-C 产品聊天助手，月 5B token 输出，DAU 10 万**

负载特征：短 prompt 短对话，并发高，TTFT 敏感。

推荐：3-6 卡 H100（按峰值），vLLM + Llama-3.1-70B FP8 + chunked prefill + multi-LoRA（不同人设）。月成本约 $15000-20000。API 方案约 $50000+。开始有自建优势。

**例子 3：金融风控批处理，每日离线扫描 1B token，无延迟要求**

负载特征：极高吞吐，低延迟无所谓。

推荐：A100 80G × 2 + vLLM + offline_inference 模式（不开 API server，直接 batch_inference）+ AWQ INT4。利用率拉满，单位 token 成本极低。这种场景**自建肯定划算**——离线推理不需要 24/7 GPU 待命，跑完关机。

### 21.9.6 自建 vs API 的成本边界

把账算清楚。

**API 方案（以 GPT-5 / Claude Sonnet 4.5 类为基准）**：
- 输入约 $3 / million token，输出约 $15 / million token。
- 平均每 1M token 约 $5（输入输出 7:3 估算）。
- 月 100M token = $500，月 1B token = $5000，月 10B token = $50000。

**自建方案（H100 FP8 跑 Qwen3-72B 或 Llama-3.1-70B）**：
- 单卡 $2.5/h，月成本 $1800。
- 单卡持续跑 5000 t/s 输出，月理论上限 ≈ 13B token。
- 实际利用率 50% 算 ≈ 6.5B token / 月。
- 折算成本 ≈ $0.28 / million token。

**break-even 点**：
- 月 1B token：API $5000 vs 自建 $1800（一卡），自建赢。
- 但加上 DevOps 成本（$10000+/月）、模型迭代成本、合规成本，**实际 break-even 在月 10-20B token 左右**。
- 月不到 5B token，几乎肯定 API 更划算。

例外：
- **强合规要求**（医疗、金融、政企）：必须自建，没得选。
- **特殊定制化**：要 fine-tune / LoRA 大量小模型，自建更灵活。
- **延迟极致敏感**：跨境 API 调用 RTT 可能 100-300ms，自建可压到 < 30ms。

实战推荐：**起步期一律 API 起跑**。流量长稳定到 10B token / 月以上、或者出现合规 / 定制化硬需求，再切自建。混合架构（自建处理稳定基线，API 处理峰值和高难度任务）是最务实的中后期形态。

## 21.10 监控与可观测性

服务上线只是开始，运维它一年才是真本事。LLM 服务的可观测性比传统 web 服务更复杂——你不仅要知道它"还活着"，还要知道它"还聪明"。

### 21.10.1 三层监控体系

**基础设施层**（cAdvisor / DCGM-Exporter）：
- GPU 利用率、显存占用、温度、SM 时钟。
- 主机 CPU / 内存 / 磁盘 / 网络。
- DCGM-Exporter 是 NVIDIA 官方的 Prometheus exporter，必装。

**服务层**（vLLM 自带 `/metrics`）：

vLLM 暴露的核心 Prometheus 指标：

| 指标 | 类型 | 用途 |
|------|------|------|
| `vllm:num_requests_running` | gauge | 当前正在生成的请求数 |
| `vllm:num_requests_waiting` | gauge | 排队等待的请求数（扩容触发器） |
| `vllm:gpu_cache_usage_perc` | gauge | KV cache 使用率（接近 1 接近 OOM） |
| `vllm:prompt_throughput_total` | counter | 累计 prompt token 处理量 |
| `vllm:generation_throughput_total` | counter | 累计生成 token 量 |
| `vllm:time_to_first_token_seconds` | histogram | TTFT 分布 |
| `vllm:time_per_output_token_seconds` | histogram | TPOT 分布 |
| `vllm:e2e_request_latency_seconds` | histogram | 端到端延迟 |
| `vllm:request_prompt_tokens` | histogram | 输入 token 数分布 |
| `vllm:request_generation_tokens` | histogram | 输出 token 数分布 |

**业务层**（自己埋）：
- 不同业务路径的 token 用量、错误率。
- 用户级别的 RPS、配额使用。
- 模型版本切换 A/B 对比。

### 21.10.2 必备 Grafana 看板

至少配置六张图：

1. **TTFT p50/p95/p99 时序**：聊天体验主线。
2. **请求队列深度时序**：扩容决策依据。
3. **KV cache 使用率时序**：OOM 预警。
4. **token 吞吐**（输入 + 输出）：成本追溯。
5. **GPU 利用率 + 显存**：硬件状态。
6. **错误率分布**（HTTP 状态、超时、OOM）：可用性。

### 21.10.3 告警规则

```yaml
groups:
  - name: vllm
    rules:
      - alert: VLLMHighQueueDepth
        expr: avg_over_time(vllm:num_requests_waiting[5m]) > 20
        for: 5m
        annotations:
          summary: "vLLM 排队深度持续 > 20，需要扩容"

      - alert: VLLMHighTTFT
        expr: histogram_quantile(0.99, rate(vllm:time_to_first_token_seconds_bucket[5m])) > 3
        for: 5m
        annotations:
          summary: "TTFT p99 > 3s，用户体验受损"

      - alert: VLLMKVCacheNearFull
        expr: vllm:gpu_cache_usage_perc > 0.95
        for: 2m
        annotations:
          summary: "KV cache 即将满，下一步 OOM"

      - alert: VLLMHighErrorRate
        expr: |
          rate(vllm:request_failure_total[5m])
          / rate(vllm:request_success_total[5m]) > 0.01
        for: 3m
        annotations:
          summary: "错误率 > 1%，检查后端"
```

### 21.10.4 模型质量监控（业务层）

服务"活着"不等于"还聪明"。LLM 推理服务有个独特问题：**模型可能悄悄变笨而你不知道**。原因：

- 量化版本切换、KV cache dtype 改了、引擎升级。
- 上游模型仓库静悄悄换了权重（开源模型也会出 bugfix）。
- 上下文截断策略变了，长 prompt 被悄悄裁掉。

应对：**回归测试集 + 定时自动评测**。
- 准备一份你的领域 200-500 题，含标准答案。
- 上线后每天定时跑一次，比较自动评分（GPT-4 当裁判 / 自家小评分模型）。
- 评分掉到一定阈值（比如比基线低 5%）触发告警。

这个机制在 OpenAI / Anthropic 等头部厂里叫 evals。自建服务不能省，否则一旦模型悄悄变笨，业务方第一个发现你还蒙在鼓里。

## 21.11 排错清单：上线后会遇到的坑

把生产中会遇到的高频问题列出来，按发生概率排序：

### 21.11.1 OOM（Out of Memory）

最常见。原因和处理：
- **`--gpu-memory-utilization` 太高**：从 0.95 降到 0.90。
- **`--max-model-len` 太大**：长 prompt 用户来了 KV cache 爆掉。设一个合理上限（比如 16K）。
- **某个请求超长**：在 router 层加 token 数限制，超过 N 直接拒绝。
- **多模态模型**：图像 token 占用比文本大几倍，max_seq_len 要给图像留空间。

### 21.11.2 TTFT 抖动剧烈

p50 200ms，p99 突然 5s。原因：
- **没开 chunked prefill**：长 prompt 阻塞 decode。打开 `--enable-chunked-prefill`。
- **批量大请求同时到达**：让上游 router 做 rate limit 平滑流量。
- **prefix cache 命中率低**：检查请求是否 system prompt 一致。

### 21.11.3 流式中断

客户端收到一半连接断了。原因：
- **超时设置太短**：长输出（几千 token）超 60s，把客户端 timeout 设到 300s+。
- **代理（Nginx）默认关闭长连接**：在 Nginx 配 `proxy_buffering off; proxy_read_timeout 600s;`。
- **K8s Ingress 默认 60s 超时**：annotation 加 `nginx.ingress.kubernetes.io/proxy-read-timeout: "600"`。

### 21.11.4 GPU 利用率低

GPU-Util 长期 < 40%，说明：
- **请求量不够**：QPS 太低，batch 攒不起来。这个时候考虑缩容。
- **`max-num-seqs` 设小了**：放大到 256+。
- **请求长度方差大**：一个 8K prompt 拖累整个 batch。开启 chunked prefill 缓解。

### 21.11.5 模型加载慢导致扩容失败

新 pod 启动 3 分钟，期间 K8s 探针失败被杀。处理：
- `startupProbe.failureThreshold` 至少给 60×10s = 10 分钟。
- 用 PVC 共享模型权重，避免每个 pod 重新拉。
- 用容器镜像预热（initContainer 把权重 warm 到 page cache）。

### 21.11.6 量化模型质量问题

上线后业务说"模型变笨了"。处理：
- 跑回归测试集（你的领域 200 题）对比 FP16 vs 量化 vs 上线模型。
- FP8 / AWQ 量化掉点 < 2% 是正常的。如果掉点 > 5%，考虑：
  - 量化数据集是否包含你的领域分布（GPTQ/AWQ 校准数据）。
  - 是否过度量化（Q3 / Q2 不要在生产上用）。
  - KV cache 也量化（fp8 kv cache）会再损失一点点，对长上下文场景明显。

## 21.12 章节小结：四个层次的能力清单

学到这里，你应该具备四个层次的部署能力：

**L1 单机本地部署**：
- Ollama 拉模型起服务，OpenAI SDK 直接调。
- llama.cpp 量化模型跑 GGUF。
- 知道选什么模型尺寸适合你的硬件。

**L2 单机生产服务**：
- vLLM 起 OpenAI 兼容服务，参数能背下来。
- FP8 量化、KV cache、PagedAttention 原理能讲清楚。
- 写过压测脚本，知道服务的 TTFT / TPOT / 吞吐边界。

**L3 多机生产服务**：
- Docker 镜像 + K8s 部署 + GPU Operator + KEDA 自动扩缩容。
- 自写或用现成 router 做多实例负载均衡。
- 监控关键指标：队列深度、KV cache 利用率、p99 TTFT。

**L4 架构决策**：
- 知道什么时候自建什么时候 API。
- 知道什么场景用 vLLM 什么场景用 SGLang / TensorRT-LLM。
- 能算出一笔月度账：硬件 + 流量 + 运维 + 风险，给出技术方案的成本曲线。

把这四个层次都摸过一遍，下次有人扔给你一个 70B 模型说"上线"，你就有完整的工具箱、完整的成本估算、完整的架构方案。

下一章讲 LLM 应用框架：LangChain、LlamaIndex、AutoGPT 这些上层工具是把模型推理服务包装成业务应用的桥梁——你已经掌握的"模型能跑"这一层，到"模型能解决业务问题"还有一段路要走。
