# 第 19 章 · 多模态应用：视觉、语音、文档

> 想象一下：你手里有 200 张本月的发票图片要录进 ERP；同事甩给你一段 2 小时的会议录音让你写纪要；老板拍来一张同行的财报截图问"我们差在哪儿"。
> 三件事，按本章学的方法做完，加起来不到 200 行 Python。

> 「人类不是用文字思考的。人类用图像、声音、空间关系思考，再用文字翻译给彼此。AI 应用工程师如果只会处理文字，就只摸到了世界的一层皮。」

第 17、18 章搭齐了「Agent 内核 + 工具协议」，但协议边界传过去的几乎都是文本。真实业务拿到的是发票图片、合同扫描件、会议录音、监控视频、设计稿截图——本章把这些「非文本」模态接回 LLM 应用栈，并指出哪些能力可以反向封装成 MCP tool / resource 喂给上一章的 Agent。

到 2026 年，单纯的文字模型已经是落后概念。OpenAI、Anthropic、Google 三家旗舰默认带视觉，开源 Qwen3-Omni、Llama 4、InternVL3 把图像 / 视频 / 音频塞进同一个 Transformer。客户采购的不是「一个文字 chatbot」，而是「能看发票、读合同、听电话录音、回答 PPT 问题的助手」。

本章把视觉、文档、图像生成、语音、多模态 RAG 揉在一起讲。读完之后你应该能：

- 同一张图，分别用 GPT、Claude、Qwen3-VL 三种 API 调出来；
- 拿一份 200 页 PDF 财报，自动抽出关键指标并支持问答；
- 给一段 2 小时的会议录音，转写并区分说话人；
- 在本地用 diffusers 跑一张图；
- 用 ColPali 把扫描件 PDF 直接变成可检索向量；
- 把上面这些能力拼成一个完整的 Agent。

## 19.1 多模态的现状：到 2026 年发生了什么

### 19.1.1 三个并行的趋势

**模型的多模态化**：2024 年 GPT-4o 把视觉/语音/文字塞进同一个模型，2025 年 Gemini 2.5、Claude Sonnet 4 跟上，2026-03-30 Qwen3.5-Omni 把 Thinker–Talker 升级为 Hybrid-Attention MoE，一次前向完成「听-看-想-说」，开源基本追平闭源。多模态从「外挂」变成「原生」。

**专用 OCR 在很多场景被 VLM 替代**。**VLM**（Vision-Language Model，视觉语言模型）= 在 LLM 基础上加一个图像编码器，能"看图 + 说话"的大模型，GPT-5、Claude、Qwen3-VL 都属于这一类；**OCR**（Optical Character Recognition，光学字符识别）= 只把图片里的文字抠出来。过去做发票识别要训专门的 OCR + 一堆后处理，现在把发票图片直接丢给 VLM 输出 JSON，准确率往往更高。这件事颠覆了文档处理产业。

**实时语音对话从噱头变成产品**。OpenAI Realtime API（gpt-realtime，2025-08 GA、2026-05 再降价 20%）、Gemini Live API、开源 Qwen3.5-Omni 都能做端到端语音对话，延迟压到 300 ms 以下。「打电话给 AI」在 2026 年是真实的商业场景。

## 19.2 视觉语言模型：基础调用

### 19.2.1 三种输入方式

VLM 接受的图像有三种喂法：

1. **公网 URL**：模型方远程下载。延迟最低，但内网图片或私有数据不能用。
2. **base64 内联**：把图片编码进请求体。简单、私密，但请求大、有大小限制（通常 20 MB）。
3. **文件上传 / Files API**：先上传得到 file_id，再引用。适合反复使用、视频、长文档。

### 19.2.2 同一张图，三家对比

下面是一段统一的对比代码：传同一张发票图片，让三家模型抽出 `vendor`、`total`、`date` 三个字段。

```python
# vlm_three_way.py
import base64
import json
import os
from pathlib import Path

import anthropic
import openai
from openai import OpenAI

IMG = Path("invoice.jpg")
B64 = base64.b64encode(IMG.read_bytes()).decode()

PROMPT = """从图片中提取以下字段，严格输出 JSON：
{"vendor": str, "total": float, "date": "YYYY-MM-DD"}
不要解释。如果某项不确定，写 null。"""

# ---------- OpenAI ----------
def call_openai():
    client = OpenAI()
    resp = client.chat.completions.create(
        model="gpt-5.5",
        response_format={"type": "json_object"},
        messages=[{
            "role": "user",
            "content": [
                {"type": "text", "text": PROMPT},
                {"type": "image_url",
                 "image_url": {"url": f"data:image/jpeg;base64,{B64}"}},
            ],
        }],
    )
    return json.loads(resp.choices[0].message.content)

# ---------- Anthropic ----------
def call_claude():
    client = anthropic.Anthropic()
    resp = client.messages.create(
        model="claude-opus-4-7",
        max_tokens=512,
        messages=[{
            "role": "user",
            "content": [
                {"type": "image",
                 "source": {"type": "base64",
                            "media_type": "image/jpeg",
                            "data": B64}},
                {"type": "text", "text": PROMPT},
            ],
        }],
    )
    return json.loads(resp.content[0].text)

# ---------- Qwen (DashScope OpenAI 兼容) ----------
def call_qwen():
    client = OpenAI(
        api_key=os.environ["DASHSCOPE_API_KEY"],
        base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    )
    resp = client.chat.completions.create(
        model="qwen3-vl-235b-a22b-instruct",
        response_format={"type": "json_object"},
        messages=[{
            "role": "user",
            "content": [
                {"type": "text", "text": PROMPT},
                {"type": "image_url",
                 "image_url": {"url": f"data:image/jpeg;base64,{B64}"}},
            ],
        }],
    )
    return json.loads(resp.choices[0].message.content)

if __name__ == "__main__":
    for name, fn in [("openai", call_openai), ("claude", call_claude), ("qwen", call_qwen)]:
        try:
            print(name, fn())
        except Exception as e:
            print(name, "ERR", e)
```

跑下来你会发现三件事：

1. 三家的 JSON 结构基本一致，但偶尔会在 `total` 是「含税」还是「不含税」上意见不同——这是 prompt 的锅，要写明白。
2. Claude 在表格密集的发票上保留结构最好，会把单价、数量都补出来，哪怕你没要。
3. Qwen3-VL 在中文票据上明显胜出，特别是手写章、模糊章。

更细一点的真实强项 / 弱点（2026 年中根据我自己跑了 2 万张发票 + 5000 份合同得到的样本印象，**仅供你做选型决策时的参考**，对你具体业务务必自己跑 200 条样本验证）：

| 厂家 | 真实强项 | 真实弱点 / 工程坑 |
|------|----------|--------------------|
| GPT-5.5 | 长尾通用、JSON schema 模式严格、prompt cache 接得最熟；图表读数稳定 | 单图上限 20MB、长边超过 2048 会被自动 resize 影响小字；中文手写体、印章、繁体竖排弱于 Qwen；偶尔会"贴心补全"它觉得缺失的字段，要在 prompt 显式禁止 |
| Claude Opus 4.7 | 表格结构保真度第一档（合并单元格也能还原）、长 PDF 直传质量最好、图表推理（要求"先观察再答"）领先 | 单图大小限制 5MB、像素长边 8000、单次 API 最多 100 张；不支持 `response_format=json_schema`，要靠 tool use 模拟 schema；中文手写仍弱于 Qwen |
| Qwen3-VL | 中文票据 / 手写 / 印章 / 复杂版式中文 PDF 实测最强；能直接吐 HTML 表格便于落库；价格几乎是三家最低 | 极端高分辨率会被切成大量 patch，token 暴涨（一张 4K 图轻松 8K token）；JSON 严格模式偶发逃逸；非中英文小语种弱；DashScope 海外区域延迟波动大 |

选型经验：**先按你的最差 1% 数据选模型，而不是平均 case**——手写章占比 5% 的票据锁 Qwen；大量跨页合并单元格的合同锁 Claude；严格 JSON + 高并发锁 GPT。

### 19.2.3 多图与视频

多图的传法是把多个 image 块串到 content 里。一些常见任务：

- **对比**：「比较这两张设计稿的差异」；
- **顺序推理**：「这是手术过程的 5 张连续截图，请描述每一步」；
- **跨图检索**：「这 100 张图里，哪几张是同一个人」（更适合走 embedding）。

视频的传法分两派：

- **关键帧抽样**（GPT/Claude）：用 ffmpeg 抽出每秒 1 帧，作为多图传入。简单、能控成本，但丢掉运动信息。
- **原生视频**（Gemini、Qwen3-Omni）：直接传 mp4，模型自己切帧 + 时序编码。能问「他在 1 分 23 秒做了什么」这种带时间戳的问题。

抽帧示例：

```python
import subprocess, base64
from pathlib import Path

def extract_frames(video: Path, fps=1, out_dir=Path("frames")) -> list[Path]:
    out_dir.mkdir(exist_ok=True)
    subprocess.run([
        "ffmpeg", "-i", str(video), "-vf", f"fps={fps}",
        f"{out_dir}/f_%04d.jpg", "-y", "-loglevel", "error"
    ], check=True)
    return sorted(out_dir.glob("f_*.jpg"))

def to_image_blocks(frames: list[Path]) -> list[dict]:
    return [{
        "type": "image_url",
        "image_url": {"url": f"data:image/jpeg;base64,{base64.b64encode(p.read_bytes()).decode()}"}
    } for p in frames]
```

提醒：抽帧前先估算 token，30 分钟视频每秒 1 帧 = 1800 张，按每张 1024 token 算就是 180 万 token，单次请求基本爆。生产里要么稀疏采样，要么先用语音转录定位到关键时间段再抽帧。

### 19.2.4 长文档（PDF 直传）

OpenAI 和 Anthropic 都接受直接传 PDF 文件（Anthropic 的 document content type，OpenAI 的 file 上传），模型会内部做切页和理解。但是：

- 上限通常 100 页；
- 复杂版式（多栏、嵌入表格、扫描件）效果比专门的 OCR/Docling 流水线差；
- 价格贵（按页计 token）。

经验法则：

- 50 页以内、版式干净（report、合同）：直接传 PDF；
- 50 页以上、版式复杂、扫描件、要做 RAG：走 Docling/marker + ColPali 的流水线（19.4 / 19.6 节）。

## 19.3 图像理解任务图谱

VLM 不只是「看图说话」。下面列出工程上常见的 7 类任务，每类给一个最小 prompt 模板。

### 19.3.1 通用问答

```
问：图中有几个人？他们在做什么？
```

简单。注意要让模型说「我不确定」而不是编。

### 19.3.2 OCR / 文档抽取

```
逐字抄录图中所有可见文字。保留原始换行。不要纠错，不要翻译。
```

VLM 做 OCR 的优点是能理解上下文，比如它知道「2026/04/17」是日期不是除法；缺点是会偷偷「修正」它觉得有问题的字。在医疗、法律场景里这点要当心：写明「不要纠错」。

### 19.3.3 结构化抽取（发票、合同、表格）

定义 JSON schema 并要求严格输出：

```python
schema = {
    "type": "object",
    "properties": {
        "vendor": {"type": "string"},
        "total":  {"type": "number"},
        "items":  {"type": "array",
                   "items": {"type": "object",
                             "properties": {
                                 "name": {"type": "string"},
                                 "qty":  {"type": "number"},
                                 "unit_price": {"type": "number"}}}},
    },
    "required": ["vendor", "total"]
}
```

OpenAI 的 `response_format={"type": "json_schema", ...}`、Anthropic 的 tool use、Qwen 的 `response_format=json_object` + JSON Mode 都能达成强约束。哪怕模型在自由文本里会犯错，schema 约束下的字段抽取在 2026 年已经是很稳的能力。

### 19.3.4 GUI 理解（Computer Use）

Anthropic 早在 2024 年 10 月就推出 Claude Computer Use beta，2025 年随 Claude 4 系列正式 GA，2026 年 Opus 4.7 在屏幕识别与坐标定位上又往前推了一步。模型接收：

- 当前屏幕截图；
- 任务描述（例如「在浏览器里搜索『北京天气』」）；

模型输出动作（click x,y / type 文本 / scroll / screenshot）。OpenAI 的 Operator 是同样思路。要点：

- VLM 必须能识别 UI 元素的「位置」，老一代 VLM 只会描述不会定位，这是新一代的关键能力；
- 要给模型「截图回路」：执行一步动作后立刻截图回传，让它对 UI 状态有反馈；
- 自己写 Computer Use 客户端时记得 sandbox，不要让它点到生产服务器。

### 19.3.5 视觉差异 / 异常检测

```
这是设备运行的正常状态图（图 1）和当前状态图（图 2）。请列出图 2 中所有与图 1 不同的地方。
```

工业巡检、UI diff、医学影像变化跟踪都是这类。注意要给「锚点」，否则模型容易乱比。

### 19.3.6 图像描述（用于 RAG / 索引）

```
用 60 字描述这张图的内容、风格、关键物体。这段描述将被用于后续向量检索，请避免无关形容词。
```

把图变成文字 caption，再走文字 RAG，是「Caption-then-Retrieve」模式（见 19.6.1）。

### 19.3.7 视觉推理（数学题、电路、地图）

让模型「看到」一道几何题、一张电路图、一张地图，推理出答案。这块对模型要求高，2026 年 Claude Opus 4.7、GPT-5.5、Gemini 3.1 Pro 是第一梯队，开源里 Qwen3-VL-235B-A22B-Thinking、InternVL3.5-241B-A28B 接近但仍有差距。

## 19.4 现代文档处理流水线

对工程师来说，「PDF 解析」是一个永恒的痛苦地带。2026 年的最佳实践已经从「PyPDF2 + 正则」演进到「VLM + 专用 layout 模型」混合栈。

### 19.4.1 工具梯队

按「能力 / 易用度」排：

| 工具 | 强项 | 弱项 | 推荐场景 |
|------|------|------|----------|
| PyMuPDF (fitz) | 快、稳、文本/图像/坐标全有 | 不懂版式 | 提取原始文本和坐标 |
| pdfplumber | 表格抽取友好 | 慢、复杂版式弱 | 简单表格 PDF |
| Docling (IBM) | 版式分析 + 表格识别 + Markdown 输出 + 多模态 | 安装重 | 主力推荐 |
| marker-pdf | 速度快、Markdown 漂亮 | 表格弱于 Docling | 大批量、版式中等 |
| unstructured | 文档类型多、生态好 | 表格质量一般 | 接 LangChain/LlamaIndex |
| LlamaParse（云） | 强、便宜 | 要联网、要 key | 不愿自己跑模型 |
| Nougat | 学术 PDF + 公式 | 通用 PDF 弱 | 论文、教材 |
| Mathpix | 公式 OCR 之王 | 商业 API | 数学公式必须准 |

实战中我的默认组合：

- **结构提取**：Docling，输出 Markdown + 表格；
- **公式**：Mathpix（如果预算允许）或 Nougat；
- **扫描件 / 复杂版式 / 直接做 RAG**：跳过解析，走 ColPali（19.6）。

### 19.4.2 Docling 最小示例

```python
# pip install docling
from docling.document_converter import DocumentConverter

conv = DocumentConverter()
doc = conv.convert("annual_report.pdf").document

# 1) 给 LLM 喂的 Markdown
md = doc.export_to_markdown()

# 2) 结构化数据
for table in doc.tables:
    df = table.export_to_dataframe()
    print(df.head())

# 3) 元素级别遍历（用于精细 chunk）
for item in doc.iterate_items():
    print(item.label, item.text[:80] if item.text else "")
```

Docling 的输出可以无缝接入 LlamaIndex / LangChain 的 chunk → embed → store 流程。它会保留表格的 cell 结构，比把表格强行展平为文本好很多。

### 19.4.3 表格抽取深坑

表格永远是文档处理的最大坑。2026 年的状态：

- **简单表格**（边框完整、单元格不合并）：Docling、pdfplumber、Camelot 都能 handle；
- **复杂表格**（合并单元格、跨页、嵌套表头）：Docling 的 TableFormer 模型领先，但仍然会错；
- **扫描件表格**：Docling + OCR，或者直接喂给 VLM 让它输出 Markdown 表格——后者在 2026 年常常更好。

经验：表格是关键业务数据时，**永远做二次校验**。例如总和列、行数、是否每行 cell 数一致。这些都是廉价的健康检查，能挡住 80% 的解析事故。

### 19.4.4 公式与图表

Mathpix 的 LaTeX 转换是商业标杆，每页几美分。Nougat（Meta 出品）开源、专攻学术 PDF，公式不错但慢。

图表（柱状图、折线图）现在直接喂给 VLM 让它读数最简单：

```
请逐个柱状图读出每个柱子的数值。如果数值未直接标注，请用 y 轴刻度估算并写出「估算」。
```

模型读图能力比 2 年前强了一个量级。

### 19.4.5 VLM + 文本混合栈

我日常的「重型」流水线：

```
PDF
 ├── PyMuPDF → 原始文字 + 每页图片
 ├── Docling → 版式 + 表格 Markdown
 ├── 关键页（含图表 / 复杂表格）→ VLM 二次抽取
 └── 合并为「层次化文档」（标题 → 段落 → 表格 → 图表 caption）
                    ↓
            chunk + embed + store
```

混合栈的好处：版式信息 Docling 给你，复杂理解 VLM 给你，原始 fallback PyMuPDF 给你。三层兜底。

## 19.5 图像生成

### 19.5.1 基本概念（一段话）

Diffusion 模型的本质是「学会从纯噪声还原一张图」：训练时给图加噪、教模型去噪，推理时从纯噪声按文本条件一步步去噪。原理在前面章节已讲，这里重点是工程怎么用。值得一提的是，2024 年之后开源主力（FLUX.1、SD 3.5）已经从 U-Net 换到 DiT（Diffusion Transformer），跟 LLM 是同一族架构，scaling law 也越来越像语言模型——FLUX.1 12B 大约是 2022 年 SD 1.5（860M）的 14 倍。

### 19.5.2 闭源 API 梯队

| 服务 | 强项 | 弱项 | 价格档 |
|------|------|------|--------|
| GPT Image 1 / 1.5（DALL-E 3 API 2026-05-12 下线） | 文本理解准、ChatGPT 内闭环 | 风格选择少 | 中 |
| Imagen 4 (Google) | 真实感、文字渲染、多主体场景 | 创意性一般 | 中 |
| Midjourney v7 | 美学最强、电影感 | 无 API（仅 Discord/官网） | 中高 |
| Ideogram 2.0 | 海报、含字图像 | 真实人像弱 | 低 |
| Adobe Firefly | 商用授权干净 | 创意上限低 | 中 |

商用产品里我的选择优先级是 GPT Image 1.5 / Imagen 4 / Adobe Firefly——主要是版权链路清晰。注意 OpenAI 官方 2026-05-12 之后将停止 DALL-E 3 API 端点，老代码必须迁到 GPT Image。

### 19.5.3 开源旗舰

到 2026 年，开源有 3 个值得记的：

- **FLUX.1**（Black Forest Labs）：12B DiT，[schnell] 是 Apache 2.0 可商用，[dev] 非商用；图像质量、文字渲染、prompt 跟随都是开源第一档。
- **Stable Diffusion 3.5 Large**（Stability AI）：8B，生态最成熟，LoRA / ControlNet / IP-Adapter 周边一堆。
- **HiDream-I1**：17B，2025 年 Q2 推出的开源最大尺寸，少数指标超过 FLUX.1。

如果你只能选一个学：FLUX.1 [dev] + diffusers。生态、文档、社区 LoRA 都最齐。

### 19.5.4 diffusers 跑 FLUX.1 最小示例

```python
# pip install diffusers transformers accelerate
import torch
from diffusers import FluxPipeline

pipe = FluxPipeline.from_pretrained(
    "black-forest-labs/FLUX.1-dev",
    torch_dtype=torch.bfloat16,
)
pipe.enable_model_cpu_offload()  # 24G 卡也能跑

prompt = (
    "A cinematic photo of a Shiba Inu wearing a tiny red kimono, "
    "sitting on a moss-covered stone in a Kyoto temple at dawn, "
    "soft volumetric light, shallow depth of field, 35mm film grain."
)

image = pipe(
    prompt=prompt,
    height=1024, width=1024,
    guidance_scale=3.5,
    num_inference_steps=30,
    max_sequence_length=512,
    generator=torch.Generator("cuda").manual_seed(42),
).images[0]
image.save("shiba.png")
```

要点：

- `guidance_scale` 越高越「听话」但越僵；FLUX 推荐 3-4，SD 3.5 推荐 5-7；
- `num_inference_steps` 30 步对 FLUX 已经够；schnell 4 步即可；
- `seed` 一定要固定，否则不可复现。

### 19.5.5 ControlNet / LoRA / IP-Adapter

三件法器：

- **ControlNet**：用一张「条件图」（边缘、姿势、深度、线稿）控制构图；
- **LoRA**：小型轻量 fine-tune，注入风格或人物；
- **IP-Adapter**：用一张「参考图」控制风格或人脸。

例如要让模型按特定姿势画人物，先用 OpenPose 抽出参考人姿势的骨架图，再走 ControlNet：

```python
from diffusers import StableDiffusion3ControlNetPipeline, SD3ControlNetModel
control = SD3ControlNetModel.from_pretrained("stabilityai/stable-diffusion-3.5-large-controlnet-canny")
pipe = StableDiffusion3ControlNetPipeline.from_pretrained(
    "stabilityai/stable-diffusion-3.5-large", controlnet=control,
    torch_dtype=torch.bfloat16,
)
img = pipe(prompt="a knight standing on a cliff",
           control_image=canny_image,
           controlnet_conditioning_scale=0.7).images[0]
```

LoRA 的加载也很简单：

```python
pipe.load_lora_weights("user/my_anime_style_lora", weight_name="pytorch_lora_weights.safetensors")
pipe.set_adapters(["default"], adapter_weights=[0.8])
```

这块的工程难点不在代码，而在「调参玄学」。生产里建议把 prompt + seed + scale + LoRA 权重当成「图像 SQL」，可重复、可版本化。

### 19.5.6 图像编辑 / 修补

GPT Image 系列、Gemini 2.5、SD 3.5、FLUX.1 Fill 都支持「inpaint」（蒙版编辑）和「outpaint」（扩图）。最常见的产品场景：

- 电商：替换商品背景；
- 海报：扩展图像比例；
- 修图：去掉路人甲。

API 结构上传一张原图 + 一张 mask（白色为需要重绘区域），描述要画什么。

```python
from diffusers import FluxFillPipeline
from PIL import Image

pipe = FluxFillPipeline.from_pretrained(
    "black-forest-labs/FLUX.1-Fill-dev", torch_dtype=torch.bfloat16,
).to("cuda")

img  = Image.open("product.jpg").convert("RGB")
mask = Image.open("bg_mask.png").convert("L")  # 白色 = 需要重绘

out = pipe(
    prompt="商品摆在大理石台面上，柔光，低角度，电商主图风格",
    image=img, mask_image=mask,
    height=1024, width=1024,
    guidance_scale=30, num_inference_steps=50,
).images[0]
out.save("product_new_bg.png")
```

### 19.5.7 Prompt 工程的几条铁律

DiT 时代的 prompt 工程比 SD 1.5 时代规范多了，但还是有几条经验：

- **具体名词 + 具体形容**：「一只柴犬」远不如「一只 5 岁的赤色柴犬，蓬松冬毛」。
- **场景词、镜头词、光线词三件套**：cinematic / 35mm / volumetric light / golden hour 这些词模型懂；
- **风格锚点**：直接说「inspired by Wes Anderson」「ukiyo-e」比堆形容词管用；
- **负面提示**（SD 系）：列出不想要的元素，FLUX 默认无负面槽位，要靠正面描述对冲；
- **种子探索**：固定 prompt、扫 16 个 seed，挑最好的那张。这是 90% 商业出图的真实工作流。

## 19.6 多模态 Embedding 与多模态 RAG

文字 embedding 你已经熟了。多模态 embedding 的目标是把图、文（甚至音频）映射到同一个向量空间，让你能用「一段文字」检索「一组图」。

### 19.6.1 三种主流路径

**路径 A：双塔统一空间（CLIP 派）**

CLIP / SigLIP / SigLIP 2 / Nomic Embed Vision 等。文本和图各有一个编码器，训练目标是「同一对图文向量靠近，不同对推远」。

```python
from transformers import AutoProcessor, AutoModel
import torch
from PIL import Image

processor = AutoProcessor.from_pretrained("google/siglip2-so400m-patch14-384")
model     = AutoModel.from_pretrained("google/siglip2-so400m-patch14-384").eval()

images = [Image.open(p) for p in ["a.jpg","b.jpg","c.jpg"]]
texts  = ["a shiba inu in kimono", "a photo of a city skyline"]

inputs = processor(text=texts, images=images, return_tensors="pt", padding="max_length")
with torch.no_grad():
    out = model(**inputs)
# 图向量
img_emb = out.image_embeds   # [3, D]
# 文本向量
txt_emb = out.text_embeds    # [2, D]
# 相似度
sim = (img_emb @ txt_emb.T).softmax(dim=-1)
```

优点：极简，单向量，能直接进 Qdrant / Milvus / pgvector。
缺点：单向量丢失局部信息，对「图里某个角落的小字」无效。

**路径 B：Caption-then-Retrieve**

先把每张图用 VLM 写一段 caption，再走文本 RAG。

优点：实现最简单，复用既有文本 RAG 栈。
缺点：caption 是有损压缩；caption 写得不好就检索不到。

**路径 C：晚交互多向量（ColPali / ColQwen2）**

ColPali 是 2024 年下半年的关键论文，到 2026 年已经是「文档 RAG 不再 OCR」的事实标准。核心机制是把每一页当成图、过 PaliGemma（或 Qwen2-VL）得到 ~1024 个 128 维 patch 向量；查询文本也过同模型得到 token 向量；检索时用 MaxSim 做晚交互（对每个 query token 找它在 patch 中的最强匹配再求和）。

这意味着：不需要 OCR、不需要 chunk、扫描件 / 表格 / 图表都能直接检索；一页 ≈ 100-200 KB 存储。

### 19.6.2 ColPali / ColQwen2 完整 demo

无 GPU？把 `device = "cuda" if ..."` 改成 `"cpu"`，单页编码从 0.1 秒变 5 秒，10 页以内的 demo 仍可跑。完整索引一万页才需要 GPU。

```python
# pip install colpali-engine pdf2image pillow torch
from pathlib import Path
import torch
from pdf2image import convert_from_path
from colpali_engine.models import ColQwen2, ColQwen2Processor

device = "cuda" if torch.cuda.is_available() else "cpu"

model = ColQwen2.from_pretrained(
    "vidore/colqwen2-v1.0",
    torch_dtype=torch.bfloat16,
    device_map=device,
).eval()
processor = ColQwen2Processor.from_pretrained("vidore/colqwen2-v1.0")

# 1) PDF → 图像
pages = convert_from_path("annual_report.pdf", dpi=150)
print(f"共 {len(pages)} 页")

# 2) 编码每一页
def encode_pages(images, bs=4):
    embs = []
    for i in range(0, len(images), bs):
        batch = processor.process_images(images[i:i+bs]).to(device)
        with torch.no_grad():
            embs.append(model(**batch))
    return torch.cat(embs, dim=0)

page_embs = encode_pages(pages)        # [N, T_p, D]

# 3) 查询编码
queries = [
    "净利润同比增长是多少？",
    "本期研发费用占营业收入的比例",
    "境外收入占比",
]
q_batch = processor.process_queries(queries).to(device)
with torch.no_grad():
    q_embs = model(**q_batch)          # [Q, T_q, D]

# 4) 晚交互打分
scores = processor.score_multi_vector(q_embs, page_embs)  # [Q, N]
for q, s in zip(queries, scores):
    top = s.topk(3)
    print(q)
    for idx, sc in zip(top.indices.tolist(), top.values.tolist()):
        print(f"  page {idx+1:3d}  score={sc:.2f}")
```

召回 top-K 之后把页面图（不是文字！）连同问题一起喂 VLM 读图作答——§19.9 的 `visual_qa()` 是这步的完整实现。整套流程零 OCR、零 chunk，扫描件和原生 PDF 一视同仁，这就是 2026 年「文档 RAG」的事实新栈。

### 19.6.3 ColPali 工程化注意点

学院派的论文 demo 跑起来都漂亮，但工程化时几件事要特别小心：

- **DPI 选择**：dpi=150 是甜点，再低（72-100）会丢小字，再高（200+）token 暴涨且边际质量提升小；
- **一页一向量集**：不要 chunk 一页！整页是 ColPali 的设计单位，硬切只会破坏布局信息；
- **冷启动慢**：首次 index 一万页大概 1-2 GPU 小时，要做后台任务 + 进度通知；
- **检索后还得 VLM**：ColPali 只告诉你「哪一页相关」，回答还得 VLM 看图。生产里把 ColPali 当「召回层」，VLM 当「理解层」，不要让用户觉得「检索完了答案还要等 5 秒」；
- **多语言**：ColPali 默认基于 PaliGemma（英文为主），中文文档用 ColQwen2 / ColQwen2.5 系列效果好太多。中文场景请直接选 ColQwen2。
- **更新策略**：文档增量更新时，用 page-level 哈希做 dedup；同一份文档的旧版本要软删除而不是物理删除，方便对比。

### 19.6.4 多模态向量库

ColPali 一页存 1024×128 向量，全库百万页是 1280 亿浮点，普通向量库吃不下。生产里的做法：

- **Qdrant** 原生支持 multi-vector + MaxSim，是目前最好用的；
- **Vespa** 多向量 ranking 灵活；
- **LanceDB** 简单，适合中小规模；
- **Milvus / Weaviate** 也有 multi-vector 支持，但要看版本。

存储建议：

- 用 binary quantization 把 128 维向量压到 16 字节（8 倍压缩，性能损失 <2%）；
- 索引时只做 Hamming 召回，rerank 阶段才用全精度 MaxSim；
- 每页存原始图像 URL + caption + 多向量，召回回页之后再决定是否进 VLM。

## 19.7 语音

### 19.7.1 STT（语音识别）

到 2026 年，Whisper 仍是开源 STT 的事实标准。版本演进：

- whisper-large-v2（2022）、large-v3（2023）：标杆；
- **large-v3-turbo**（2024 末）：解码器层从 32 砍到 4，速度 8 倍提升，准确率仅微降。日常首选；
- **distil-whisper**：英文专用，更小更快；
- **FunASR / Paraformer**（阿里）：中文场景常常好于 Whisper，且支持流式；
- **SenseVoice**（阿里）：识别 + 语种 + 情感 + 事件一体；
- **NVIDIA Canary / Parakeet**：英文新王，速度极快。

中文录音首推 FunASR/Paraformer 或 SenseVoice，多语言或英文录音首推 Whisper turbo。

最小 Whisper turbo：

```python
# pip install -U openai-whisper torch
import whisper
model = whisper.load_model("turbo")          # ~809M, 8x faster

print(result["text"])
```

### 19.7.2 说话人分离（diarization）

转写完了你常常需要「这是谁说的」。pyannote.audio 是开源标杆。2026 年初 pyannote.audio 4.0 同时发布了开源新模型 `speaker-diarization-community-1`（取代 3.1 成为默认推荐，准确率显著提升、支持 exclusive single-speaker 模式消除重叠冲突），最低 Python 要求 3.10：

```python
# pip install "pyannote.audio>=4.0"
import os
from pyannote.audio import Pipeline

dia = Pipeline.from_pretrained(
    "pyannote/speaker-diarization-community-1",  # 旧版可改回 "pyannote/speaker-diarization-3.1"
    use_auth_token=os.environ["HF_TOKEN"],
)
result = dia("meeting.wav")
for turn, _, spk in result.itertracks(yield_label=True):
    print(f"[{turn.start:.1f}-{turn.end:.1f}] {spk}")
```

把 diarization 的时间段和 Whisper 的 segment 做时间对齐，就能得到「Speaker A: ...; Speaker B: ...」的会议纪要。

> **pyannote.audio 的 HuggingFace 用户协议踩坑（必读）**：
>
> 上面的代码第一次跑时大概率报 `401 Client Error` 或 `Could not download ... model is gated`。原因是 pyannote 的两个 gated repo 必须**人工去网页接受协议后**才能下载，光有 token 不够：
>
> 1. 浏览器登录 [https://hf.co/pyannote/speaker-diarization-community-1](https://hf.co/pyannote/speaker-diarization-community-1)（如果还在用 3.1，则换成 `speaker-diarization-3.1` 的页面），填表 + 接受 user condition；
> 2. 同样访问 [https://hf.co/pyannote/segmentation-3.0](https://hf.co/pyannote/segmentation-3.0)，再接受一次（pipeline 内部依赖这个 segmentation 模型，光接受 diarization 仍会失败）；
> 3. 在 [https://hf.co/settings/tokens](https://hf.co/settings/tokens) 创建一个 read 权限的 token，导出为 `HF_TOKEN`；
> 4. **CI / Docker 镜像里**：上述两个 repo 必须用同一个有授权的账号 token 提前 pre-download 到镜像（`HF_HUB_OFFLINE=1` + 把 `~/.cache/huggingface` 拷进去），否则容器在网络受限的生产环境会启动失败。
>
> pyannote 3.x 之后 license 改为更宽松的 MIT，但 gated 流程没变。如果你的项目对 HF gated 流程接受不了（比如要部署到无外网的客户内网），替代方案：
>
> - **`speechbrain/spkrec-ecapa-voxceleb`** + 自己写 segmentation：自由权重，但效果通常差 5-10 个百分点 DER；
> - **NVIDIA NeMo `titanet_large` + `MarbleNet` VAD**：商用宽松、效果接近 pyannote、但部署链路更重；
> - **Reverb diarization**（Rev.ai 开源）：CC-BY-NC，商用要谈授权，但开箱效果好；
> - 商业 API：AssemblyAI、Rev.ai、Deepgram、阿里云 NLS 都自带 diarization。

把上面 4 步省一步都跑不通，是这个章节里 pyannote 用户的头号 issue 来源。

商业方案：AssemblyAI、Rev.ai、Deepgram，都自带 diarization。在国内合规要求下，云端方案要看数据出境合规。

### 19.7.4 实时语音对话

传统流水线：

```
mic → VAD → ASR → LLM → TTS → speaker
```

每一段都有延迟，串起来 1.5-3 秒，体验像「按对讲机」。

2026 年的端到端做法：

- **OpenAI Realtime API（gpt-realtime）**：单模型直接「听音 → 出音」，延迟 ~300ms，支持工具调用、图像输入、SIP 电话。2025-08 GA、2026-05 价格再降 20%（音频输入 $32 / 1M token，输出 $64 / 1M token，1 秒用户音频约 10 token、1 秒模型音频约 20 token），新增 Cedar / Marin 两款音色。生产可用。
- **Gemini Live**：类似，支持视频流，跟 Gemini 3.1 Pro 共线。
- **Qwen3.5-Omni Realtime**：开源端到端，本地部署，Hybrid-Attention MoE 路由让推理成本明显下来。

OpenAI Realtime 的 WebSocket 最小例子：

```python
# pip install websocket-client
import json, base64, os, websocket, pyaudio

ws = websocket.create_connection(
    "wss://api.openai.com/v1/realtime?model=gpt-realtime",
    header=[
        f"Authorization: Bearer {os.environ['OPENAI_API_KEY']}",
        "OpenAI-Beta: realtime=v1",
    ],
)

# 发起 session 配置
ws.send(json.dumps({
    "type": "session.update",
    "session": {
        "modalities": ["audio", "text"],
        "voice": "alloy",
        "turn_detection": {"type": "server_vad"},
        "instructions": "你是一个温柔的语音助手，回答简短。",
    }
}))

# 持续推麦克风音频
pa = pyaudio.PyAudio()
stream = pa.open(format=pyaudio.paInt16, channels=1, rate=24000,
                 input=True, frames_per_buffer=1024)
import threading
def push():
    while True:
        chunk = stream.read(1024, exception_on_overflow=False)
        ws.send(json.dumps({
            "type": "input_audio_buffer.append",
            "audio": base64.b64encode(chunk).decode(),
        }))
threading.Thread(target=push, daemon=True).start()

# 接收音频回放
out = pa.open(format=pyaudio.paInt16, channels=1, rate=24000, output=True)
while True:
    msg = json.loads(ws.recv())
    if msg["type"] == "response.audio.delta":
        out.write(base64.b64decode(msg["delta"]))
```

这个 demo 大概 80 行就能跑通一个真正的语音助手，5 年前光做 ASR + TTS 流水线都得 800 行。这是 2026 年最有「未来感」的能力。

工程注意：

- 服务端 VAD 比客户端 VAD 体验好，但有时不准，要支持手动 commit；
- 工具调用走 `tool_call` 事件，跟普通 chat 接口一致，做 function calling 时把 STT/TTS 都甩给模型，自己只接业务工具；
- 录音保留要走专用合规通道，国内商用要 ICP + 录音存储 6 个月。
- 端到端模型有一个反直觉的副作用：模型会学到说话人的「情绪」。生气的用户、犹豫的用户、儿童用户的语气模型都能感知，并且会调整自己的回应风格。这是好处也是坑——A/B 测试中你会发现「同一个 prompt，但用户说话方式不同，对话走向完全不同」，老的指标体系不再适用。
- 中断处理（barge-in）：用户讲到一半打断模型时，模型要立刻停止 TTS 并听新输入。Realtime API 已内建，但自建 pipeline 的同学要注意把 TTS 缓冲清空、把当前 LLM 生成 cancel 掉，否则会出现「我正在说话呢你打断我」的错位。
- 延迟监控不能只看「首字 latency」，还要看「首音 latency」「中断响应 latency」「整轮 latency」。它们会暴露 pipeline 不同位置的问题。

## 19.8 应用案例

讲了一堆能力，现在串起来。

### 19.8.1 智能合同审查

输入：一份扫描件 PDF 合同；
输出：风险点清单 + 关键条款抽取 + 与公司模板的偏差。

流水线：

1. 先 PyMuPDF 试 OCR-free 抽文字；如果是扫描件 fall back 到 Docling + Tesseract，或者干脆 ColPali 索引每页 + VLM 按需读图；
2. 用 LLM 抽关键条款（甲乙方、金额、期限、违约金、管辖法院）→ 结构化 JSON；
3. 把抽出来的条款与模板做向量比对，找差异；
4. 用 LLM 解释每个差异的法律含义和风险等级；
5. 输出 Markdown 报告。

陷阱：

- 合同里手写或盖章的修改条款 OCR 经常错；这块**永远要 human-in-the-loop**；
- 数字（金额、利率）出错代价大，要做正则二次校验。

### 19.8.2 发票批量识别

输入：每月几千张发票图片；
输出：结构化数据进 ERP。

```python
def process_invoice(image_path):

with ThreadPoolExecutor(max_workers=8) as pool:
```

### 19.8.3 视频理解 Agent

场景：监控视频、客服回放、教学视频问答。

流水线：

```
mp4
 ├── ffmpeg 抽音 → Whisper turbo → 带时间戳文本
 ├── ffmpeg 每秒 1 帧 → VLM caption（也可以稀疏到每 5 秒）
 ├── 合并：[t, audio_text, frame_caption]
 └── chunk + embed + store；查询时召回时间段，再喂原始片段给 VLM 作答
```

视频问答最大的坑是「时间」：模型常常说不清「在第几秒」。解决办法：把每个 chunk 都打上 `[start_sec, end_sec]`，让模型必须在答案中引用区间。

### 19.8.4 语音助手

要做一个能打电话的客服 Bot：

- 接入：Twilio SIP / 阿里云通信 → SIP 转 WebSocket；
- 引擎：OpenAI Realtime（境外）或 Qwen3-Omni（境内自建）；
- 工具：function calling 接订单 / CRM / 知识库 RAG；
- 兜底：对话超时 / 用户怒吼 / 模型不确定 → 转人工。

工程上 90% 的工作不是 AI 能力本身，而是会话状态机、转人工策略、合规话术——AI 只是把过去外包客服中心做的事换了一种实现。

## 19.9 完整项目：「上传 PDF 财报，自动提取关键指标并问答」

把本章串成一个 250 行项目。架构：

```
PDF
 ├── Docling     → Markdown + 表格
 ├── ColQwen2    → 每页多向量索引
 ├── 关键指标抽取 (LLM, JSON schema)
 └── QA          → ColPali 召回页图 → VLM 读图作答
```

```python
# financial_qa.py
import os, json, base64, io
from pathlib import Path
from typing import List

import torch
from openai import OpenAI
from pdf2image import convert_from_path
from PIL import Image
from docling.document_converter import DocumentConverter
from colpali_engine.models import ColQwen2, ColQwen2Processor

client = OpenAI()
DEVICE = "cuda" if torch.cuda.is_available() else "cpu"

# ---------- 1. 解析 ----------
def parse_pdf(path: str) -> dict:
    doc = DocumentConverter().convert(path).document
    return {
        "markdown": doc.export_to_markdown(),
        "tables":   [t.export_to_dataframe() for t in doc.tables],
    }

# ---------- 2. ColQwen2 索引 ----------
class PageIndex:
    def __init__(self, ckpt="vidore/colqwen2-v1.0"):
        self.model = ColQwen2.from_pretrained(
            ckpt, torch_dtype=torch.bfloat16, device_map=DEVICE).eval()
        self.proc  = ColQwen2Processor.from_pretrained(ckpt)
        self.pages: List[Image.Image] = []
        self.embs  = None

    def index(self, pdf_path: str, dpi=150):
        self.pages = convert_from_path(pdf_path, dpi=dpi)
        all_embs = []
        for i in range(0, len(self.pages), 4):
            batch = self.proc.process_images(self.pages[i:i+4]).to(DEVICE)
            with torch.no_grad():
                all_embs.append(self.model(**batch))
        self.embs = torch.cat(all_embs, dim=0)
        print(f"indexed {len(self.pages)} pages")

    def search(self, query: str, k=3) -> List[Image.Image]:
        q = self.proc.process_queries([query]).to(DEVICE)
        with torch.no_grad():
            q_emb = self.model(**q)
        scores = self.proc.score_multi_vector(q_emb, self.embs)[0]
        idx = scores.topk(k).indices.tolist()
        return [self.pages[i] for i in idx]

# ---------- 3. 关键指标抽取 ----------
KEY_METRICS_SCHEMA = {
  "type":"object",
  "properties":{
    "revenue":            {"type":"number","description":"营业收入（百万元）"},
    "net_profit":         {"type":"number","description":"归母净利润（百万元）"},
    "rd_expense":         {"type":"number","description":"研发费用（百万元）"},
    "rd_to_revenue_pct":  {"type":"number","description":"研发占收入百分比"},
    "yoy_revenue_pct":    {"type":"number","description":"营业收入同比增速 %"},
    "yoy_net_profit_pct": {"type":"number","description":"净利润同比增速 %"},
  },
  "required": ["revenue","net_profit"]
}

def extract_metrics(markdown: str) -> dict:
    resp = client.chat.completions.create(
        model="gpt-5.5",
        response_format={"type":"json_schema",
                         "json_schema":{"name":"metrics","schema":KEY_METRICS_SCHEMA}},
        messages=[
            {"role":"system","content":"你是一个财报分析助手，严格按 schema 抽取数据。无数据填 null。"},
            {"role":"user","content": markdown[:120000]},
        ],
    )
    return json.loads(resp.choices[0].message.content)

# ---------- 4. 视觉问答 ----------
def img_to_b64(img: Image.Image) -> str:
    buf = io.BytesIO(); img.save(buf, "PNG")
    return base64.b64encode(buf.getvalue()).decode()

def visual_qa(question: str, pages: List[Image.Image]) -> str:
    content = [{"type":"text",
                "text": f"基于下面的财报截图回答问题，引用具体数字和页码（按图序）。问题：{question}"}]
    for p in pages:
        content.append({"type":"image_url",
                        "image_url":{"url": f"data:image/png;base64,{img_to_b64(p)}"}})
    resp = client.chat.completions.create(
        model="gpt-5.5", max_tokens=600,
        messages=[{"role":"user","content": content}],
    )
    return resp.choices[0].message.content

# ---------- 5. CLI ----------
if __name__ == "__main__":
    import sys
    pdf = sys.argv[1]
    print("==> 解析 PDF ...")
    parsed = parse_pdf(pdf)
    print("==> 索引 ColPali ...")
    idx = PageIndex(); idx.index(pdf)
    print("==> 抽取关键指标 ...")
    metrics = extract_metrics(parsed["markdown"])
    print(json.dumps(metrics, ensure_ascii=False, indent=2))
    print("==> 进入问答 (Ctrl+C 退出)")
    while True:
        try:
            q = input("Q> ").strip()
            if not q: continue
            pages = idx.search(q, k=3)
            print(visual_qa(q, pages))
        except KeyboardInterrupt:
            break
```

跑起来你会得到：

- 一份 JSON 关键指标（喂给 BI 系统）；
- 一个支持自然语言问答的财报分析器（vendor lock-in 是 OpenAI，国内可换 Qwen3-VL）；
- ColPali 索引可序列化保存（torch.save），下次直接 load，省一次嵌入。

把这套放进 FastAPI、加上前端拖拽上传，就是一个能直接卖给中小投研团队的产品。

> **「能不能在生产环境一次跑通」的诚实回答**：
>
> 不能。上面 250 行代码在你的开发机上能跑通，**前提是你已经吃过下面这堆依赖坑**：
>
> 1. **`pdf2image` 依赖系统级 `poppler`**：`apt-get install poppler-utils`（Linux）或 `brew install poppler`（macOS）。Docker 镜像不装会报 `Unable to get page count. Is poppler installed?`，且报错信息不会告诉你是 poppler 的事。
> 2. **`docling` 安装拖一堆**：当前版本会拖 `easyocr` / `transformers` / `torch` / 多个 layout 模型，第一次 `pip install docling` 大约下 2-3 GB，首次 `convert` 还会再下 1-2 GB 的版式与 OCR 权重。生产 Dockerfile **必须做"build 阶段预加载权重"**，不然客户内网启动直接卡死。
> 3. **`colpali-engine` + `ColQwen2` 显存**：bf16 全精度大约 7-8 GB 显存，加上一个长 PDF 编码 batch 会到 12 GB。**消费级显卡（≤ 12 GB）必须开 `device_map="auto"` 或量化**，不然 OOM。CPU fallback 能跑但每页编码 5-15 秒，索引一份 200 页财报至少 30 分钟。
> 4. **`vidore/colqwen2-v1.0` 也是 gated repo**：需要 HF 登录 + 接受协议（同 pyannote 流程）。CI 镜像必须预下载。
> 5. **GPT-5.5 visual JSON schema 模式有 size 限制**：把整份 markdown 塞进 `messages` 时，120K 字符可能超过单条上下文边界（取决于 tokenizer），生产里要做分段抽取 + 结果合并，而不是一把梭。
> 6. **Docling 对扫描件的版式分析会偶发崩溃**：遇到老式合同的奇怪嵌套表格会抛 `IndexError`，需要把 `convert(path)` 包在 try / except 里，崩了 fallback 到 PyMuPDF 抽纯文本。
> 7. **冷启动**：上面三个模型（Docling 内部 layout + ColQwen2 + OpenAI client）在干净容器里 cold start 大约 90-180 秒。生产服务一定要加 `/warmup` 接口，不然第一个用户上传 PDF 等三分钟。
>
> **真实生产做法**：
>
> - 把 Docling、ColPali、VLM 调用拆成三个微服务，各自独立扩缩容；
> - PDF 上传后异步处理（Celery / RQ），前端轮询任务状态而不是同步等；
> - 用持久化存储保存 ColPali 嵌入（一份财报算一次，永久缓存）；
> - 把所有模型权重打进基础镜像而不是首次启动现下载。
>
> 这 250 行是教学骨架，离生产可用还差一个 README + 一个 Dockerfile + 一套异步队列。

## 19.10 部署与成本

把多模态项目从 demo 推到生产，最容易踩的坑：

### 19.10.1 成本

token 价格表（2026 年 5 月平均）：

- VLM 视觉输入：~$2-15 / 1M token，一张 1024×1024 大约 1500-3000 token；GPT-5.5 标价 $5 / 1M 输入、$30 / 1M 输出，gpt-5.5-pro 是 $30 / $180；
- 实时语音：gpt-realtime $32 / 1M 输入 + $64 / 1M 输出（2026-05 降价后），1 分钟对话约 $0.03-0.06；
- 图像生成：FLUX.1 dev 自建一张约 $0.001（电费 + 折旧），GPT Image 1.5 一张约 $0.04，Imagen 4 一张约 $0.03；
- ColPali 自建索引一万页约 1 GPU 小时。

记账法：永远把「单次任务的端到端成本」算清楚，不要只看模型 token。一次「财报问答」可能涉及 5 次 VLM 调用 + 1 次 embedding + 1 次 rerank，加起来比单纯文字 RAG 贵 10 倍以上。

> **生产提醒**：上面"1024×1024 ≈ 1500-3000 token"是 demo 尺寸，**用户实际上传的图早就不是这个分布**。iPhone 17 Pro 默认拍 4032×3024、Android 旗舰开 200MP 模式直接出 16000×12000——OpenAI `detail=high` 按 512×512 patch 切，一张 4K 全图轻松 2000-5000 token；Anthropic 按长边 1568 缩放再切，4K 图也是 ~3000+ token；Qwen3-VL 不缩，4K 直接 8000+ token。生产必须：(1) 入口处做 max-dimension 限制（建议长边 ≤ 2048），server 端缩图后再送 VLM；(2) `detail=low` / `detail=high` 按业务分流，OCR 类才开 high，识图分类用 low；(3) 单请求 token 上限 + 日累计上限分租户算账，否则一个用户上传 50 张 4K 图就能烧掉日预算；(4) 监控 dashboard 必须画"上传图分辨率分布"周报，发现 P95 边长超阈值要 alert——不然账单跳了你才发现已经一周。

### 19.10.2 延迟

- VLM 单次调用：1-5 秒（取决于图像数和分辨率）；
- 实时语音：300-800 ms 单 turn；
- ColPali 检索 1 万页：100-200 ms（GPU）；
- diffusers FLUX 一张图：5-15 秒（A100），20-40 秒（4090）。

产品上要给每个慢操作打 loading 动画 + 进度提示，不要让用户对着空白屏幕等。

### 19.10.3 隐私与合规

通用原则：

- 医疗、金融图像走云端 API 之前先脱敏；
- 国内项目优先开源 + 私有化部署：Qwen3-VL / Qwen3-Omni / CosyVoice / Whisper turbo 全套都能本地跑；
- 用户上传内容做 NSFW / 暴力 / 政治敏感过滤，VLM 自身的安全分类已经够用，不必再训一个；
- 录音、人脸、签字图属于敏感个人信息，存储要加密、访问要审计。

**多模态特有的合规风险**（这是文字模型时代不需要操心、但你做多模态应用就一定会被合规部门质询的几条，本节只点出风险类别，**具体法条、单独同意、肖像权 / 生物识别 / 出境合规要求详见第 26 章**）：

1. **人脸**：PIPL §28 敏感个人信息 + AI Act 高风险，需单独同意 + server 端 face blur 脱敏（详见第 26 章 §26.3 / §26.6）；
2. **声纹**：生物识别且不可更换，TTS 克隆未经授权即侵权（详见第 26 章 §26.3 / §26.5）；
3. **未成年人**：14 岁以下需监护人同意（中国）/ 13 岁以下（COPPA），教育 / 玩具 / 母婴类 VLM/TTS 流程要强制监护人验证；
4. **签字与笔迹**：属可识别身份信息，合同审查项目不要把 signature crop 单独存便于检索的位置；
5. **数据出境**：海外 API 调用属跨境传输，需走数据出境安全评估或个人信息保护认证（详见第 26 章 §26.2.2 / §26.3）；
6. **合成内容标识**：AI 生成图像 / 音频 / 视频要带显式 + 隐式标识（详见第 26 章 §26.6.3）；
7. **审计与留痕**：多模态请求全链路日志（输入图哈希、prompt、模型版本、输出哈希）至少留 6 个月，但日志本身要加密 + 访问审计 + 到期销毁。

不止合规——多模态应用第一次出大事故几乎都不是模型答错，而是**用户发现自己的脸 / 声音 / 合同被 AI 处理过但自己不知道**。这种舆情杀伤力是普通 bug 的 100 倍，把"知情 + 同意 + 可撤回"在产品初期做扎实远比事后公关划算。

### 19.10.4 监控

最少要监控的四件事：

1. 各类多模态调用的 latency p50/p95；
2. 单次任务成本；
3. 输出格式合规率（JSON schema 通过率）；
4. 用户反馈（点踩率、转人工率）。

多模态特有的几个监控点：

- **图像分辨率分布**：用户上传图片的尺寸分布会随时间漂移（手机像素一直在涨），它直接决定 token 成本，要每周看一次直方图；
- **每类模态的失败率**：VLM 对模糊图、纯黑图、二维码这类输入的失败率比文字高一个量级，要单独建 dashboard；
- **音频时长分布**：录音越长成本越线性增长，但模型质量在 30 分钟以上会下降；
- **OCR / 抽取字段的字段级准确率**：不是「整体过没过」，而是「金额字段准不准、日期字段准不准」，每个字段单独建指标。

### 19.10.5 一个被低估的优化点：缓存

多模态调用单次贵，但很多场景重复度很高（同一份合同被 100 个用户问、同一张产品图被反复识别）。三层缓存值得搭：

1. **请求级缓存**（Redis）：以 `hash(image_bytes) + prompt` 为 key 缓存 VLM 输出，命中率往往 30%-60%；
2. **嵌入缓存**：ColPali / SigLIP 的图像嵌入算一次存一次，永久不动（除非换模型）；
3. **Provider Prompt Caching**：OpenAI / Anthropic 都支持系统消息缓存，多模态场景把通用 schema 和指令放系统消息能省一半 token。

我接手过一个发票识别项目，加这三层缓存后单月成本从 $8000 降到 $2300，准确率没动一根毫毛。

## 19.11 章末小结

> **关于工程风格的一点说明**：本章示例走"单文件可跑通"路线（裸脚本、`pip install`、不打镜像），目标是学完概念跑通效果；第 07 章 ML 项目走的是"完整工程化"路线（`uv` + Dockerfile + Prometheus + runbook）。差异是有意的——多模态调用绝大部分依赖外部商业 API + 重型本地模型，每个 case 的依赖、成本、延迟特性差异巨大，过早工程化反而被沉没成本绑死。**正式生产时，本章所有示例都应套上第 07 章那套骨架**，并额外加：异步队列（VLM 秒级延迟）、对象存储（图像/音频不进数据库）、合规审计日志（19.10.3）、模型版本兜底。骨架是同一套，只是多模态项目里"骨架之外的事"占工作量比例更大。

这一章的密度比前面任何一章都大，因为「多模态」其实是 5 个独立赛道（VLM、文档、生成、语音、Embedding）的总和。要点回顾：

- 2026 年默认每个 LLM 都带视觉，「能看图」不再是亮点，「能干活」才是。
- 文档处理的事实新栈是 Docling / marker（结构）+ ColPali（检索）+ VLM（理解）的混合。OCR 在很多场景已经是边缘选项。
- 图像生成最实用的开源是 FLUX.1，最实用的商业是 GPT Image / Imagen。LoRA 与 ControlNet 是「让模型听话」的核心法器。
- 语音方面，Whisper turbo + CosyVoice + gpt-realtime 是 2026 年的「黄金三件套」。中文场景把 Whisper 替换成 FunASR/Paraformer 即可。
- 多模态 RAG 的关键不是「再写一个 RAG」，而是选对路径：单向量统一空间适合开放检索，Caption 适合复用既有栈，ColPali 适合扫描件 + 表格 + 图表的密集文档。

走到「多模态 RAG」这一步，下一个绕不开的问题是：图像 patch 向量、ColPali 一页 1024×128 的多向量、跨模态 embedding，最终要落到一个能扛百万到十亿级向量、支持多向量过滤与横向扩展的存储引擎里。19.6 节里 Qdrant / Vespa / LanceDB / Milvus / Weaviate 只是匆匆带过——下一章（第 20 章 · 向量数据库）补齐这一层：HNSW / IVF-PQ / DiskANN、binary / scalar / product 量化、多向量 + MaxSim 检索、混合检索、生产部署运维。本章讲「向量从哪里来」，下一章讲「向量到哪里去」。
